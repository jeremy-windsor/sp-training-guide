# 3.3 — eBGP Peering: IX, Transit & Customers — Answer Key

## Question 1
> Design the complete BGP peering policy for an SP with: 2 transit providers, 3 PNI peers, 50 IX peers via route server, and 200 customers. Specify LOCAL_PREF values, community tagging, and outbound filtering rules.

### Answer

**LOCAL_PREF hierarchy (Step 2 of best path — the primary traffic engineering knob):**

| Peer Type | LOCAL_PREF | Rationale |
|-----------|-----------|-----------|
| Customer routes | 150 | Always prefer customer-learned routes (you get paid to carry this traffic) |
| PNI peer routes | 120 | Settlement-free peering — preferred over transit (no per-bit cost) |
| IX peer routes (route server) | 110 | Still settlement-free, but less reliable than dedicated PNI |
| Transit routes | 80 | Last resort — you pay for every bit. Only used when no customer/peer path exists |

**Community tagging scheme:**

Use a structured community format: `<your-ASN>:<type><value>`

```
65000:1100  — learned from Customer
65000:1200  — learned from PNI Peer
65000:1300  — learned from IX Peer
65000:1400  — learned from Transit A (AS 3356)
65000:1500  — learned from Transit B (AS 174)

Action communities (customer-triggered):
65000:666   — RTBH (blackhole)
65000:9001  — do not advertise to any peer
65000:9002  — do not advertise to Transit A
65000:9003  — do not advertise to Transit B
65000:9100  — prepend 1x to all peers
65000:9200  — prepend 2x to all peers
65000:9300  — prepend 3x to all peers
```

**Inbound policy per peer type:**

*Customer inbound:*
- Accept only customer's registered prefixes (prefix-list from IRR/RPKI)
- Set LOCAL_PREF 150
- Tag with community 65000:1100
- Apply max-prefix limit (customer-specific, typically 2x their announced count)
- Validate RPKI: reject Invalid, accept Valid and NotFound

*PNI peer inbound:*
- Accept routes per negotiated prefix list or IRR-generated filter
- Set LOCAL_PREF 120
- Tag with community 65000:1200
- Max-prefix: based on peer's typical announcement count + buffer
- RPKI: reject Invalid

*IX route server inbound:*
- Accept with IRR-based filtering (or RPKI-only if route server does filtering)
- Set LOCAL_PREF 110
- Tag with community 65000:1300
- Max-prefix: liberal (route servers carry many peers' routes)
- RPKI: reject Invalid

*Transit inbound:*
- Accept full table (or partial, depending on design)
- Set LOCAL_PREF 80
- Tag with community 65000:1400 or 65000:1500
- Max-prefix: ~1.1M (current full table ~950K + headroom)
- RPKI: reject Invalid

**Outbound filtering rules:**

| To Peer Type | What You Advertise | What You Filter |
|-------------|-------------------|-----------------|
| Transit providers | Your prefixes + customer prefixes only | Never send peer-learned or other transit-learned routes (you're not a free transit) |
| PNI peers | Your prefixes + customer prefixes | Never send other peer or transit routes |
| IX peers | Your prefixes + customer prefixes | Same as PNI |
| Customers | Full table (or default + customer routes, per agreement) | Filter based on customer contract |

**Critical outbound safeguards:**
- Always apply `as-path access-list` outbound to prevent becoming an accidental transit (only advertise routes with empty AS_PATH or your customer's ASNs)
- Explicit export policy to prevent peer-learned routes from leaking to transit providers (don't blanket NO_EXPORT on IX routes — that also blocks advertisement to customers, which is usually desired)
- Max-prefix on every session — prevents route leaks from causing memory exhaustion

## Question 2
> A customer reports they can reach most of the internet but a specific /24 is unreachable. You find the prefix in your BGP table with an RPKI state of "Invalid." The customer insists the route is legitimate. What do you do?

### Answer

**Do NOT simply whitelist or bypass RPKI. Follow the investigation chain.**

**Step 1: Verify the RPKI state yourself.**

```
IOS-XR: show bgp <prefix> | include RPKI|Origin-AS|validity
Junos:  show route <prefix> detail | match "validation|as-path"
```

Check which origin AS is advertising this prefix and what ROA (Route Origin Authorization) exists. An "Invalid" state means either:
- The prefix is covered by a ROA, but the advertising AS doesn't match the ROA's authorized AS
- The prefix is more specific than the ROA allows (e.g., ROA authorizes /20 but someone is announcing a /24 from it)

**Step 2: Look up the ROA in a public RPKI validator.**

Use `rpki-client`, `routinator`, or check on bgp.tools/rpki or rpki-validator.ripe.net. Find:
- What ROA exists for this prefix or its covering aggregate?
- Which AS is authorized?
- What max-length is specified?

**Step 3: Determine if it's a legitimate ROA misconfiguration or a hijack.**

- **Scenario A:** The prefix owner created a ROA for their /20 with max-length /20, then later started announcing a /24 disaggregate without updating the ROA. This is the most common case — legitimate route, misconfigured ROA. The fix is on the prefix owner's side: update the ROA to include max-length /24.

- **Scenario B:** The origin AS in the BGP announcement doesn't match the ROA at all. This is a potential route hijack. Do NOT accept the route.

- **Scenario C:** No VRP/ROA covers the announced prefix at all. Per RFC 6811, the validation state is **NotFound** (not Invalid). Invalid requires at least one covering VRP with no matching origin AS. NotFound prefixes are typically accepted but without the preference boost given to Valid routes.

**Step 4: Communicate with the customer.**

If Scenario A: Explain that the prefix owner needs to update their ROA in their RIR portal (ARIN, RIPE, APNIC). Provide the specific ROA that needs updating. This is a 5-minute fix on the RIR's website.

If the customer is the prefix owner: walk them through updating the ROA. If the customer receives the route from a third party: contact that third party's NOC.

**Step 5: Temporary relief (if business-critical).**

While the ROA is being fixed, you have options:
- **Preferred:** Set RPKI policy to `allow Invalid with lower LOCAL_PREF` instead of `reject Invalid`. This deprioritizes the route but doesn't blackhole it. The customer reaches the destination via a longer path if a valid alternative exists.
- **Acceptable:** Add a temporary static route or prefix-specific exception. Document it with a ticket and expiry date.
- **Not recommended:** Disable RPKI validation entirely. Never do this — it defeats the entire purpose.

**Key principle:** RPKI Invalid means something is wrong. Your job is to figure out WHAT is wrong, not to assume the route is good because the customer says so.

## Question 3
> Explain why GTSM (`ttl-security hops 1`) and `ebgp-multihop` are mutually exclusive. When would you choose multihop over GTSM, and what compensating security controls would you add?

### Answer

**They appear contradictory but are not mutually exclusive at the protocol level.**

**GTSM (Generalized TTL Security Mechanism, RFC 5082):** The sender sets TTL=255; the receiver checks that the incoming TTL is ≥ (256 - expected hop count). For directly connected peers (1 hop), the receiver requires TTL ≥ 254. This prevents remote attackers from spoofing BGP packets — they can't forge a high-enough TTL from multiple hops away.

**eBGP-multihop:** Allows BGP sessions to span multiple IP hops by setting the TTL high enough to survive the hop count.

**The nuance:** RFC 5082 supports GTSM for bounded multi-hop sessions — the receiver sets its acceptance threshold based on the expected hop count (e.g., 2 hops → require TTL ≥ 253). This works because GTSM always sends TTL=255 and validates based on expected distance, not a fixed threshold. Some vendor CLIs make `ttl-security` and `ebgp-multihop` mutually exclusive — that's a platform limitation, not a protocol constraint. For directly connected peers, always prefer GTSM.

**When to choose multihop over GTSM:**

1. **Peering via loopbacks over multiple physical paths.** Two routers connected by 2+ parallel links, peering via loopback addresses for session resilience. The BGP packets traverse at least 1 extra hop (loopback → physical → remote physical → remote loopback), so TTL decrements more than once. GTSM's `hops 1` would reject these packets. Use `ebgp-multihop 2`.

2. **eBGP sessions through a firewall or load balancer.** If a transparent firewall or policy device sits between the BGP peers, TTL decrements through that device. GTSM fails. Use `ebgp-multihop` with the appropriate hop count.

3. **BGP peering between VMs/containers on different hosts.** Virtualized environments may add extra L3 hops between BGP speakers that aren't visible at the physical topology level.

**Compensating security controls when using multihop:**

Since GTSM's anti-spoofing protection is gone, you must add:

1. **TCP MD5 authentication (RFC 2385) or TCP-AO (RFC 5925):**
```
IOS-XR: neighbor <ip> password encrypted <key>
Junos:  neighbor <ip> authentication-key "<key>"
```
This prevents spoofed TCP/179 connections — an attacker would need the shared key.

2. **Infrastructure ACLs:** Explicitly permit TCP/179 only from the specific peer IP on your edge ACLs. Block TCP/179 from all other sources.

3. **Control-plane policing (CoPP):** Rate-limit TCP/179 to prevent SYN flood DoS against the BGP process.

4. **uRPF (Unicast Reverse Path Forwarding):** Enable on the interface facing the peer. Drops packets with spoofed source addresses that don't match the expected reverse path.

5. **Max-prefix limits:** Always configured regardless, but especially important when multihop expands the attack surface.

**Bottom line:** Use GTSM for all directly-connected eBGP sessions (the common case). Only use multihop when the topology requires it, and always add MD5/TCP-AO + ACLs as compensating controls.
