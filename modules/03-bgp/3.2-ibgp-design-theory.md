# 3.2 — iBGP Design — Protocol Theory

## Protocol Overview

iBGP (Internal BGP) is BGP peering between routers within the same Autonomous System. While eBGP builds the Internet's inter-domain routing table, iBGP distributes that information internally and carries VPN routing across the SP backbone.

iBGP has a fundamental constraint that shapes everything: **iBGP split horizon**. A route learned from one iBGP peer is NOT re-advertised to another iBGP peer. This prevents routing loops within the AS (since iBGP doesn't modify AS_PATH for internal routes, there's no loop detection mechanism like eBGP's "reject if my AS is in the path"). But it means every iBGP router must learn every route — either directly from the originator or through a route reflector.

The split horizon rule means that in a network of N iBGP speakers, you need either N(N-1)/2 iBGP sessions (full mesh), route reflectors, or confederations. For an SP with 200 PE routers, full mesh means 19,900 sessions. That doesn't scale. Route reflectors and confederations were invented to solve this.

---

## Core Mechanisms

### 1. Full Mesh iBGP

In a full mesh, every iBGP speaker peers with every other iBGP speaker. Every router receives every route directly from the originator (or from the eBGP ingress point).

**Session count**: N(N-1)/2 for N routers.

**Why it doesn't scale:**
- 10 routers: 45 sessions (manageable)
- 50 routers: 1,225 sessions (painful)
- 200 routers: 19,900 sessions (unmanageable)
- 1,000 routers: 499,500 sessions (absurd)

Each session consumes memory (TCP buffers, BGP adj-RIB-in, adj-RIB-out), CPU (UPDATE processing, best path calculation), and generates operational complexity (monitoring, troubleshooting). Full mesh is only viable for small networks (under ~20-30 iBGP speakers).

**iBGP behavior in full mesh:**
- NEXT_HOP is not changed (preserves the eBGP next-hop or originator's address).
- LOCAL_PREF is carried (set at eBGP ingress, propagated via iBGP).
- MED is carried (from the eBGP peer, propagated via iBGP).
- AS_PATH is NOT modified (no AS prepending within iBGP — the AS number is not added).
- Routes are NOT re-advertised to other iBGP peers (split horizon).

### 2. Route Reflectors (RFC 4456)

Route reflectors break the full mesh requirement by allowing one or more routers to "reflect" routes between iBGP peers.

#### Core Concept

A Route Reflector (RR) receives routes from its clients and re-advertises them to other clients and non-client peers. This violates the normal iBGP split horizon rule — the RR is explicitly permitted to re-advertise iBGP routes it learned from iBGP peers.

**Terminology:**
- **RR (Route Reflector)**: A router configured to reflect routes.
- **Client**: An iBGP peer of the RR that is in the RR's client group. The RR reflects routes to/from clients.
- **Non-client**: An iBGP peer of the RR that is NOT a client. Non-clients must still be fully meshed with each other (or be clients of another RR).

**Reflection rules (what the RR does with a route):**

| Route learned from | Reflected to |
|---|---|
| Client | All other clients + all non-client iBGP peers + eBGP peers |
| Non-client iBGP peer | All clients only (NOT to other non-clients) |
| eBGP peer | All clients + all non-client iBGP peers |

The key rule: routes from non-clients go to clients only. Routes from clients go everywhere. This is how the RR replaces the full mesh — clients don't need to peer with each other; the RR distributes routes between them.

#### Loop Prevention

Since RR reflection violates split horizon, new loop prevention is needed. Two attributes handle this:

**ORIGINATOR_ID (Type 9):**
- Set by the RR to the Router ID of the route's originator (the router that first injected the route into iBGP).
- ORIGINATOR_ID is only added when the RR *reflects* a client's route — it identifies the original client. A route locally originated by the RR itself is not reflected and does not carry ORIGINATOR_ID.
- If a router receives a route with its own Router ID in ORIGINATOR_ID, it discards the route. This prevents the route from looping back to the originator.

**CLUSTER_LIST (Type 10):**
- Each RR has a Cluster ID (defaults to its Router ID; can be explicitly configured).
- When an RR reflects a route, it prepends its Cluster ID to the CLUSTER_LIST.
- If an RR receives a route with its own Cluster ID already in CLUSTER_LIST, it discards the route. This prevents reflection loops between RRs.

**Loop scenarios prevented:**
1. **Client → RR → same client**: ORIGINATOR_ID prevents this. The client sees its own Router ID and drops it.
2. **RR1 → RR2 → RR1**: CLUSTER_LIST prevents this. RR1 sees its own Cluster ID in the list and drops it.
3. **Client → RR1 → RR2 → RR1 → client**: CLUSTER_LIST catches the RR loop; ORIGINATOR_ID catches the client loop.

#### Cluster ID

A Cluster ID identifies a route reflector cluster (an RR and its clients). By default, the Cluster ID equals the RR's Router ID. When deploying redundant RRs for the same set of clients, you MUST configure the same Cluster ID on all RRs in the cluster. Otherwise, each RR treats the others' reflections as new routes (different Cluster IDs = no loop detection between them), leading to routing loops.

**Single RR cluster:**
```
Clients: PE1, PE2, PE3
RR: RR1 (Cluster ID = RR1's Router ID, implicit)
```

**Redundant RR cluster (correct):**
```
Clients: PE1, PE2, PE3
RR1: Cluster ID = 1.1.1.1 (explicit)
RR2: Cluster ID = 1.1.1.1 (same — they share the cluster)
```

**Redundant RR cluster (incorrect — creates loops):**
```
RR1: Cluster ID = RR1's Router ID (implicit, e.g., 10.0.0.1)
RR2: Cluster ID = RR2's Router ID (implicit, e.g., 10.0.0.2)
→ Different Cluster IDs: RR1 reflects to RR2, RR2 reflects back, loop!
```

#### RR and Best Path

Route reflectors run the standard best path algorithm. When an RR receives multiple paths to the same prefix (from multiple clients or non-clients), it selects the best path and reflects only that one.

**This has implications:**
- Clients only see the RR's best path, not all available paths. If the RR's best path is suboptimal from a client's perspective (e.g., the client is closer to a different exit), the client is forced to use the RR's choice.
- This is the "RR optimal path problem." It's addressed by ADD-PATH or by careful RR placement (place RRs at the network's topological center).

#### RR Placement

**Physical RR** (RR runs on an inline router, typically a P-router or PE):
- Simplest deployment. The RR is in the forwarding path.
- Risk: RR failure affects both routing and forwarding.
- Advantage: The RR's IGP metric to next-hops matches the real forwarding metric.

**Logical/virtual RR** (RR runs on a dedicated server or VM, not in the forwarding path):
- RR is out-of-band. Its failure doesn't affect forwarding (routes are already installed on PEs).
- Risk: The RR's IGP metric to next-hops may not match what PEs see. The RR may select a different "best path" than what a PE would choose if it saw all paths (because step 8 — lowest IGP metric — differs from the RR's perspective).
- Mitigation: Use ADD-PATH to send multiple paths, letting PEs decide.

### 3. Confederations (RFC 5065)

Confederations partition a single AS into sub-ASes, each running iBGP internally. Between sub-ASes, a modified form of eBGP is used.

#### Concept

The AS is divided into sub-ASes (also called member-ASes or confederation member ASes). To the outside world, the confederation appears as a single AS. Internally, sub-ASes peer with each other using confederation eBGP (which uses private AS numbers for the sub-ASes).

**Key properties:**
- Each sub-AS runs normal iBGP internally (full mesh or RR within the sub-AS).
- Between sub-ASes, confederation eBGP sessions exchange routes with AS_CONFED_SEQUENCE/AS_CONFED_SET path segments.
- Confederation AS path segments are stripped when routes are advertised to true eBGP peers. The outside world sees only the confederation's public AS number.
- LOCAL_PREF and MED are preserved across confederation eBGP boundaries (unlike regular eBGP where LOCAL_PREF is stripped). This is because the confederation is logically one AS.
- NEXT_HOP is changed at confederation eBGP boundaries (same as regular eBGP behavior).

#### Loop Prevention

- **AS_CONFED_SEQUENCE**: Similar to AS_SEQUENCE but for sub-ASes. A sub-AS number in the path prevents loops between sub-ASes.
- **AS_CONFED_SET**: Unordered set variant for aggregation within confederations.
- Confederation path segments do NOT count toward AS_PATH length for best path calculation. This prevents suboptimal path selection based on internal confederation structure.

#### Confederation vs Route Reflectors

| Property | Route Reflectors | Confederations |
|----------|-----------------|----------------|
| Complexity | Lower | Higher (sub-AS management, private ASNs) |
| Configuration | Per-RR, minimal client config | Every router needs confederation config |
| Path diversity | Limited without ADD-PATH | Better (sub-AS eBGP sees multiple paths) |
| NEXT_HOP | Preserved (iBGP behavior) | Changed at sub-AS boundary |
| Deployment | Incremental (add RR, make clients) | Big-bang (partition entire AS) |
| Industry preference | Dominant | Rare (legacy deployments, specific vendors) |

**In practice**: Route reflectors win. They're simpler to deploy, easier to troubleshoot, and can be added incrementally. Confederations are used primarily in legacy networks (some large European carriers) or where sub-AS topological isolation is specifically desired.

### 4. ADD-PATH (RFC 7911)

ADD-PATH allows a BGP speaker to advertise multiple paths for the same prefix to the same peer, instead of only the best path.

#### Mechanism

Normally, BGP sends one path per prefix per peer. ADD-PATH adds a Path Identifier (4-byte integer) to each NLRI, allowing multiple distinct paths for the same prefix.

**NLRI encoding with ADD-PATH:**
```
├─ Path Identifier (4 bytes): locally assigned, unique per prefix
├─ Prefix Length (1 byte)
└─ Prefix (variable)
```

**Capability negotiation**: ADD-PATH is negotiated per AFI/SAFI in the OPEN message. Each side advertises whether it can Send, Receive, or both:
- **Send**: "I can send multiple paths"
- **Receive**: "I can receive multiple paths"
- **Send/Receive**: Both

**How many paths?** The sender decides which paths to advertise. Options include:
- All paths (memory intensive)
- Best N paths
- Paths that are "diverse" (different next-hops, different ASes)
- Implementation-specific selection algorithms

#### Why ADD-PATH Matters for RR

Without ADD-PATH, an RR reflects only its best path. Clients have no visibility into alternative paths. If the RR's best path fails, clients must wait for the RR to re-run best path and reflect the new winner.

With ADD-PATH:
- RR sends multiple paths to clients.
- Clients run their own best path selection across all received paths.
- If one path fails, the client immediately switches to the next-best path from its local RIB — no wait for RR reconvergence.
- This also solves the "RR optimal path problem" — clients can choose the path that's best from their own IGP perspective.

**ADD-PATH + BFD + prefix-independent convergence (PIC)**: The combination provides sub-second BGP failover at scale. BFD detects failure, PIC switches to the pre-computed backup, and ADD-PATH ensures the backup was available without waiting for the RR.

### 5. Outbound Route Filtering (ORF, RFC 5291)

ORF allows a BGP speaker to send filter criteria to its peer, telling the peer "only send me routes matching these criteria." The filtering happens at the sender, reducing UPDATE traffic and memory on the receiver.

**Mechanism:**
1. Both sides negotiate ORF capability in OPEN (per AFI/SAFI, per ORF type).
2. The receiver sends ORF entries to the sender via ROUTE-REFRESH message.
3. The sender applies the filters to its outbound policy for that peer.
4. Only matching routes are sent.

**ORF types:**
- **Address Prefix ORF (RFC 5292)**: Filter based on prefix/mask. "Only send me routes for 10.0.0.0/8 and 172.16.0.0/12."
- **Community ORF**: Filter based on community values (non-standard, vendor-specific).
- For VPN route-target scaling, the standardized mechanism is **RT-Constrain (RFC 4684)**, not extended community ORF. RT-Constrain uses AFI 1/SAFI 132 to signal RT interest.

**Use case**: In VPN environments, a PE only needs routes for VRFs it hosts. ORF allows the PE to tell the RR "only send me routes with RT 65000:100 and RT 65000:200" — reducing the number of routes the PE receives from potentially millions to just the relevant ones.

**RT-Constrained Route Distribution (RFC 4684)**: A more elegant solution to the same problem. Instead of ORF, each PE advertises its interested RTs as BGP routes in the RT-Constrain AFI/SAFI (AFI 1, SAFI 132). RRs use these advertisements to filter VPN route distribution automatically.

## Protocol Interactions

**RR ↔ IGP**: The RR's position in the IGP topology affects best path selection (step 8 — IGP metric to NEXT_HOP). A poorly placed RR may select a path that's optimal from its own location but suboptimal for its clients. Place RRs at the topological center of the network, or use ADD-PATH to push the decision to clients.

**RR ↔ BFD**: BFD can be enabled on iBGP sessions to the RR for fast failure detection. If the RR fails, clients detect via BFD and switch to backup RR (if redundant RRs are deployed). Without BFD, clients wait for Hold Timer expiry (default 90s).

**Confederation ↔ IGP**: Each sub-AS typically runs its own IGP instance. NEXT_HOP changes at sub-AS boundaries, so the remote sub-AS's IGP must know how to reach the local sub-AS's border routers. This adds IGP-BGP interaction complexity that RR deployments avoid.

**ADD-PATH ↔ PIC (Prefix-Independent Convergence)**: ADD-PATH provides multiple paths; PIC pre-installs a backup path in the FIB. When the primary path fails, PIC switches to the backup instantly (data plane operation, no control plane reconvergence). Together, they provide sub-second BGP failover for all prefixes sharing a next-hop — "prefix-independent" because you don't need to re-converge each prefix individually.

## Edge Cases & Gotchas

**1. Cluster ID misconfiguration**: Redundant RRs for the same client set MUST share the same Cluster ID. If they don't, each RR reflects routes with a different Cluster ID. The other RR doesn't see its own Cluster ID in the CLUSTER_LIST, so it reflects again. Routes bounce between RRs, CLUSTER_LIST grows, and you get a reflection loop that wastes CPU and memory until CLUSTER_LIST depth limits kick in.

**2. RR hides paths from clients**: An RR selects one best path and reflects it. If two clients originate the same prefix with different attributes, the RR picks one and the other client's route is invisible to the rest of the network. This matters for VPN anycast (same prefix from multiple PEs) or backup path visibility. ADD-PATH solves this.

**3. Third-party next-hop**: On multi-access networks (IXP LANs), BGP may set NEXT_HOP to a third-party address — not the advertising router's address, but the true origin's address on the shared segment. This optimization avoids an extra hop but requires all routers on the segment to have ARP/ND entries for the third-party next-hop. Misconfigured filtering can break this silently.

**4. Confederation eBGP next-hop change**: Unlike regular iBGP, confederation eBGP changes the NEXT_HOP at sub-AS boundaries (it behaves like eBGP in this regard). Routes from one sub-AS need the border router's address as next-hop, resolvable by the receiving sub-AS's IGP. If the IGP domains don't overlap at the boundary, next-hop resolution fails.

**5. RR client-to-client reflection and policies**: By default, an RR reflects routes between clients without modification. If you apply outbound policy on the RR toward a client, you're filtering reflected routes — this breaks the principle that reflected routes should be unmodified. Some designs intentionally use RR policies (e.g., RT-based filtering at hierarchical RRs), but it must be done with full understanding of the consequences.

**6. ORIGINATOR_ID and multi-homed clients**: If a client is a client of two different RR clusters (dual-homed for redundancy), ORIGINATOR_ID correctly prevents the route from looping back. But if the client uses different Router IDs toward different RRs (misconfiguration), loop prevention breaks. Always use a single Router ID per device.
