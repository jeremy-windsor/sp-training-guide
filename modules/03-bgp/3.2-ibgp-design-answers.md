# 3.2 — iBGP Design: Route Reflectors & Confederations — Answer Key

## Question 1
> You have 4 RRs in two clusters (2 per cluster). A PE is a client of both clusters. Draw the session topology and explain how CLUSTER_LIST prevents loops.

### Answer

**Session topology:**

```
        Cluster 1              Cluster 2
     (cluster-id: 1)        (cluster-id: 2)
    ┌──────┐ ┌──────┐     ┌──────┐ ┌──────┐
    │ RR1  │─│ RR2  │─────│ RR3  │─│ RR4  │
    └──┬───┘ └──┬───┘     └──┬───┘ └──┬───┘
       │  ╲   ╱  │            │  ╲   ╱  │
       │   ╲ ╱   │            │   ╲ ╱   │
       │    ╳    │            │    ╳    │
       │   ╱ ╲   │            │   ╱ ╲   │
    ┌──┴──┐ ┌─┴───┐       ┌──┴──┐ ┌─┴───┐
    │ PE1 │ │ PE-X │       │ PE3 │ │ PE-X │  ← PE-X is client of BOTH clusters
    └─────┘ └─────┘        └─────┘ └─────┘
```

**Sessions for PE-X (the dual-homed client):**
- PE-X → RR1 (iBGP, client of cluster 1)
- PE-X → RR2 (iBGP, client of cluster 1)
- PE-X → RR3 (iBGP, client of cluster 2)
- PE-X → RR4 (iBGP, client of cluster 2)

**Inter-cluster sessions:** RR1/RR2 peer with RR3/RR4 as non-client iBGP peers (standard iBGP mesh between RRs across clusters).

**How CLUSTER_LIST prevents loops:**

When an RR reflects a route, it prepends its own cluster-id to the CLUSTER_LIST attribute. Before reflecting, the RR checks the incoming CLUSTER_LIST — if its own cluster-id is already present, it drops the route (loop detected).

Walk through a reflection cycle:

1. PE1 originates a prefix, sends to RR1 (cluster 1). CLUSTER_LIST: empty.
2. RR1 reflects to all clients (including PE-X) and to non-client peers (RR3, RR4). CLUSTER_LIST: `[1]`.
3. RR3 (cluster 2) receives the route. CLUSTER_LIST `[1]` doesn't contain cluster-id 2, so RR3 reflects to its clients. CLUSTER_LIST: `[2, 1]`.
4. PE-X receives the route from **both** RR1 (CLUSTER_LIST `[1]`) and RR3 (CLUSTER_LIST `[2, 1]`). PE-X runs best path selection — both are valid, shorter CLUSTER_LIST (from RR1) may influence tie-breaking.
5. RR2 (cluster 1) also receives the route from RR1 via iBGP. RR2 checks CLUSTER_LIST `[1]` — its own cluster-id is 1. **Loop detected. Route dropped.** RR2 does NOT re-reflect this route within cluster 1.

**Key insight:** CLUSTER_LIST prevents intra-cluster loops (RR2 won't re-reflect what RR1 already reflected within cluster 1). ORIGINATOR_ID prevents the originating router (PE1) from accepting its own route back. Together, they ensure no routing loops regardless of how many RR tiers or clusters exist.

## Question 2
> With ADD-PATH configured to send 3 paths on the RR, a client PE receives 3 paths for a prefix but still selects a suboptimal exit. What could cause this and what additional information would you check?

### Answer

ADD-PATH (RFC 7911) solves the "hidden path" problem — instead of the RR reflecting only its best path, it sends multiple paths so clients can make better-informed decisions. If the client receives 3 paths and still picks a suboptimal exit, the problem isn't path visibility — it's what happens AFTER the paths arrive.

**Cause 1: The truly optimal path isn't in the 3 sent by the RR.**

ADD-PATH configured to send 3 doesn't mean the 3 best paths from the client's perspective. The RR selects which 3 paths to send based on its own best path calculation. If there are 8 total paths and the one optimal for this client (based on IGP cost to next-hop) is path #5 from the RR's perspective, the client never sees it.

Check: On the RR, `show bgp <prefix> all` — how many total paths exist? Are there paths ranked 4+ that would be better for this client's topology position?

**Cause 2: IGP metric to the next-hop (Step 8) is making the "wrong" choice.**

The client selects the path with the lowest IGP cost to the BGP next-hop (hot-potato routing). If the client is topologically closer to an exit point that's suboptimal from a global perspective, BGP will still prefer it. The client is doing exactly what BGP says — it just doesn't align with what the operator considers "optimal."

Check:
```
IOS-XR: show bgp <prefix> bestpath-compare
        show route <next-hop-1>    ← compare IGP costs to each next-hop
        show route <next-hop-2>
        show route <next-hop-3>
Junos:  show route <prefix> detail extensive
        show route <next-hop> | match metric
```

**Cause 3: LOCAL_PREF or MED is overriding AS_PATH-based selection.**

If the 3 paths have different LOCAL_PREF values (set by import policy at different ingress points), LOCAL_PREF wins at Step 2 regardless of what Step 8 would prefer. Similarly, if MED comparison is active between same-AS paths, it could override the IGP-cost-based selection.

Check: `show bgp <prefix>` — examine LOCAL_PREF, MED, and AS_PATH for all 3 received paths.

**Cause 4: Stale path information after topology change.**

If the IGP topology recently changed (link failure, metric change), the BGP next-hop resolution may be based on stale IGP data. BGP scanner interval (default 60s on many platforms) means BGP doesn't immediately re-evaluate best path after IGP convergence.

Check: `show bgp <prefix> bestpath` — when was best path last recalculated? Force with `clear bgp <prefix>` if needed.

**What "additional information" to check:** The IGP topology map showing the client's position relative to all 3 BGP next-hops, the import policies at each ingress point that set LOCAL_PREF, and the total path count on the RR for this prefix.

## Question 3
> An operator proposes using confederations to split a 200-router network into 4 sub-ASes of 50 routers each. Each sub-AS would still need internal full-mesh or RR. Argue for or against this design vs. a flat AS with hierarchical RRs.

### Answer

**Against. Use hierarchical RRs instead. Confederations add complexity without meaningful benefit at this scale.**

**The confederation proposal analyzed:**

4 sub-ASes × 50 routers = 200 routers. Each sub-AS needs its own iBGP mesh or RR infrastructure. Between sub-ASes, you need eBGP-style peering (confederation eBGP). So the total architecture becomes:

- 4 internal iBGP domains (each needing 2+ RRs anyway for redundancy = 8 RRs minimum)
- Inter-sub-AS eBGP peering at every sub-AS boundary
- Confederation-specific attributes (AS_CONFED_SEQUENCE, AS_CONFED_SET) that add path selection complexity
- Every engineer must understand confederation semantics, not just standard iBGP

**Why hierarchical RRs are better for this use case:**

1. **Simpler operations.** One AS, one BGP process, one set of policies. Every engineer already understands iBGP + RR. Confederations introduce a pseudo-eBGP concept that looks like eBGP but behaves differently (MED is preserved across confederation boundaries, AS_PATH for loop detection includes CONFED segments but not for path length comparison in some implementations). This creates subtle bugs.

2. **Equivalent scaling.** Hierarchical RR with 4 regional RRs (one per "region" of 50) + 2 top-level RRs gives you the same session reduction as confederations. Each PE has 2 iBGP sessions (to its regional RR pair). Each regional RR pair peers with the top-level RR pair. Total iBGP sessions: ~400 (200 PEs × 2 regional RRs) + 16 (8 regional RRs × 2 top-level RRs) + a few (inter-RR). Clean.

3. **No boundary complexity.** Confederations require careful next-hop handling at sub-AS boundaries (`next-hop-self` is typically needed on confederation eBGP). If you forget it on one border router, next-hop resolution breaks for that sub-AS's clients. With RRs, next-hop propagation is standard iBGP behavior — no special handling.

4. **Policy flexibility.** RRs can selectively reflect routes using outbound policies. Confederations give you inter-sub-AS policy points, but those policies affect an entire sub-AS boundary, not individual clients. RR gives finer-grained control.

5. **Path diversity.** With ADD-PATH on hierarchical RRs, clients can receive multiple paths. Confederations inherently hide paths — each sub-AS border router selects its best path before advertising to the other sub-AS, reducing path diversity.

**When confederations ARE justified:** Networks with 1000+ routers where the administrative or organizational boundary naturally aligns with sub-ASes (e.g., different business units, different countries with different NOCs). At 200 routers, you haven't reached that threshold.

**Recommendation:** Flat AS 65000 with 2 top-level RRs + 4 regional RR pairs (8 RRs total). Each PE peers with its 2 regional RRs. Regional RRs peer with top-level RRs. Clean, proven, maintainable.
