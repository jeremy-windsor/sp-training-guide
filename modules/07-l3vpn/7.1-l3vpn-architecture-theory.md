# 7.1 — MPLS L3VPN Architecture — Protocol Theory

## Protocol Overview

MPLS L3VPN (RFC 4364) is the most commercially successful protocol architecture in service provider networking. It virtualizes the IP routing table — each customer gets an isolated Virtual Routing and Forwarding instance (VRF) on the Provider Edge router, customer routes are exchanged between PEs via MP-BGP, and traffic is transported across the MPLS backbone using a two-label stack: a transport label (to reach the remote PE) and a VPN label (to identify the customer VRF at the remote PE).

The architecture was originally defined in RFC 2547 (1999) by Eric Rosen and Yakov Rekhter, then significantly revised as RFC 2547bis, which became RFC 4364 (2006). The design was driven by a fundamental economic requirement: service providers needed to carry thousands of customers' routing tables across a shared infrastructure without any customer being aware of — or able to interfere with — any other customer. The solution had to scale horizontally (more customers, more PEs) without creating O(n²) complexity.

The genius of RFC 4364 is the separation of concerns: VRFs handle isolation, Route Distinguishers handle uniqueness, Route Targets handle policy, MP-BGP handles distribution, and the MPLS data plane handles forwarding. Each component does one thing, and the combination produces a system that scales to tens of thousands of VPNs on a single provider backbone.

## Core Mechanisms

### 1. VRF — Virtual Routing and Forwarding

A VRF is a separate routing table instance on a PE router. Each VRF maintains its own RIB, FIB, and set of routing protocol instances.

**Properties:**
- **Isolation**: Routes in VRF-A are invisible to VRF-B. Two customers can use the same private address space (10.0.0.0/8) without conflict.
- **Per-VRF routing**: Each VRF can run its own PE-CE routing protocol independently. Standards-based options: static, RIP, BGP, and OSPF (RFC 4577). IS-IS and EIGRP as PE-CE protocols are vendor-specific extensions.
- **Interface binding**: Physical or logical interfaces are assigned to a VRF. Traffic arriving on a VRF-bound interface is looked up in that VRF's FIB, not the global table.
- **Not a router**: A VRF is not a separate logical router. It shares the PE's hardware, control plane CPU, and management. Only the routing and forwarding tables are separated.

**VRF-lite vs MPLS VRF**: VRF-lite uses VRFs without MPLS — useful for local traffic separation but doesn't scale across a backbone. MPLS L3VPN uses VRFs at the edge combined with MP-BGP and MPLS transport for end-to-end VPN service.

### 2. Route Distinguisher (RD)

The Route Distinguisher is an 8-byte value prepended to an IPv4 or IPv6 prefix to create a VPN-unique address: the VPNv4 or VPNv6 prefix.

**Why RDs exist**: BGP can't carry duplicate prefixes. If Customer A and Customer B both advertise 10.0.0.0/24, BGP would see them as the same prefix and apply best-path selection — keeping one, discarding the other. The RD makes them unique:
- RD1:10.0.0.0/24 (Customer A)
- RD2:10.0.0.0/24 (Customer B)

These are now distinct VPNv4 prefixes. BGP carries both without conflict.

**RD formats (3 types):**

| Type | Format | Example | Use Case |
|------|--------|---------|----------|
| 0 | 2-byte ASN : 4-byte number | 65000:100 | Most common, simple numbering |
| 1 | IPv4 address : 2-byte number | 10.0.0.1:100 | When ASN space is limited or per-PE uniqueness is needed |
| 2 | 4-byte ASN : 2-byte number | 4200000000:100 | 4-byte ASN environments |

**Critical point — RD does NOT determine route import/export**: The RD only provides uniqueness. Route Target extended communities (not the RD) determine which VRFs import which routes. This is the most common misconception in L3VPN.

**RD assignment strategies:**
- **Unique RD per VRF per PE**: Most common. PE1's VRF-A uses RD 65000:1001; PE2's VRF-A uses RD 65000:1002. Enables per-PE best-path selection on route reflectors.
- **Same RD per VPN everywhere**: Simpler to manage but prevents route reflectors from maintaining multiple paths for the same customer prefix (since RD+prefix would be identical).

### 3. Route Target (RT)

Route Targets are BGP Extended Communities (8 bytes) that control VPN route distribution. They are the policy mechanism — the RD handles uniqueness, but RTs determine who gets what routes.

**Mechanics:**
- **Export RT**: When a PE exports a VPN route from a VRF into MP-BGP, it attaches one or more Export RT extended communities.
- **Import RT**: Each VRF is configured with Import RT values. When a VPN route arrives via MP-BGP, the PE checks: does any RT on this route match any Import RT on this VRF? If yes, import the route into the VRF. If no, discard it.

**RT formats**: Same three formats as RD (Type 0, 1, 2).

**Common RT topologies:**

**Full-mesh VPN (any-to-any):**
- All sites export RT 65000:100
- All sites import RT 65000:100
- Every site sees every other site's routes

**Hub-and-spoke VPN:**
- Spoke sites export RT 65000:101 (spoke RT)
- Hub site imports RT 65000:101 (receives all spoke routes)
- Hub site exports RT 65000:102 (hub RT)
- Spoke sites import RT 65000:102 (receive only hub routes)
- Result: spokes reach hub, hub reaches spokes, spokes don't directly reach each other

**Extranet:**
- VRF-A exports RT 65000:100 AND RT 65000:999
- VRF-B imports RT 65000:999
- VRF-B now receives VRF-A's routes. Controlled route leaking.

### 4. VPNv4 and VPNv6 Address Families

MP-BGP carries VPN routes in dedicated address families:
- **VPNv4**: AFI 1 (IPv4), SAFI 128 (MPLS-labeled VPN). Prefix = 8-byte RD + 4-byte IPv4 prefix.
- **VPNv6**: AFI 2 (IPv6), SAFI 128 (MPLS-labeled VPN). Prefix = 8-byte RD + 16-byte IPv6 prefix.

**NLRI encoding for VPNv4:**
```
├─ Prefix Length (1 byte): in bits, including RD (add 64 to IPv4 mask)
├─ Label (3 bytes): MPLS VPN label (20-bit label + 3 bits EXP + 1 bit S)
├─ Route Distinguisher (8 bytes)
└─ IPv4 Prefix (variable)
```

So a VPNv4 /24 has prefix length = 64 (RD) + 24 (IPv4) = 88 bits in the BGP UPDATE.

**VPN label**: Each VPN route carries an MPLS label allocated by the advertising PE. This is the VPN label — the inner label in the two-label stack. When the remote PE receives a packet with this VPN label, it knows which VRF to look up the destination in.

### 5. Two-Label Stack Forwarding

L3VPN forwarding uses a two-label MPLS stack:

```
┌─────────────────┐
│ Transport Label  │  ← Outer: gets packet to remote PE (LDP, RSVP-TE, or SR)
├─────────────────┤
│   VPN Label      │  ← Inner: identifies customer VRF at remote PE
├─────────────────┤
│  Customer IP     │  ← Original customer packet
└─────────────────┘
```

**Forwarding lifecycle:**
1. **Ingress PE**: Customer packet arrives on VRF interface. PE looks up destination in VRF FIB. Finds BGP next-hop (remote PE) and VPN label. Pushes VPN label (inner), then pushes transport label (outer). Forwards into MPLS core.
2. **P-routers**: Swap transport label hop by hop. Never see or touch the VPN label. Don't know VPNs exist.
3. **Penultimate P-router**: PHP pops the transport label. Packet now has only the VPN label.
4. **Egress PE**: Receives packet with VPN label. Looks up VPN label — maps to a VRF and outgoing interface/next-hop. Pops VPN label, forwards customer IP packet to CE.

**Why two labels and not one**: The transport label provides reachability to the remote PE across any MPLS core topology. The VPN label provides service context at the remote PE. Separating them means the core doesn't need any VPN state — pure transport.

### 6. PE-CE Routing Protocols

The PE-CE link connects the provider's VRF to the customer's network. Multiple routing protocols are supported:

**BGP (most scalable):**
- PE runs eBGP with CE in the VRF context.
- Customer routes are received as BGP routes, converted to VPNv4 with RD and RT, sent to MP-BGP.
- Preserves BGP attributes end-to-end (AS-path, communities, MED).
- Supports SOO (Site of Origin) to prevent routing loops in multi-homed sites.

**OSPF (RFC 4577):**
- PE runs OSPF with CE in the VRF.
- Customer OSPF routes are redistributed into MP-BGP as VPNv4.
- Special handling: OSPF Domain ID, Route Type, and OSPF Extended Community are carried in BGP to reconstruct OSPF route attributes at the remote PE.
- Remote PE can re-originate the route as an OSPF inter-area (Type 3) LSA to the remote CE, preserving the customer's OSPF domain.

**Static routing:**
- Simplest PE-CE method.
- PE has static routes in the VRF pointing to the CE.
- Redistributed into MP-BGP.
- No dynamic failover without BFD or IP SLA.

**IS-IS, RIP, EIGRP**: Also supported but less common in SP deployments.

## Protocol Interactions

**L3VPN ↔ MP-BGP**: MP-BGP is the control plane for L3VPN. VPN routes are distributed as VPNv4/VPNv6 NLRI with RT extended communities. Without MP-BGP, L3VPN doesn't function. Period.

**L3VPN ↔ MPLS transport**: The MPLS data plane carries customer traffic with the two-label stack. The transport label can come from LDP, RSVP-TE, or SR-MPLS — L3VPN is agnostic to the transport signaling protocol.

**L3VPN ↔ Route Reflectors**: In any non-trivial deployment, PEs peer with route reflectors rather than full-mesh iBGP. RRs must carry VPNv4/VPNv6 address families. RT-constrained distribution (RFC 4684) prevents RRs from flooding all VPN routes to all PEs.

**L3VPN ↔ IGP**: The PE's IGP (IS-IS or OSPF for the SP backbone) provides the transport label binding (via LDP or SR) and next-hop reachability. VPN routes resolve their BGP next-hops via the IGP.

## Edge Cases & Gotchas

**1. RD is NOT an RT**: RDs provide uniqueness; RTs provide policy. Configuring them identically is common but coincidental — the functions are completely different. Changing the RD doesn't change which VRFs receive routes.

**2. Same RD across PEs breaks RR path diversity**: If PE1 and PE2 both advertise 10.0.0.0/24 with the same RD, the RR sees them as the same VPNv4 prefix and selects only one (best-path). To maintain path diversity, use unique RDs per PE.

**3. VPN label allocation mode matters at scale**: Per-prefix label allocation creates one VPN label per VPN route. With 50,000 VPN prefixes on a PE, that's 50,000 labels. Per-VRF or per-CE label allocation uses one label per VRF or CE — far fewer labels, but requires an IP lookup at the egress PE after label pop.

**4. PHP and VPN label**: When the penultimate P-router pops the transport label, the egress PE receives the packet with only the VPN label. If PHP is disabled (or explicit-null is used), the PE receives both labels. This matters for QoS (TC bits in the transport label are lost after PHP).

**5. Maximum VRFs per PE**: Platform-dependent but typically 500-4000 VRFs per PE. The limit is usually RIB/FIB memory and per-VRF control plane overhead, not a protocol limitation.
