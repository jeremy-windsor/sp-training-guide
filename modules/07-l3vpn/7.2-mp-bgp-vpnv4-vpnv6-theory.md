# 7.2 — MP-BGP for VPNv4/VPNv6 — Protocol Theory

## Protocol Overview

Multiprotocol BGP (MP-BGP) is the control plane engine that makes L3VPN scale. RFC 4760 extended BGP to carry arbitrary address families beyond IPv4 unicast. For L3VPN, the relevant address families are VPNv4 (AFI 1, SAFI 128) and VPNv6 (AFI 2, SAFI 128). The VPN prefix — RD plus customer prefix — is encoded as MP-BGP NLRI and distributed between PEs, typically via route reflectors.

The elegance of using BGP for VPN route distribution is twofold. First, BGP already solves the scaling problem — route reflectors eliminate full-mesh requirements, and BGP's inherent policy mechanisms (communities, AS-path, local preference) provide fine-grained control. Second, BGP attributes travel with the VPN routes: the VPN label, next-hop, Route Target extended communities, and any operator-defined communities are all carried in a single UPDATE message.

MP-BGP for VPN services was formalized alongside RFC 4364. The AFI/SAFI model allows a single BGP session between PEs (or PE-to-RR) to carry multiple address families simultaneously — VPNv4, VPNv6, EVPN, IPv4 unicast, IPv6 unicast — each with independent RIB and policy.

## Core Mechanisms

### 1. AFI/SAFI and Capability Negotiation

BGP speakers advertise which address families they support during the OPEN message using the Multiprotocol Extensions Capability (RFC 4760).

**VPN-relevant AFI/SAFI pairs:**

| AFI | SAFI | Address Family | Purpose |
|-----|------|----------------|---------|
| 1 | 1 | IPv4 Unicast | Regular IPv4 routing |
| 1 | 4 | IPv4 Labeled Unicast | IPv4 with MPLS labels (Inter-AS Option C) |
| 1 | 128 | VPNv4 | MPLS L3VPN for IPv4 |
| 2 | 1 | IPv6 Unicast | Regular IPv6 routing |
| 2 | 128 | VPNv6 | MPLS L3VPN for IPv6 |
| 25 | 65 | EVPN | Ethernet VPN |
| 1 | 73 | SR Policy | SR-TE policy distribution |

**Negotiation**: If PE1 sends OPEN with VPNv4 capability and PE2 doesn't include VPNv4 in its OPEN, VPNv4 routes will not be exchanged on that session. Both sides must agree.

**Multiple address families on one session**: A single BGP session between PE1 and RR1 can carry VPNv4, VPNv6, EVPN, and IPv4 unicast simultaneously. Each AF has its own Adj-RIB-In, Loc-RIB, and Adj-RIB-Out. Route policy can be applied per AF.

### 2. VPNv4/VPNv6 NLRI Encoding

**VPNv4 UPDATE message structure:**
```
MP_REACH_NLRI Attribute:
├─ AFI: 1 (IPv4)
├─ SAFI: 128 (MPLS-labeled VPN)
├─ Next Hop: 12 bytes for VPNv4 (8-byte zero RD + 4-byte IPv4 next-hop)
└─ NLRI:
    ├─ Prefix Length (1 byte): includes label (24 bits) + RD (64 bits) + IPv4 prefix
    ├─ Label (3 bytes): 20-bit label + 3 EXP + 1 S-bit
    ├─ Route Distinguisher (8 bytes)
    └─ IPv4 Prefix (variable, 0-4 bytes based on prefix length)
```

**Next-hop encoding quirk**: The VPNv4 next-hop includes a dummy RD (all zeros) prepended to the actual next-hop IPv4 address. This is because MP-BGP requires the next-hop to be in the same address family as the NLRI. The RD 0:0 is a placeholder — only the IPv4 address matters for resolution.

**VPNv6 NLRI**: Same structure but with AFI 2 and IPv6 prefix/next-hop. The next-hop can be an IPv6 address or, in common practice, an IPv4-mapped IPv6 address (::FFFF:a.b.c.d) when the PE-PE transport is IPv4 MPLS.

### 3. Extended Communities for VPN

VPN routes carry BGP Extended Communities (8 bytes each, defined in RFC 4360) for policy and metadata.

**Route Target (RT):**
- Type: 0x0002 (2-byte ASN) or 0x0102 (IPv4 address) or 0x0202 (4-byte ASN)
- Carried in the Extended Community attribute
- Multiple RTs can be attached to a single route
- Import/export matching is the core VPN policy mechanism

**Site of Origin (SOO):**
- Prevents routing loops in multi-homed VPN sites
- The PE tags routes received from a CE with the SOO value
- If a route arrives via MP-BGP with the same SOO as the local CE, the PE does not advertise it to that CE
- Critical for dual-homed sites where the same prefix could loop back

**OSPF Domain ID:**
- Used when OSPF is the PE-CE protocol
- Identifies the OSPF domain the route originated from
- Remote PEs use it to decide whether to generate Type 3 (inter-area) or Type 5 (external) LSAs

**OSPF Route Type:**
- Encodes the original OSPF route type (intra-area, inter-area, external Type 1/2, NSSA)
- Allows the remote PE to reconstruct appropriate OSPF LSAs

### 4. Label Allocation Schemes

The VPN label carried in the VPNv4/VPNv6 NLRI tells the remote PE which VRF and next-hop to use. How labels are allocated impacts scale and convergence.

**Per-prefix allocation:**
- One unique label per VPN route
- Egress PE receives packet, pops label, and knows the exact VRF + outgoing interface + next-hop
- No IP lookup needed at egress — pure label-switched forwarding
- Scales poorly: 50,000 VPN routes = 50,000 labels
- Enables Prefix-Independent Convergence (PIC) — backup labels can be pre-installed

**Per-VRF allocation:**
- One label per VRF (or per VRF per next-hop)
- Egress PE receives packet, pops label, identifies the VRF, then does an IP lookup in the VRF FIB
- Far fewer labels (one per VRF instead of one per prefix)
- Requires an IP lookup at egress — slightly slower forwarding
- Does NOT support PIC (because the label doesn't identify the specific prefix)

**Per-CE allocation:**
- One label per CE next-hop within a VRF
- Middle ground: more labels than per-VRF, fewer than per-prefix
- Egress PE identifies both the VRF and the CE — still needs an IP lookup to determine the specific prefix
- Some platforms support PIC with per-CE labels

**Vendor defaults:**
- Cisco IOS-XR: per-prefix (default)
- Junos: per-CE (default, can be changed with `vrf-table-label` for per-VRF)

### 5. Next-Hop Resolution

VPN routes have a BGP next-hop that must be resolved to a transport path.

**Resolution chain:**
1. MP-BGP VPNv4 route has next-hop = remote PE's loopback (e.g., 10.0.0.2)
2. PE looks up 10.0.0.2 in the global routing table
3. Finds an IGP route to 10.0.0.2 with an associated transport label (LDP, SR, or RSVP-TE)
4. Installs a two-label stack: transport label (outer) + VPN label (inner)

**next-hop-self**: Route reflectors typically don't change the BGP next-hop (preserving the originating PE's loopback). The receiving PE resolves the next-hop directly. If ASBR or inter-area BGP is involved, next-hop-self may be needed at boundaries.

**Recursive next-hop resolution**: In hierarchical designs (RR clusters, inter-AS), the VPN route's next-hop may resolve through multiple levels: VPNv4 next-hop → labeled unicast next-hop → IGP next-hop. Each level adds a label to the stack.

### 6. Route Reflector Design for VPN

VPN route reflectors have unique scaling considerations:

**VPN RR state**: An RR must carry ALL VPN routes from ALL PEs in its cluster. With 1000 PEs and 50,000 VPN routes per PE, the RR holds 50 million routes. Memory and CPU become the bottleneck.

**RT-constrained distribution (RFC 4684):**
- PEs advertise which RTs they are interested in (via AFI 1, SAFI 132)
- RR only sends VPN routes matching the PE's RT interest set
- Massively reduces VPN route distribution — a PE with 10 VRFs (10 import RTs) only receives routes matching those 10 RTs, not all 50 million routes
- Essential for VPN RR scaling

**RR placement for VPN:**
- Dedicated VPN RRs (not shared with IPv4 unicast RRs) for operational isolation
- Out-of-path RRs (RR not in the forwarding path) to avoid traffic tromboning
- Hierarchical RR designs for multi-region providers

## Protocol Interactions

**MP-BGP VPN ↔ IGP**: IGP provides next-hop resolution for VPN routes. The BGP next-hop (remote PE loopback) must be reachable via IGP. Transport labels from LDP/SR/RSVP-TE are resolved through the IGP.

**MP-BGP VPN ↔ PE-CE BGP**: When BGP is the PE-CE protocol, customer BGP routes are imported into the VRF, then exported as VPNv4/VPNv6 into MP-BGP. The two BGP instances (PE-CE eBGP and PE-PE iBGP VPNv4) are distinct but connected through redistribution.

**MP-BGP VPN ↔ Route Reflectors**: RRs are the scaling mechanism. VPNv4/VPNv6 sessions between PEs and RRs replace full-mesh iBGP. RT-constrained distribution prevents RR state explosion.

## Edge Cases & Gotchas

**1. Next-hop encoding in VPNv4**: The VPNv4 next-hop is NOT just an IPv4 address — it's RD(0:0) + IPv4 address (12 bytes). Packet captures and debug output may show the full 12-byte encoding.

**2. Label in NLRI vs label in tunnel**: The VPN label in the NLRI is the inner label. The transport label (outer) is NOT in the BGP UPDATE — it's resolved separately via IGP/LDP/SR. Don't confuse them.

**3. RT import order of operations**: When a VPN route arrives, the PE first checks RT match, then installs in the VRF. If RT policy changes (new import RT added), the PE must re-evaluate existing VPN routes. Some platforms require `clear bgp` or soft-reconfiguration for this.

**4. Dual-stack VPN**: A VRF can support both VPNv4 and VPNv6 simultaneously. The same RT values can be used for both address families, but the address families are processed independently. Ensure both are configured on the MP-BGP session.

**5. Maximum-prefix limits**: VPNv4/VPNv6 sessions should have maximum-prefix limits configured. A misbehaving PE advertising millions of VPN routes can overwhelm an RR. This is both a security and stability control.
