# 7.3 — Inter-AS L3VPN — Answer Key

---

## Question 1
> **Scenario:** An SP running Option B notices that adding ASBR memory (128GB → 256GB) only delayed the scaling problem by 6 months. VPNv4 table is growing 40% annually. What's the migration path, and what are the risks during transition?

### Answer

**Diagnosis: Option B is Inherently Memory-Bound**

Option B (ASBR VPNv4 Exchange) requires ASBRs to hold the **entire VPNv4 table** for the inter-AS VPN routes. With a 40% annual growth rate compounding, adding memory is a treadmill — you're buying 6 months at a time against exponential growth. Memory is not the right solution; the architecture is the problem.

The correct migration target is **Option C (Multihop MP-eBGP + Labeled Unicast)**. Option C removes VPNv4 routes from the ASBRs entirely — ASBRs carry only PE loopback routes (labeled-unicast, AFI 1 SAFI 4), which scale at O(PE count), not O(VPN prefix count). VPNv4 routes move to RR-to-RR multihop eBGP, where they belong.

**Migration Path: Option B → Option C**

The migration is performed in phases to maintain service continuity. This is a live network migration, not a maintenance window event.

**Phase 1: Deploy Inter-RR Multihop eBGP VPNv4 Session (No Traffic Impact)**

Establish the RR-to-RR multihop eBGP VPNv4/VPNv6 session between the two ASes:

```cisco
! IOS-XR (RR1, AS1):
router bgp 65000
 neighbor 10.254.1.1                    ! RR2 loopback in AS2
  remote-as 65100
  update-source Loopback0
  ebgp-multihop 10
  address-family vpnv4 unicast
   next-hop-unchanged                   ! Critical: preserve original PE next-hop
  !
  address-family vpnv6 unicast
   next-hop-unchanged
```

```junos
! Junos (RR2, AS2):
! next-hop unchanged requires an export policy — there is no inline knob
policy-statement PRESERVE-NEXT-HOP {
    then {
        next-hop unchanged;
    }
}

protocols bgp {
    group AS1-RR {
        type external;
        peer-as 65000;
        multihop { ttl 10; }
        local-address 10.254.1.1;
        export PRESERVE-NEXT-HOP;      ! Apply before configuring neighbor
        neighbor 10.254.0.1 {           ! RR1 loopback in AS1
            family inet-vpn { unicast; }
            family inet6-vpn { unicast; }
        }
    }
}
```

**Critical Junos note:** There is no `no-modify-nh` configuration option in Junos BGP. Next-hop preservation on eBGP sessions requires an explicit export policy with `next-hop unchanged` in the `then` clause. Without this policy, Junos (like all eBGP speakers by default) rewrites the next-hop to the local BGP address — which would break Option C by making the RR appear as the VPN next-hop instead of the originating PE.

At this stage, VPNv4 routes flow between RRs but can't be used yet — PE1's loopback isn't resolvable in AS2 (no labeled-unicast path exists). Routes sit in RIBs without becoming active.

**Phase 2: Establish ASBR Labeled-Unicast eBGP Sessions**

Configure eBGP labeled-unicast (AFI 1, SAFI 4) between the ASBRs to exchange PE loopback routes with labels:

```cisco
! IOS-XR (ASBR1, AS1):
router bgp 65000
 neighbor 198.51.100.2                  ! ASBR2 interface (directly connected)
  remote-as 65100
  address-family ipv4 labeled-unicast
   route-policy PE-LOOPBACKS-OUT out    ! Export only PE loopback /32s, not full tables
   route-policy PE-LOOPBACKS-IN in
```

Redistribute PE loopback routes into iBGP labeled-unicast within each AS so all PEs can resolve the remote PE next-hops:

```cisco
! IOS-XR (ASBR1): redistribute PE loopbacks from AS2 into AS1's iBGP labeled-unicast
router bgp 65000
 address-family ipv4 labeled-unicast
  redistribute connected route-policy ASBR-LOOPBACKS  ! local PE loopbacks from inter-AS
```

**Phase 3: Enable next-hop Resolution via Labeled-Unicast in PEs**

PEs need to resolve VPNv4 next-hops (remote PE loopbacks from AS2) via the labeled-unicast table, not just inet.0:

```cisco
! IOS-XR (each PE in AS1):
router bgp 65000
 address-family vpnv4 unicast
  nexthop resolution prefix-length-minimum 30   ! accept /32 labeled-unicast routes
```

```junos
! Junos (each PE in AS1): inet.3 automatically picks up labeled-unicast routes
! No additional config needed if ASBR is redistributing into iBGP properly
```

**Phase 4: Verify Option C is Carrying Traffic Before Removing Option B**

With labeled-unicast in place and RR-to-RR VPNv4 sessions active, Option C should now be fully functional. Verify:

```cisco
! IOS-XR (PE in AS1): check if remote PE loopbacks resolve with labels
show bgp ipv4 labeled-unicast 10.255.2.1/32     ! Remote PE in AS2
! Should show label, next-hop via ASBR, recursive resolution

show route vrf CUST-X 10.2.0.0/24              ! VPN route via Option C path
! Should show with 3-label stack (transport + stitching + VPN)
```

**Phase 5: Decommission Option B ASBR eBGP VPNv4 Sessions**

Once Option C is confirmed operational and carrying production traffic, shut down the ASBR eBGP VPNv4 sessions. Monitor for any VPN drops. Remove the VPNv4 address-family from the ASBR-to-ASBR BGP session.

**Risks During Transition**

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| **VPN black hole during switchover** | Medium | Keep Option B active until Option C fully verified; brief parallel operation is safe |
| **Three-label stack drops on legacy hardware** | Low-Medium | Audit all hardware in the forwarding path for 3-label support before enabling Option C |
| **Labeled-unicast route leakage** | Medium | Apply strict route-policy on ASBR to export ONLY PE /32 loopbacks — never customer prefixes or full routing tables |
| **next-hop-unchanged misconfiguration** | High risk if missed | Without `next-hop-unchanged` route-policy on the inter-RR eBGP session (IOS-XR and Junos both use export policy with `next-hop unchanged`), PE next-hops get rewritten to the RR, breaking VPN label resolution |
| **RR loopback reachability for multihop eBGP** | Medium | RR loopbacks must be mutually reachable across ASes — either advertise via labeled-unicast or use a static route to the inter-AS links |

## Question 2
> **Design:** You're designing inter-AS L3VPN between an acquired regional network (AS 65100, 200 VPNs) and your national backbone (AS 65000, 8,000 VPNs). Only 50 VPNs need to span both ASes today, growing to 500 over 2 years. Which option do you deploy and why? What changes if the acquired network runs Junos and your backbone is IOS-XR?

### Answer

**Recommended Option: Option B (ASBR eBGP VPNv4)**

At 50 VPNs today growing to 500 in 2 years, the ASBR VPNv4 table for inter-AS routes will be a small fraction of the regional network's 200-VPN table. This is firmly within Option B's comfort zone.

**Why NOT Option A:**
- 50 VPNs today × 2 VRFs per ASBR pair (one per AS) = 100 VRFs on each ASBR pair plus 100 eBGP sessions. Manageable now, but growing to 500 VPNs = 500 VRFs + 500 sessions. Option A stops scaling before you reach the 2-year target.

**Why NOT Option C:**
- Option C's complexity (multihop eBGP, labeled-unicast, 3-label stack) is justified when VPN prefix counts in the millions are at stake. At 500 VPNs total (a generous estimate for inter-AS), even with 1,000 routes per VPN that's 500K prefixes — well within Option B's range on modern ASBRs.
- Option C also requires both networks to support labeled-unicast (AFI 1, SAFI 4) with label stitching. In an acquisition scenario with heterogeneous gear, this is an unnecessary risk.

**Option B is the right answer** for this scale, this timeline, and this acquisition scenario. Design it properly with RT filtering so ASBRs don't accidentally import all 8,000 VPNs from the backbone:

```cisco
! IOS-XR (ASBR, national backbone side):
! Define explicit extcommunity-set — inline ranges with "through" are not supported
extcommunity-set rt INTER-AS-ALLOWED-RTS
  65100:1,
  65100:2,
  65100:3,
  ! ... add each inter-AS VPN RT explicitly ...
  65100:50
end-set

route-policy INTER-AS-VPN-IN
  if extcommunity rt matches-any INTER-AS-ALLOWED-RTS then
    pass
  endif
  drop
end-policy

router bgp 65000
 neighbor 203.0.113.1                   ! Regional ASBR
  address-family vpnv4 unicast
   route-policy INTER-AS-VPN-IN in
   next-hop-self                        ! ASBR rewrites next-hop to itself
```

Plan for Option C migration at the 18-month mark if growth trends toward 1,000+ inter-AS VPNs.

**Impact of IOS-XR (national) vs Junos (regional) Heterogeneity**

Option B works cleanly in a dual-vendor environment. The key platform-specific differences to resolve:

**1. VPN Label Allocation — Behavioral Difference**

IOS-XR default: per-prefix label allocation.  
Junos default (without `vrf-table-label`): per-next-hop allocation.

The ASBR receives VPN routes from the remote AS and allocates **new** VPN labels for re-advertisement into the local AS. Label allocation mode on the ASBR must be consistent with what the rest of the local AS expects. No cross-vendor issue here — each AS handles its own labels independently. The ASBR label-swaps at the boundary, so both ASes use their own label spaces.

**2. ASBR next-hop-self Behavior**

Both platforms support ASBR next-hop-self for VPNv4 eBGP. Verify the exact command:

```cisco
! IOS-XR:
router bgp 65000
 neighbor <ASBR2-IP>
  address-family vpnv4 unicast
   next-hop-self                        ! Mandatory for Option B
```

```junos
! Junos:
protocols bgp {
    group INTER-AS {
        neighbor 198.51.100.1 {
            family inet-vpn {
                unicast;
            }
        }
    }
}
! next-hop-self is implicit for eBGP on Junos — verify with:
show bgp neighbor <IP> | match "Next hop"
```

**3. Extended Community Format — No Issue**

Route Targets are carried in BGP Extended Communities (RFC 4360), which both IOS-XR and Junos implement identically. The ASIC-level formats are vendor-specific but the on-wire format is standard. No interop issue.

**4. PE-CE Protocol Interop**

Verify the regional Junos PEs can peer PE-CE with existing customer CPE. Junos supports eBGP, OSPF (with sham-link), static, and RIPv2 as PE-CE protocols — the same set as IOS-XR. No issues expected.

**5. RT-Constraint Across eBGP**

RT-Constraint (RFC 4684) does NOT cross eBGP boundaries natively. Each AS runs RT-Constraint internally. The ASBRs handle inter-AS VPN route filtering via explicit RT-based route-policy on the eBGP VPNv4 session, which you've already planned above.

## Question 3
> **Troubleshooting:** Option C deployment. `show bgp vpnv4 unicast` on PE2 shows routes for Customer-A with next-hop 10.0.0.1 (PE1 in AS1), but pings from CE2 to CE1 fail. `show bgp ipv4 labeled-unicast` on ASBR2 shows 10.0.0.1/32 with a label from ASBR1. Where do you look next, and what's the most likely cause?

### Answer

**Reading the Clue**

The presence of `10.0.0.1/32` in `show bgp ipv4 labeled-unicast` on ASBR2 confirms that labeled-unicast eBGP between the ASBRs is functioning — ASBR2 has the stitching label for PE1's loopback. The VPNv4 route is in PE2's BGP table. So the control plane looks correct at first glance.

The failure is in the **data plane label forwarding chain**. Somewhere between CE2 and CE1, a label is wrong, unresolved, or the label stack isn't being built correctly.

**Step 1: Verify the Full Label Stack on PE2**

Option C uses a 3-label stack. Confirm PE2 is building all three labels:

```cisco
! IOS-XR (PE2):
show bgp vpnv4 unicast vrf CUST-A 10.1.0.0/24 detail
! Look for: 
!   - Remote PE next-hop (10.0.0.1)
!   - VPN label (inner, allocated by PE1)
!   - Stitching label (from labeled-unicast, how to get from ASBR2 to PE1)
!   - Transport label (outer, how to get from PE2 to ASBR2 via AS2 LDP/SR)

show cef vrf CUST-A 10.1.0.0/24 detail
! Should show 3-label stack: [transport-to-ASBR2][stitching][VPN]
```

```junos
! Junos (PE2):
show route table CUST-A.inet.0 10.1.0.0/24 detail
show route 10.0.0.1 table inet.3 detail    ! Check stitching label resolution
```

**Most Likely Cause: Stitching Label Not Redistributed from ASBR2 into AS2 iBGP**

ASBR2 has the stitching label in its own labeled-unicast table, but **PE2 doesn't know about it**. For PE2 to resolve VPN next-hop 10.0.0.1, it needs a labeled route to 10.0.0.1 in its own routing table. This comes from ASBR2 redistributing the labeled-unicast routes it learned from ASBR1 into AS2's iBGP labeled-unicast.

If ASBR2 accepted the route from ASBR1 but didn't redistribute it into iBGP, PE2 has no labeled path to PE1's loopback. PE2 might have a BGP VPNv4 route pointing to 10.0.0.1, but `10.0.0.1/32` is unreachable (or only reachable via the global table without a label), so the VPN route is installed in BGP but never becomes active in the VRF.

Verify on PE2:
```cisco
! IOS-XR:
show bgp ipv4 labeled-unicast 10.0.0.1/32
! If this shows nothing, ASBR2 isn't redistributing labeled-unicast into AS2 iBGP
```

Fix on ASBR2 — the eBGP labeled-unicast routes from ASBR1 are already in ASBR2's BGP table. They need to be re-advertised to AS2's PEs and RR via iBGP labeled-unicast. The fix is ensuring the labeled-unicast address-family is activated on the iBGP sessions:

```cisco
! IOS-XR (ASBR2): ensure labeled-unicast AF is active on iBGP sessions toward AS2
router bgp 65100
 neighbor-group AS2-RR
  address-family ipv4 labeled-unicast   ! Must be present — carries PE /32 routes with labels
 !
 neighbor-group AS2-PEs
  address-family ipv4 labeled-unicast   ! PEs need to receive labeled routes to resolve VPN next-hops
```

The `redistribute static` approach doesn't help here — ASBR2's routes to PE1's loopback were learned via eBGP labeled-unicast, not as static routes. BGP routes redistribute into other BGP AFIs only if the address-family sessions are properly activated and the routes are redistributed/readvertised via BGP (not via `redistribute static`).

**Step 2: Check ASBR1 Label Stitching**

ASBR2 should swap the stitching label and forward to ASBR1. ASBR1 pops the stitching label (which maps to "forward toward PE1") and imposes transport toward PE1 within AS1. If ASBR1's stitching label doesn't map correctly:

```cisco
! IOS-XR (ASBR1):
show mpls forwarding labels <stitching-label>
! Should show action: Pop (or Swap to transport), output interface toward PE1
```

**Step 3: Verify PHP Interaction with 3-Label Stack**

Penultimate Hop Popping in AS2 removes the transport label before ASBR2, so ASBR2 receives `[stitching][VPN]`. ASBR2 swaps the stitching label to ASBR1's label. The inter-ASBR link then carries `[ASBR1-label][VPN]`. ASBR1 sees this and must know what to do with its own label — specifically, it should pop its label and impose transport toward PE1 in AS1.

Check ASBR1's LFIB for its own stitching label:
```cisco
show mpls forwarding labels <ASBR1-label>
! Should show: Pop label, push transport-to-PE1
```

If the label is missing or maps to an unexpected destination, the labeled-unicast session between ASBRs may have flapped and labels were reallocated without ASBR2 being updated. A soft-reset of the labeled-unicast session will resync labels:

```cisco
! IOS-XR:
clear bgp ipv4 labeled-unicast <ASBR2-IP> soft
```

**Summary of Diagnosis Order:**
1. `show bgp ipv4 labeled-unicast 10.0.0.1/32` on PE2 — confirm stitching label is known
2. `show cef vrf CUST-A 10.1.0.0/24 detail` on PE2 — confirm 3-label stack in FIB
3. `show mpls forwarding` on ASBR1 and ASBR2 — confirm label swap actions
4. `show bgp ipv4 labeled-unicast` on ASBR2 — confirm stitching label is being redistributed into AS2 iBGP
