# 7.3 — Inter-AS L3VPN — Protocol Theory

## Protocol Overview

When a VPN customer spans multiple Autonomous Systems — different providers, different regions of the same provider with separate AS numbers, or acquired networks not yet merged — standard iBGP VPNv4 breaks at the AS boundary. iBGP sessions carry VPNv4 routes between PEs within an AS, but iBGP doesn't cross AS boundaries. The customer needs end-to-end VPN connectivity. The provider needs to stitch the VPN across the boundary.

RFC 4364 Section 10 defines three canonical approaches, universally known as Option A, Option B, and Option C. Each makes a different trade-off between simplicity, scalability, and inter-provider trust. This is one of the most heavily examined topics in both CCIE-SP and JNCIE-SP — not because the concepts are complex, but because the operational implications of choosing the wrong option at scale are severe.

---

## Core Mechanisms

### 1. Option A — Back-to-Back VRF

**Architecture:**
The ASBRs behave as if the inter-AS link is a PE-CE link. ASBR1 has VRFs. ASBR2 has VRFs. Each VPN is connected via a per-VPN sub-interface between the ASBRs.

**How it works:**
1. PE1 (AS1) sends VPNv4 route to RR1 (AS1) via iBGP.
2. RR1 sends to ASBR1 (AS1). ASBR1 imports into local VRF.
3. ASBR1 runs a PE-CE protocol (BGP, OSPF, static) with ASBR2 across the inter-AS link, per VRF.
4. ASBR2 receives the route in its VRF, exports as VPNv4 into AS2's iBGP.
5. RR2 (AS2) distributes to PE2 (AS2).

**Properties:**
- **Simple**: No VPNv4 eBGP needed. Standard PE-CE routing.
- **Full isolation**: Each AS manages its own VPN independently. No trust required.
- **Doesn't scale**: Each VPN needs a dedicated sub-interface on the ASBR link. 500 VPNs = 500 sub-interfaces. ASBR becomes the bottleneck.
- **Control plane overhead**: ASBR runs VRF routing instances for every VPN. Memory and CPU intensive.

**When to use**: Small number of VPNs crossing the boundary. Inter-provider peering where trust is low. Merging acquired networks temporarily.

### 2. Option B — ASBR VPNv4/VPNv6 Exchange

**Architecture:**
ASBRs run eBGP VPNv4 sessions directly with each other. VPN routes cross the AS boundary as VPNv4 NLRI. The ASBR performs next-hop-self and label swapping.

**How it works:**
1. PE1 (AS1) sends VPNv4 route (label L1, next-hop PE1) to RR1.
2. RR1 sends to ASBR1. ASBR1 holds the route with label L1.
3. ASBR1 sends VPNv4 route via eBGP to ASBR2. ASBR1 does next-hop-self, allocates a new label L2, and advertises {VPNv4 prefix, label L2, next-hop ASBR1}.
4. ASBR2 receives, does next-hop-self again, allocates label L3. Sends to RR2 with {VPNv4 prefix, label L3, next-hop ASBR2}.
5. PE2 resolves via RR2.

**Label stitching at the ASBR:**
When traffic flows PE2 → ASBR2 → ASBR1 → PE1:
- ASBR2 receives packet with transport label (to ASBR2) + VPN label L3.
- ASBR2 swaps VPN label L3 → L2, pushes transport label for ASBR1.
- ASBR1 receives, swaps VPN label L2 → L1, pushes transport label for PE1.

Each ASBR maintains a label mapping: incoming label → outgoing label + next-hop.

**Properties:**
- **Scales better than A**: No per-VPN sub-interfaces. VPNv4 eBGP carries all VPNs on one session.
- **ASBR holds VPN state**: Both ASBRs must carry the full VPNv4 table. This can be millions of routes on a large provider's ASBR.
- **Next-hop-self breaks end-to-end label path**: Each ASBR rewrites the next-hop and allocates new labels. Three label swaps per direction minimum.
- **Moderate trust**: ASBRs exchange VPN routes but not internal topology.
- **RT filtering at ASBR**: ASBRs should filter which RTs are accepted to prevent one AS from injecting routes into unauthorized VPNs.

**When to use**: Medium-scale inter-AS. Single provider with multiple ASes. Moderate trust between the AS operators.

### 3. Option C — Multihop MP-BGP + Labeled Unicast

**Architecture:**
VPNv4 routes are exchanged directly between PEs (or RRs) across AS boundaries via multihop eBGP. The ASBRs only exchange labeled IPv4 routes (the PE loopbacks), providing transport reachability without carrying any VPN state.

**How it works:**
1. ASBR1 and ASBR2 exchange labeled unicast routes (AFI 1, SAFI 4) for PE loopbacks via eBGP. ASBR1 advertises PE1's loopback with a label; ASBR2 advertises PE2's loopback with a label.
2. PE1 and PE2 (or RR1 and RR2) establish multihop eBGP VPNv4 sessions directly.
3. VPNv4 routes are exchanged PE-to-PE with the original VPN labels and next-hops.
4. Traffic from PE2 to PE1: PE2 pushes VPN label + transport label toward ASBR2. ASBR2 swaps transport label (labeled unicast label). ASBR1 swaps again. Final transport label delivers to PE1. PE1 processes VPN label.

**The key insight**: ASBRs carry only PE loopback routes with labels (a few hundred entries), not the full VPNv4 table (potentially millions of entries). VPN state is end-to-end between PEs.

**Properties:**
- **Maximum scale**: ASBRs carry zero VPN state. VPN scale is limited only by PE/RR capacity.
- **End-to-end labels**: VPN label from originating PE is preserved end-to-end. No label stitching at ASBRs.
- **Higher trust required**: RRs or PEs from different ASes peer directly. This exposes internal PE loopbacks across AS boundaries.
- **Labeled unicast on ASBRs**: ASBRs must support AFI 1/SAFI 4 (labeled unicast BGP) and program label swaps for PE loopback reachability.
- **Complex troubleshooting**: Three levels of label (VPN label, inter-AS labeled unicast, intra-AS transport). Debugging requires understanding the full label stack.

**When to use**: Large-scale deployments. Same provider, multiple regions. When ASBR VPN state is the bottleneck.

### 4. Option AB and Hybrid Approaches

Real deployments often combine options:

**Option AB (Cisco terminology):**
- ASBR terminates VRFs like Option A (per-VPN interfaces)
- But uses VPNv4 eBGP like Option B on the inter-AS link
- VRFs on the ASBR provide isolation; VPNv4 eBGP provides efficient route exchange
- Better scaling than pure Option A, better isolation than pure Option B

**Option C with RR-to-RR peering:**
- Instead of PE-to-PE multihop eBGP, RR1 peers with RR2
- Reduces the number of multihop sessions
- RRs relay VPN routes between ASes
- Most common Option C deployment model

### 5. Label Stack Depth Across Inter-AS

Each Inter-AS option has different label stack requirements:

**Option A**: Two labels per AS (transport + VPN), but labels are independent per AS. No cross-AS label stack.

**Option B**: Three labels minimum in transit: transport (to ASBR) + ASBR-allocated VPN label. ASBRs swap VPN labels. Stack depth at any point is 2.

**Option C**: Three or more labels: intra-AS transport + inter-AS labeled unicast + VPN label. At transit points, the stack can be 3 deep. With segment routing, additional SIDs may be needed.

## Protocol Interactions

**Inter-AS ↔ Labeled Unicast BGP**: Option C depends on labeled unicast BGP (RFC 3107) for inter-AS PE reachability. Without it, PE loopbacks can't be reached across AS boundaries with MPLS labels.

**Inter-AS ↔ RT filtering**: In Option B, the ASBR must filter RTs on the eBGP VPNv4 session. Without RT filtering, one AS could inject routes into any VPN in the remote AS. This is both a security and scaling concern.

**Inter-AS ↔ Route Reflectors**: Option C's multihop eBGP sessions often terminate on RRs rather than individual PEs. This keeps the number of cross-AS sessions manageable but concentrates VPN state on the RRs.

## Edge Cases & Gotchas

**1. Option B ASBR memory**: In large deployments, Option B ASBRs must hold the full VPNv4 table from both ASes. This can exceed hardware capacity on older platforms.

**2. Option C requires MPLS on the inter-AS link**: The labeled unicast routes exchanged between ASBRs need MPLS forwarding on the inter-AS link. If the link is a raw IP link without MPLS, Option C doesn't work.

**3. TTL propagation across AS boundaries**: MPLS TTL behavior may differ between ASes. Some providers disable TTL propagation (no-propagate-ttl) to hide internal topology. This can cause asymmetric TTL behavior in traceroute across inter-AS boundaries.

**4. SOO in multi-homed inter-AS sites**: If a VPN site is dual-homed to PEs in different ASes, SOO must be coordinated across both ASes to prevent routing loops. This requires operational agreement between the ASes.

**5. Option A sub-interface scale on ASBR**: The per-VPN sub-interface requirement means ASBR platforms need to support thousands of logical interfaces. Some platforms hit ASIC limits before software limits.
