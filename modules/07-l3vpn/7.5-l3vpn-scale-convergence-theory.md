# 7.5 — L3VPN Scale & Convergence — Protocol Theory

## Protocol Overview

L3VPN convergence is where theory meets thermodynamics. A provider with 50,000 VPN prefixes per PE, 400 VRFs, and a core link failure needs sub-second recovery — not the 45-second reconvergence event that makes the NOC phone ring at 2am. This section covers the protocol-level mechanisms that determine how fast L3VPN recovers from failures: BGP convergence mechanics, label allocation's impact on forwarding table updates, Prefix-Independent Convergence (PIC), and the interplay between IGP convergence, MPLS transport convergence, and BGP VPN route convergence.

The fundamental scaling dimensions of L3VPN are: number of VRFs per PE, number of VPN routes per VRF, number of PEs per route reflector, total VPN routes in the network, and convergence time as a function of all the above. Each dimension has protocol-level implications.

## Core Mechanisms

### 1. L3VPN Convergence Layers

An L3VPN failure triggers convergence at three layers, each with different timescales:

**Layer 1 — Failure detection:**
- Interface down: immediate (milliseconds).
- BFD: configurable, typically 50-150ms (3 × 50ms intervals).
- IGP hello timeout: seconds (IS-IS hello interval × multiplier).

**Layer 2 — Transport convergence (IGP + MPLS):**
- IGP SPF recomputation: 10-200ms depending on SPF timers and network size.
- LDP/SR label convergence: follows IGP, typically adds minimal delay with SR (label is index-based), more with LDP (session-based).
- RSVP-TE FRR: pre-installed backup, switchover in milliseconds.
- TI-LFA: pre-installed backup segment list, switchover in microseconds.
- **Total**: 50-500ms for transport to reconverge.

**Layer 3 — VPN convergence (BGP):**
- BGP detects next-hop unreachable (triggered by IGP convergence).
- BGP withdraws VPN routes or selects alternate paths.
- BGP best-path recalculation for affected VPN routes.
- RIB/FIB reprogramming for affected entries.
- **Total**: 200ms to several seconds depending on number of affected prefixes and platform.

**The gap**: Transport converges in milliseconds. VPN convergence can take seconds. This gap is what PIC addresses.

### 2. Prefix-Independent Convergence (PIC)

PIC is the key mechanism for achieving fast L3VPN convergence. The idea: separate the forwarding entry (prefix → next-hop) from the next-hop resolution (next-hop → outgoing interface/label). When a failure affects a next-hop, update only the next-hop resolution — don't walk through every prefix individually.

**PIC Edge (PE failure):**

Without PIC: PE1 has 50,000 VPN routes pointing to PE2 as next-hop. PE2 fails. PE1 must update all 50,000 FIB entries individually. Time = O(number of prefixes).

With PIC: PE1 has 50,000 VPN routes pointing to a next-hop indirection object for PE2. When PE2 fails, PE1 updates the single indirection object to point to the backup next-hop (PE3). All 50,000 prefixes instantly follow. Time = O(1).

**How PIC Edge works:**
1. BGP installs both primary and backup paths for VPN prefixes during steady state.
2. Primary path: next-hop PE2 (via MPLS transport). Backup path: next-hop PE3 (alternate PE for the same VPN site).
3. Both paths share an indirection structure keyed by next-hop.
4. On PE2 failure: the indirection for PE2 switches to PE3. One FIB update covers all 50,000 prefixes.

**Label allocation considerations**: Per-prefix label allocation provides natural path distinction at egress PEs and simplifies PIC implementation. Per-VRF labeling may limit PIC on some platforms, but this is implementation-dependent — PIC is fundamentally about preinstalled alternate path recursion/indirection, not label granularity.

**PIC Core (P-router/link failure):**
- A core link or P-router failure doesn't change the BGP next-hop (the remote PE is still up, just reachable via a different path).
- Transport convergence (IGP + MPLS) handles this. The transport label path to the remote PE updates; VPN routes don't need to reconverge.
- PIC Core is essentially "transport convergence is independent of VPN convergence." SR and TI-LFA make PIC Core near-instant.

### 3. BGP Bestpath and VPN Route Processing

When a PE failure or next-hop change triggers BGP reconvergence, the processing cost depends on the number of affected routes and the bestpath algorithm.

**BGP scanner**: Most implementations use a periodic BGP scanner (every 60 seconds by default) to validate next-hop reachability. When a next-hop becomes unreachable between scanner runs, the router may not detect it until the next scan — delaying VPN convergence.

**Event-driven next-hop tracking**: Modern implementations use event-driven tracking. When the IGP withdraws a route to a PE loopback, the BGP process is immediately notified. All VPN routes with that next-hop are reconverged without waiting for the scanner.

**BGP bestpath computation cost**: For N affected VPN routes, bestpath must be recomputed for each. On modern platforms: 10,000-50,000 bestpath computations per second. With 100,000 affected routes: 2-10 seconds for bestpath alone.

**Batched FIB updates**: After bestpath, RIB entries must be programmed into hardware FIB. Modern platforms batch FIB updates for efficiency. Programming rate: 50,000-200,000 entries per second depending on platform and ASIC.

### 4. Route Reflector Convergence Impact

Route reflectors introduce additional convergence delay:

**RR processing path on PE failure:**
1. Originating PE goes down.
2. Adjacent IGP routers detect the failure and flood updated LSPs/LSAs.
3. RR's IGP reconverges. RR detects that PE's loopback is unreachable.
4. RR marks all VPN routes from that PE as having an unreachable next-hop.
5. RR runs bestpath for affected routes. If backup paths exist (from other PEs), RR selects and advertises them.
6. RR sends BGP UPDATE/WITHDRAW to all client PEs.
7. Client PEs receive, run local bestpath, update FIB.

**RR-added latency**: Each step adds time. In practice, RR convergence adds 1-5 seconds to VPN convergence compared to PIC (which switches locally in milliseconds).

**BGP Add-Path on RR**: Without Add-Path, the RR only advertises the single best path per VPN prefix. If the best path fails, the RR must wait for a withdrawal, select a new best path, and advertise it — a serial process. With Add-Path, the RR sends multiple paths to client PEs. The PE has both primary and backup paths pre-installed, enabling local PIC switchover without waiting for the RR.

### 5. VPN Scale Dimensions

**VRFs per PE:**
- Each VRF consumes: routing protocol instances (PE-CE), RIB memory, FIB TCAM entries, label space.
- Typical limits: 500-4000 VRFs per PE (platform-dependent).
- Scaling concern: per-VRF control plane overhead (OSPF hello/dead timers, BGP sessions per VRF).

**VPN routes per PE:**
- Total routes across all VRFs.
- Typical limits: 1-4 million VPN routes per PE on high-end platforms.
- Scaling concern: RIB memory, FIB TCAM capacity, bestpath computation time during convergence.

**VPN routes per RR:**
- RR holds routes from all PEs in the cluster.
- Can reach tens of millions of VPN routes.
- RT-constrained distribution (RFC 4684) is essential to limit what the RR sends to each PE.

**BGP sessions per PE:**
- PE-to-RR: typically 2-4 sessions (redundant RRs).
- PE-CE BGP sessions: one per VRF with BGP PE-CE. Hundreds to thousands.
- Each session: TCP state, BGP FSM, keepalive timers.

### 6. Graceful Restart and NSR

**BGP Graceful Restart (RFC 4724):**
- When a PE's BGP process restarts (software upgrade, crash), GR preserves the forwarding state.
- The restarting PE tells neighbors "I'm restarting — keep my routes in your RIB and keep forwarding."
- Neighbors hold routes for the GR restart timer (typically 120 seconds).
- After restart, the PE re-exchanges routes with neighbors.
- Traffic continues forwarding during the restart — no VPN outage.

**Non-Stop Routing (NSR):**
- BGP state is replicated to a standby route processor.
- On RP failover, the standby takes over with full BGP state — no restart needed.
- Neighbors don't even detect the failover.
- NSR provides faster recovery than GR (no re-exchange of routes).

**NSR vs GR trade-off:**
- NSR: zero convergence impact but requires dual RP and state synchronization overhead.
- GR: works with single RP but has a convergence window during restart.

### 7. Label Space and FIB Scale

**Transport labels**: LDP or SR allocates transport labels for PE loopbacks. With SR (prefix-SID), the number is small — one per node. With LDP, one per prefix per neighbor — can be large.

**VPN labels**: With per-prefix allocation, one label per VPN route. 100,000 VPN routes = 100,000 labels. Each label is a FIB entry. Some platforms share FIB entries between the IP FIB and the label FIB; others have separate label FIBs with independent limits.

**ECMP and labels**: With per-prefix VPN labels and ECMP to the next-hop PE, the FIB must program multiple paths per label entry. This multiplies FIB resource consumption.

## Protocol Interactions

**VPN Convergence ↔ IGP Convergence**: IGP convergence triggers transport reconvergence. If the IGP is slow (large network, slow SPF), VPN convergence waits. Fast IGP convergence (sub-100ms SPF, BFD) is prerequisite for fast VPN convergence.

**VPN Convergence ↔ PIC**: PIC decouples VPN convergence from the number of affected prefixes. Without PIC, convergence time scales linearly with prefix count. With PIC, it's constant.

**VPN Scale ↔ RR Design**: The number of VPN routes determines RR memory and CPU requirements. RT-constrained distribution and Add-Path are essential at scale.

**VPN Convergence ↔ Label Allocation**: Per-prefix labels enable PIC. Per-VRF labels save label space but sacrifice PIC capability. The trade-off is convergence speed vs. resource efficiency.

## Edge Cases & Gotchas

**1. BGP scanner delay**: If event-driven next-hop tracking is not enabled, VPN convergence waits for the BGP scanner (up to 60 seconds). Always enable event-driven tracking in production.

**2. PIC requires backup paths**: PIC only helps when a backup path exists in BGP. For single-homed sites (one PE), there is no backup path. PIC doesn't apply. Multi-homing is the prerequisite.

**3. FIB programming rate as the bottleneck**: Even with fast bestpath computation, FIB programming rate may limit convergence. 200,000 FIB updates at 50,000/second = 4 seconds. Platform ASIC generation matters.

**4. RR cluster split and VPN blackholing**: If the RR cluster loses connectivity between sub-clusters, VPN routes from PEs in one sub-cluster may become unreachable to PEs in the other. Redundant RRs in diverse locations mitigate this.

**5. GR during VPN convergence**: If a PE is restarting with GR while a remote PE failure occurs, the restarting PE can't update its VPN routes — it's still re-exchanging state. Traffic may blackhole until GR completes. The interaction between GR and independent failures is a known operational risk.
