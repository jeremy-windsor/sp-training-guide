# 7.4 — Extranet & Shared Services — Answer Key

---

## Question 1
> **Design:** A service provider has 50 VPN customers who all need access to a centralized internet breakout and DNS service. Five specific customer pairs also need bilateral extranet connectivity. Design the RT allocation scheme. How many unique RTs do you need? How do you prevent the shared services hub from creating unintended connectivity between extranet pairs?

### Answer

**Classifying the Requirements**

Two distinct connectivity patterns are present:
1. **Shared services** (hub-and-spoke): All 50 customers → centralized internet + DNS hub. O(n) scaling — adding customer N requires one RT import change.
2. **Extranet** (bilateral pairs): 5 customer pairs need full bidirectional connectivity. Only 5 pairs → O(n²) at this scale is 10 RT configurations.

**RT Allocation Scheme**

| RT | Purpose | Exported By | Imported By |
|----|---------|------------|-------------|
| `65000:100` through `65000:150` | Base VPN RT per customer | Each customer's VRF (unique) | Same customer's VRFs only |
| `65000:9000` | Shared services — forward (hub → spokes) | Hub VRF only | ALL 50 customer VRFs |
| `65000:9001` | Shared services — return (spokes → hub) | ALL 50 customer VRFs | Hub VRF only |
| `65000:8001` through `65000:8005` | Extranet pair RTs (one per pair) | Both VRFs in each pair | Both VRFs in each pair |

**Count:** 50 (base) + 1 (shared-fwd) + 1 (shared-return) + 5 (extranet pairs) = **57 unique RTs**

**Hub VRF Configuration (Shared Services):**

```cisco
! IOS-XR:
vrf INTERNET-HUB
 address-family ipv4 unicast
  import route-target
   65000:9001                           ! Accept return routes from all customers
  !
  export route-target
   65000:9000                           ! Advertise internet/DNS prefixes to all customers
```

```junos
! Junos:
routing-instances INTERNET-HUB {
    instance-type vrf;
    vrf-import HUB-IMPORT;
    vrf-export HUB-EXPORT;
}
policy-statement HUB-IMPORT {
    term from-spokes {
        from community rt-9001;
        then accept;
    }
    then reject;
}
policy-statement HUB-EXPORT {
    term to-spokes {
        then { community set rt-9000; accept; }
    }
}
community rt-9000 members target:65000:9000;
community rt-9001 members target:65000:9001;
```

**Spoke VRF Configuration (Customer 1 example, no extranet):**

```cisco
! IOS-XR:
vrf CUST-1
 address-family ipv4 unicast
  import route-target
   65000:100                            ! Base VPN RT
   65000:9000                           ! Internet/DNS routes from hub
  !
  export route-target
   65000:100                            ! Base VPN export
   65000:9001                           ! Return route to hub (for internet traffic)
```

**Extranet VRF Configuration (Customer 1 and Customer 2 bilateral pair):**

Customer 1 and Customer 2 form extranet pair 1, using dedicated RT 65000:8001:

```cisco
! IOS-XR (CUST-1 — in extranet pair 1):
vrf CUST-1
 address-family ipv4 unicast
  import route-target
   65000:100                            ! Own VPN routes
   65000:9000                           ! Shared services
   65000:8001                           ! Extranet pair 1 (receives CUST-2 routes)
  !
  export route-target
   65000:100                            ! Base VPN
   65000:9001                           ! Shared services return
   65000:8001                           ! Extranet pair 1 (advertises to CUST-2)
```

```cisco
! IOS-XR (CUST-2 — in extranet pair 1):
vrf CUST-2
 address-family ipv4 unicast
  import route-target
   65000:101                            ! Own VPN
   65000:9000                           ! Shared services
   65000:8001                           ! Extranet pair 1 (receives CUST-1 routes)
  !
  export route-target
   65000:101                            ! Base VPN
   65000:9001                           ! Shared services return
   65000:8001                           ! Extranet pair 1 (advertises to CUST-1)
```

**Preventing Unintended Connectivity Between Extranet Pairs**

The structural protection is already built into the dedicated-RT design: CUST-3/CUST-4 (extranet pair 2) use RT 65000:8002, which CUST-1 and CUST-2 neither export nor import. No route leakage is possible between pairs without explicit configuration.

The real danger is the **hub VRF creating transitive leakage**. If the hub VRF blindly re-exports everything it imports (i.e., export policy adds RT 65000:9000 to ALL routes in the VRF including imported customer routes), every spoke VRF would receive every other customer's prefixes via the hub. This turns a hub-and-spoke into an accidental full mesh.

**Prevention: Hub Export Policy Must Only Export Hub-Originated Prefixes**

The hub export policy must be selective — only service prefixes (internet default route, DNS, NTP) get exported with RT 65000:9000. Customer-originated routes imported via RT 65000:9001 must NOT be re-exported:

```cisco
! IOS-XR: Hub export policy — only export hub's own service prefixes
prefix-set HUB-SERVICES
  0.0.0.0/0,               ! Internet default
  10.99.0.0/24,            ! DNS service
  10.99.1.0/24             ! NTP service
end-set

route-policy HUB-EXPORT
  if destination in HUB-SERVICES then
    set extcommunity rt (65000:9000) additive
    pass
  else
    drop                                ! Suppress customer routes from being re-exported
  endif
end-policy

vrf INTERNET-HUB
 address-family ipv4 unicast
  export route-policy HUB-EXPORT
```

```junos
! Junos: equivalent prefix-matching export policy
policy-statement HUB-EXPORT {
    term hub-services-only {
        from {
            route-filter 0.0.0.0/0 exact;
            route-filter 10.99.0.0/24 exact;
            route-filter 10.99.1.0/24 exact;
        }
        then {
            community set rt-9000;
            accept;
        }
    }
    term block-customer-routes {
        then reject;
    }
}
```

With this policy, no customer prefix ever leaves the hub VRF with RT 65000:9000. Each customer exports their routes with RT 65000:9001 (visible only to the hub for return routing) and imports RT 65000:9000 (visible only from the hub's service prefixes). The isolation is mathematically enforced by the export policy.

## Question 2
> **Troubleshooting:** Customer reports they can reach the shared DNS server (10.99.0.53) from Site 1 but not from Site 2. Both sites are in the same VRF. Site 2's PE is in a different AS (Inter-AS Option B). What are the three most likely causes?

### Answer

**Context**

Site 1 works = the shared services architecture itself is functional. The shared DNS route (10.99.0.53 in hub VRF, exported with RT 65000:9000) is reaching PE1 (Site 1's PE) correctly. The problem is isolated to Site 2's PE, which is in a different AS connected via Option B.

**Cause 1: Option B RT Filtering Is Blocking the Shared Services RT**

The ASBR running Option B has explicit RT-based route-policy to control which VPNv4 routes cross the AS boundary. If that policy was configured to allow only the customer's base VPN RT (e.g., 65000:100) and not the shared services RT (65000:9000), the DNS route never crosses into AS2.

**Diagnosis:**
```cisco
! IOS-XR (ASBR in AS1 or AS2 — check both):
show bgp vpnv4 unicast rd 65000:9000 10.99.0.53/32
! If missing here, the ASBR is filtering it

show bgp vpnv4 unicast neighbors <ASBR-peer> advertised-routes | include 10.99.0
! Check if DNS route is being sent across the ASBR boundary
```

**Fix:** Update the inter-AS RT filter to permit RT 65000:9000 in addition to the customer base RT:

```cisco
! IOS-XR:
route-policy INTER-AS-VPN-IN
  if extcommunity rt matches-any (65000:100, 65000:9000) then  ! Add shared services RT
    pass
  endif
  drop
end-policy
```

**Cause 2: Site 2's VRF is Missing the Shared Services RT Import**

The VRF configuration on Site 2's PE (in AS2) may have been created without the shared services import RT. If the PE in AS1 has `import route-target 65000:9000` but the PE in AS2 doesn't, the shared services routes never get imported into Site 2's VRF.

**Diagnosis:**
```cisco
! IOS-XR (PE in AS2):
show vrf CUST-A detail
! Check import route-targets — should include 65000:9000

show route vrf CUST-A 10.99.0.0/24
! If missing, RT import is the issue
```

**Fix:**
```cisco
! IOS-XR (PE in AS2):
vrf CUST-A
 address-family ipv4 unicast
  import route-target
   65000:100
   65000:9000   ! Add missing shared services RT
```

**Cause 3: Site 2's VRF Not Exporting Return RT — Hub Has No Return Path to Site 2**

Even if Site 2 receives the DNS route, the DNS server (or any host at the shared hub) needs a return route to Site 2's subnet. The hub VRF imports return routes via RT 65000:9001. If Site 2's VRF is not exporting RT 65000:9001, the hub has no path back to Site 2.

This manifests as: Site 2 can **reach** the DNS server (send packets), but the DNS server's **responses** are dropped (no return path). The customer reports "can't reach DNS" — which could mean either direction is broken.

**Diagnosis:**
```cisco
! IOS-XR (Hub PE):
show route vrf INTERNET-HUB 10.2.0.0/24   ! Site 2's subnet
! If missing, hub has no return path to Site 2

! On PE in AS2:
show bgp vpnv4 unicast vrf CUST-A 10.2.0.0/24 detail
! Check: does it export with RT 65000:9001?
```

**Fix:**
```cisco
! IOS-XR (PE in AS2):
vrf CUST-A
 address-family ipv4 unicast
  export route-target
   65000:100
   65000:9001   ! Add shared services return RT
```

Also verify the ASBR's inter-AS RT filter permits RT 65000:9001 in addition to 65000:9000 — the return path route must also cross the AS boundary.

## Question 3
> **Security:** During a penetration test, the tester discovers that VRF-A can reach prefixes in VRF-C, even though no extranet relationship exists between them. VRF-A has extranet connectivity to VRF-B, and VRF-B has shared services access. Walk through the route-leaking chain that could cause this and describe how to fix it.

### Answer

**The Transitive Leaking Chain**

This is the classic "RT leaking loop through a shared hub" failure, and it's one of the most dangerous misconfiguration patterns in L3VPN. Here's the step-by-step route propagation that created unauthorized A→C connectivity:

```
Step 1: VRF-C exports routes with RT 65000:300 (its base VPN RT)
        VRF-C also exports with RT 65000:9001 (shared services return RT — 
        because it has shared services access like all customers)

Step 2: Hub VRF imports RT 65000:9001 → hub now has routes to VRF-C's subnets
        Hub VRF exports ALL its routes with RT 65000:9000
        (This is the bug: hub should only export its own service prefixes,
         not re-export customer-originated routes)

Step 3: VRF-B imports RT 65000:9000 (shared services) → VRF-B receives 
        VRF-C's subnets (because the hub is re-exporting them with 65000:9000)

Step 4: VRF-A has extranet with VRF-B via RT 65000:8001
        VRF-A imports RT 65000:8001 → receives everything VRF-B has, 
        including the transitive VRF-C routes leaked in Step 3

Result: VRF-A → VRF-B (extranet) → Hub VRF → VRF-C (unintended)
```

The chain required two leaking vectors: (1) the hub's unfiltered re-export of customer routes, and (2) the extranet from VRF-A into VRF-B spreading the contaminated routes further.

**Root Cause: Hub Export Policy Missing Prefix-Match Filter**

The hub VRF's export route-policy is applying RT 65000:9000 to all routes in the VRF, including routes imported from customer VRFs. Every customer's routes that entered the hub via RT 65000:9001 are being re-exported with RT 65000:9000 to all other spokes.

**Fix 1: Restrict Hub Export to Hub-Originated Prefixes Only (Primary Fix)**

The hub export policy must only export routes that originated within the hub VRF (DNS, NTP, internet prefixes). Any route imported from a customer VRF must be suppressed at export:

```cisco
! IOS-XR:
prefix-set HUB-OWN-SERVICES
  10.99.0.0/24,            ! DNS
  10.99.1.0/24,            ! NTP
  0.0.0.0/0                ! Internet default route
end-set

route-policy HUB-EXPORT-SAFE
  if destination in HUB-OWN-SERVICES then
    set extcommunity rt (65000:9000) additive
    pass
  else
    drop
  endif
end-policy

vrf INTERNET-HUB
 address-family ipv4 unicast
  export route-policy HUB-EXPORT-SAFE
```

```junos
! Junos:
policy-statement HUB-EXPORT-SAFE {
    term hub-services {
        from {
            route-filter 10.99.0.0/24 exact;
            route-filter 10.99.1.0/24 exact;
            route-filter 0.0.0.0/0 exact;
        }
        then { community set rt-9000; accept; }
    }
    term block-all-else {
        then reject;
    }
}
```

**Fix 2: Tag Customer Routes with a Custom "Do-Not-Re-Export" Community at Hub Import**

As a defense-in-depth measure, tag customer-originated routes as they enter the hub with a private community signaling "do not re-export." The hub export policy checks for this community and suppresses those routes.

**Important:** Do NOT use the well-known `no-export` community here. `no-export` (RFC 1997) prevents routes from being advertised to eBGP peers — it has no effect on VRF route export into MP-iBGP VPNv4. Use a private community instead, and explicitly check it in the export policy:

```cisco
! IOS-XR: Apply custom do-not-re-export community on import
community-set DO-NOT-REEXPORT
  65000:666
end-set

route-policy HUB-IMPORT-CUSTOMER
  set community (65000:666) additive    ! Tag customer routes on entry to hub
  pass
end-policy

! Hub export policy — suppress anything tagged with the do-not-re-export community
route-policy HUB-EXPORT-SAFE
  if community matches-any DO-NOT-REEXPORT then
    drop                                ! Customer-originated — don't leak
  endif
  if destination in HUB-OWN-SERVICES then
    set extcommunity rt (65000:9000) additive
    pass
  else
    drop
  endif
end-policy

vrf INTERNET-HUB
 address-family ipv4 unicast
  import route-policy HUB-IMPORT-CUSTOMER
  export route-policy HUB-EXPORT-SAFE
```

**Fix 3: Audit and Remediate the Extranet Contamination**

VRF-B has already received the contaminated routes. After fixing the hub export policy, trigger a BGP route refresh to clear stale routes:

```cisco
! IOS-XR — on all affected PEs:
clear bgp vpnv4 unicast vrf VRF-B soft in

! Junos:
clear bgp neighbor * soft-in
```

Verify VRF-A no longer has routes to VRF-C's subnets after the refresh:
```cisco
! IOS-XR:
show route vrf VRF-A 10.3.0.0/24    ! VRF-C's subnet should be absent
ping vrf VRF-A 10.3.1.1              ! Should fail
```

**Structural Lesson**

This vulnerability is a design error, not an operator error. The hub VRF should never be configured with a blanket "export everything" policy. The correct design for shared services is:

- Hub exports: **only hub-originated prefixes** with RT 65000:9000
- Hub imports: **customer routes** via RT 65000:9001 for return routing
- Hub never re-exports what it imports

Enforce this at the route-policy layer AND document it in the VRF template so every new PE deployment inherits the correct export filter.
