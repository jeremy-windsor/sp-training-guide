# 7.2 — MP-BGP for VPNv4/VPNv6 — Answer Key

---

## Question 1
> **Scenario**: You enable RT-Constraint on all PEs and the RR. After enabling, 200 of 500 VRFs lose connectivity. The RR logs show no errors. What happened, and how do you fix it without disabling RT-Constraint?

### Answer

**What Happened: The RR Withheld Routes Before PEs Subscribed**

When RT-Constraint (RFC 4684, AFI 1, SAFI 132) is enabled, the RR enters a new mode: it withholds VPN routes for any RT until it receives an explicit subscription from a PE client. The subscription is sent when the PE's RT-Constraint session comes up and the PE advertises which RTs it imports.

The 200 VRFs that lost connectivity have one thing in common: their PEs failed to successfully advertise RT subscriptions to the RR. This happens for several reasons:

1. **RT-Constraint session not established on those PEs** — some PEs may not have fully converged the new `address-family ipv4 rt-filter` (IOS-XR) or `family route-target` (Junos) session before the RR started filtering
2. **PE configuration incomplete** — the PE has the VRFs configured but didn't have RT-Constraint enabled or the address family wasn't activated under the BGP neighbor group
3. **RR-side hold-down** — the RR's RT-Constraint implementation enforces a brief holddown where it treats PEs as "no subscriptions" until the AFI 132 session fully converges. During the holddown, routes for those PEs' RTs are suppressed.
4. **Router-ID or session flap** — if any PE-to-RR BGP session reset during the migration, the RT subscription must be re-sent. PEs that had sessions reset during the cutover had a gap in subscription coverage.

The absence of errors in the RR logs is expected — RFC 4684 defines this behavior as correct operation, not an error. Withholding routes for unsubscribed RTs is the entire point of RT-Constraint.

**Verification**

On the RR, check which RT subscriptions have been received from affected PEs:

```cisco
! IOS-XR (RR):
show bgp ipv4 rt-filter
show bgp ipv4 rt-filter neighbors <PE-IP> advertised-routes
```

```junos
! Junos (RR):
show route table bgp.rtarget.0
show bgp neighbor <PE-IP> 
```

On an affected PE:
```cisco
! IOS-XR (PE):
show bgp ipv4 rt-filter summary          ! Should show session Up
show bgp ipv4 rt-filter                  ! Should show locally originated RT NLRIs
```

**Fix: Ensure RT-Constraint Sessions Are Active on All PEs**

Without disabling RT-Constraint, the fix is to ensure all PEs have the `ipv4 rt-filter` address family activated and are actively advertising their RT subscriptions:

**IOS-XR (on each affected PE):**
```cisco
router bgp 65000
 neighbor-group RR-GROUP
  address-family ipv4 rt-filter
  !
 !
```

After committing, the PE establishes an AFI 132 session with the RR and sends RT subscription updates. The RR then allows those routes through within one BGP advertisement cycle (typically seconds).

**Junos (on each affected PE):**
```junos
protocols bgp {
    group RR-PEERS {
        family route-target;
    }
}
```

If sessions are already configured but the routing is still missing, trigger a soft reset to force RT subscription re-advertisement:

```cisco
! IOS-XR:
clear bgp ipv4 rt-filter <RR-IP> soft out
```

```junos
! Junos:
clear bgp neighbor <RR-IP> soft-out
```

**Migration Best Practice (After the Fact)**

The safe migration sequence is: (1) enable RT-Constraint on PEs first, (2) wait for all RT subscriptions to populate on the RR, (3) only then enable RT-Constraint filtering on the RR. Enabling the RR first before PEs subscribed caused the 200-VRF outage.

## Question 2
> **Design**: A customer is dual-homed to PE1 and PE2 in your network. Both PEs advertise the same prefix with the same RD. You're using a single RR. Engineers report that traffic always enters via PE1 even when PE2 is closer to the source. The RR is co-located with PE1. What are three solutions, ranked by operational complexity?

### Answer

**Root Cause**

With identical RDs on PE1 and PE2, both PEs advertise the same VPNv4 NLRI (`RD:prefix`). The RR receives two paths for the same NLRI and runs BGP best-path. Because the RR is co-located with PE1, PE1's route has a **lower IGP metric** from the RR's perspective. The RR picks PE1 as best path and advertises only that path to all client PEs. Other PEs always prefer PE1 regardless of their own topology.

**Solution 1: BGP ADD-PATH for VPNv4 (Lowest Operational Complexity)**

ADD-PATH (RFC 7911) allows the RR to advertise **multiple paths** for the same NLRI to each client. Each receiving PE then runs its own best-path selection against the local topology — picking PE1 or PE2 based on IGP metric to each PE's loopback.

This is the modern, recommended solution. It requires only RR and PE configuration changes; no VRF RT changes needed.

```cisco
! IOS-XR (RR):
router bgp 65000
 address-family vpnv4 unicast
  additional-paths send receive
 !
 neighbor-group PE-CLIENTS
  address-family vpnv4 unicast
   additional-paths send
```

```junos
! Junos (RR):
protocols bgp {
    group PE-CLIENTS {
        family inet-vpn {
            unicast {
                add-path {
                    send { path-count 6; }
                    receive;
                }
            }
        }
    }
}
```

After enabling, each PE receives both the PE1 path and the PE2 path and selects based on local IGP cost. Sources closer to PE2 now use PE2. This is exactly the intended behavior for dual-homed customers.

**Solution 2: BGP Optimal Route Reflection (ORR, RFC 9107) (Medium Complexity)**

ORR enables the RR to compute best-path **from each client PE's perspective** rather than from the RR's own perspective. The RR uses the IGP topology (via BGP-LS or IS-IS/OSPF database exchange) to simulate each PE's view of the network and picks the best path as that PE would.

This solves the suboptimal routing without requiring PEs to receive multiple paths. The RR advertises one path per client, but the correct one.

```cisco
! IOS-XR (RR):
router bgp 65000
 address-family vpnv4 unicast
  optimal-route-reflection GROUP-NAME
```

Requires IGP topology information be available to the RR (BGP-LS from the IGP, or direct IGP adjacency). More complex to deploy than ADD-PATH, but more memory-efficient for large-scale deployments (PEs don't need to store multiple paths).

**Solution 3: Unique RDs per PE (Highest Operational Complexity — Requires VRF Changes)**

Assign different RDs on PE1 and PE2 for the same customer VRF:
- PE1: `export RD 65000:1001` 
- PE2: `export RD 65000:1002`

Now the RR sees two **different** VPNv4 NLRIs (`65000:1001:prefix` and `65000:1002:prefix`). Best-path runs independently per NLRI, and the RR advertises both. All PEs receive both paths and pick based on BGP best-path (IGP metric is the tiebreaker when all other attributes are equal).

This works, but it's operationally complex:
- Requires VRF config changes on both PEs
- Doubles the number of VPNv4 NLRIs in the RIB (mild scale impact)
- New RD allocation must be tracked and managed going forward
- If the customer has multiple sites, each PE pair for each site needs unique RDs

This was the standard workaround before ADD-PATH and ORR existed. ADD-PATH is the better answer today.

## Question 3
> **Troubleshooting**: A Junos PE shows the VPNv4 route in `show route table bgp.l3vpn.0` but the route does NOT appear in the VRF routing table (`show route table CUSTOMER-A.inet.0`). The RT matches. What is the most likely cause?

### Answer

**Most Likely Cause: Next-hop Not Resolved in `inet.3`**

On Junos, VPN next-hop resolution is strictly constrained. VPNv4 routes in `bgp.l3vpn.0` have a next-hop of the remote PE's loopback (e.g., `10.255.0.2`). Before a VPNv4 route can be installed in the VRF routing table (`CUSTOMER-A.inet.0`), that next-hop **must be resolvable in `inet.3`**.

`inet.3` is Junos's MPLS-eligible routing table. It contains only routes learned via MPLS signaling protocols: LDP, RSVP-TE, or SR-MPLS. Routes that exist only in `inet.0` (the global IPv4 table, reachable via IGP or static) do **not** qualify — Junos will not resolve a VPN next-hop against an unlabeled route.

This means if PE2's loopback (10.255.0.2) is reachable via IS-IS in `inet.0` but LDP is down, the RSVP tunnel is missing, or SR is not configured for that prefix, `inet.3` will have no entry for 10.255.0.2. The VPNv4 route stays in `bgp.l3vpn.0` but never gets imported into the VRF.

**Verification**

```junos
show route 10.255.0.2 table inet.3     ! Should show LDP/RSVP/SR route
show ldp database                       ! Check label bindings for remote PE loopback
show rsvp session                       ! If RSVP-TE, check tunnel to remote PE
show route protocol bgp table inet.3    ! Check SR-MPLS labeled routes if applicable
```

If the inet.3 route is missing, that's your answer.

**Fix**

Restore the MPLS path to the remote PE:

```junos
! Check if LDP is running and has a session to the next-hop router
show ldp session                        ! Session to P router should be "Operational"
show ldp neighbor                       ! Verify LDP hellos

! If SR-MPLS: ensure prefix-SID is being received and installed
show route protocol isis table inet.3   ! Should show remote PE loopback with label
```

If LDP is configured but `inet.3` is still empty for the PE loopback, check for LDP session issues or LDP tunnel-to-next-hop misconfiguration.

**Secondary Causes (Eliminate After Verifying inet.3)**

If `inet.3` has the PE loopback with a valid label, check these in order:

1. **VRF import policy isn't actually matching the RT** — the RT might display in `show route table bgp.l3vpn.0` but be rejected by a stricter import policy applied at the VRF level. Inspect `show route table bgp.l3vpn.0 <prefix> detail` for the exact RT, then check the VRF import policy:
   ```junos
   show policy CUSTOMER-A-IMPORT
   ```

2. **Route origin or community causing policy rejection** — if the VRF import policy has additional terms beyond the RT match (e.g., rejecting routes with certain BGP communities, or requiring specific ORIGIN type), a correctly-tagged route could still be rejected.

3. **`resolution-map` misconfiguration** — on platforms using explicit resolution maps (less common), the VRF might be configured to resolve next-hops from a table that doesn't include `inet.3`.

4. **`vrf-table-label` and label resolution conflict** — in rare cases on older Junos versions, enabling or disabling `vrf-table-label` while routes are active can leave stale state. A routing table reset (`restart routing`) resolves this but is disruptive.

The canonical first check remains: **`show route 10.255.0.2 table inet.3`**. If it's empty, LDP/SR to the remote PE is broken. If it has a route, the problem is in the import policy or VRF configuration.
