# 7.4 — Extranet & Shared Services — Protocol Theory

## Protocol Overview

Extranet and shared services are controlled route leaking between VRFs. The base L3VPN architecture (RFC 4364) provides strict isolation — VRF-A can't see VRF-B. But real service provider customers inevitably need controlled connectivity between VPNs: a shared DNS/NTP infrastructure, a common internet gateway, partner-to-partner connectivity, or centralized security services accessible by all tenants.

The mechanism is entirely RT-based. No new protocols, no new RFCs — just creative use of Route Target import/export policies that already exist in the L3VPN framework. The power is in the policy; the danger is in getting it wrong. A misconfigured RT import can silently merge two customers' routing tables, violating the isolation guarantee that L3VPN was built to provide.

---

## Core Mechanisms

### 1. Extranet — Controlled Inter-VPN Connectivity

Extranet means allowing specific routes from one VPN to be visible in another VPN. It's selective route leaking, not full merger.

**Mechanism:**
1. VRF-A exports routes with RT 65000:100 (its normal RT) AND RT 65000:999 (an extranet RT).
2. VRF-B is configured to import RT 65000:999.
3. VRF-B now receives VRF-A's routes. VRF-A doesn't receive VRF-B's routes (unless VRF-B also exports an RT that VRF-A imports).

**Bidirectional extranet**: If both VRFs export and import each other's extranet RTs, connectivity is mutual. If only one direction is configured, the extranet is asymmetric — one VRF can reach the other but not vice versa (though return traffic may fail without bidirectional route visibility).

**Selective extranet with route-maps:**
Not all of VRF-A's routes should necessarily leak to VRF-B. Route-maps on the VRF import can filter by prefix, community, or other BGP attributes:
- Import RT 65000:999 but only accept 10.0.0.0/24 (the shared database server)
- Reject everything else with RT 65000:999

This provides fine-grained control — leak specific services, not entire routing tables.

### 2. Shared Services VPN — The Central Services VRF

Shared services is a pattern where a dedicated VRF provides services (DNS, NTP, internet gateway, authentication, monitoring) to multiple customer VPNs.

**Architecture:**
- **Shared Services VRF**: Contains the actual services infrastructure. Exports routes with a shared RT (e.g., RT 65000:888).
- **Customer VRFs**: Import RT 65000:888 to receive the shared services routes. Export their own RT (e.g., RT 65000:100 for Customer A).
- **Shared Services VRF imports**: The shared services VRF imports RTs from all customers that need return reachability. Or it imports a common "all customers" RT.

**Example:**
```
Shared Services VRF:
  Export RT: 65000:888
  Import RT: 65000:100, 65000:200, 65000:300 (each customer's RT)

Customer A VRF:
  Export RT: 65000:100
  Import RT: 65000:888

Customer B VRF:
  Export RT: 65000:200
  Import RT: 65000:888
```

Result: All customers can reach the shared services. The shared services can reach all customers. But Customer A still can't reach Customer B — their RTs don't cross.

### 3. Internet Access via Shared Services

One of the most common shared services use cases: centralized internet access.

**Design pattern:**
1. Internet VRF on the PE connects to the internet gateway / NAT device.
2. Internet VRF exports a default route (0.0.0.0/0) with RT 65000:777.
3. Customer VRFs import RT 65000:777 to receive the default route.
4. Customer traffic to the internet follows the default route into the Internet VRF.
5. Return traffic: Internet VRF imports customer-specific RT to route return traffic back to the correct customer VRF.

**NAT consideration**: Without NAT, the internet sees the customer's private addressing. With NAT in the Internet VRF, source addresses are translated before hitting the public internet. Return traffic NAT-translates back and routes into the correct customer VRF.

### 4. Route Leaking Without MP-BGP (Local VRF Leaking)

On some platforms, route leaking between VRFs can happen locally on the PE without going through MP-BGP:

**Import from global table to VRF**: A VRF can import routes from the global routing table (and vice versa). This doesn't involve VPNv4 at all — it's a local operation.

**VRF-to-VRF import**: Some platforms support direct VRF-to-VRF route import on the same PE. Again, local — no MP-BGP exchange.

**When local leaking is used**: When the shared service and the customer VRF are on the same PE. Avoids the MP-BGP round-trip (PE → RR → PE) for routes that are locally present.

**Danger**: Local leaking bypasses MP-BGP policy. Changes to local leak configuration don't propagate to remote PEs. If the shared service moves to a different PE, local leaking breaks.

### 5. Preventing Route Leaking Disasters

The most dangerous aspect of extranet/shared services is unintended route leaking.

**Common mistakes:**
- Importing a customer's RT into the shared services VRF without prefix filtering — now ALL of Customer A's routes are visible in shared services, not just the routes needed for return traffic.
- Two customer VRFs accidentally importing the same RT — they can now reach each other. VPN isolation is broken.
- Shared services VRF exporting all routes when only the service prefix should be exported — customers see infrastructure routes they shouldn't.

**Defensive patterns:**
- **Prefix-based filtering on import**: Only accept specific prefixes, not everything with the matching RT.
- **Dedicated extranet RTs**: Don't reuse customer VPN RTs for extranet. Create separate extranet-specific RTs.
- **Audit RT assignments**: Maintain a central RT allocation registry. This sounds bureaucratic; the alternative is a production outage.
- **SOO for loop prevention**: In multi-homed extranet scenarios, Site of Origin prevents routes from looping back.

### 6. Hierarchical Shared Services

Large providers may have multiple levels of shared services:

**Regional shared services**: Each region has its own shared services VRF with regional DNS, monitoring, etc.

**Global shared services**: A global VRF provides services across all regions.

**Interaction**: Regional VRFs import both regional and global shared services RTs. Global services VRF imports from all regional VRFs for return reachability.

This creates a hierarchy of RT imports that must be carefully designed to avoid unintended cross-region leaking.

## Protocol Interactions

**Extranet ↔ RT Policy**: The entire extranet and shared services model is built on RT import/export. There are no additional protocols — it's pure BGP extended community policy.

**Shared Services ↔ Route Reflectors**: When shared services routes are distributed via MP-BGP, RRs carry them like any other VPN route. RT-constrained distribution (RFC 4684) ensures only PEs with matching VRFs receive them.

**Extranet ↔ PE-CE Routing**: When an extranet route leaks into a VRF, the PE may need to advertise it to the CE via the PE-CE routing protocol (BGP, OSPF, static). The leaked route appears as a VPN route in the VRF and is distributed normally to the CE.

## Edge Cases & Gotchas

**1. Asymmetric extranet and return traffic**: If VRF-A exports to VRF-B but VRF-B doesn't export to VRF-A, VRF-B can route TO VRF-A's destinations but VRF-A can't route return traffic back. The connection appears to work (SYN reaches), then fails (SYN-ACK can't return). Bidirectional leaking is almost always required.

**2. Overlapping address space in extranet**: If VRF-A and VRF-B both use 10.0.0.0/24, leaking routes between them creates ambiguity. The receiving VRF has two paths to the same prefix — its own and the leaked one. Best-path selection resolves it, but the result may not be what you intended.

**3. Shared services VRF as a traffic magnet**: If the shared services VRF imports ALL customer RTs, it holds every customer route in the network. This VRF becomes massive. Only import what's needed for return reachability — or use static routes for return.

**4. RT import on PE with many VRFs**: Each imported RT increases the PE's route processing load. A PE with 500 VRFs all importing a shared services RT means every shared services route update triggers import evaluation in 500 VRFs.

**5. Internet VRF and full table**: If the Internet VRF imports a full BGP internet table (800K+ routes), that table exists in the VRF context alongside the global table. Memory doubles. Most designs use a default route in the Internet VRF, not a full table.
