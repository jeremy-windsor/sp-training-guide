# 7.5 — L3VPN Scale & Convergence — Answer Key

---

## Question 1
> **Scenario**: Your RR is consuming 14GB of memory and CPU spikes to 100% during PE failures. You have 400K VPNv4 prefixes and 40 PE clients. What three changes would you make, in what order?

### Answer

**Diagnosis**

The two failure modes have different root causes:
- **14GB memory**: The RR holds all 400K VPNv4 routes × multiple paths in the Adj-RIB-In. Without RT-Constraint, it may be holding paths for routes PEs don't need.
- **CPU 100% during PE failures**: When a PE goes down, its 400K/40 = ~10K routes must be withdrawn, best-path re-run for each, and updates sent to all 40 PE clients. At 10K routes × 40 clients = 400K update messages, plus the bestpath computation — that's a significant CPU burst.

**Change 1: Enable RT-Constraint (RFC 4684) — Memory Fix**

RT-Constraint dramatically reduces the RR's memory burden. Currently the RR receives and stores **all paths** for all 400K VPNv4 routes from every PE, regardless of whether those routes will ever be sent to a particular client. With 40 PEs and no RT-Constraint, Adj-RIB-In can be 40 × 400K = 16 million route-path entries (before compression).

After enabling RT-Constraint, the RR only keeps paths for routes that at least one client has subscribed to. In practice, not all 40 PEs import all 400K routes — if each PE has 50 VRFs out of a total pool of 2,000 VRFs, it only needs 2.5% of the VPNv4 table. RT-Constraint eliminates the other 97.5% from each PE's view.

Memory reduction: proportional to VRF distribution density. A well-deployed RT-Constraint implementation typically reduces RR memory 50–90%.

**IOS-XR (RR):**
```cisco
router bgp 65000
 address-family ipv4 rt-filter
 !
 neighbor-group PE-CLIENTS
  address-family ipv4 rt-filter
```

**Junos (RR):**
```junos
protocols bgp {
    group PE-CLIENTS {
        family route-target;
    }
}
```

Deploy this first. It reduces ongoing memory pressure and gives headroom for the next change.

**Change 2: Enable ADD-PATH for VPNv4 + Pre-Install PIC on PEs — CPU Fix**

The CPU spike during PE failures happens because the RR must process all affected prefixes serially: withdraw routes, run bestpath, generate updates. Without pre-computed backup paths, this is unavoidable — the RR doesn't know the backup until it runs bestpath after the failure.

ADD-PATH (RFC 7911) allows the RR to advertise **multiple paths** for each VPNv4 prefix. PEs receive both the primary and backup paths and install both in the FIB (via PIC — Prefix Independent Convergence). When a PE fails, PEs with PIC already have the backup path installed and can switch at the FIB level without waiting for RR convergence.

The CPU spike on the RR is also reduced: instead of a burst of per-prefix updates after failure detection, the RR can leverage graceful-restart and route-refresh to smooth the update load. But the real fix is that PEs no longer depend on the RR's update cycle for fast failover.

**IOS-XR (RR — enable ADD-PATH for VPNv4):**
```cisco
router bgp 65000
 address-family vpnv4 unicast
  additional-paths send receive
 !
 neighbor-group PE-CLIENTS
  address-family vpnv4 unicast
   additional-paths send
```

**IOS-XR (PEs — enable PIC for VPN):**
```cisco
router bgp 65000
 address-family vpnv4 unicast
  additional-paths receive
```

On IOS-XR 7.x+, VPN PIC is enabled by default once ADD-PATH provides multiple paths. On older IOS-XR (pre-7.0), enable next-hop trigger delay to near-zero:
```cisco
! Pre-7.x IOS-XR only:
router bgp 65000
 nexthop trigger-delay critical 0
```

**Junos (RR):**
```junos
protocols bgp {
    group PE-CLIENTS {
        family inet-vpn {
            unicast {
                add-path {
                    send { path-count 6; }
                    receive;
                }
            }
        }
    }
}
```

**Junos PEs — PIC requires explicit configuration** (not automatic):
```junos
routing-options {
    forwarding-table {
        indirect-next-hop;
        chained-composite-next-hop {
            ingress {
                l3vpn;
            }
        }
    }
}
```

Without `indirect-next-hop` and `chained-composite-next-hop ingress l3vpn`, Junos does not enable L3VPN PIC regardless of how many BGP paths are available.

**Junos PEs — verify PIC is explicitly enabled and functional per platform/release:**

**Change 3: Migrate to Dedicated RR Hardware with BGP Graceful Restart**

A 14GB memory footprint and CPU-intense bestpath computation needs dedicated hardware. If the RR is running on a P router (sharing memory/CPU with data-plane forwarding), memory pressure from 400K VPNv4 routes competes with packet forwarding memory. During PE failure events, RR CPU competition with forwarding disrupts both functions.

A dedicated RR (physical or VM with adequate resources: 64GB+ RAM, 8+ cores) eliminates this conflict. Additionally, enabling BGP Graceful Restart (RFC 4724) on the RR prevents cascading route withdrawals when the RR itself is upgraded or restarted:

**IOS-XR (RR and PEs):**
```cisco
router bgp 65000
 bgp graceful-restart
```

**Junos (RR and PEs):**
```junos
protocols bgp {
    graceful-restart;
}
```

With graceful restart, RR maintenance events don't trigger a full VPNv4 route purge and re-advertisement cycle — PEs continue forwarding based on stale routes until the RR comes back.

**Priority Order and Expected Impact:**

| Change | Addresses | Memory Impact | CPU Impact |
|--------|-----------|--------------|------------|
| 1. RT-Constraint | Memory first | −50 to −90% | Minor improvement (fewer routes to process) |
| 2. ADD-PATH + PIC | CPU during failures | Slight increase (+backup paths stored) | Eliminates RR burst on PE failure |
| 3. Dedicated RR + GR | Long-term stability | No more data/control contention | Maintenance without disruption |

## Question 2
> **Design**: A customer asks for sub-second L3VPN failover for dual-homed sites. They have 200 sites, each with 2 CEs connecting to different PEs. Walk through the end-to-end design: RD allocation, label mode, ADD-PATH, PIC, BFD timers. What's the expected convergence time?

### Answer

**RD Allocation: Per-PE Unique RDs (Recommended for PIC)**

For PIC to work at the ingress PE, that PE must receive **two distinct paths** for the customer's prefixes — one via each egress PE. Unique RDs per PE create distinct NLRIs, which helps RRs send both paths without ADD-PATH. With ADD-PATH (RFC 7911), multiple paths for the same NLRI can be delivered even with identical RDs — but unique RDs remain the simpler, more widely deployed approach.

Allocation scheme for 200 sites × 2 PEs each:
- PE-A1 at site 1: RD `65000:10001` (site-id × 10 + local-pe-offset)
- PE-A2 at site 1: RD `65000:10002`
- PE-B1 at site 2: RD `65000:20001`
- PE-B2 at site 2: RD `65000:20002`
- ... and so on

This ensures the RR sees two different VPNv4 NLRIs per customer prefix per dual-homed site, can advertise both via ADD-PATH, and ingress PEs can install both in FIB for PIC.

**Label Mode: Per-CE or Per-VRF**

With 200 sites × 2 PEs × N customer VRFs, per-prefix label allocation would consume significant FIB label space. Per-VRF is the most memory-efficient. Per-CE is a reasonable middle ground if FIB allows it — it enables direct CE targeting without a second IP lookup while consuming far fewer labels than per-prefix.

**Recommendation:** Per-CE for this scenario. With 2 CEs per site, each site pair contributes 2 labels (one per CE) regardless of prefix count. 200 sites × 2 PEs × 2 CEs = 800 labels total — trivial.

**IOS-XR:**
```cisco
router bgp 65000
 address-family vpnv4 unicast
  label mode per-ce
```

**Junos:** Per-CE is the default behavior when each CE is a separate next-hop in the VRF. Verify with `show route table CUST-X.inet.0 detail` — routes with the same CE next-hop share a label.

**ADD-PATH: Enable on RR and All PEs**

The RR must advertise both egress PE paths (PE-A1 and PE-A2 for site 1) to every ingress PE. Ingress PEs must accept and process multiple paths per VPNv4 NLRI.

**IOS-XR (RR):**
```cisco
router bgp 65000
 address-family vpnv4 unicast
  additional-paths send receive
 !
 neighbor-group ALL-PES
  address-family vpnv4 unicast
   additional-paths send
   maximum-paths ibgp 8    ! Allow up to 8 paths per prefix
```

**IOS-XR (all PEs):**
```cisco
router bgp 65000
 address-family vpnv4 unicast
  additional-paths receive
 !
 vrf CUST-X
  address-family ipv4 unicast
   maximum-paths ibgp 8
```

**Junos (RR):**
```junos
protocols bgp {
    group ALL-PES {
        family inet-vpn {
            unicast {
                add-path { send { path-count 8; } receive; }
            }
        }
    }
}
```

**PIC: Enable VPN Prefix Independent Convergence**

PIC pre-installs backup paths in the FIB so failover is a pointer update, not a per-prefix reprogram. The FIB maintains a next-hop group referencing both PE-A1 and PE-A2; when PE-A1's transport path fails, the next-hop group switches to PE-A2 in O(1) time.

**IOS-XR (PEs):**
```cisco
! PIC for VPN prefixes (enabled via cef config):
router bgp 65000
 vrf CUST-X
  address-family ipv4 unicast
   maximum-paths ibgp 8   ! Allow up to 8 equal-cost iBGP paths
```

Note: `unequal-cost` is not a valid modifier for `maximum-paths ibgp` on IOS-XR. For unequal-cost BGP multipath, use DMZ Link Bandwidth. On IOS-XR 7.x+, VPN PIC is active by default once ADD-PATH delivers multiple paths with resolvable next-hops. Verify with:
```cisco
show cef vrf CUST-X 10.1.0.0/24 detail
! Should show: nexthop group with primary + backup path, "PIC" annotation
```

**Junos:** PIC requires explicit configuration — it does NOT activate automatically. Add the forwarding-table config to every PE (see Change 2 in 7.5 Q1 for the full config block):
```junos
routing-options {
    forwarding-table {
        indirect-next-hop;
        chained-composite-next-hop {
            ingress { l3vpn; }
        }
    }
}
```

**BFD Timers**

BFD is needed at two points in the convergence chain:

**1. PE-CE BFD (detect CE/CE-link failure):**
- Timer: 3 × 50ms = 150ms detection
- Interval: 50ms transmit/receive

```cisco
! IOS-XR PE-CE eBGP:
router bgp 65000
 vrf CUST-X
  neighbor 10.0.1.2
   bfd fast-detect
   bfd minimum-interval 50
   bfd multiplier 3
```

```junos
protocols bgp {
    group CUST-X-CE {
        bfd-liveness-detection {
            minimum-interval 50;
            multiplier 3;
        }
    }
}
```

**2. PE-RR BFD (detect RR/PE session failure for control-plane convergence):**
- Timer: 3 × 100ms = 300ms (multihop BFD — more conservative than single-hop)
- This triggers BGP session teardown and RR to withdraw/re-advertise routes

```cisco
! IOS-XR PE-to-RR:
router bgp 65000
 neighbor <RR-IP>
  bfd fast-detect
  bfd minimum-interval 100
  bfd multiplier 3
```

**Also enable transport-layer fast detection** (TI-LFA from Module 6.3 provides sub-50ms transport protection for core link failures without requiring BFD on every PE-to-PE path).

**Expected Convergence Time**

| Failure Scenario | Detection | IGP SPF + Next-Hop Track | FIB Switch | Total |
|-----------------|-----------|--------------------------|-----------|-------|
| PE-CE link down (Fast External Fallover) | Immediate (interface down) | n/a (CE failure, not core) | ~50ms (PIC pointer update) | ~50ms |
| PE-CE BFD timeout | 3 × 50ms = 150ms | n/a | ~50ms (PIC pointer update) | ~200ms |
| PE failure (BFD on PE-RR + IGP SPF) | 3 × 100ms = 300ms | ~200–400ms (SPF recompute + next-hop tracking) | ~50ms (PIC, if backup path pre-installed) | ~550–750ms |
| Core link failure (TI-LFA transport) | ~50ms (TI-LFA repair path) | n/a (TI-LFA pre-computes backup) | ~50ms (label forwarding switch) | ~100ms |

**Important:** PE node failure is NOT sub-second in all cases. The IGP SPF recomputation step (after detecting the PE's loopback withdrawal) adds 200–400ms depending on SPF throttle timers (typically `initial-wait 50ms, min-wait 200ms, max-wait 5000ms` on IOS-XR). The full chain is: BFD fires → IGP adj down → SPF runs → PE loopback withdrawn from topology → next-hop tracking detects unresolvable VPN next-hop → PIC pointer switches. Total: **~550–750ms for PE node failure with PIC + BFD.**

Core link failures (not PE failures) handled by TI-LFA remain sub-100ms because TI-LFA's repair path is pre-computed and activated without waiting for BGP re-convergence.

**Overall: sub-200ms for link failures (TI-LFA + PIC), sub-second for PE node failures.**

The key insight: the "magic" of PIC is that the 400K-routes-per-PE reprogram bottleneck is eliminated. The backup path is already in the FIB. Convergence becomes limited by **detection time** (BFD timers), not **processing time** (bestpath + FIB programming).

**Design Checklist for Production Deployment:**
- [ ] Unique RD per PE per VRF (not per customer) ✓
- [ ] Per-CE label mode (or per-VRF if FIB is constrained) ✓
- [ ] ADD-PATH enabled on RR (send) and all PEs (receive) ✓
- [ ] PIC enabled and verified via `show cef detail` ✓
- [ ] BFD on PE-CE eBGP sessions (50ms × 3) ✓
- [ ] BFD on PE-RR iBGP sessions (100ms × 3) ✓
- [ ] TI-LFA enabled in the core IGP for transport protection ✓
- [ ] RT-Constraint enabled (reduces RR processing time during scale-up) ✓

## Question 3
> **Troubleshooting**: After enabling PIC, you see that convergence improved for some VPN prefixes but not others. `show cef vrf CUST-A 10.1.0.0/24 detail` shows no backup path. The prefix is advertised by two PEs. What's wrong?

### Answer

**PIC Requires Both Paths in the Local BGP RIB**

PIC pre-installs backup paths in the FIB. For a backup to exist in the FIB, two things must both be true:
1. The PE must have received **two paths** for the VPNv4 prefix from the RR (requires ADD-PATH)
2. Both paths' next-hops must be resolvable and the second path must not be filtered by policy

If `show cef vrf CUST-A 10.1.0.0/24 detail` shows no backup path, the FIB never had a backup to install. The root cause is almost certainly one of the following:

**Cause 1: ADD-PATH Not Fully Enabled — Most Likely**

ADD-PATH must be enabled on both the RR (send) and the PE (receive). If the PE isn't receiving the second path, it can't install it in the FIB.

**Verification:**
```cisco
! IOS-XR (affected PE):
show bgp vpnv4 unicast rd 65000:1001 10.1.0.0/24
show bgp vpnv4 unicast rd 65000:1002 10.1.0.0/24
! If only ONE of these shows up, ADD-PATH isn't delivering the second path

show bgp vpnv4 unicast neighbors <RR-IP> detail | include "additional"
! Should show: "Sends 6 paths, Receives 6 paths" or similar
```

If the PE is only receiving one path, check the RR:
```cisco
! IOS-XR (RR):
show bgp vpnv4 unicast neighbors <PE-IP> advertised-routes | include 10.1.0.0
! Should show two entries: one for each RD (65000:1001 and 65000:1002)
```

If the RR is only advertising one path, verify ADD-PATH is enabled on the RR's neighbor statement toward this PE:
```cisco
! IOS-XR (RR):
router bgp 65000
 neighbor-group PE-CLIENTS
  address-family vpnv4 unicast
   additional-paths send   ! Must be present
```

**Cause 2: Same RD on Both PEs Without ADD-PATH — Silent Path Suppression**

If both PEs advertising the prefix are using the same RD, the RR receives two paths for the same VPNv4 NLRI. Without ADD-PATH, BGP best-path selection picks ONE and advertises only that one.

**Important clarification:** ADD-PATH (RFC 7911) exists *specifically* to advertise multiple paths for the **same NLRI** — paths are distinguished by a Path Identifier. So ADD-PATH IS the correct solution for same-RD scenarios. The issue described here occurs only if ADD-PATH is not configured or not negotiated between the RR and this PE.

If ADD-PATH is properly configured on both the RR and this PE, same-RD paths should work fine — both paths arrive at the PE with different Path IDs and can both be installed in FIB for PIC. If same-RD paths aren't arriving with ADD-PATH enabled, the problem is in ADD-PATH negotiation (see Cause 1), not the RD design.

Using unique RDs per PE is still recommended as defense-in-depth — it ensures path diversity even on platforms or sessions where ADD-PATH isn't negotiated, and avoids best-path suppression on the RR when ADD-PATH isn't available.

**Verification:**
```cisco
! IOS-XR (RR):
show bgp vpnv4 unicast 10.1.0.0/24
! If both paths show the SAME RD prefix, that's the problem
! You should see:
!   65000:1001:10.1.0.0/24    ← PE1
!   65000:1002:10.1.0.0/24    ← PE2
! Not:
!   65000:1001:10.1.0.0/24    ← PE1 (only this one, PE2's path suppressed)
```

**Fix:** Assign unique RDs per PE for this customer VRF (see 7.1 design recommendations).

**Cause 3: Policy Filtering the Second Path Before FIB Installation**

An import route-policy on the VRF might be filtering the second path. If the policy accepts one path and rejects another based on prefix length, community, or other attribute, only the accepted path makes it to the VRF RIB and FIB.

**Verification:**
```cisco
! IOS-XR:
show bgp vpnv4 unicast vrf CUST-A 10.1.0.0/24
! If BGP table shows two paths but VRF RIB only shows one, check import policy

show bgp vpnv4 unicast vrf CUST-A 10.1.0.0/24 detail | include "Status"
! Look for "Not selected" on the backup path with a reason code
```

Common policy issues:
- `maximum-paths ibgp 1` under the VRF — limits to single path, preventing backup installation
- Community-based policy that accepts routes with a specific community only present on one PE
- Local-preference manipulation making one path permanently non-eligible

**Fix:** 
```cisco
! IOS-XR (PE, VRF):
router bgp 65000
 vrf CUST-A
  address-family ipv4 unicast
   maximum-paths ibgp 8   ! Allow multiple equal-cost paths
```

**Cause 4: Backup Path Next-Hop Not Resolvable**

Even if the second path is in the BGP VPN RIB, the FIB won't install it as a backup if the next-hop (the second PE's loopback) isn't resolvable via the transport (LDP/SR). If PE2's loopback is unreachable or lacks an MPLS label binding, the backup path is "present but unusable."

**Verification:**
```cisco
! IOS-XR:
show route 10.255.0.2/32    ! PE2 loopback — must be reachable
show mpls ldp bindings 10.255.0.2/32   ! Or SR prefix-SID
! If the label binding is missing, the backup next-hop can't be installed
```

**Systematic Troubleshooting Order:**
1. Confirm ADD-PATH sends two paths: `show bgp vpnv4 unicast vrf CUST-A <prefix>` — see two RDs
2. Confirm different RDs on the two advertising PEs
3. Check `maximum-paths ibgp` is > 1 in the VRF address-family
4. Verify both PE loopbacks are reachable with MPLS labels
5. Check import route-policy for unexpected filtering
6. If all above pass, check `show bgp vpnv4 unicast vrf CUST-A <prefix> detail` for "path not selected" reason codes

After fixing the root cause, trigger a soft-reset to force BGP to re-evaluate paths and push updates to CEF:
```cisco
! IOS-XR:
clear bgp vpnv4 unicast vrf CUST-A soft

! Verify PIC is now active:
show cef vrf CUST-A 10.1.0.0/24 detail
! Should now show: nexthop-group with primary + backup entries
```
