# 7.1 — MPLS L3VPN Architecture — Answer Key

---

## Question 1
> **Design:** A customer has 4 sites, each dual-homed to two PEs. They use AS 65100 at all sites. Design the RD, RT, and loop-prevention strategy. What happens if you use the same RD on both PEs at a site?

### Answer

**RD Design — Use Unique RDs per PE per VRF (not per customer)**

The correct production answer is to assign a **per-PE unique RD** for each VRF/customer. Because this customer is dual-homed (each site has two PEs), using the same RD on both PEs for a given site triggers silent best-path suppression at the Route Reflector — a classic and dangerous failure mode.

Example allocation scheme (4 sites × 2 PEs each = 8 PE/VRF instances):

| PE | VRF | RD | Notes |
|----|-----|----|-------|
| PE-A1 | CUST-X | 65000:1001 | Site 1, first PE |
| PE-A2 | CUST-X | 65000:1002 | Site 1, second PE |
| PE-B1 | CUST-X | 65000:1003 | Site 2, first PE |
| PE-B2 | CUST-X | 65000:1004 | Site 2, second PE |
| PE-C1 | CUST-X | 65000:1005 | Site 3, first PE |
| PE-C2 | CUST-X | 65000:1006 | Site 3, second PE |
| PE-D1 | CUST-X | 65000:1007 | Site 4, first PE |
| PE-D2 | CUST-X | 65000:1008 | Site 4, second PE |

You can also use the PE loopback IP as the Type 1 RD: `192.0.2.1:100`, which self-documents the originating PE.

**RT Design — Symmetric Import/Export for Full Mesh**

All VRFs for this customer share the same RT. Full-mesh connectivity between all 4 sites:

```
All CUST-X VRFs:
  export route-target 65000:100
  import route-target 65000:100
```

All 8 PE/VRF instances export the same RT and import the same RT. Every site can reach every other site.

**Loop Prevention Strategy — SOO + eBGP AS-path**

The customer uses AS 65100 at all sites with eBGP PE-CE. Standard eBGP loop prevention (`as-path` contains the CE's own AS → reject) prevents basic loops but breaks dual-homing: PE-B1 advertises Site 2 routes to PE-B2 via MP-BGP, PE-B2 re-advertises them to Site 2's CE2 — which rejects them because it sees AS 65100 in the path. Site 2 loses the backup path.

Fix: **`as-override`** on the PE-CE eBGP session. The PE replaces the customer AS 65100 in the AS_PATH with the SP's own ASN before advertising to the CE. The CE no longer sees its own ASN and accepts the route.

**Mandatory companion:** Configure **Site of Origin (SOO)** per site to prevent routing loops that `as-override` introduces:

```
! IOS-XR: Apply SOO on PE-CE eBGP neighbor
route-policy SOO-SITE1-IN
  set extcommunity soo (65000:1001)
  pass
end-policy

router bgp 65000
 vrf CUST-X
  neighbor 10.0.1.2
   address-family ipv4 unicast
    route-policy SOO-SITE1-IN in
    as-override
```

```
! Junos: Apply SOO via import policy on PE-CE BGP group
policy-statement SOO-SITE1-IN {
    term add-soo {
        then {
            community add soo-site1;
            accept;
        }
    }
}
community soo-site1 members origin:65000:1001;

protocols bgp {
    group CUST-X-CE {
        import SOO-SITE1-IN;
        as-override;
    }
}
```

Each site gets a unique SOO value. When PE-B2 receives a route tagged with SOO 65000:1001 and is about to re-advertise it to Site 1's CE, the SOO match causes the route to be rejected. Loop prevented.

**What Happens with Duplicate RDs?**

If PE-A1 and PE-A2 both use RD 65000:1001, both advertise the VPNv4 prefix `65000:1001:10.1.0.0/24` to the Route Reflector. The RR sees two paths with **identical NLRI** (same RD + prefix). BGP best-path selection runs and picks **one** — typically based on router-ID tiebreaker (the PE with the lower router-ID wins). The RR then advertises only that one path to all other PEs.

The result: **one path silently disappears**. Every other PE in the network only receives routes to Site 1 via one PE, even though two PEs are reachable. Dual-homing is functionally broken — the customer's redundancy is invisible to the SP network. The failure mode is completely silent: no BGP error, no log message, just suboptimal forwarding and hidden redundancy.

The fix is unique RDs per PE (as designed above) so the RR sees two genuinely different NLRIs and can advertise both.

## Question 2
> **Troubleshooting:** PE1 shows VPNv4 routes from PE2 in `show bgp vpnv4 unicast`, but `show route vrf CUST-A` on PE1 is empty. PE1's VRF imports RT 65000:100. PE2's VRF exports RT 65000:200. What's wrong, and what's the one-command fix?

### Answer

**Root Cause: Route Target Mismatch**

PE2 is exporting routes with RT `65000:200`. PE1 is importing routes with RT `65000:100`. These values don't match — PE1 will never import a route tagged with RT 65000:200 because its import policy requires RT 65000:100.

This explains the symptom precisely:
- `show bgp vpnv4 unicast` on PE1 shows the routes **because MP-BGP accepts all received VPNv4 routes into the Adj-RIB-In regardless of RT** — the BGP session itself doesn't filter
- `show route vrf CUST-A` is empty **because the RT import policy on the VRF rejects routes tagged with 65000:200** — they never get installed in the VRF RIB

You can confirm this on IOS-XR by looking at the extended community on the received VPNv4 routes:

```
PE1# show bgp vpnv4 unicast rd 65000:200 10.1.0.0/24 detail
  Extended Community: RT:65000:200
```

PE1's VRF import says "I want 65000:100" — the route has 65000:200. No match. Route discarded.

**The One-Command Fix**

Correct PE2's export RT to match what PE1 imports (65000:100), or alternatively add 65000:200 to PE1's import RT. The canonical fix is correcting PE2:

**IOS-XR (on PE2):**
```cisco
vrf CUST-A
 address-family ipv4 unicast
  export route-target
   65000:100   ← Change 65000:200 to 65000:100
```

**Junos (on PE2):**
```junos
routing-instances CUST-A {
    vrf-export CUST-A-EXPORT;
}
policy-statement CUST-A-EXPORT {
    term set-rt {
        then {
            community set rt-65000-100;   # Change from 65000:200 to 65000:100
            accept;
        }
    }
}
community rt-65000-100 members target:65000:100;
```

After the RT is corrected, PE2's routes get re-exported with the new RT. PE1's VRF import policy matches, and the routes populate `show route vrf CUST-A` immediately (within BGP advertisement interval, typically seconds).

**Why This Misconfig Happens in Production**

Copy-paste errors when creating new VRFs, or someone changed the import RT on PE1 for a policy reason and forgot to update PE2's export, or vice versa. The silence of the failure is what makes it dangerous — BGP stays up, VPNv4 routes are present in the BGP table, but the VRF is dark. Most monitoring checks BGP session state; this fails at the RT level, above what most monitors check.

## Question 3
> **Scale:** Your network has 2,000 VRFs and 500K VPN routes. PEs are running out of memory. What three changes would you make, in priority order?

### Answer

**Priority 1: Enable RT-Constraint (RFC 4684) on All PEs and the RR**

This is the single highest-leverage change. Without RT-Constraint, every PE receives every VPN route from the RR regardless of whether it has any VRF that imports those routes. With 500K routes and 2,000 VRFs, a PE with only 50 VRFs is receiving 500K routes but actually using perhaps 25K of them. It receives, processes, and stores 475K routes it immediately discards — 95% waste.

RT-Constraint lets each PE advertise the exact set of RTs it imports. The RR only sends matching routes. Memory consumption drops proportionally to VRF density per PE.

**IOS-XR:**
```cisco
router bgp 65000
 address-family ipv4 rt-filter
 !
 neighbor-group RR-GROUP
  address-family ipv4 rt-filter
```

**Junos:**
```junos
protocols bgp {
    group RR-PEERS {
        family route-target;
    }
}
```

Expected impact: if each PE has 50 VRFs out of 2,000, and routes are roughly uniformly distributed, PE memory drops from 500K routes to ~12.5K routes — a 97.5% reduction.

**Priority 2: Switch VPN Label Allocation to Per-VRF (or Per-CE)**

With 500K VPN routes and per-prefix label allocation (IOS-XR default), you're consuming 500K label entries in the FIB — one per route. On most ASICs this is between 500K and 1M label-forwarding entries per PE, which competes directly with IPv4/IPv6 FIB space and can exhaust hardware forwarding tables.

Switch to **per-VRF** allocation: one label per VRF regardless of how many prefixes are in it. 2,000 VRFs = 2,000 labels. The trade-off is an additional IP lookup on egress (the arriving label identifies the VRF, then a normal IP lookup finds the CE next-hop). On modern hardware this is negligible.

**IOS-XR** — configure per-VRF under each VRF's address-family (not the global vpnv4 AF):
```cisco
router bgp 65000
 vrf CUST-A
  address-family ipv4 unicast
   label mode per-vrf
 !
! Repeat for each VRF. On IOS-XR 7.5+, a global default under
! address-family vpnv4 unicast may also be supported, but per-VRF
! placement is the canonical and backward-compatible approach.
```

**Junos:** Per-VRF is achieved by enabling `vrf-table-label` on each routing-instance:
```junos
routing-instances CUST-A {
    vrf-table-label;
}
```

**Priority 3: Deploy Dedicated Route Reflectors with ADD-PATH**

If the RR is running on a P or PE router (a common cost-saving mistake), it's doing double duty: forwarding production traffic AND holding 500K VPNv4 routes in memory for all 2,000 VRFs. The control plane and data plane are competing for the same memory and CPU.

Dedicated RR hardware removes this competition. A dedicated RR (Cisco NCS 540 or 5000 series, Juniper PTX10001 in RR mode, or even a high-memory x86 server running IOS-XR-v or Junos VM) holds all VPN state without affecting forwarding.

Add **ADD-PATH** for VPNv4 on the RR simultaneously — this allows the RR to advertise multiple paths per VPN prefix to each PE client, enabling prefix-independent convergence (PIC) and optimal path selection without the single-best-path bottleneck:

**IOS-XR (RR):**
```cisco
router bgp 65000
 address-family vpnv4 unicast
  additional-paths send receive
 !
 neighbor-group PE-CLIENTS
  address-family vpnv4 unicast
   additional-paths send
```

**Junos (RR):**
```junos
protocols bgp {
    group PE-CLIENTS {
        family inet-vpn {
            unicast {
                add-path {
                    send {
                        path-count 6;
                    }
                }
            }
        }
    }
}
```

**Summary of impact:**
1. RT-Constraint: ~90-97% memory reduction per PE (eliminates irrelevant VPN routes)
2. Per-VRF labels: ~250× reduction in FIB label entries (500K → 2K)
3. Dedicated RR + ADD-PATH: removes control/data-plane resource contention, enables PIC
