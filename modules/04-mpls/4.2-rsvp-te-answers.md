# 4.2 — RSVP-TE — Answer Key

## Question 1
> You configure an RSVP-TE tunnel with 2 Gbps bandwidth and priority 4/4. The tunnel fails to come up, and the log shows "admission control failure." The link you expect it to traverse is a 10G link with `ip rsvp bandwidth 8000000`. What's the most likely cause, and how do you diagnose it?

### Answer

**Most likely cause: there is not 2 Gbps of *unreserved* RSVP bandwidth left on one of the links in the computed path.**

`ip rsvp bandwidth 8000000` on a 10G link means the operator made **8 Gbps reservable**, not 10 Gbps. RSVP-TE admission control compares the new tunnel's requested bandwidth (2 Gbps) against the **remaining unreserved bandwidth at the relevant priority level** on every hop in the LSP path. If any one hop has less than 2 Gbps available at priority 4, CSPF may still compute the path, but RSVP signaling will fail at reservation time with admission control failure.

Common reasons:
1. **Existing tunnels already consumed the reservable pool.** Example: three existing 2.5 G tunnels = 7.5 G reserved. Only 0.5 G remains — your 2 G tunnel can't be admitted.
2. **A downstream link on the path has a lower RSVP bandwidth than the 10G link you checked.** Operators often assume the obvious core link is the bottleneck, but the failure could be on a 1G or 2.5G link later in the explicit path.
3. **Priority interaction.** RSVP tracks unreserved bandwidth per priority level (0-7). A higher-priority LSP can preempt lower-priority LSPs if configured, but your 4/4 tunnel cannot displace other 4/4 or higher-priority LSPs.

**How to diagnose:**

```
IOS-XE:
  show ip rsvp interface                  ← reservable vs reserved BW per link
  show mpls traffic-eng tunnels tunnel10 detail
  show ip rsvp reservation detail         ← who is already consuming bandwidth
  show mpls traffic-eng topology path <src> <dst>

Junos:
  show rsvp interface detail
  show rsvp session detail
  show mpls lsp extensive name <lsp-name>
```

Look specifically for:
- **Unreserved bandwidth at priority 4** on each hop
- **The RRO / explicit path** to identify the exact hop failing admission
- Whether **preemption** is allowed — with setup/hold 4/4, the tunnel can only preempt lower-priority LSPs with hold priority >4

**Likely fix:** reduce requested bandwidth, increase reservable RSVP bandwidth on the limiting link, or allow a dynamic path option so CSPF can find an alternate route with enough headroom.

## Question 2
> You have a 20-node core with full-mesh TE tunnels between 50 PE routers. Calculate the approximate number of RSVP-TE states that each P router must maintain. Why is this a scaling concern, and what alternatives exist?

### Answer

A full mesh of **50 PE routers** means every PE builds an RSVP-TE tunnel to every other PE.

Number of unidirectional tunnels:

\[
50 \times 49 = 2450
\]

(If you count bidirectional intent as separate unidirectional LSPs — which RSVP-TE does — it's 2450. If someone means unique PE pairs only, it's \(50 \times 49 / 2 = 1225\), but RSVP state is per direction, so 2450 is the operational number.)

**RSVP-TE is per-LSP, per-hop state.** Every transit P router on the path stores at least:
- a **PATH state block**
- a **RESV state block**
- refresh state / timer metadata
- FRR-related local repair state if enabled

So a P router that sits near the middle of the core and carries a large fraction of the mesh may maintain **roughly 2 × 2450 = 4900 core RSVP state entries**, and realistically more once you count detours/bypass participation, refresh reduction bookkeeping, and reoptimization state.

If FRR is enabled, add bypass tunnel state. If multiple path options exist, add reoptimization churn. If this becomes 100 PEs instead of 50, you're looking at ~9900 unidirectional LSPs — RSVP state growth is quadratic with tunnel endpoints.

**Why this is a scaling concern:**
1. **Control-plane memory and CPU.** Every P router refreshes, maintains, and ages state for LSPs it does not originate or terminate.
2. **Failure churn.** A core failure causes mass PATH/RESV refreshes and potential re-signaling of hundreds or thousands of LSPs.
3. **Operational complexity.** Full-mesh RSVP-TE is harder to reason about than hop-by-hop LDP or SR policies targeted only where needed.

**Alternatives:**

1. **Segment Routing (SR-MPLS or SRv6).** This is the dominant modern answer. No per-LSP state on transit nodes. The head-end pushes the SID stack; core nodes just forward labels.
2. **Targeted TE tunnels only where needed.** Don't full-mesh every PE. Build RSVP-TE only for high-value traffic flows or congested corridors.
3. **Hierarchical / loose-hop TE.** Use fewer inter-POP tunnels and let IGP/LDP or SR handle intra-POP forwarding.
4. **BGP LU / SR-TE policies.** For inter-domain or policy-based steering without full RSVP state explosion.

**Bottom line:** 2450 RSVP-TE LSPs is survivable, but it's exactly the kind of design that made operators fall in love with Segment Routing.

## Question 3
> After enabling FRR node-protection on your TE tunnels, you notice that some LSPs show "FRR: None" in their status. What are two reasons a bypass tunnel might not be available for a given LSP?

### Answer

**1. The PLR cannot find a valid bypass path to the merge point that avoids the protected node.**

Node protection is stricter than link protection. To protect against node C failing, the bypass from the PLR must reach a merge point downstream **without traversing C at all**. In sparse topologies, rings with insufficient alternate paths, or designs where the next-next-hop is reachable only through the protected node, there is simply no node-disjoint backup path.

Example:

```
A — B — C — D
      \       \
       \-- E --\
```

If B is the PLR and C is the protected node, B needs a bypass to D that avoids C. If E→D doesn't exist or doesn't have TE/SR reachability, node protection fails. Link protection might still be possible (protect only B→C), but node protection won't.

**2. The required bypass tunnel does not exist or is not eligible.**

Facility backup depends on a bypass tunnel being present on the PLR toward the appropriate merge point. Reasons it may be unavailable:
- no bypass tunnel configured/signaled on that interface
- bypass exists but doesn't reach the correct merge point
- bandwidth / affinity / admin-group constraints prevent its use
- bypass is only link-protect, while the primary LSP requested node-protect

**How to verify:**

```
IOS-XE:
  show mpls traffic-eng fast-reroute database
  show mpls traffic-eng tunnels <id> detail
  show ip rsvp interface detail

Junos:
  show rsvp session detail | match bypass
  show mpls lsp extensive name <lsp-name>
  show rsvp interface detail
```

Look for whether the PLR has a bypass with the right protection type and whether the topology actually supports node-disjoint repair.

**Practical note:** If the topology only supports link protection for some segments, seeing `FRR: None` on node-protect requests is not a software failure — it's the network honestly admitting it cannot perform magic.

## Question 4
> Explain the difference between facility backup and one-to-one backup FRR. Why is facility backup dominant in production? What's the one advantage of one-to-one backup?

### Answer

**Facility backup (bypass tunnels):** One pre-signaled backup tunnel protects **many** primary LSPs that share the same protected link or node. It's an **N:1** model. If link B→C or node C fails, the PLR pushes one extra bypass label and steers all protected traffic through the same repair tunnel.

**One-to-one backup (detour LSPs):** Every protected RSVP-TE LSP gets its **own dedicated detour** from the PLR to the tail-end or merge point. It's a **1:1** model. Ten protected LSPs = ten detours.

**Why facility backup dominates:**
1. **Scales far better.** One bypass can protect dozens or hundreds of primary LSPs.
2. **Much less control-plane state.** You avoid exploding RSVP state with per-LSP detours.
3. **Simpler operations.** Easier to understand, provision, and troubleshoot.
4. **Good enough for most SPs.** Most operators care about fast restoration, not per-LSP reserved bandwidth on the backup path.

That's why facility backup is what you find in production and in most reference designs.

**The one real advantage of one-to-one backup:**

**Per-LSP bandwidth guarantees on the backup path.** Because each detour is a real RSVP-TE LSP, it can reserve bandwidth independently. If you have a small number of extremely high-value tunnels and must guarantee bandwidth even during failure, one-to-one can enforce that. Facility backup generally treats the bypass as shared/best-effort from the perspective of the protected LSPs.

In practice, operators looked at that advantage, looked at the scaling bill, and chose facility backup almost every time.

## Question 5
> During a maintenance window, you need to move all TE tunnels off a specific link without dropping traffic. Describe the steps using RSVP-TE features (not by shutting the link).

### Answer

Use **administrative metric manipulation and make-before-break**, not brute-force link shutdown.

### Clean maintenance workflow

**1. Mark the link as undesirable for CSPF.**
Increase the TE metric or remove available reservable bandwidth on that link so CSPF stops selecting it for new paths.

```
IOS-XE:
  interface GigabitEthernet0/0/0
    mpls traffic-eng administrative-weight 65535
  ! or temporarily reduce rsvp bandwidth to 0 / minimal

Junos:
  set protocols mpls interface ge-0/0/0.0 administrative-group MAINT
  ! then exclude MAINT in LSP path constraints
```

If existing LSPs have explicit paths, also update those path options so the maintained link is no longer part of the primary path.

**2. Trigger reoptimization / reroute using MBB.**
RSVP-TE's make-before-break signals the new path first, switches traffic, then tears down the old path. That's the whole reason this protocol got invited to the party.

```
IOS-XE:
  clear mpls traffic-eng tunnel <id> path-option
  ! or reoptimize the tunnel / wait for reoptimization timer

Junos:
  clear mpls lsp name <lsp-name> statistics
  request mpls lsp reroute name <lsp-name>
```

**3. Verify traffic moved.**
Check the RRO / active path on each tunnel and confirm the maintained link is no longer traversed.

```
IOS-XE: show mpls traffic-eng tunnels detail
Junos:  show mpls lsp extensive
```

**4. Drain residual protected traffic.**
If any tunnels remain because of explicit path constraints or lack of alternate feasible path, manually edit those LSP definitions and reroute them. Don't proceed until the link is empty.

**5. Perform maintenance.**
At this point, the link can be taken out with minimal or zero traffic impact.

**Key principle:** Don't shut the link first and let FRR save you. FRR is your safety net, not your maintenance plan. For planned work, use MBB and drain the link gracefully.
