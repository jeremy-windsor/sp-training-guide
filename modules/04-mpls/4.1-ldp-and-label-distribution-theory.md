# 4.1 — LDP and Label Distribution — Protocol Theory

## Protocol Overview

LDP (Label Distribution Protocol) is the protocol that binds MPLS labels to FECs (Forwarding Equivalence Classes). In simple terms, LDP is how routers agree on which label to use for which destination prefix. It is defined in RFC 5036 (2007, updating the original RFC 3036 from 2001).

LDP operates alongside the IGP. The IGP builds the IP forwarding table; LDP mirrors it with labels. For every IGP-learned prefix, LDP assigns a local label and distributes it to neighbors. The result is a Label Switched Path (LSP) that follows the IGP shortest path — no traffic engineering, no explicit path setup, just label-switched forwarding along the IGP-computed route.

LDP runs over TCP (port 646) for reliable session communication and UDP (port 646) for neighbor discovery. This split is important: discovery uses multicast UDP to find adjacent LDP speakers, then a TCP session is established for label exchange. The TCP session carries all label bindings and is maintained with keepalives.

**LDP vs RSVP-TE**: LDP creates best-effort LSPs that follow IGP shortest paths. RSVP-TE creates traffic-engineered LSPs along explicitly constrained paths with bandwidth reservations. LDP is simpler; RSVP-TE is more powerful. Many SP networks use both: LDP for general IP reachability (BGP next-hop resolution) and RSVP-TE for traffic-engineered tunnels.

**LDP vs Segment Routing**: SR-MPLS replaces LDP entirely. Instead of a separate label distribution protocol, SR encodes labels directly in the IGP (via IS-IS/OSPF extensions). Each prefix gets a globally significant Prefix-SID; each adjacency gets a local Adj-SID. No LDP sessions, no LDP state, no LDP-IGP sync issues. SR is the modern replacement. LDP is still widely deployed in brownfield networks.

## Core Mechanisms

### 1. LDP Message Format

All LDP messages share a common structure:

**LDP PDU Header (10 bytes):**
```
├─ Version (2 bytes): always 1
├─ PDU Length (2 bytes): length of rest of PDU
├─ LDP Identifier (6 bytes):
│   ├─ LSR ID (4 bytes): Router ID of the LDP speaker
│   └─ Label Space ID (2 bytes): identifies the label space (0 for per-platform)
```

The LDP Identifier is how routers identify each other. The LSR ID is typically the same as the IGP Router ID / loopback address. Label Space ID 0 means the router uses a single platform-wide label space (the common case).

**LDP Message Header:**
```
├─ U-bit (1 bit): Unknown message bit. If set, silently ignore unknown messages.
├─ Message Type (15 bits): identifies the message
├─ Message Length (2 bytes)
├─ Message ID (4 bytes): unique identifier for request/response correlation
└─ TLVs: message-specific parameters
```

**Message Types:**

| Type | Name | Function |
|------|------|----------|
| 0x0100 | Notification | Error and advisory messages |
| 0x0200 | Hello | Neighbor discovery |
| 0x0300 | Initialization | Session parameter negotiation |
| 0x0301 | KeepAlive | Session maintenance |
| 0x0400 | Address | Advertise interface addresses |
| 0x0401 | Address Withdraw | Remove interface addresses |
| 0x0402 | Label Mapping | Bind label to FEC |
| 0x0403 | Label Request | Request label for FEC |
| 0x0404 | Label Withdraw | Remove label binding |
| 0x0405 | Label Release | Release label no longer needed |
| 0x0406 | Label Abort Request | Abort pending label request |

### 2. LDP Discovery

LDP discovery uses UDP multicast to find neighboring LDP speakers.

**Basic Discovery (directly connected):**
- Send Hello messages to multicast address 224.0.0.2 (All Routers), UDP port 646
- Hello contains: Hold Time, Transport Address, and optionally Configuration Sequence Number
- Default Hello interval: 5 seconds
- Default Hold Time: 15 seconds (3 × Hello)
- Any LDP speaker receiving a Hello from an unknown source creates a "Hello adjacency"

**Extended Discovery (targeted):**
- Send Hello messages to a specific unicast IP address, UDP port 646
- Used for non-directly-connected LDP peers (targeted LDP sessions)
- Common use case: LDP sessions between PEs for L2VPN/VPLS pseudowire signaling
- Must be explicitly configured on both sides

**Transport Address:**
The Hello message carries a Transport Address — the IP address the router will use as the source for the TCP session. This is typically the loopback address. The transport address determines which IP the remote peer connects to for the LDP TCP session. If this doesn't match what the remote peer expects, the TCP connection fails.

### 3. Session Establishment

After discovery, LDP establishes a TCP session:

1. **TCP connection**: The router with the higher Transport Address initiates the TCP connection to port 646 on the peer. This deterministic rule prevents both sides from simultaneously connecting.

2. **Initialization**: Both sides exchange Initialization messages containing:
   - Session Keep Alive Time (negotiated: minimum of both proposals)
   - Label Distribution Method (Downstream Unsolicited or Downstream on Demand)
   - Max PDU Length
   - Receiver's LDP Identifier

3. **KeepAlive**: After Initialization, both sides send KeepAlive messages. Receipt of the peer's KeepAlive confirms the session is established.

4. **Address**: Each side sends Address messages listing all its interface IP addresses. This is critical — the peer uses these addresses to determine which FEC-label bindings map to which IGP next-hops.

5. **Label Mapping**: Each side sends Label Mapping messages for all FECs in its label database. After this exchange, both sides have a complete label binding table from the peer.

**Session State Machine:**
```
Non-Existent → Initialized → OpenRec → OpenSent → Operational
```

- **Non-Existent**: No session. Waiting for discovery.
- **Initialized**: TCP connected, Initialization message sent, waiting for peer's Initialization.
- **OpenRec**: Peer's Initialization received, own Initialization not yet sent (race condition handling).
- **OpenSent**: Own Initialization sent, waiting for peer's KeepAlive confirming acceptance.
- **Operational**: Session up. Label Mapping exchange begins.

### 4. FEC and Label Binding

A FEC (Forwarding Equivalence Class) defines a set of packets that receive the same forwarding treatment. In practice, a FEC is usually an IP prefix.

**FEC Element Types (RFC 5036 §3.4.1):**
- **Wildcard FEC Element (0x01)**: Matches all FECs. Used in Label Withdraw/Release to withdraw all bindings at once.
- **Prefix FEC Element (0x02)**: An IP prefix (address family + prefix length + prefix). This is the primary FEC type — one label per IGP prefix.
- **Typed Wildcard FEC Element (RFC 5918)**: Extension that matches all FECs of a given type — more selective than the base wildcard.

**Label Binding**: A Label Mapping message associates a label with a FEC. Example: "For prefix 10.0.0.1/32, use label 24001." The peer stores this binding and uses it when forwarding traffic toward that prefix through this LDP neighbor.

### 5. Label Distribution Modes

LDP supports two label distribution modes, negotiated during Initialization:

**Downstream Unsolicited (DU):**
- The default and dominant mode in SP networks.
- A router assigns labels for all its FECs and pushes them to all LDP peers without being asked.
- Every peer gets every label binding, regardless of whether that peer is the next-hop for the FEC.
- Simple, fast convergence, but distributes more state than strictly necessary.

**Downstream on Demand (DoD):**
- A router requests labels from the downstream (next-hop) router for specific FECs.
- Labels are distributed only when requested, only to the requestor.
- Less state, but slower convergence (requires request-response exchange on topology change).
- Rarely used in modern SP networks.

### 6. Label Retention Modes

Independent of distribution mode, LDP speakers can retain labels differently:

**Liberal Label Retention:**
- Keep ALL label bindings received from ALL peers, even if the peer is not the current IGP next-hop for that FEC.
- Advantage: faster convergence. When the IGP next-hop changes, the new next-hop's label is already in the label table — no need to wait for a new Label Mapping.
- Disadvantage: more memory (labels from every peer for every prefix).
- This is the most common mode in practice and improves convergence, though RFC 5036 defines both modes without mandating a default.

**Conservative Label Retention:**
- Keep label bindings ONLY from the current IGP next-hop for each FEC.
- Advantage: less memory (only one label per FEC).
- Disadvantage: slower convergence. When the IGP next-hop changes, the router must wait for a Label Mapping from the new next-hop before it can forward labeled traffic.
- Used in memory-constrained environments.

### 7. Label Allocation Modes

**Per-Platform Label Space:**
- One label space shared across all interfaces. Label value X means the same thing regardless of which interface it arrives on.
- Label Space ID = 0.
- The standard for most SP deployments.

**Per-Interface Label Space:**
- Different label spaces per interface. The same label value can mean different things on different interfaces.
- Required for certain L2VPN/ATM-over-MPLS scenarios.
- Rare in modern deployments.

### 8. Loop Detection

LDP has two loop detection mechanisms, both optional:

**Hop Count:**
- A hop count field in Label Mapping messages, incremented at each hop.
- If it exceeds a configured maximum, the binding is rejected.
- Simple but rarely enabled (IGP loop-free convergence is usually sufficient).

**Path Vector:**
- A list of LSR IDs in the Label Mapping message, similar to BGP AS_PATH.
- If a router sees its own LSR ID in the path vector, the binding is rejected.
- More precise than hop count but adds overhead to every Label Mapping.

**In practice**: LDP loop detection is almost never enabled. The IGP is assumed to be loop-free (because SPF computes loop-free paths), and LDP follows the IGP. If the IGP has a transient loop, LDP will too — but the IGP loop resolves via SPF convergence, not LDP loop detection.

### 9. LDP-IGP Synchronization (RFC 5443)

A critical operational mechanism. When an LDP session goes down on a link but the IGP adjacency stays up, traffic sent over that link would be IP-forwarded (no label) — potentially breaking MPLS VPN services that require label-switched transport.

**LDP-IGP Sync mechanism:**
1. When an LDP session on a link goes down, the router advertises maximum IGP metric on that link.
2. Other routers' SPF avoids the link (it looks infinitely expensive).
3. When the LDP session re-establishes and label bindings are exchanged, the router restores the normal IGP metric.

This prevents traffic blackholing during LDP session flaps. It's a critical feature for VPN-carrying networks and should always be enabled.

### 10. Targeted LDP Sessions

Targeted LDP sessions are non-directly-connected LDP sessions, typically between PE routers for L2VPN signaling.

**Use cases:**
- L2VPN pseudowire signaling (LDP-based VPLS, VPWS)
- Remote LFA repair tunnels
- Inter-area LDP (though this is increasingly rare with SR)

**How they work:**
- Both PEs configure a targeted Hello toward each other's loopback.
- Extended Discovery Hellos are sent unicast (not multicast).
- TCP session establishment follows the same rules as basic LDP.
- Targeted sessions commonly carry pseudowire FEC bindings (PW ID FEC, Type 128/129), but RFC 5036 also allows targeted LDP for prefix-label use cases such as LDP-over-RSVP or LDP-over-TE tunnels.

## Protocol Interactions

**LDP ↔ IGP**: LDP follows the IGP. For every IGP-learned prefix, LDP creates a label binding. When the IGP next-hop changes, LDP adjusts which label binding is active (liberal retention makes this fast). LDP-IGP sync prevents blackholes when LDP is down but IGP is up.

**LDP ↔ BGP**: In MPLS VPN networks, BGP next-hops are resolved via IGP, and the transport label to reach those next-hops comes from LDP. The label stack is: outer label from LDP (transport to BGP next-hop) + inner label from BGP (VPN label). If LDP doesn't have a label for the BGP next-hop, the VPN route is unusable.

**LDP ↔ RSVP-TE**: LDP and RSVP-TE can coexist. RSVP-TE provides TE tunnels; LDP provides general label-switched reachability. LDP-over-RSVP-TE tunneling allows LDP to use RSVP-TE tunnels as transport, combining TE benefits with LDP's simplicity for VPN label distribution.

**LDP ↔ Segment Routing**: SR replaces LDP. During migration, both can run simultaneously. SR provides prefix-SIDs via IGP extensions; LDP provides traditional label bindings. Migration is typically done per-prefix or per-region, with SR-prefer mode causing the router to prefer SR labels over LDP labels when both exist for the same prefix.

## Edge Cases & Gotchas

**1. Transport address mismatch**: If Router A's Hello advertises transport address 10.0.0.1 but Router B can't reach 10.0.0.1 (IGP issue, ACL, wrong loopback), the TCP session fails. Discovery succeeds (multicast Hello received) but the session never establishes. This is the most common LDP troubleshooting scenario.

**2. Liberal retention hides convergence issues**: With liberal retention, a router has labels from all peers. When the IGP next-hop changes, the new label is immediately available — convergence appears instantaneous. But if the new label is wrong (stale, from a previous topology), traffic gets mis-labeled. This is rare but devastating when it happens.

**3. Label space exhaustion**: Each router has a finite label space (platform-dependent, typically 16,000 to 1,000,000+). In large networks with many prefixes and liberal retention, the label table can grow large. Monitor label utilization.

**4. LDP session over multihop**: Basic LDP sessions require direct connectivity. For targeted sessions, the underlying IP path must be stable. If the IGP path between PEs changes, the targeted LDP TCP session may briefly flap if TCP retransmits are delayed.

**5. Implicit NULL vs Explicit NULL**: By default, the egress LER (Label Edge Router) advertises Implicit NULL (label value 3) for directly connected prefixes, triggering PHP at the penultimate hop. If MPLS QoS requires the label to be preserved to the egress (for EXP bit inspection), configure Explicit NULL (label value 0) instead.
