# 4.1 — LDP & Label Distribution — Answer Key

## Question 1
> You enable MPLS on a new core link. IGP adjacency comes up immediately, but L3VPN traffic that should traverse the link is being dropped for about 10 seconds after the link comes up. What feature would prevent this, and how does it work?

### Answer

**LDP-IGP Synchronization.**

This is the classic IGP-before-LDP race condition. When a link comes up, IGP (IS-IS/OSPF) converges within seconds and installs the route with a normal metric. Traffic immediately starts flowing toward the link. But LDP needs additional time to establish a TCP session (port 646), exchange label bindings, and program the LFIB. During that gap — typically 5-15 seconds — labeled traffic (L3VPN, L2VPN, any MPLS service) arrives at the link with no outgoing label, and gets dropped.

**How LDP-IGP sync fixes it:** It tells the IGP to advertise the link with **maximum metric** (IS-IS: 2^24 - 2 = 16777214; OSPF: 65535) until LDP is fully operational on that interface. Other routers see the link as infinitely expensive and route around it. Once LDP converges (session up + labels exchanged), the IGP metric drops to the configured value, and traffic shifts to the link only after labels are ready.

```
IOS-XR:
  router isis CORE
    interface GigabitEthernet0/0/0/0
      address-family ipv4 unicast
        mpls ldp sync

Junos:
  protocols isis {
    interface ge-0/0/0.0 {
      ldp-synchronization;
    }
  }
  # or globally:
  protocols ldp {
    igp-synchronization holddown-interval 30;
  }
```

**Important nuance:** LDP-IGP sync also has a holddown timer. If LDP goes down on an interface, the IGP immediately raises the metric to maximum. The holddown controls how long to wait before declaring LDP "down" — too short and you get false positives during brief LDP flaps.

This feature should be enabled on **every MPLS-enabled interface** in a production SP network. Its absence is a guaranteed source of transient blackholes during link flaps, maintenance, and router reboots.

## Question 2
> Your network has 500 routers and carries a full Internet routing table (950K+ prefixes). You notice the LFIB on your P routers is consuming significant memory. What LDP configuration change would reduce memory usage, and what's the trade-off?

### Answer

**LDP label filtering — advertise labels only for loopback/infrastructure prefixes, not the entire IGP table.**

By default, LDP assigns a local label to **every** IGP prefix and advertises those bindings to all peers (Downstream Unsolicited mode). In a network with 950K+ IGP routes, every P router maintains 950K+ label bindings in its LFIB. Most of these labels are never used — P routers only need labels for BGP next-hops (PE loopbacks), not for every prefix in the internet table.

**The fix — label filter policy:**

```
IOS-XR:
  mpls ldp
    address-family ipv4
      label local allocate for LOOPBACKS-ACL
  !
  ipv4 access-list LOOPBACKS-ACL
    10 permit 10.0.0.0/16       ← infrastructure /32 loopbacks
    20 deny 0.0.0.0/0 any

Junos:
  protocols ldp {
    egress-policy LABEL-FILTER;
  }
  policy-statement LABEL-FILTER {
    term loopbacks {
      from {
        route-filter 10.0.0.0/16 orlonger;
        protocol isis;
      }
      then accept;
    }
    then reject;
  }
```

This tells LDP to only allocate and advertise labels for infrastructure loopbacks (where BGP next-hops resolve). LFIB drops from 950K+ entries to a few hundred.

**The trade-off:**

1. **ICMP TTL-expired responses from P routers break.** When a labeled packet's TTL expires on a P router, the P router needs to send an ICMP Time Exceeded back to the source. Without a label for the source IP's prefix, the ICMP response can't be sent back via the label-switched path — it falls back to IP forwarding, which may not work if the P router doesn't have the internet route in its FIB (which it might not, if it's a pure P router with only a default route).

2. **Traceroute through the MPLS core shows `* * *` at P-router hops** instead of revealing hop-by-hop latency.

3. **Any service that relies on label-switched paths to arbitrary prefixes** (not just PE loopbacks) will break. This is rare in well-designed SP networks but can bite during migrations.

**Recommendation:** Apply label filtering on P routers (they only need labels for PE loopbacks). Leave PEs with full label allocation if they need it for internet traffic engineering or VPN services.

## Question 3
> `show mpls ldp discovery` shows Hello adjacency with a peer, but `show mpls ldp neighbor` shows no session. The peer's transport address is 10.0.0.5. What are the three most likely causes, and how do you verify each?

### Answer

LDP discovery (Hello adjacency) uses UDP multicast/unicast on the link. LDP sessions use TCP/646 to the peer's **transport address** (typically the loopback). Discovery working but no session means UDP Hellos are exchanged, but the TCP session can't establish.

**1. No IGP route to the transport address 10.0.0.5.**

LDP Hellos contain the transport address in the Hello message. The receiving router tries to establish a TCP connection to that address. If there's no route to 10.0.0.5 in the routing table (IGP adjacency not yet converged, or loopback not advertised in IGP), the TCP SYN has no path.

```
Verify:
  IOS-XR: show route 10.0.0.5     ← does an IGP route exist?
  Junos:  show route 10.0.0.5
```

**2. Transport address mismatch — local router is using the wrong source for TCP.**

LDP defaults to using the router's highest loopback IP as its transport address (IOS-XR) or the `router-id` / configured `transport-address` (Junos). If the local router's transport address doesn't have a reverse path from 10.0.0.5, the TCP SYN-ACK can't return.

```
Verify:
  IOS-XR: show mpls ldp discovery | include Transport
          ← check what transport address this router is advertising
  Junos:  show ldp interface detail
          show ldp session detail
```

Common mistake: transport address set to a physical interface IP instead of loopback. If that physical link goes down, all LDP sessions using it as transport fail.

**3. TCP port 646 is blocked by an ACL or control-plane policy.**

An infrastructure ACL on either router (or an intermediate firewall if the transport path traverses multiple hops) is blocking TCP/646. UDP/646 (used for Hello discovery) passes, but TCP/646 (used for the session) doesn't.

```
Verify:
  IOS-XR: telnet 10.0.0.5 646      ← test TCP/646 reachability
          show lpts bindings brief   ← verify LPTS allows LDP
  Junos:  show firewall log          ← check for rejected TCP/646
```

Also check: some platforms require the LDP transport address to be explicitly configured. If both peers are using physical interface IPs as transport (non-loopback), and the TCP connection source doesn't match the expected transport address, the session will be rejected.

## Question 4
> Explain the difference between liberal and conservative label retention. Which is the default on IOS-XE? On Junos? In what scenario would you change the default?

### Answer

**Liberal label retention:** Store ALL received label bindings from ALL LDP peers, even if the peer is not the current IGP best-path next-hop for that prefix. Memory usage is higher because you maintain label bindings from every peer, but convergence after a topology change is faster — the backup label is already in the LIB, so when IGP shifts the best path to a new next-hop, the LFIB can immediately install the new outgoing label without waiting for an LDP exchange.

**Conservative label retention:** Store ONLY the label binding from the current IGP best-path next-hop. Bindings from non-best-path peers are discarded. Lower memory usage, but on IGP topology change, the router must request/receive a new label binding from the new next-hop before MPLS forwarding works on the new path. This adds a small delay (typically <1 second) to MPLS convergence.

**Defaults:**
- **IOS-XE / IOS-XR:** Liberal retention (default). Stores all received bindings.
- **Junos:** Conservative retention (default). Only stores the active binding.

**When to change the default:**

**Change IOS-XR from liberal to conservative** when the LFIB memory is a problem — specifically in large networks (500+ routers) where each router carries bindings from many peers. If a router has 10 LDP peers and 100K prefixes, liberal retention stores 1M label bindings (10 × 100K). Conservative stores 100K. On memory-constrained platforms, this matters.

```
IOS-XR: mpls ldp
          address-family ipv4
            label local advertise
              explicit-null
            !
          !
```

**Change Junos from conservative to liberal** if you need the fastest possible MPLS convergence and have memory to spare. In practice, Junos LDP convergence is fast enough with conservative retention that this is rarely needed — LDP re-signaling takes <1 second in most topologies.

**Reality check:** With modern platforms having 64-128GB of RAM, memory pressure from liberal retention is rarely the actual bottleneck. Most SPs leave the default and use TI-LFA (Segment Routing) for fast convergence instead of relying on LDP label retention mode for backup paths. This question matters more for exam knowledge than daily operations in 2026.

## Question 5
> You need to establish an L2VPN pseudowire between two PE routers that are 5 hops apart. Regular LDP uses link-local discovery. What LDP feature do you need, and what's the additional requirement for it to work?

### Answer

**Targeted LDP (also called Extended Discovery or Directed LDP).**

Standard LDP uses **Basic Discovery** — it sends Hello packets as UDP multicast (224.0.0.2) on directly connected interfaces. This discovers only directly connected LDP peers (one hop away). For a pseudowire between PEs that are 5 hops apart, the PEs are not directly connected — they need an LDP session without link-local adjacency.

Targeted LDP sends Hello packets as **UDP unicast** to a specific peer's transport address. This establishes an LDP session between non-adjacent routers over multiple hops, using the IGP-routed path between them.

```
IOS-XR:
  mpls ldp
    router-id 10.0.0.1
    neighbor 10.0.0.5              ← explicit targeted neighbor
      targeted

Junos:
  protocols ldp {
    targeted-hello {
      10.0.0.5;                    ← remote PE loopback
    }
  }
```

**The additional requirement: an end-to-end LSP between the two PEs.**

Targeted LDP doesn't create its own transport — it rides on top of an existing MPLS label-switched path. The pseudowire needs TWO label levels:

1. **Outer label (transport):** An LSP from PE1 to PE2 — either via standard hop-by-hop LDP (which requires LDP on every intermediate link) or via RSVP-TE tunnel or Segment Routing
2. **Inner label (pseudowire/VC label):** Signaled by the Targeted LDP session between PE1 and PE2

So the complete requirement is:
1. **IGP reachability** between the two PE loopbacks (so targeted Hello packets can reach each other)
2. **An MPLS transport LSP** between the PEs (standard LDP on all intermediate hops, or RSVP-TE, or SR)
3. **Targeted LDP session** configured on both PEs pointing to each other's loopback

Without the transport LSP, the targeted LDP session establishes (it's TCP-based, works over IP) but the pseudowire has no label-switched path to carry encapsulated L2 frames — packets would be IP-routed instead of label-switched, which breaks the pseudowire.
