# 4.4 — MPLS OAM & Troubleshooting — Answer Key

## Question 1
> You run `ping 10.0.0.5` from PE1 and it succeeds (5/5). You then run `ping mpls ipv4 10.0.0.5/32` and it fails (0/5 timeout). What does this tell you, and what are the three most likely causes?

### Answer

This tells you something extremely useful:

**The plain IP path works, but the MPLS data plane for the FEC 10.0.0.5/32 does not.**

An ordinary IP ping only proves there is L3 reachability to 10.0.0.5. It does **not** prove labels are correct. MPLS LSP ping sends a specially crafted labeled packet for that exact FEC through the MPLS forwarding path. If IP ping succeeds but MPLS ping fails, the fault is specifically in the label-switched path — not generic routing.

**Three most likely causes:**

**1. Missing or incorrect LFIB entry / label binding on one of the transit routers.**
A router may have IGP reachability to 10.0.0.5 but no correct LDP/RSVP label mapping, or a stale label in hardware.

Check:
```
IOS-XE/XR: show mpls forwarding-table 10.0.0.5 32
           show mpls ldp bindings 10.0.0.5 32
Junos:     show route forwarding-table destination 10.0.0.5/32 detail
           show ldp database 10.0.0.5/32
```

**2. LDP/RSVP session is up somewhere but the end-to-end LSP is incomplete.**
A partial MPLS core means some hops have labels and some don't. IP forwarding masks the problem because traffic can still route natively.

Check:
```
IOS-XE/XR: show mpls ldp neighbor
           show mpls traffic-eng tunnels detail   (if RSVP-TE)
Junos:     show ldp session
           show rsvp session
```

**3. MTU issue in the labeled path.**
IP ping with small payloads succeeds, but MPLS Echo packets or real VPN traffic fail once labels are imposed. This is especially likely if the path includes deeper stacks or one core hop has a lower MPLS MTU than expected.

Check:
```
IOS-XE/XR: ping mpls ipv4 10.0.0.5/32 size 1472 repeat 5
Junos:     ping mpls ldp fec 10.0.0.5/32 size 1472 count 5
```

**What it does NOT mean:** It does not automatically mean the far-end router is down. The egress is clearly reachable by IP. This is a label-path or MPLS-forwarding problem until proven otherwise.

## Question 2
> LSP traceroute to 10.0.0.5/32 shows `L` for the first two hops, then `N` at the third hop, then timeout. What does return code `N` mean, and what should you check on the third hop?

### Answer

In MPLS OAM / LSP traceroute notation, **`N` means "mapping for this FEC is not associated with the receiving interface"** (or equivalently, the packet arrived on the wrong interface for the expected FEC-to-label mapping). In other words, the third hop received a labeled packet for the FEC, but from the perspective of its control plane, that packet did **not** arrive the way it expected.

This is a huge clue. It points to a mismatch between the control-plane label mapping and the actual incoming labeled data plane.

**What to check on the third hop:**

1. **LFIB vs LIB consistency.**
Does the third hop have the expected incoming label and outgoing action for 10.0.0.5/32?

```
IOS-XE/XR: show mpls forwarding-table 10.0.0.5 32 detail
           show mpls forwarding-table labels <incoming-label>
Junos:     show route forwarding-table destination 10.0.0.5/32 detail
           show route label <label>
```

2. **LDP or RSVP neighbor state on the incoming interface.**
If the router expects the label to arrive from neighbor A but it actually arrives from neighbor B, you may have an IGP/LDP desynchronization, stale label binding, or ECMP asymmetry the router didn't anticipate.

```
IOS-XE/XR: show mpls ldp neighbor detail
           show isis/ospf neighbor
Junos:     show ldp session detail
           show isis adjacency / show ospf neighbor
```

3. **IGP next-hop alignment.**
The control plane may think the FEC should be reached through interface X, but due to stale hardware programming or a recent topology change, traffic is arriving on Y.

4. **Possible routing loop or unexpected PHP behavior upstream.**
A loop or incorrect penultimate-hop action can send the packet to the wrong place with the wrong label context.

**Interpretation of the traceroute:**
- first two `L` hops = normal label switching in transit
- third hop `N` = the break point where control plane and data plane disagree
- timeout after that = packet is dropped or misforwarded beyond that point

The third hop is your crime scene. Start there, not at the endpoints.

## Question 3
> You need sub-second MPLS failover detection in your core. Compare three approaches: RSVP Hello, BFD for MPLS, and BFD for IGP. Which would you recommend and why?

### Answer

### 1. RSVP Hello
- Designed to detect failure of RSVP neighbor/control-plane adjacency
- Default timers are relatively slow (often seconds)
- Good for detecting RSVP session liveliness, not best for fast data-plane failover
- Tied specifically to RSVP-TE deployments

**Pros:** Native to RSVP, useful for control-plane health
**Cons:** Slower than BFD, RSVP-specific, not ideal as the primary fast-fail detector

### 2. BFD for MPLS (RFC 5884)
- Sends BFD control packets over the actual LSP being monitored
- Verifies the **labeled data path itself**, not just underlying IP reachability
- Can achieve 100ms × 3 = 300ms or 50ms × 3 = 150ms detection if the platform supports it cleanly

**Pros:** Tests the real MPLS data plane, catches label-path failures that plain IGP BFD might miss
**Cons:** More operational overhead, per-LSP monitoring can scale poorly, especially in large RSVP/LDP meshes

### 3. BFD for IGP
- Runs on the IGP adjacencies (IS-IS/OSPF) underneath MPLS
- Detects link/node failure quickly, triggers IGP reconvergence or FRR
- Scales better because it is per adjacency, not per LSP

**Pros:** Operationally simple, scalable, widely supported, drives fast convergence for all services using the link
**Cons:** Does not directly test the labeled path — if the IGP path is healthy but label programming is broken, BFD-for-IGP may still stay up

### Recommendation

**Recommend BFD for IGP as the default core fast-failure detector**, paired with FRR / TI-LFA / RSVP FRR depending on the transport design.

Why:
1. **Scale.** In a core, you have far fewer IGP adjacencies than MPLS LSPs. Monitoring the underlay once is cheaper than monitoring every label-switched path individually.
2. **Operational simplicity.** NOC and engineering teams already understand BFD on adjacencies. Troubleshooting is cleaner.
3. **Service breadth.** One IGP BFD session protects LDP, RSVP-TE, VPNs, and any other service using that link.

**When to add BFD for MPLS:** For high-value LSPs or when you specifically need end-to-end labeled path verification beyond adjacency health.

**When RSVP Hello matters:** Mostly as supplemental RSVP control-plane monitoring, not the primary sub-second failover mechanism.

So the real SP answer is: **BFD for IGP first, FRR for fast traffic switchover, BFD for MPLS only where its extra fidelity is worth the state overhead.**

## Question 4
> Explain why LSP Ping uses destination addresses in 127.0.0.0/8. What would happen if it used the actual destination prefix instead?

### Answer

LSP Ping uses **127.0.0.0/8** because those addresses are guaranteed to be **locally terminated and never normally routed end-to-end**. That is exactly what MPLS OAM needs.

The ingress PE builds an MPLS Echo Request, pushes the real label stack for the FEC under test, and sends the packet through the MPLS data plane. When the packet reaches the egress and labels are popped, the egress sees the destination 127.x.x.x and punts it to the local control plane, where the MPLS OAM process validates the FEC Stack TLV and generates an Echo Reply.

**Why this is brilliant:** The packet follows the real data path **as labeled traffic**, but once it reaches the egress, it is guaranteed to stop there and be processed locally.

**If LSP Ping used the actual destination prefix instead:**
1. The packet might simply be forwarded onward as ordinary IP traffic instead of being intercepted by the MPLS OAM process.
2. The egress would have no guaranteed way to distinguish an OAM probe from real payload traffic to that destination.
3. You could accidentally deliver OAM packets to customer systems or arbitrary infrastructure addresses — which is messy, dangerous, and stupid.
4. The sender would lose deterministic knowledge that a reply means "the labeled path reached the egress and was validated." It would instead mean only "some IP host responded," which defeats the purpose.

So the 127/8 destination is not random trivia. It is what guarantees that MPLS Echo tests the real label-switched path while still terminating cleanly and safely at the egress control plane.

## Question 5
> You suspect an MTU issue is causing VPN packet loss for a customer sending 1400-byte packets. Describe the exact commands you'd run on IOS-XE and Junos to confirm the hypothesis, including the specific `size` parameter you'd use and why.

### Answer

A customer sending **1400-byte IP packets** over an MPLS VPN will incur label overhead:
- **L3VPN normal stack:** transport label + VPN label = **8 bytes**
- So the MPLS packet becomes **1408 bytes** before considering any extra repair / TE labels

To test the real forwarding condition, I would use a probe size that reproduces the customer's payload while also pushing the labeled path.

### IOS-XE / IOS-XR

**1. Test MPLS OAM path with matching payload size:**
```
ping mpls ipv4 10.0.0.5/32 size 1400 repeat 5 timeout 1
```
Why `size 1400`? Because I want the MPLS Echo payload to match the customer's IP packet size under suspicion. If the issue is MTU-related, reproducing the same payload size through the labeled path is the fastest way to catch it.

**2. Then test the margin / break point:**
```
ping mpls ipv4 10.0.0.5/32 size 1472 repeat 5 timeout 1
```
Why `1472`? Because 1472-byte IP payload + 20-byte IPv4 header + 8-byte ICMP header = 1500-byte IP packet. Add 8 bytes of MPLS labels and you're at **1508** — which will fail on a strict 1500 MPLS core. This is the classic MTU cliff test.

**3. Check interface MPLS MTU / counters:**
```
show mpls interfaces detail
show interfaces GigabitEthernet0/0/0 | include MTU|drops|giants
show mpls forwarding-table <customer-prefix>
```

### Junos

**1. Probe the labeled path with the customer-sized payload:**
```
ping mpls ldp fec 10.0.0.5/32 size 1400 count 5 rapid
```

**2. Probe the failure threshold:**
```
ping mpls ldp fec 10.0.0.5/32 size 1472 count 5 rapid
```

**3. Validate interface MTU and counters:**
```
show interfaces ge-0/0/0 extensive | match "MTU|giants|errors|drops"
show route forwarding-table destination <customer-prefix> detail
show ldp database 10.0.0.5/32
```

### Interpretation

- **1400 succeeds, 1472 fails:** likely MTU is fine for this customer, but you are near the ceiling
- **1400 fails on MPLS ping but plain IP ping works:** strong evidence of MPLS label-stack overhead causing drops
- **Failures only during FRR/TE events:** you may have deeper temporary stacks (bypass label + transport + VPN label), so the normal 2-label path works but repair path exceeds MTU

**Recommended operational fix:** raise core MTU, not customer expectations.
