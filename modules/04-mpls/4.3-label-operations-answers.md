# 4.3 — Label Operations — Answer Key

## Question 1
> A customer using your L3VPN service reports that traceroute from their CE router shows all the P routers in your core with their loopback addresses. Your security team is concerned. What's happening and how do you fix it on IOS-XE and Junos?

### Answer

**What's happening:** MPLS TTL propagation is enabled, so the customer's IP TTL is being copied into the MPLS label TTL at the ingress PE. Each P router decrements the MPLS TTL. When the TTL expires at a P router, that router generates an ICMP Time Exceeded message sourced from its loopback or outgoing interface. Traceroute sees each P hop and prints it.

This is normal MPLS behavior when **TTL propagation is on**. It makes the MPLS core visible to customer traceroute. Operationally useful, but many SPs consider it an information leak because it exposes core topology and infrastructure addressing.

**How to fix it:** Disable TTL propagation so the ingress PE does **not** copy the customer's IP TTL into the MPLS label stack. Instead, the MPLS domain uses a local TTL (typically 255), and the customer's IP TTL is decremented only at the ingress and egress PEs. The P routers become invisible to traceroute.

**IOS-XE / IOS-XR:**
```
mpls ip propagate-ttl forwarded disable
```
This disables TTL propagation for forwarded traffic (not locally originated control traffic).

**Junos:**
```
set protocols mpls no-propagate-ttl
```

**What changes after the fix:**
- Customer traceroute will typically show:
  - hop 1 = customer-facing PE
  - then a gap / hidden MPLS core
  - final hop(s) near egress PE/CE
- The core no longer leaks P-router loopback addresses

**Trade-off:** Troubleshooting gets harder. With TTL propagation disabled, customer-facing traceroutes no longer reveal where inside the MPLS core delay or loss is occurring. NOC must rely on LSP traceroute / MPLS OAM instead of customer traceroute.

**Security team's concern is valid, but this is not a breach.** It's a configurable visibility choice. Most internet-facing SPs disable TTL propagation. Internal enterprise MPLS often leaves it on because troubleshooting convenience wins.

## Question 2
> You're deploying Inter-AS L3VPN Option C, which requires a 3-label stack. Your core MTU is 1500 bytes. What's the maximum customer IP packet size that can traverse the network, and what's your recommended fix?

### Answer

A 3-label MPLS stack adds **12 bytes** of overhead (3 labels × 4 bytes each).

If the core interface MTU is **1500 bytes total**, then the maximum payload that can fit without fragmentation/drop is:

\[
1500 - 12 = 1488\text{ bytes}
\]

So the **maximum customer IP packet size is 1488 bytes**.

That assumes the 1500 MTU refers to the IP/MPLS payload after L2 framing. If you include Ethernet overhead on the wire, the situation is even worse, but operationally we usually talk about the L3 MTU here.

**Why this matters in Inter-AS Option C:**
Typical stack depth:
1. **Inter-AS / BGP-LU transport label**
2. **iBGP / next-hop label or intra-AS transport label**
3. **VPN label**

That 3-label stack is the baseline. Add FRR, TE, or SR repair labels and the stack gets deeper. A customer sending full-size 1500-byte IP packets will exceed the core MTU and get dropped unless fragmentation occurs before MPLS imposition — which usually it doesn't for transit traffic.

**Recommended fix:**

**Raise the core MTU.** That's the only adult answer.

- Minimum practical fix for 3-label stacks: **1530 bytes**
- Modern best practice: **jumbo MTU everywhere in the core (typically 9100-9216)**

```
IOS-XE:
  interface GigabitEthernet0/0/0
    mtu 9216
    mpls mtu 9216

Junos:
  set interfaces ge-0/0/0 mtu 9192
```

**Why not just tell customers to lower MTU?** Because that's fragile and stupid at scale. One forgotten server NIC and you're back in ticket hell. The MPLS core should be engineered to carry labeled traffic with headroom for realistic label depths, not balanced on a knife edge.

**Operational note:** Test the real path MTU with MPLS-aware pings and DF-bit set before declaring victory.

## Question 3
> `show mpls forwarding-table` on a P router shows `Pop Label` for prefix 10.0.0.5/32, but you expected to see a label swap. Is this a problem? Explain.

### Answer

**Probably not. This is Penultimate Hop Popping (PHP), and it's the default behavior.**

When the egress LER advertises **Implicit NULL (label 3)** for its own loopback/prefix, it is telling the upstream neighbor: "For packets to this FEC, don't send me a real transport label — pop it before it reaches me." The upstream (the penultimate hop) therefore installs a forwarding action of **Pop Label** instead of **Swap**.

So if this P router is the penultimate hop for 10.0.0.5/32, `Pop Label` is exactly what you should expect.

**Why PHP exists:** It saves the egress router from doing two lookups:
1. MPLS label lookup to pop transport label
2. IP or VPN lookup to process the payload

With PHP, the egress receives the packet already unlabeled (or with only the inner service label in VPN scenarios) and does one less operation.

**When it would be a problem:**
- If you expected **Explicit NULL** because you need MPLS TC bits preserved for QoS on the final hop
- If this router is **not** the penultimate hop and is still popping labels unexpectedly — then yes, that's wrong
- If a service label gets popped when only the transport label should be popped — that would break VPN forwarding

**How to verify it's normal:**

```
IOS-XE/XR:
  show mpls forwarding-table 10.0.0.5 32 detail
  show mpls ldp bindings 10.0.0.5 32

Junos:
  show route forwarding-table destination 10.0.0.5/32 detail
  show ldp database 10.0.0.5/32
```

If the downstream/egress advertised label **3 (Implicit NULL)**, then `Pop Label` is correct. This is not a fault. It's MPLS doing exactly what it was raised to do.

## Question 4
> Explain why Explicit NULL exists when PHP is more efficient. Give a specific scenario where Explicit NULL is required.

### Answer

PHP is more efficient because it removes the top label at the penultimate hop, sparing the egress from one MPLS lookup. But **efficiency isn't the only requirement**. Sometimes the egress PE needs to see the MPLS label header — specifically the **Traffic Class (TC/EXP) bits**.

**Explicit NULL** exists to preserve MPLS QoS semantics to the egress hop.

With PHP:
- Penultimate hop pops the transport label
- Egress receives a plain IP packet (or just the inner service label in VPN cases)
- The outer MPLS TC bits are gone

With Explicit NULL:
- Egress advertises label **0** (IPv4 Explicit NULL) or **2** (IPv6 Explicit NULL) instead of Implicit NULL
- Penultimate hop swaps the transport label to 0/2 instead of popping it
- Egress receives a labeled packet, reads the TC bits, pops the Explicit NULL, and applies correct QoS handling on the final hop

**Specific scenario where Explicit NULL is required:**

**MPLS VPN QoS where ingress PE maps customer traffic classes into MPLS TC bits and the egress PE must honor those markings.**

Example:
- Customer voice traffic enters ingress PE with DSCP EF
- Ingress PE maps it into MPLS TC=5 across the core
- Intermediate P routers schedule based on TC
- At the egress PE, you want to preserve that premium treatment onto the egress CE-facing interface

If PHP is used, the outer transport label — and its TC bits — are popped one hop early. The egress may only see the IP DSCP or inner label semantics, which may not reflect the intended QoS behavior. With Explicit NULL, the egress PE still sees the MPLS TC bits and can classify correctly.

**Configuration:**

```
IOS-XE/XR:
  mpls ldp explicit-null

Junos:
  set protocols ldp explicit-null
```

So yes, PHP is more efficient. Explicit NULL exists because operators care about **correctness at the egress** more than shaving one lookup off a router that's already a giant money furnace.

## Question 5
> You have 8 equal-cost paths between two P routers, but labeled traffic is only using 2 of them. What's the likely cause and how do you fix it?

### Answer

**Most likely cause: the platform is hashing only on top-label fields or using a limited ECMP label-aware hashing profile, so MPLS load-balancing entropy is poor.**

Classic MPLS ECMP problems happen because many routers historically hashed only on:
- top label
- source/destination IP of the payload only in limited cases
- or a small subset of packet fields

If many flows share the same outer transport label (common in L3VPN or RSVP/LDP transport), the hash collapses and traffic lands on only a subset of the available ECMP paths — here, 2 out of 8.

**Likely causes:**
1. **MPLS entropy-label support is missing or disabled.** Entropy Labels (RFC 6790) give the network a flow-aware label specifically for better load balancing.
2. **CEF / forwarding ASIC hash profile is too limited.** The router may only be doing 2-way or 4-way load balancing for labeled traffic unless label-aware load sharing is enabled.
3. **Per-platform ECMP limit for MPLS forwarding is lower than the IGP ECMP count.** Some platforms can compute 8-way IP ECMP but only install 2 or 4 labeled next-hops in hardware.

**How to verify:**

```
IOS-XE/XR:
  show cef <prefix> detail
  show mpls forwarding-table <prefix> detail
  show platform hardware qfp active feature cef ecmp

Junos:
  show route forwarding-table destination <prefix> detail
  show chassis forwarding-options hash-key
  show route extensive <prefix>
```

Check how many next-hops are actually installed in hardware for the labeled FEC, not just what the IGP says is available.

**Fix options:**

1. **Enable entropy labels** where supported:
```
IOS-XR/SR environments: segment-routing mpls entropy-label
Junos: set protocols mpls entropy-label
```
2. **Enable label-aware load balancing / deeper payload hashing** on the platform.
3. **Verify hardware ECMP limits** — if the ASIC only supports 2-way MPLS ECMP, the fix may be a platform upgrade, not a config knob.

**Bottom line:** The network *topology* has 8 equal-cost paths. The *forwarding plane* is only exploiting 2. That's usually a hashing/ASIC reality problem, not an IGP problem.
