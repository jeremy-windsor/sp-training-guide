# 4.4 — MPLS OAM and Troubleshooting — Protocol Theory

## Protocol Overview

MPLS OAM (Operations, Administration, and Maintenance) provides tools to test, monitor, and troubleshoot MPLS Label Switched Paths. Because MPLS transit routers don't process IP headers, standard IP tools like ping and traceroute cannot validate the MPLS data plane. MPLS OAM fills this gap with purpose-built mechanisms.

The primary MPLS OAM tools are LSP Ping and LSP Traceroute (RFC 8029, originally RFC 4379), which inject test packets into the MPLS data plane and validate forwarding at each hop. BFD for MPLS (RFC 5884) provides continuous proactive monitoring. The Generic Associated Channel (G-ACh, RFC 5586) provides a general in-band OAM framework.

These tools solve a specific problem: an MPLS LSP may look healthy from the control plane (LDP/RSVP sessions up, labels exchanged) while the data plane is broken (labels mis-programmed in hardware, LFIB inconsistency, MTU issue). MPLS OAM tests the actual forwarding path, not just the control plane state.

## Core Mechanisms

### 1. LSP Ping (RFC 8029)

LSP Ping sends an MPLS Echo Request packet along the label-switched path and expects an MPLS Echo Reply from the target.

#### MPLS Echo Request

The Echo Request is an IP/UDP packet that is label-switched rather than IP-routed:

**Encapsulation:**
```
┌─────────────────┐
│   MPLS Labels    │  Same label stack as the LSP being tested
├─────────────────┤
│   IP Header      │  src = sender, dst = 127.0.0.0/8 (loopback)
├─────────────────┤
│   UDP Header     │  dst port = 3503
├─────────────────┤
│  MPLS Echo TLVs  │  Target FEC Stack, etc.
└─────────────────┘
```

**Why the IP destination is 127/8**: The IP destination address is set to a loopback address (127.0.0.0/8) so that if the MPLS labels are all popped prematurely (label mis-programming), the packet is delivered to the local router's loopback rather than being routed across the network. This prevents test packets from accidentally reaching customer destinations.

**Why UDP port 3503**: Identifies the packet as an MPLS Echo Request to the receiving router's control plane.

**Key TLVs in Echo Request:**

**Target FEC Stack TLV**: Describes the FEC being tested. This allows the responder to verify that the label stack matches the expected FEC:
- LDP IPv4/IPv6 FEC (prefix)
- RSVP-TE LSP FEC (tunnel ID, sender, endpoint)
- VPN FEC (RD, prefix)
- Pseudowire FEC (PW ID, PW type)

**Reply Mode**: How the responder should send the Echo Reply:
- Mode 1: Do not reply
- Mode 2: Reply via IPv4/IPv6 UDP packet (IP-routed, not label-switched)
- Mode 3: Reply via IPv4/IPv6 UDP packet with Router Alert
- Mode 4: Reply via application-level control channel

#### MPLS Echo Reply

The Echo Reply is a UDP packet sent back to the originator. It contains:

**Return Code**: The result of the validation.

| Code | Meaning |
|------|---------|
| 0 | No return code |
| 1 | Malformed Echo Request |
| 2 | Unsupported TLV |
| 3 | Replying router is egress for the FEC |
| 4 | Replying router has no FEC mapping |
| 5 | Downstream Mapping Mismatch |
| 8 | Label switched at stack-depth X |
| 10 | Mapping for this FEC not the given label at stack-depth X |
| 12 | See DDMAP TLV for return code and further details |

**Return Code 3** ("Replying router is egress") = success. The packet reached the tail-end of the LSP via the MPLS data plane.

**Return Code 8** ("Label switched at stack-depth X") = the packet was correctly label-switched at this hop. Used in traceroute mode.

#### Validation Process at Responder

When a router receives an MPLS Echo Request:
1. Check the Target FEC Stack TLV.
2. Verify that the incoming label(s) match the expected FEC in the local LFIB.
3. If mismatch: return code indicates the error.
4. If match and this is the egress: return code 3.
5. If match and this is transit: return code 8 (traceroute) or forward (ping).

This validation catches data plane issues that control plane monitoring misses: hardware LFIB inconsistencies, label mis-programming, and forwarding path errors.

### 2. LSP Traceroute

LSP Traceroute uses the same MPLS Echo Request/Reply mechanism but with incrementing TTL values, similar to IP traceroute.

**Mechanism:**
1. Send Echo Request with MPLS TTL = 1.
2. First transit router: TTL expires, generates Echo Reply with its identity and return code 8.
3. Send Echo Request with MPLS TTL = 2.
4. Second transit router: TTL expires, generates Echo Reply.
5. Continue incrementing TTL until the egress router responds with return code 3.

**Each hop's response includes:**
- IP address of the responding interface
- MPLS label received and forwarded
- Downstream mapping information (DDMAP TLV): what the router would do with the FEC — which label it would swap to, which interface it would forward on.

**DDMAP (Downstream Detailed Mapping) TLV**: Contains:
- Downstream IP address
- Downstream interface index
- Downstream label (what this router would swap to)
- MTU
- Address type and DS flags

This information allows the operator to verify the complete label path, hop by hop, and identify exactly where a forwarding discrepancy exists.

### 3. BFD for MPLS (RFC 5884)

BFD for MPLS provides continuous, proactive monitoring of LSP data plane health.

**Encapsulation:**
BFD packets are sent with the same label stack as the LSP being monitored. The inner IP packet uses destination 127/8 (same as LSP Ping) with UDP port 3784 (standard BFD).

**Session establishment:**
- The head-end of the LSP initiates a BFD session.
- The tail-end validates the session against the expected FEC.
- BFD packets traverse the actual MPLS data plane, testing real forwarding.

**Detection:**
- If BFD packets stop arriving (detection timer expires), the LSP data plane is broken.
- The head-end is notified and can trigger fast reroute or re-signal the LSP.

**BFD vs LSP Ping for monitoring:**
- BFD: continuous, sub-second detection, low overhead per session.
- LSP Ping: on-demand or periodic (seconds-minutes interval), more detailed diagnostic info.
- Use BFD for real-time health monitoring; use LSP Ping for diagnostics and validation.

**Bootstrapping:**
The BFD session for MPLS is bootstrapped using an initial LSP Ping exchange. The Echo Request carries a BFD Discriminator TLV, telling the tail-end which BFD session to associate with this LSP. After bootstrapping, BFD packets flow continuously without further LSP Ping involvement.

### 4. Generic Associated Channel (G-ACh) — RFC 5586

G-ACh provides a generic in-band channel for MPLS OAM, running alongside (associated with) the LSP data traffic.

**GAL (Generic Associated Channel Label, label 13):**
The GAL is pushed below the LSP labels. When a router pops the LSP label and sees label 13, it knows the next data is an Associated Channel Header (ACH), not a customer packet.

**ACH (Associated Channel Header):**
```
├─ First Nibble: 0001 (identifies this as ACH, not IP)
├─ Version: 0
├─ Reserved: 0
├─ Channel Type: identifies the OAM protocol
│   ├─ 0x0007: BFD (control, RFC 5885)
│   ├─ 0x0021: IPv4 (for LSP Ping)
│   ├─ 0x0057: IPv6 (for LSP Ping)
│   ├─ 0x8902: Performance Monitoring (loss, delay)
│   └─ ... others
```

**G-ACh enables:**
- BFD for pseudowires (VCCV, RFC 5885)
- Performance monitoring (loss measurement, delay measurement)
- Connectivity verification
- Route tracing for pseudowires
- OAM for MPLS-TP (Transport Profile)

### 5. Performance Monitoring

Beyond connectivity testing, MPLS OAM includes performance measurement:

#### Loss Measurement (LM)

Counts packets entering and exiting an LSP/PW to detect loss.

**Mechanism:**
- Each endpoint maintains packet counters for the monitored path.
- Periodic LM messages exchange counter values.
- Difference between sent and received counters = packet loss.

**Two modes:**
- **Synthetic LM**: Inject OAM packets and count arrivals. Measures OAM path loss.
- **Inferred LM**: Use actual traffic counters. Measures real traffic loss. More accurate but requires hardware counter support.

#### Delay Measurement (DM)

Measures one-way or two-way delay across an LSP/PW.

**Two-way delay:**
1. Send a DM message with a timestamp.
2. Responder timestamps receipt and sends reply with both timestamps.
3. Originator timestamps reply receipt.
4. Round-trip delay = (T4 - T1) - (T3 - T2), where T1=send, T2=remote receive, T3=remote send, T4=local receive.

**One-way delay:**
Requires clock synchronization (NTP, PTP) between endpoints. T2 - T1 = one-way delay. More useful for asymmetric path analysis but harder to implement accurately.

### 6. MPLS-TP OAM

MPLS-TP (Transport Profile) is a subset of MPLS designed for transport networks. It adds connection-oriented OAM similar to SDH/SONET:

- **Continuity Check (CC)**: Periodic hello-like messages verifying path connectivity.
- **Connectivity Verification (CV)**: Validates the correct endpoint identity.
- **Alarm Indication Signal (AIS)**: Signal to downstream that upstream is broken.
- **Remote Defect Indication (RDI)**: Signal to upstream that downstream detected a defect.
- **Lock Report (LKR)**: Indicates intentional maintenance lockout.

MPLS-TP OAM uses G-ACh for transport. It is primarily deployed by transport-oriented operators (telcos, submarine cable operators) who need SDH-like OAM in an MPLS network.

## Protocol Interactions

**MPLS OAM ↔ Label Operations**: LSP Ping and BFD for MPLS rely on correct label stack encoding. The test packet must carry the exact same label stack as production traffic to test the real forwarding path. Any discrepancy (wrong label, missing label, wrong stack depth) means you're testing a different path than traffic actually uses.

**MPLS OAM ↔ PHP**: PHP complicates LSP traceroute. The penultimate hop pops the label, so the Echo Request arrives at the egress as an IP packet (to 127/8). The egress must recognize this as an MPLS Echo Request based on the UDP port (3503), not based on the label.

**MPLS OAM ↔ FRR**: When FRR activates, the label stack changes (bypass tunnel label added). LSP Ping/BFD packets should follow the same path as data traffic. BFD for MPLS will detect if the primary path breaks and FRR activates, because the BFD session was monitoring the primary path — but the forwarding continues on the bypass.

**MPLS OAM ↔ QoS**: OAM packets should have the same QoS treatment as the traffic they're monitoring. If OAM packets use a different TC value, they may traverse a different queue and give misleading results (OAM succeeds but real traffic is dropped in a congested queue).

## Edge Cases & Gotchas

**1. 127/8 destination and line cards**: Some older line cards or NPUs don't punt 127/8 traffic correctly when it arrives with MPLS labels. The packet may be dropped instead of being sent to the CPU for Echo Reply processing. Verify LSP Ping functionality per platform.

**2. BFD for MPLS bootstrap failure**: If the initial LSP Ping (carrying the BFD Discriminator TLV) fails, the BFD session never establishes. The LSP may look unmonitored — check that the bootstrap exchange completed.

**3. ECMP and OAM path pinning**: LSP Ping/traceroute follows one ECMP path (determined by the test packet's hash). It may show all hops healthy while a different ECMP path is broken. For thorough testing, send multiple probes with different entropy (source port, entropy label) to hit all ECMP paths.

**4. Performance monitoring clock skew**: One-way delay measurement requires tight clock synchronization. NTP accuracy of 1-10ms may be insufficient for measuring sub-millisecond delay. PTP (IEEE 1588) with hardware timestamping provides the accuracy needed.

**5. GAL and transit routers**: A transit router should not inspect label 13 (GAL) unless it has popped all LSP labels above it. If a transit router's LFIB is programmed to pop a label and sees GAL, it must process the ACH — not forward it as data. Mis-configured LFIB entries can cause OAM packets to be silently dropped.

**6. Rate limiting of OAM responses**: Most routers rate-limit ICMP and OAM responses to protect the control plane. If you send many LSP Ping probes rapidly, some replies may be dropped by the rate limiter, creating false negative results. Space probes appropriately.
