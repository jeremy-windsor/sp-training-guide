# 9.3 — OTN (Optical Transport Network) — Protocol Theory

## Protocol Overview

OTN (Optical Transport Network) is the digital framing and multiplexing layer that sits between client signals (Ethernet, FC, SDH/SONET) and the DWDM optical layer. If DWDM provides the highway lanes, OTN provides the standardized containers that ride in those lanes — with built-in performance monitoring, fault management, and protection switching.

OTN was standardized by ITU-T in the G.709 family of recommendations, first published in 2001 and significantly expanded since. It was designed as the successor to SDH/SONET for the optical transport era, inheriting the operations-friendly features of SDH (tandem connection monitoring, protection switching, comprehensive overhead) while supporting modern payloads at modern speeds.

The key value proposition of OTN: **it wraps any client signal in a standardized digital wrapper** that provides operations, administration, maintenance, and performance monitoring (OAM/PM) capabilities end-to-end, hop-by-hop, and across tandem connections — without touching the client payload. The client signal is opaque to OTN; OTN treats it as a container to be transported, monitored, and protected.

## Core Mechanisms

### 1. OTN Frame Hierarchy

OTN uses a layered structure of containers and wrappers:

**OPU (Optical Channel Payload Unit):**
The innermost container. Carries the client payload plus adaptation-specific overhead.
- OPU overhead: mapping type, payload structure, justification (for rate adaptation)
- Client signal is mapped into the OPU payload area

**ODU (Optical Channel Data Unit):**
Wraps the OPU. Adds path-layer overhead for end-to-end monitoring and management.
- ODU overhead: PM (path monitoring), TCM (tandem connection monitoring), APS (automatic protection switching), fault type/fault location
- Each ODU has its own trail trace identifier (TTI), BIP-8 error monitoring, and status/fault fields

**OTU (Optical Channel Transport Unit):**
Wraps the ODU. Adds section-layer overhead for span-by-span monitoring and FEC.
- OTU overhead: section monitoring (SM), FEC
- OTU is the entity transmitted on a single wavelength

**Hierarchy summary:**
```
┌─────────────────────────────────────┐
│  OTU (section layer + FEC)          │
│  ┌─────────────────────────────┐    │
│  │  ODU (path layer + TCM)     │    │
│  │  ┌─────────────────────┐    │    │
│  │  │  OPU (payload + map) │    │    │
│  │  │  ┌───────────────┐   │    │    │
│  │  │  │ Client Signal  │   │    │    │
│  │  │  └───────────────┘   │    │    │
│  │  └─────────────────────┘    │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

### 2. OTN Rate Hierarchy

OTN defines standard rates that map to common client signals:

| Level | Line Rate | Typical Client | Introduced |
|-------|-----------|---------------|------------|
| OTU1 | 2.66 Gbps | GbE, STM-16/OC-48 | G.709 (2001) |
| OTU2 | 10.71 Gbps | 10GbE, STM-64/OC-192 | G.709 (2001) |
| OTU2e | 11.10 Gbps | 10GbE LAN PHY (exact rate) | G.Sup43 |
| OTU3 | 43.02 Gbps | 40GbE, STM-256/OC-768 | G.709 (2001) |
| OTU4 | 111.81 Gbps | 100GbE | G.709 (2009 rev) |
| OTUCn | n × 100G | 200GbE, 400GbE, 800GbE | G.709 (2016 rev) |

**OTUCn (FlexO)**: The modern scalable OTN. "C" stands for "beyond 100G" and "n" is the number of 100G building blocks. OTUC4 = 400G. OTUC8 = 800G. This replaced the fixed-rate model with a flexible, scalable approach.

### 3. ODU Multiplexing

One of OTN's most powerful features: lower-rate ODUs can be multiplexed into higher-rate ODUs.

**Example multiplexing:**
- 4 × ODU1 → 1 × ODU2
- 4 × ODU2 → 1 × ODU3
- 10 × ODU2 → 1 × ODU4
- 80 × ODU0 → 1 × ODU4

**Tributary slots**: Higher-rate ODUs are divided into tributary slots. Lower-rate ODUs are mapped into specific slots. Each tributary has its own overhead and can be independently monitored.

**ODUflex**: A flexible-rate ODU that can carry any client rate. The number of tributary slots assigned to an ODUflex is variable, determined by the client signal's bandwidth requirement. This enables efficient transport of non-standard rates (e.g., 25GbE, Fibre Channel 32G).

**Why multiplexing matters**: A provider can carry a mix of 1G, 10G, and 100G services on a single 400G wavelength. Each service has its own ODU with independent monitoring, fault isolation, and protection — true TDM-style grooming at optical speeds.

### 4. OTN Overhead Fields

OTN overhead is divided by layer and provides comprehensive OAM.

**OTU Overhead (Section Layer):**
- **SM (Section Monitoring)**: TTI (trail trace identifier — 64 bytes, identifies the OTU trail), BIP-8 (bit-interleaved parity for error detection), BEI (backward error indication — tells the far end about errors detected here), IAE (incoming alignment error), BIAE
- **GCC0 (General Communication Channel 0)**: In-band management channel. 2 bytes per frame = ~1.2 Mbps. Used for control plane signaling, management traffic between adjacent OTN nodes.

**ODU Overhead (Path Layer):**
- **PM (Path Monitoring)**: TTI, BIP-8, BEI, status — end-to-end path monitoring between source and destination ODU termination points.
- **TCM1-6 (Tandem Connection Monitoring)**: Six independent monitoring levels, each with its own TTI, BIP-8, BEI, and status. Enables monitoring across intermediate domains (e.g., one TCM per carrier in a multi-carrier path).
- **APS/PCC (Automatic Protection Switching / Protection Communication Channel)**: Signaling for protection switching — which path is active, switchover coordination.
- **GCC1/GCC2 (General Communication Channels)**: Two additional in-band management channels at the path layer.
- **EXP (Experimental)**: Reserved for vendor-specific or future use.
- **FTFL (Fault Type Fault Location)**: 256-byte field for fault identification — what failed and where.

### 5. Tandem Connection Monitoring (TCM)

TCM is what makes OTN operationally superior to raw Ethernet transport. It provides segment-by-segment monitoring without terminating the end-to-end ODU path.

**The problem TCM solves**: In a multi-operator path (Customer → Carrier A → Carrier B → Customer), each carrier needs to monitor errors on their segment without seeing the full end-to-end path. Without TCM, the only monitoring points are the endpoints.

**How TCM works:**
- Six TCM levels (TCM1-6) are available in the ODU overhead.
- Each carrier assigns a TCM level to their segment.
- At the segment ingress, the carrier inserts BIP-8 calculated over the ODU payload.
- At the segment egress, the carrier checks BIP-8 and reports errors.
- The end-to-end PM runs independently across the entire path.
- All six TCMs and PM operate simultaneously, providing seven independent monitoring domains on a single ODU.

**Example multi-domain path:**
```
Customer Site A → [TCM1: Carrier X] → [TCM2: Carrier Y] → [TCM3: Carrier Z] → Customer Site B
                                                                                 [PM: end-to-end]
```

Each carrier monitors their segment independently. If errors occur in Carrier Y's domain, TCM2 catches them — even if PM (end-to-end) also shows errors. This enables fault sectionalization without finger-pointing.

### 6. Forward Error Correction (FEC)

OTN includes standardized FEC in the OTU layer, adding redundancy that allows the receiver to correct bit errors without retransmission.

**FEC types in OTN:**

**GFEC (Generic FEC, ITU-T G.709):**
- Reed-Solomon RS(255,239)
- 7% overhead (16 bytes redundancy per 239 bytes payload)
- Coding gain: ~6.2dB at BER = 10⁻¹²
- Pre-FEC BER threshold: ~10⁻⁴ (corrected to <10⁻¹⁵ post-FEC)
- Standard for OTU1/OTU2/OTU3

**EFEC (Enhanced FEC):**
- Concatenated coding (e.g., BCH + interleaver + RS)
- Same 7% overhead but better coding gain: ~8-9dB
- Proprietary implementations (vendor-specific)
- Used for extended reach applications

**oFEC / SD-FEC (Soft-Decision FEC):**
- Modern soft-decision decoding (LDPC or turbo codes)
- 15-25% overhead (higher overhead enables better correction)
- Coding gain: ~10-12dB
- Pre-FEC BER threshold: ~10⁻² (1 in 100 bits can be wrong and still corrected)
- Essential for coherent 100G+ systems
- Standardized in OIF (Optical Internetworking Forum) implementation agreements

**FEC and system design**: FEC coding gain directly translates to additional reach or margin. A 6dB coding gain means the system can tolerate 6dB more loss — roughly 30km of additional fiber at 0.2 dB/km. SD-FEC at 12dB gain enables reaches that would be physically impossible without it.

### 7. Client Signal Mapping

OTN maps various client signals into OPU containers:

**GFP (Generic Framing Procedure, ITU-T G.7041):**
- Used for packet-based clients (Ethernet, IP)
- GFP-F (frame-mapped): each Ethernet frame is individually mapped. Variable rate, slight overhead.
- GFP-T (transparent): 8B/10B encoded stream mapped directly. Preserves timing but less efficient.

**BMP (Bit-synchronous Mapping Procedure):**
- Used for constant-bit-rate clients (SDH/SONET, Fibre Channel)
- Client signal is bit-synchronized into the OPU payload
- No framing overhead beyond OPU

**AMP (Asynchronous Mapping Procedure):**
- Used when client and OPU clock rates differ slightly
- Justification bytes accommodate rate differences (positive/negative/zero justification)

**IEEE 802.3 100GbE mapping into OPU4**: Direct mapping using 66B encoding. 100GbE's 103.125 Gbps line rate fits into OPU4's 104.794 Gbps payload capacity using GMP (Generic Mapping Procedure) for rate adaptation. AMP/BMP are legacy mapping methods.

## Protocol Interactions

**OTN ↔ DWDM**: Each OTU is transmitted on a single DWDM wavelength. The OTU's FEC corrects errors from optical impairments. DWDM provides the amplified optical path; OTN provides the digital framing and monitoring.

**OTN ↔ Ethernet**: Ethernet frames are mapped into OPU via GFP-F or direct mapping. OTN transports the Ethernet signal with full OAM/PM. The Ethernet client is unaware of OTN wrapping.

**OTN ↔ GMPLS**: OTN switching can be controlled by GMPLS (Generalized MPLS). GMPLS signals ODU cross-connections through the transport network, providing automated provisioning and restoration at the OTN layer.

## Edge Cases & Gotchas

**1. OTU2 vs OTU2e**: OTU2 maps 10GbE WAN PHY (9.953 Gbps). OTU2e maps 10GbE LAN PHY (10.3125 Gbps). Using the wrong mapping causes client signal errors. Know which 10GbE flavor your routers output.

**2. TCM level coordination across carriers**: If Carrier A uses TCM1 and Carrier B also uses TCM1, they overwrite each other's monitoring. TCM levels must be coordinated across all domains in the path. Without coordination, you lose segment monitoring.

**3. ODU multiplexing granularity waste**: An ODU2 (10G) in an ODU4 (100G) consumes 10 tributary slots. If the client signal is 8G, you still use 10G of capacity. ODUflex improves this but adds complexity.

**4. FEC interoperability**: GFEC is standardized; EFEC/SD-FEC are often vendor-specific. Two transponders using different SD-FEC implementations won't interoperate, even if both are "SD-FEC." Check OIF implementation agreements for multi-vendor compatibility.

**5. GCC channel bandwidth**: GCC0/GCC1/GCC2 channels are narrow (~1.2 Mbps each). They're designed for management and signaling, not data transport. Don't try to run meaningful traffic over them.
