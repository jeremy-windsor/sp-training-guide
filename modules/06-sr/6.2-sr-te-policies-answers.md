# 6.2 — SR-TE Policies — Answer Key

## Question 1
> You configure an SR-TE policy with color 100 and endpoint 10.0.0.5 on PE1. A BGP route arrives with next-hop 10.0.0.5 and color 100, but traffic still uses the IGP path. What are the top 3 things to check?

### Answer

**1. Is the SR-TE policy operationally up?**

A configured policy can be admin-up but operationally down if no candidate path is valid (segment list failed validation, PCE returned nothing, or all SIDs are unresolvable).

```
IOS-XR: show segment-routing traffic-eng policy color 100 endpoint ipv4 10.0.0.5
        ! Look for: Admin: up, Operational: up
        ! If Operational: down → check candidate path status

Junos:  show spring-traffic-engineering lsp color 100 detail
```

If the policy is down, BGP has no valid SR-TE next-hop to steer into — it falls back to IGP.

**2. Does the BGP route actually carry the color extended community?**

The route must have the exact color community (color:0:100) attached. If the originating PE isn't setting it, or a route-policy strips it in transit, steering won't trigger.

```
IOS-XR: show bgp vpnv4 unicast <prefix> detail
        ! Look for: Extended Community: Color:100

Junos:  show route <prefix> extensive
        ! Look for: communities: color:0:100
```

**3. Is automatic steering / ODN enabled for the address-family?**

If a matching SR-TE policy already exists, color-based steering can work without ODN. ODN is specifically for on-demand policy instantiation when no matching policy exists. Verify both: is there a matching policy, and if not, is ODN configured?

```
IOS-XR:
  segment-routing traffic-eng
   on-demand color 100
    dynamic
     ! or: pcep / metric type igp / etc.

  ! Also verify under BGP:
  router bgp 65000
   address-family vpnv4 unicast
    ! Steering is implicit when SR-TE policies exist in XR
    ! But ODN templates must be configured for on-demand creation
```

**Additional checks if all three are fine:**
- Endpoint mismatch: the BGP next-hop is 10.0.0.5 but the policy endpoint resolves to a different address or address-family
- A higher-preference candidate path exists pointing to wrong color
- The policy has a binding SID but forwarding isn't using it (check CEF/FIB)

## Question 2
> You have 50 PE routers and want to deploy centralized TE using a PCE. Each PE has ~200 VPN services. What redundancy model should you use for the PCE, and why?

### Answer

**Deploy a stateful HA PCE pair in active/standby configuration.**

### Why stateful HA

With 50 PEs × potentially hundreds of SR-TE policies (driven by ODN for 200 VPN services per PE), the PCE holds significant state:
- Computed paths for all active policies
- Topology via BGP-LS
- PCEP sessions to all 50 PEs
- Delegation state (which policies are PCE-controlled vs locally computed)

If the PCE fails without redundancy, all ODN-created policies lose their computation source. Existing policies continue forwarding (segment lists are installed locally), but:
- New VPN routes with color communities can't trigger ODN (no PCE to compute paths)
- Reoptimization stops
- Policy recovery after topology changes stops

### Recommended model

**Active/Standby with state synchronization:**

1. **Primary PCE** handles all PCEP sessions and computations
2. **Standby PCE** maintains synchronized state (topology, policies, computed paths)
3. Each PE router is configured with both PCE addresses (primary + secondary)
4. On primary failure, PEs detect via PCEP keepalive timeout and reconnect to standby
5. Standby already has the state — failover is fast (seconds, not minutes)

```
IOS-XR (PE config):
  segment-routing traffic-eng
   pce
    peer ipv4 10.0.0.100
     precedence 100
    peer ipv4 10.0.0.101
     precedence 200
    ! Lower precedence = preferred
```

### Why not active/active?

Active/active PCE (split-brain) is possible but adds complexity:
- Need consistent path computation across both PCEs
- Risk of conflicting policy updates
- State synchronization becomes bidirectional

For 50 PEs and ~10,000 policies, active/standby is sufficient and operationally simpler. Active/active becomes relevant at 500+ PEs or when geographic PCE placement matters (e.g., PCE per region).

### Sizing

50 PEs × ~200 policies each = up to **10,000 SR-TE policies**. Modern PCE platforms (Cisco XTC, Juniper Paragon) handle this comfortably on a single server. The redundancy isn't about capacity — it's about availability.

## Question 3
> A policy shows `Operational: up`, but traceroute shows traffic taking the IGP path. The policy uses a segment list of 3 Prefix-SIDs. What could cause this?

### Answer

The policy is up and has a valid segment list, but traffic isn't being steered into it. This is a **forwarding-plane / steering disconnect**, not a policy computation problem.

**Most likely causes:**

**1. The policy isn't installed as the forwarding next-hop for the relevant prefixes.**

The policy exists, but the routes that should use it aren't resolving through it. Check whether BGP routes have the correct color community and whether the forwarding table actually points to the policy's binding SID.

```
IOS-XR: show cef <vpn-prefix> detail
        ! Does it show the SR-TE policy as next-hop?
        ! Or does it show the IGP next-hop?

        show segment-routing traffic-eng forwarding policy color 100
        ! Are prefixes associated with this policy?
```

**2. Traceroute is misleading you.**

Standard IP traceroute may not reflect the SR label path accurately. If the policy pushes a 3-label stack and PHP occurs, intermediate hops may respond with their interface IPs (which happen to match the IGP path). The traffic might actually BE on the SR path, but traceroute makes it look like IGP.

Use SR-aware traceroute instead:
```
IOS-XR: traceroute segment-routing traffic-eng policy name <policy>
        traceroute sr-mpls <endpoint> color <color>
```

**3. The binding SID is allocated but not programmed in hardware.**

The policy is operationally up in the control plane, but the BSID label isn't installed in the FIB/LFIB. This can happen due to:
- Hardware table exhaustion
- Platform bug
- Label conflict with dynamic range

```
IOS-XR: show mpls forwarding labels <bsid-label>
        ! Is the label present and pointing to the correct output?
```

**4. Another resolution path has higher priority.**

If a static route, autoroute from an RSVP tunnel, or a more-specific route exists, it may win over the SR-TE color steering. Check the routing table for the specific prefix — what's actually the active next-hop?

**Bottom line:** "Policy up but traffic on IGP" is almost always a steering/resolution problem, not a policy problem. Check the forwarding table, not just the policy state.
