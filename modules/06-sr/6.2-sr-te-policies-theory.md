# 6.2 — SR-TE Policies — Protocol Theory

## Protocol Overview

SR-TE Policies are the traffic engineering construct for Segment Routing. Where Section 5.3 covered the SR-TE architecture conceptually alongside RSVP-TE comparison, this section focuses on the detailed protocol mechanics of how SR-TE policies are instantiated, how steering works, and how BGP distributes SR policies.

An SR-TE policy is identified by the tuple (headend, color, endpoint). It contains one or more candidate paths, each with one or more weighted segment lists. The active candidate path (highest preference, valid) determines forwarding. Traffic is steered into policies via BGP color communities, static routes, or direct policy binding.

The SR-TE policy model is defined in RFC 9256 (SR Policy Architecture). Policy distribution via BGP uses the SR Policy SAFI (AFI 1, SAFI 73), defined in draft-ietf-idr-sr-policy-safi (progressing toward RFC).

## Core Mechanisms

### 1. Policy Instantiation and Lifecycle

**Policy creation sources:**
- **Static configuration**: Operator manually defines headend, color, endpoint, and segment lists on the router.
- **On-Demand Next-Hop (ODN)**: Router automatically creates a policy when a BGP route with a matching color community arrives and no existing policy serves it.
- **PCE-initiated (RFC 8281)**: An external PCE creates the policy via PCEP and pushes it to the headend.
- **BGP SR Policy SAFI**: A controller distributes the policy via BGP to the headend.

**Policy lifecycle states:**
- **Active**: Highest-preference valid candidate path is installed in forwarding. Traffic is steered.
- **Inactive**: No valid candidate path exists. Traffic falls back to IGP shortest path or is dropped (depending on config).
- **Transition**: When candidate paths change (new path computed, old path invalidated), MBB-like behavior ensures hitless switchover.

**Validity of a candidate path**: A candidate path is valid if at least one segment list within it is valid. A segment list is valid if every SID in the list resolves in the forwarding plane (the node/adjacency exists and is reachable).

### 2. Candidate Path Selection

When multiple candidate paths exist for a policy, the selection algorithm is:

1. **Highest preference wins**: Each candidate path has a preference value (architecturally a 32-bit unsigned integer; some CLIs constrain it to smaller ranges such as 1-65535). Higher = more preferred. If one candidate has preference 200 and another has 100, the 200 wins.

2. **Within same preference — origin tiebreak**: If multiple candidates have the same preference:
   - PCEP-originated > BGP-originated > locally configured
   - Within same origin, lower originator ID wins

3. **Validity check**: Only valid candidates (all SIDs resolvable) are considered.

**Failover**: When the active candidate path becomes invalid (a SID becomes unreachable), the next highest-preference valid candidate activates automatically.

### 3. Traffic Steering Mechanisms

**Color-based steering (primary method):**
1. BGP route arrives with Color Extended Community (e.g., color 100).
2. Router checks: does an SR-TE policy exist for (self, color=100, endpoint=BGP-next-hop)?
3. If yes: install BGP route with the policy as the resolution.
4. Traffic matching the route enters the policy's active segment list.

**Automatic steering**: When BGP next-hop resolution finds an SR-TE policy, no explicit steering configuration is needed. The BGP route "automatically" steers into the policy via the color+endpoint match.

**Static binding**: A static route or PBR rule can explicitly reference an SR-TE policy by name or BSID.

**BSID-based steering**: Any packet arriving with the policy's Binding SID as the top label enters the policy. This enables policy chaining and inter-domain stitching.

**Steering preference**: When both an SR-TE policy and a regular IGP/LDP path exist for a destination, the SR-TE policy takes precedence if the BGP color matches. If no color matches, traffic uses the default IGP path.

### 4. Weighted ECMP Across Segment Lists

A candidate path can contain multiple segment lists, each with a weight.

**Behavior:**
- Traffic is distributed across segment lists proportional to their weights.
- Weight 1 + weight 1 = 50/50 split.
- Weight 3 + weight 1 = 75/25 split.
- If a segment list becomes invalid, traffic shifts to remaining valid lists (weights recalculated among valid lists).

**Use case**: Load balancing across diverse paths. Segment list 1 goes through the northern path; segment list 2 goes through the southern path. Both serve the same policy with weighted distribution.

### 5. BGP SR Policy SAFI

SR-TE policies can be distributed via BGP using a dedicated SAFI.

**Address family**: AFI 1 (IPv4) or AFI 2 (IPv6), SAFI 73 (SR Policy).

**NLRI encoding:**
```
├─ Distinguisher (4 bytes): for uniqueness in RIB
├─ Policy Color (4 bytes)
├─ Endpoint (4 or 16 bytes)
```

**Attributes carried:**
- **Tunnel Encapsulation Attribute (Type 23, RFC 9012)**: Contains the SR Policy sub-TLV with:
  - Preference
  - Binding SID
  - Segment List(s) with SID entries
  - Policy Name
  - Policy Candidate Path Name

**Why BGP for SR Policy**: Allows centralized controllers to push SR-TE policies to routers without PCEP. BGP is already present everywhere; reusing it avoids deploying another protocol. Also enables policy distribution across AS boundaries via standard BGP mechanisms.

### 6. PCEP for SR-TE Policy Management

PCEP (RFC 8664 for SR extensions) provides a richer interface for policy management than BGP:

**Stateful PCE operations:**
- **Report**: PCC reports all local SR-TE policies to PCE (state synchronization).
- **Update**: PCE modifies an existing policy's segment list.
- **Initiate**: PCE creates a new policy on the PCC.
- **Delete**: PCE removes a policy from the PCC.

**Delegation**: A PCC can delegate a policy to the PCE, giving the PCE authority to update it. Non-delegated policies are reported but not PCE-modifiable.

**Advantages over BGP SR Policy SAFI:**
- Bidirectional: PCE can query, update, and manage policies (BGP is mostly push).
- Stateful: PCE knows the full network policy state and can optimize globally.
- Faster: PCEP updates are targeted to specific policies (BGP reconverges per-prefix).

### 7. Segment List Validation

Before a segment list is installed in forwarding, each SID is validated:

**Node SID validation:**
- Is the Prefix-SID index present in the IGP?
- Is the associated prefix reachable?
- Does the SRGB resolve to a valid label?

**Adjacency SID validation:**
- Is the Adj-SID still allocated by the neighbor?
- Is the adjacency up?

**Binding SID validation:**
- Does the BSID reference a valid, active policy?

If any SID fails validation, the segment list is invalid. If all segment lists in a candidate path are invalid, the candidate path is invalid. This cascading validation ensures that traffic is never steered into a broken path.

## Protocol Interactions

**SR-TE Policy ↔ BGP**: BGP color communities trigger policy selection. BGP SR Policy SAFI distributes policies from controllers. The coupling is tight — SR-TE's primary use case is service-aware traffic steering driven by BGP VPN routes.

**SR-TE Policy ↔ PCEP**: PCE provides global optimization, inter-domain computation, and dynamic policy management. PCEP is the preferred control interface for large-scale SR-TE deployments.

**SR-TE Policy ↔ ODN**: ODN automates policy creation from templates. Combined with Flex-Algo, ODN enables zero-touch intent-based TE.

**SR-TE Policy ↔ TI-LFA**: TI-LFA protects individual segments within a policy's segment list. If a link/node fails, TI-LFA's backup segment list kicks in for the affected segment, maintaining the overall policy path.

## Edge Cases & Gotchas

**1. Color 0 is not a color**: Color value 0 typically means "no color" or "default." Depending on implementation, a BGP route with no Color Extended Community may or may not match a policy with color 0.

**2. Endpoint mismatch with BGP next-hop**: The SR-TE policy endpoint must exactly match the BGP next-hop address. If BGP next-hop is changed (next-hop-self, route-map), the endpoint in the policy must match the new next-hop, not the original.

**3. Segment list ordering matters**: SIDs in a segment list are processed left to right (top to bottom in the label stack). Reversing two SIDs changes the path entirely.

**4. BSID stability during candidate path changes**: When the active candidate path changes, the BSID remains stable. This is by design — upstream references to the BSID don't break. But traffic may take a different path without any upstream notification.

**5. PCE and BGP policy conflict**: If both PCE and BGP distribute a candidate path for the same policy with the same preference, the origin tiebreaker applies (PCEP wins over BGP). This can cause unexpected behavior if both sources are active.
