# 6.4 — SRv6 Fundamentals — Answer Key

These questions are IE-SP caliber — scenario-based, requiring you to apply multiple concepts.

---

## Question 1
> You're evaluating SRv6 for a greenfield metro network. Your hardware team confirms the routers support SRv6 (NCS 5500 with J2 ASICs). Your backbone MTU is 9000 bytes and customer CE-facing MTU is 1500 bytes. The design calls for L3VPN services using SRv6 with up to 4 SRv6 SIDs in the segment list. Will you hit MTU problems? Show your math.

### Answer

Yes, you will hit MTU problems in certain scenarios. Here's the math.

A customer sends a 1500-byte IPv4 packet into the VPN. At the PE, SRv6 encapsulation adds:

**Outer IPv6 header:** 40 bytes (fixed — source is the encapsulating PE, destination is the first SID in the segment list).

**SRH (Segment Routing Header):** 8 bytes base header + (N × 16 bytes) for the segment list, where N is the number of SIDs. With 4 SIDs: 8 + (4 × 16) = 72 bytes.

**Total overhead:** 40 + 72 = 112 bytes.

**Encapsulated packet size:** 1500 + 112 = 1612 bytes.

Your backbone MTU is 9000, so 1612 bytes fits easily on backbone links. No problem there.

But here's the catch: if the customer sends a full 9000-byte jumbo frame (which your CE-facing interface accepts), the encapsulated packet becomes 9000 + 112 = 9112 bytes. That exceeds your 9000-byte backbone MTU. Black hole.

With micro-SID (uSID) encoding, you can compress multiple SIDs into fewer 128-bit containers — typically 4 uSIDs fit in a single 128-bit SID, reducing the SRH to just 8 + (1 × 16) = 24 bytes, total overhead 64 bytes. That's much better, but you still need to account for it.

**Comparison to SR-MPLS:** The same 4-segment path in SR-MPLS adds only 4 × 4 = 16 bytes of overhead. SRv6's 112 bytes is 7× more overhead. This is SRv6's fundamental trade-off — richer SID semantics at the cost of MTU budget.

**Recommendation:** Set backbone MTU to 9216 or higher (most modern optics support this). Alternatively, enforce a path MTU on customer VRFs that accounts for SRv6 overhead. And seriously consider uSID — it's not optional for production SRv6 deployments; it's the mechanism that makes the overhead manageable.

## Question 2
> Explain the SRv6 locator structure. Your network has 3 regions (US-East, US-West, Europe) with approximately 200 routers total. Design a locator addressing plan that supports per-region summarization and allows future growth. Show the block allocation.

### Answer

An SRv6 locator is a /48 IPv6 prefix that identifies a node's SRv6 SID space. Every SID allocated on that node falls within this /48. The locator is advertised via the IGP, making the node's SIDs routable.

The structure: `<Locator Block>:<Node ID>:<Function>:<Arguments>`

Typically: the first 32 or 40 bits are the "block" (identifying the SR domain), the next 8–16 bits identify the node, and the remaining bits encode the function (END, END.DT4, END.DX6, etc.) and optional arguments.

**Design for 3 regions, 200 routers:**

Allocate from `fd00::/8` as the SRv6 block (locally-assigned ULA per RFC 4193 — fc00::/8 is reserved for future use).

| Region | Locator Range | Per-Node Allocation | Capacity |
|--------|--------------|-------------------|----------|
| US-East | `fc00:0:1::/40` | `fc00:0:1:XX::/48` per router | 256 routers |
| US-West | `fc00:0:2::/40` | `fc00:0:2:XX::/48` per router | 256 routers |
| Europe | `fc00:0:3::/40` | `fc00:0:3:XX::/48` per router | 256 routers |
| Reserved | `fc00:0:4-F::/40` | Future regions | 12 more regions |

Each router gets a /48 from its regional /40. Within the /48, functions are allocated automatically by the platform (END gets one SID, each VRF's END.DT4 gets another, etc.).

**Summarization:** At regional boundaries, advertise the /40 summary instead of individual /48s. US-East ABRs advertise `fc00:0:1::/40` toward other regions. This keeps the global IGP table lean — 3 summaries instead of 200 individual locators.

**Growth:** The /40 per region supports 256 nodes. The /32 block supports 16 regions. If a region exceeds 256 routers, split it into sub-regions using /44 boundaries within the /40. The hierarchy is clean and extensible.

**Critical rule:** Never overlap locators. Each router's /48 must be unique. Treat locator allocation like loopback IP allocation — centralized, documented, version-controlled.

## Question 3
> A colleague argues that SRv6 eliminates the need for MPLS entirely — "just rip out LDP and RSVP-TE and use SRv6 for everything." You disagree. What are the three strongest technical arguments against a full SRv6 migration today, and when would you recommend SRv6 over SR-MPLS?

### Answer

Your colleague is directionally correct but operationally premature. Three arguments against full SRv6 migration today:

**1. Hardware support is not universal.** SRv6 requires specific ASIC capabilities — the ability to parse SRH, process uSID, and perform H.Encaps/H.Decaps at line rate. As of 2026, only newer-generation ASICs support this: Memory Broadwell (Barefoot/Intel Tofino), Memory Memory Memory Broadcom Memory Memory (Memory Memory Silicon One Q200), and Juniper's Express 5 / Trio 6. Older platforms — ASR 9000 with Typhoon line cards, MX with Trio 5 — can't do SRv6 at all or only with severe performance limitations. A backbone with mixed-generation hardware can't do pure SRv6. You'd have SR-MPLS on old boxes and SRv6 on new ones, with interworking gateways in between. That's *more* complexity, not less.

**2. MTU overhead is real and painful.** As shown in Question 1, SRv6 adds 64–112 bytes of encapsulation overhead per packet. In a network carrying customer jumbo frames or stacking multiple service layers (VPN inside VPN), this erodes usable MTU significantly. SR-MPLS adds 4 bytes per label. For deep label stacks (inter-AS VPN, 6PE, hierarchical TE), the MTU math on SRv6 gets ugly fast. uSID helps but doesn't eliminate the problem.

**3. Operational maturity gap.** NOC engineers know MPLS. They know `show mpls forwarding`, they know `traceroute mpls`, they know how to read a label stack. SRv6 troubleshooting requires reading IPv6 extension headers, understanding SRH semantics, and using tools that are still maturing. TAC knowledge bases, vendor support playbooks, and third-party NMS tools are all MPLS-first. Deploying SRv6 into production means your NOC is debugging with less institutional knowledge and fewer tools.

**When SRv6 is the right answer:**
- Greenfield IPv6-native network with no legacy MPLS (mobile backhaul, 5G transport)
- Network programming use cases that need SRv6's richer function model (service chaining with END.AD, END.AM)
- Environments where MPLS label space is genuinely constrained (extremely large-scale DC fabrics with millions of paths)
- Strategic investment when the hardware is already SRv6-capable and the team has time to build operational expertise

The answer isn't "never SRv6." It's "not yet for most production SP backbones, but plan your hardware refresh to be SRv6-ready."

## Question 4
> Walk through SRv6 packet processing for an L3VPN scenario. PE-1 (fc00:0:1::/48) needs to send a customer IPv4 packet (10.0.0.1 → 10.100.1.5) to PE-2 (fc00:0:2::/48) through intermediate router R5 (fc00:0:5::/48). PE-2 has an END.DT4 SID for the customer VRF at fc00:0:2:d400::. Describe every header operation at every hop.

### Answer

This is the full SRv6 forwarding walk-through — the equivalent of tracing an MPLS label stack hop by hop.

**At PE-1 (ingress):**
The customer IPv4 packet arrives on the VRF interface. PE-1 does a VRF route lookup for 10.100.1.5 — the BGP VPN route says the next-hop is PE-2, with SRv6 SID `fc00:0:2:d400::` (END.DT4). The SR policy says to route via R5.

PE-1 performs H.Encaps (Headend Encapsulation):
- Builds an outer IPv6 header: Source = `fc00:0:1::1` (PE-1's address), Destination = `fc00:0:5::` (R5's END SID — first segment to visit)
- Builds an SRH with Segment List = [`fc00:0:2:d400::`, `fc00:0:5::`] (reverse order per RFC 8754 — index 0 is the final segment, index 1 is the first), Segments Left = 1
- The original IPv4 packet becomes the inner payload
- Packet size: original + 40 (IPv6) + 40 (SRH with 2 segments) = original + 80 bytes

**At transit routers between PE-1 and R5:**
Each router sees an IPv6 packet with destination `fc00:0:5::`. This matches a route in the IPv6 FIB (R5's locator, learned via IS-IS). Standard IPv6 forwarding — decrement hop limit, forward toward R5. These routers don't look at the SRH at all. They just route IPv6.

**At R5:**
R5 receives the packet with IPv6 DA = `fc00:0:5::`. This matches R5's own locator — it's an END function. R5 processes the SRH:
1. Segments Left = 1, so decrement to 0
2. Copy Segment List[0] (`fc00:0:2:d400::`) into the IPv6 Destination Address
3. Forward the packet with new DA = `fc00:0:2:d400::`

The SRH stays in the packet but Segments Left is now 0. The outer IPv6 DA is now PE-2's END.DT4 SID.

**At transit routers between R5 and PE-2:**
Same as before — standard IPv6 forwarding toward `fc00:0:2:d400::`, which falls within PE-2's locator prefix `fc00:0:2::/48`. Route via IS-IS.

**At PE-2:**
PE-2 receives the packet with IPv6 DA = `fc00:0:2:d400::`. This matches its END.DT4 SID. PE-2 processes:
1. Segments Left = 0, confirmed — this is the final segment
2. END.DT4 behavior: strip the outer IPv6 header and SRH entirely (decapsulation)
3. Look up the inner IPv4 destination (10.100.1.5) in the VRF "cust-a" routing table
4. Forward the original IPv4 packet to the customer CE

**Key differences from SR-MPLS:** No label swap operations. Transit routers do plain IPv6 forwarding. Only the SRv6-aware endpoints (PE-1, R5, PE-2) process the SRH. This is both SRv6's strength (simpler transit) and weakness (more header overhead, larger encapsulation).

## Question 5
> Your SP is running SR-MPLS today and planning an eventual migration to SRv6. The CTO asks you to design an interworking strategy so both can coexist during the multi-year transition. What are the interworking options, what are the trade-offs, and which would you recommend?

### Answer

Three interworking approaches exist, each with different complexity and capability trade-offs:

**Option 1: SRv6/SR-MPLS Gateway Nodes.** Designate specific routers at domain boundaries as "gateway" nodes running both SR-MPLS and SRv6 simultaneously. These routers translate between the two data planes — they receive an SR-MPLS labeled packet, pop the MPLS labels, re-encapsulate into SRv6 (or vice versa). This is the most common approach.

Trade-offs: The gateway adds latency (two encapsulation operations) and becomes a potential bottleneck. You need at least two gateways per boundary for redundancy. The gateway must maintain state mapping between MPLS labels and SRv6 SIDs. Configuration complexity is high — you're running two SR control planes on the same box.

**Option 2: BGP-based interworking with per-VPN SID translation.** Use BGP to advertise VPN routes with both MPLS labels and SRv6 SIDs. PEs in the SR-MPLS domain use MPLS labels; PEs in the SRv6 domain use SRv6 SIDs. Route reflectors carry both attributes. Each PE uses whichever encapsulation its domain supports.

Trade-offs: Cleanest separation — no gateway needed for VPN traffic. But it requires BGP extensions (RFC 9252) and both domains must have independent transport reachability. Inter-domain TE is limited. Best for VPN services; doesn't help with core transport interworking.

**Option 3: Dual-stack SR (run both everywhere).** Every router runs SR-MPLS for MPLS-based services and SRv6 for IPv6-native services simultaneously. No interworking needed because both planes coexist.

Trade-offs: Doubles the control plane complexity. Every router needs SRGB and SRv6 locators. IGP carries both SR-MPLS SIDs and SRv6 locators. Operationally expensive but provides the most flexibility during transition.

**Recommendation:** Option 2 (BGP-based) for VPN services combined with Option 1 (gateway nodes) for a small number of core transit boundaries. This lets you migrate region by region — deploy SRv6 in new metro builds, keep SR-MPLS in the existing backbone, and use BGP to bridge VPN services across both. Gateway nodes handle the handful of cases where end-to-end transport TE is needed across domains.

The key principle: migrate services first (VPNs can use either encapsulation via BGP), then migrate transport last (backbone SR-MPLS to SRv6 is the final step, years down the road). Don't try to boil the ocean.
