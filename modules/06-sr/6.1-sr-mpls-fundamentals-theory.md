# 6.1 — SR-MPLS Fundamentals — Protocol Theory

## Protocol Overview

Segment Routing (SR) is a source-routing architecture where the head-end encodes the forwarding path as an ordered list of instructions — segments — in the packet header. Each segment is identified by a Segment Identifier (SID). In the MPLS data plane instantiation (SR-MPLS), SIDs are encoded as MPLS labels.

SR was conceived by Clarence Filsfils and colleagues at Cisco, formalized in RFC 8402 (SR Architecture, 2018). The motivation was straightforward: RSVP-TE and LDP had scaling problems. RSVP-TE maintained per-LSP state at every transit router and required continuous refresh. LDP required a separate protocol session alongside the IGP. SR eliminates both by distributing label bindings through the IGP itself and pushing forwarding instructions into the packet at the source.

SR-MPLS uses the existing MPLS data plane — the same 4-byte label stack entries, the same push/swap/pop operations, the same hardware forwarding. What changes is the control plane: instead of LDP or RSVP-TE distributing labels, the IGP (IS-IS or OSPF) carries SID information in its existing LSPs/LSAs. No new protocol, no new sessions, no new state.

**The fundamental SR principle**: the network advertises segments (capabilities); the head-end combines them into paths (instructions). Transit routers don't know about end-to-end paths — they just process the top segment and forward. This separation of intent (head-end) from execution (transit) is what makes SR scalable.

## Core Mechanisms

### 1. Segment Types

SR defines several segment types. In SR-MPLS, each is represented as an MPLS label.

#### Prefix Segment (Node SID)

A Prefix Segment identifies a specific prefix (typically a node's loopback) in the SR domain. It is globally significant — the same SID index maps to the same destination everywhere in the domain.

**Properties:**
- Allocated per prefix, usually per /32 loopback.
- Expressed as an index into the SRGB (Segment Routing Global Block).
- Actual MPLS label = SRGB base + index.
- Traffic following a Prefix-SID takes the ECMP-aware IGP shortest path to that prefix.
- Survives topology changes — as long as the prefix is reachable, the SID works.

**Example**: Node A has loopback 10.0.0.1/32 with Prefix-SID index 1. SRGB starts at 16000. The MPLS label for this SID is 16001 everywhere in the domain.

#### Adjacency Segment (Adj-SID)

An Adjacency Segment identifies a specific link (adjacency) from a specific router. It is locally significant — only meaningful at the router that allocated it.

**Properties:**
- Allocated per adjacency, from the router's local label space (SRLB or dynamic range).
- Forces traffic onto a specific link, regardless of IGP shortest path.
- Not ECMP-aware — it identifies exactly one link.
- Used for strict path control when Prefix-SIDs don't provide enough specificity.

**Example**: Router B has two links to Router C. Adj-SID 24001 forces traffic onto link 1; Adj-SID 24002 forces traffic onto link 2. A segment list {Prefix-SID(B), Adj-SID(24001)} routes to B via shortest path, then exits on link 1 specifically.

#### Anycast Segment

An Anycast Segment is a Prefix-SID shared by multiple nodes. Traffic follows the IGP shortest path to the nearest node advertising that SID.

**Use cases:**
- Anycast gateway / service redundancy.
- TE to "nearest instance of service X" without specifying which node.

**Properties:**
- Multiple nodes advertise the same Prefix-SID index for different loopbacks.
- The A-flag (Anycast flag) is set in the IGP advertisement.
- IGP naturally selects the closest instance.

### 2. SRGB (Segment Routing Global Block)

The SRGB is a contiguous range of MPLS labels reserved for SR global segments (Prefix-SIDs).

**Properties:**
- Configured on every SR-capable router.
- Default SRGB varies by vendor: Cisco IOS-XR defaults to 16000-23999 (8000 labels). Junos defaults to 16000-16999 (but configurable).
- All routers in the domain SHOULD use the same SRGB for operational simplicity (not strictly required, but strongly recommended).
- A Prefix-SID index of N maps to MPLS label SRGB_base + N.

**Why same SRGB everywhere matters**: If all routers use SRGB 16000-23999, then Prefix-SID index 1 = label 16001 everywhere. A packet with label 16001 always means "go to the node with index 1." This makes troubleshooting trivial — `show mpls forwarding` shows the same label at every hop.

**What happens with mismatched SRGBs**: If Router A has SRGB 16000-23999 and Router B has SRGB 32000-39999, Prefix-SID index 1 = label 16001 on A but label 32001 on B. The IGP handles this — each router computes the correct label for each hop based on the downstream router's SRGB. But it makes operational debugging harder because labels change at every hop (like LDP behavior).

### 3. SRLB (Segment Routing Local Block)

The SRLB is a range of labels reserved for locally significant segments (Adj-SIDs, Binding SIDs).

**Properties:**
- Separate from SRGB — must not overlap.
- Labels in the SRLB are only meaningful on the allocating router.
- Size depends on how many adjacencies and local policies the router needs.

### 4. IGP Extensions for SR

SR-MPLS distributes SID information through IGP extensions. No separate protocol needed.

#### IS-IS Extensions (RFC 8667)

**Prefix-SID Sub-TLV (type 3, under TLV 135 Extended IP Reachability):**
```
├─ Flags (1 byte):
│   ├─ R: Re-advertisement flag
│   ├─ N: Node-SID flag (this prefix represents a node, not a stub)
│   ├─ P: No-PHP flag (don't pop at penultimate hop)
│   ├─ E: Explicit NULL flag (use Explicit NULL instead of Implicit NULL)
│   ├─ V: Value flag (SID is absolute value, not index)
│   └─ L: Local flag (SID has local significance)
├─ Algorithm (1 byte): 0 = SPF, 128-255 = Flex-Algo
└─ SID/Index/Label (variable): 3 or 4 bytes
```

**Adj-SID Sub-TLV (type 31, under TLV 22 Extended IS Reachability):**
```
├─ Flags (1 byte):
│   ├─ F: Address Family flag (0=IPv4, 1=IPv6)
│   ├─ B: Backup flag (this is a backup Adj-SID for FRR)
│   ├─ V: Value flag (SID is absolute label value)
│   ├─ L: Local flag
│   ├─ S: Set flag (part of a set of Adj-SIDs)
│   └─ P: Persistent flag (survives IS-IS restart)
├─ Weight (1 byte): for weighted ECMP across parallel Adj-SIDs
└─ SID/Label (3 or 4 bytes)
```

**Router Capability TLV (242) — SR sub-TLV:**
Advertises the router's SR capabilities:
- SRGB range(s): base label + range size
- SR algorithm(s): which algorithms this router supports (0=SPF, 128-255=Flex-Algo)

**LAN Adj-SID Sub-TLV (type 32):** For broadcast/LAN segments where multiple neighbors share a segment. Includes a Neighbor System ID to disambiguate.

#### OSPF Extensions (RFC 8665)

SR information in OSPF uses Extended Prefix/Link Opaque LSAs (RFC 7684):

**Extended Prefix LSA (Type 7, Opaque):**
- Carries Prefix-SID sub-TLV: flags, algorithm, SID index.

**Extended Link LSA (Type 8, Opaque):**
- Carries Adj-SID sub-TLV: flags, weight, SID label.

**Router Information LSA (Type 4/9/10/11, Opaque):**
- Carries SR-Algorithm TLV and SID/Label Range TLV (SRGB).

The information content is identical to IS-IS — same SID types, same flags, same semantics. Only the encoding and flooding mechanism differ.

### 5. SR-MPLS Forwarding

SR-MPLS forwarding uses standard MPLS label operations. No new data plane required.

**Ingress (head-end):**
1. Classify the packet (destination, VPN, SR-TE policy).
2. Look up the segment list.
3. Push the label stack: one label per segment, outermost = first segment.
4. Forward based on the top label.

**Transit:**
1. Receive MPLS packet.
2. Look up the top label in the LFIB.
3. If it's a Prefix-SID and this router is NOT the node identified by the SID: swap to the downstream router's label for the same SID, decrement TTL, forward.
4. If it's a Prefix-SID and this router IS the node: pop the label (or swap to next segment if PHP is disabled).
5. If it's an Adj-SID: pop the label and forward on the specified adjacency.

**Egress (tail-end):**
1. Receive packet with bottom-of-stack label.
2. Pop the label.
3. Process the inner packet (IP lookup, VPN lookup, etc.).

**PHP behavior in SR**: By default, Prefix-SIDs use PHP (penultimate hop pops the label). The N-flag (Node-SID) and P-flag (no-PHP) in the Prefix-SID sub-TLV control this:
- N=1, P=0: Node-SID with PHP (default for loopbacks).
- N=1, P=1: Node-SID without PHP (label preserved to egress).
- E=1: Use Explicit NULL instead of Implicit NULL.

### 6. SR and LDP Interworking

During migration from LDP to SR, both may coexist.

**SR-prefer mode**: When both SR and LDP have a label for the same prefix, prefer the SR label. This allows gradual migration — enable SR on all routers, then remove LDP.

**LDP-to-SR stitching**: At the boundary between SR and LDP domains, a router may receive a packet with an LDP label and need to forward it into the SR domain (or vice versa). The boundary router performs label stitching — swapping the LDP label for the corresponding SR label.

**Mapping Server (RFC 8661)**: For routers that don't support SR (legacy), a Mapping Server advertises Prefix-SID bindings on behalf of non-SR routers. SR-capable routers can then build end-to-end SR-MPLS LSPs through the mixed network. The Mapping Server doesn't need to be in the forwarding path — it just advertises bindings via IGP.

## Protocol Interactions

**SR-MPLS ↔ IGP**: SR-MPLS is fully integrated with the IGP. SIDs are advertised in IS-IS TLV 135 sub-TLVs or OSPF Extended Prefix/Link LSAs. No separate protocol. IGP convergence = SR convergence. When the IGP topology changes, SR forwarding updates automatically because the LFIB is derived from IGP + SID information.

**SR-MPLS ↔ BGP**: BGP VPN routes resolve their next-hops via SR labels instead of LDP labels. The transport label stack uses Prefix-SIDs to reach the BGP next-hop PE. BGP doesn't know or care whether transport uses SR or LDP — it just resolves the next-hop.

**SR-MPLS ↔ TI-LFA**: TI-LFA is the native FRR mechanism for SR. It computes backup segment lists using post-convergence SPF. No bypass tunnels, no RSVP-TE FRR — just pre-installed segment lists in the FIB.

**SR-MPLS ↔ LDP**: During migration, both coexist. SR-prefer mode ensures SR labels are used when available. The Mapping Server bridges the gap for non-SR-capable nodes.

## Edge Cases & Gotchas

**1. SRGB mismatch across domain**: Mismatched SRGBs work (the IGP handles label computation per hop) but make troubleshooting painful. Standardize SRGB across all routers before deploying SR at scale.

**2. SID index collision**: Two routers advertising the same Prefix-SID index for different prefixes creates ambiguity. The IGP detects this (SID conflict) but behavior varies — some implementations prefer the lower Router ID, others reject both. Monitor for SID conflicts.

**3. Adj-SID and LAG**: On a LAG interface, an Adj-SID typically identifies the aggregate, not individual members. Traffic to the Adj-SID is hashed across LAG members. If you need per-member steering, use per-member Adj-SIDs (supported on some platforms).

**4. PHP and MPLS QoS**: Same as LDP — PHP pops the label at the penultimate hop, losing TC bits. Use no-PHP (P-flag) or Explicit NULL (E-flag) if MPLS QoS at the egress matters.

**5. Mapping Server is not in the forwarding path**: A common misconception. The Mapping Server only advertises SID bindings via IGP — it doesn't forward traffic. If the Mapping Server fails, existing bindings remain in the IGP (they age out normally). But new bindings for legacy routers won't be advertised until the Mapping Server returns.
