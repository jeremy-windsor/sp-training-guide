# 6.3 — TI-LFA — Protocol Theory

## Protocol Overview

TI-LFA (Topology-Independent Loop-Free Alternate) is the fast reroute mechanism for Segment Routing. It provides 100% node and link protection coverage for any network topology by pre-computing backup segment lists using post-convergence SPF. When a failure occurs, the protecting router instantly switches to the backup segment list — no signaling, no bypass tunnels, no RSVP-TE FRR.

TI-LFA was standardized in RFC 9855 (2025, building on earlier drafts). It solves the fundamental coverage limitation of classic LFA (RFC 5286) and Remote LFA (RFC 7490): those mechanisms could only protect when the topology happened to provide a loop-free alternate, achieving 40-95% coverage depending on topology. TI-LFA achieves 100% coverage by using SR segment lists to express any backup path that exists in the post-convergence topology.

The name "Topology-Independent" means the algorithm works regardless of network topology — rings, meshes, hubs, or any combination. If a backup path physically exists (the network isn't partitioned), TI-LFA will find it and encode it.

## Core Mechanisms

### 1. The TI-LFA Algorithm

TI-LFA computes backup paths by answering one question: "After this failure, what path would SPF compute?" Then it pre-installs that path as a segment list.

**Step-by-step:**

**Step 1 — Run pre-convergence SPF:**
Compute the normal SPF tree on the current topology. This gives the primary paths.

**Step 2 — For each protected element, run post-convergence SPF:**
Remove the protected link (or node) from the topology. Run SPF again. The result is the path the network would converge to after the failure.

**Step 3 — Compare pre and post-convergence paths:**
Find the first point where the post-convergence path diverges from the pre-convergence path. This is where the backup must steer traffic differently.

**Step 4 — Encode the backup as a segment list:**
Express the post-convergence path as the shortest possible sequence of SIDs (Prefix-SIDs and/or Adj-SIDs) that forces traffic onto the correct backup path.

**Step 5 — Install in FIB:**
Pre-install the backup segment list as the alternate next-hop in the FIB. When the primary next-hop fails (detected by BFD or interface down), traffic switches to the backup instantly — a data-plane operation, no control-plane involvement.

### 2. P-Space, Q-Space, and PQ Nodes

TI-LFA uses the P-space and Q-space concepts (originally from Remote LFA) to construct backup paths efficiently.

**Definitions (protecting router S, failed link S→E, destination D):**

**P-Space of S**: The set of all routers reachable from S without using the protected link S→E (or protected node E for node protection). Formally: router P is in P-space if the shortest path from S to P does not traverse S→E.

Intuitively: P-space is "where can S send traffic without going through the failure?"

**Extended P-Space**: Includes routers reachable from S's neighbors (other than E) without traversing S→E. This extends reach through neighbors that might have direct paths to distant routers.

**Q-Space of E (or D)**: The set of all routers from which D is reachable on the post-convergence shortest path without traversing the failed element. Formally: router Q is in Q-space if the post-convergence shortest path from Q to D does not traverse the failed link/node.

Intuitively: Q-space is "from where can traffic reach the destination without going through the failure?"

**PQ Node**: A router in the intersection of P-space and Q-space. Traffic can reach it from S without the failure, and it can reach D without the failure. A PQ node is an ideal repair point.

### 3. Segment List Construction

Based on P-space and Q-space analysis, TI-LFA constructs the backup segment list:

**Case 1 — Direct LFA exists (PQ node is a direct neighbor of S):**
- The neighbor is in both P-space and Q-space.
- Backup segment list: just the neighbor's Prefix-SID (or no segment needed — plain IP LFA).
- Simplest case. No additional labels required.

**Case 2 — PQ node exists but isn't directly connected:**
- A remote router is in both P-space and Q-space.
- Backup segment list: {Prefix-SID(PQ-node)}. Traffic is sent to the PQ node via SR, then follows normal forwarding to D.
- One additional label in the backup stack.

**Case 3 — No single PQ node exists:**
- No router is in both P-space and Q-space simultaneously.
- TI-LFA finds a P-node and a Q-node separately, then bridges them.
- Backup segment list: {Prefix-SID(P-node), Adj-SID(P→Q link) or Prefix-SID(Q-node)}.
- Two or more additional labels.

**Case 4 — Node protection requires path around the node:**
- For node protection, both the link and the node are excluded.
- The Q-space is computed relative to paths that avoid node E entirely.
- This may require longer segment lists if the topology has limited diversity around E.

### 4. Segment List Optimization

TI-LFA doesn't blindly encode every hop on the post-convergence path. It finds the minimum set of segments needed to force traffic onto the correct path.

**Optimization principle**: At each point in the backup path, check if normal IGP shortest-path forwarding (without segments) would follow the post-convergence path. If yes, no segment is needed. Only insert a segment where forwarding would diverge.

**Example**: Post-convergence path is S → A → B → C → D.
- From S, forwarding naturally goes to A (S's alternate next-hop). No segment needed.
- From A, forwarding naturally goes to B (A's shortest path to D avoids the failure). No segment needed.
- From B, forwarding naturally goes to C and then D. No segment needed.
- Result: zero extra segments. The backup is just "forward to A instead of E."

**More complex example**: Post-convergence path is S → A → B → C → D, but from A, IGP shortest path goes to X (not B).
- Need to force A→B: insert Prefix-SID(B) or Adj-SID(A→B).
- From B, forwarding naturally reaches D. Done.
- Result: one extra segment.

### 5. Protection Types

**Link Protection:**
- Protects against failure of the link S→E.
- Post-convergence SPF removes only the link S→E.
- Backup path may still transit node E (via a different link).
- Fastest computation, minimum segment depth.

**Node Protection:**
- Protects against failure of node E entirely.
- Post-convergence SPF removes node E and ALL its links.
- Backup path must go around E completely.
- Stronger protection, potentially deeper segment lists.
- Subsumes link protection (if E fails, all its links fail).

**SRLG Protection:**
- Protects against failure of all links in a Shared Risk Link Group.
- Post-convergence SPF removes all SRLG member links.
- Backup path avoids all shared-risk infrastructure.
- Strongest protection, may require the deepest segment lists.
- Requires accurate SRLG data in the IGP.

### 6. TI-LFA and Micro-Loop Prevention

During convergence (after failure, before all routers complete SPF), micro-loops can form between routers that have updated their FIB and those that haven't yet.

**TI-LFA's role**: The protecting router (adjacent to the failure) immediately uses the pre-installed backup segment list. Because this is an SR explicit path, traffic from S follows the post-convergence path regardless of what other routers are doing. This prevents micro-loops at the protecting router.

**Remaining exposure**: Other routers in the network (not adjacent to the failure) may still experience micro-loops during convergence. TI-LFA protects the PLR's traffic; it doesn't prevent all micro-loops globally. For global micro-loop prevention, additional mechanisms are needed (ordered FIB, microloop avoidance using segment lists at remote routers).

**Microloop Avoidance (RFC 8333)**: An extension where, during IGP convergence, all routers temporarily use SR segment lists to route traffic along post-convergence paths. This eliminates micro-loops network-wide, not just at the PLR. More complex to implement and deploy.

### 7. Computation Complexity and Scale

**Per-prefix, per-failure computation**: For each protected adjacency (link or node), TI-LFA must compute a backup for every prefix reachable through that adjacency. In a network with K adjacencies and N prefixes, the worst case is K × N backup computations.

**Optimization**: Most implementations share backup computations across prefixes with the same backup next-hop. If 1000 prefixes all use the same backup segment list (because they all go through the same PQ node), only one computation is needed.

**Incremental computation**: When a topology change occurs, only affected backups need recomputation — not all backups for all prefixes.

**Real-world performance**: On modern routing platforms, TI-LFA computation for a network with 500 nodes and 50,000 prefixes completes in milliseconds. The computation is done proactively (before failure) and is not in the failure detection → repair critical path.

## Protocol Interactions

**TI-LFA ↔ IGP**: TI-LFA uses the IGP LSDB to compute both primary and post-convergence SPF. IGP convergence speed determines how long the backup path is active before the network fully reconverges.

**TI-LFA ↔ BFD**: BFD provides the fast failure detection that triggers TI-LFA backup activation. BFD detects in ~150ms; TI-LFA switches in microseconds. Combined: sub-200ms total repair.

**TI-LFA ↔ SR-TE**: TI-LFA protects individual segments within SR-TE policy segment lists. If a segment (Prefix-SID hop) fails, TI-LFA's backup segment list handles the local repair while the overall SR-TE policy path is maintained.

**TI-LFA ↔ Flex-Algo**: TI-LFA can compute backups per Flex-Algo. A prefix with Algo 128 SID gets a backup computed on the Algo 128 topology. The backup respects the same constraints as the primary.

## Edge Cases & Gotchas

**1. No backup possible if network partitions**: TI-LFA guarantees 100% coverage only when a backup path physically exists. If removing the failed element partitions the network, no backup path exists. This is a topology limitation, not a TI-LFA limitation.

**2. Deep segment lists on sparse topologies**: In sparse topologies (few alternate paths), TI-LFA may require 3-5 segments for backup paths. Verify platform label stack depth limits.

**3. Asymmetric metrics and backup path selection**: If IGP metrics are asymmetric (different cost in each direction), the post-convergence path may traverse links with unexpected costs. TI-LFA computes correctly based on actual metrics, but the resulting path may surprise operators who assumed symmetric routing.

**4. Node protection may fail when link protection succeeds**: If all paths around node E are longer than the segment list depth limit, node protection may be infeasible while link protection (which only avoids the link, not the node) succeeds. Platforms typically fall back to link protection when node protection is impossible.

**5. SRLG data accuracy**: SRLG-protected TI-LFA is only as good as the SRLG data. Missing or incorrect SRLG assignments render SRLG protection ineffective. This is an operational data quality problem.
