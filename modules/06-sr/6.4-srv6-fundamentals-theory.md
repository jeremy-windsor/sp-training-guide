# 6.4 — SRv6 Fundamentals — Protocol Theory

## Protocol Overview

SRv6 (Segment Routing over IPv6) implements Segment Routing using the IPv6 data plane instead of MPLS. In SRv6, segments are represented as 128-bit IPv6 addresses rather than 20-bit MPLS labels. The segment list is carried in an IPv6 extension header called the Segment Routing Header (SRH), defined in RFC 8754.

The architectural shift from SR-MPLS to SRv6 is fundamental. SR-MPLS piggybacks on the existing MPLS data plane and requires MPLS-capable hardware at every hop. SRv6 uses native IPv6 forwarding — any IPv6-capable router can forward SRv6 packets, even if it doesn't understand the SRH (it just forwards based on the IPv6 destination address). Only routers that need to process segments must be SRv6-aware.

SRv6 was designed to unify networking and services on a single protocol plane. Instead of layering MPLS labels for transport, VPN labels for services, and additional encapsulations for service chaining, SRv6 encodes everything in IPv6 addresses and the SRH. A single IPv6 header can express: transport path, VPN context, service function, and network programming instruction — all in the same address space.

The trade-off is overhead. An MPLS label is 4 bytes. An SRv6 SID is 16 bytes. A segment list of 4 hops costs 16 bytes in SR-MPLS but 64 bytes (plus SRH overhead) in SRv6. This overhead concern drove the development of Micro-SID (uSID), covered in Section 6.5.

## Core Mechanisms

### 1. SRv6 SID Structure

An SRv6 SID is a 128-bit IPv6 address with internal structure:

```
 ┌──────────────────┬─────────────┬──────────────────┐
 │     Locator       │  Function   │     Arguments    │
 │   (variable)      │ (variable)  │   (variable)     │
 └──────────────────┴─────────────┴──────────────────┘
 |←────── Locator Block ─────────→|
```

**Locator**: An IPv6 prefix routable via the IGP. It identifies the node that owns the SID. The IGP advertises the locator as a reachable prefix; normal IPv6 forwarding delivers packets to the right node.

**Function**: Identifies the specific behavior to execute at the node. The function field is local to the node — the same function value can mean different things on different nodes.

**Arguments**: Optional parameters for the function. Can carry flow identifiers, VPN context, or other per-packet data.

**Example SID**: `fc00:0:1::100`
- Locator: `fc00:0:1::/48` (routed to Node 1)
- Function: `::100` (e.g., END.DT4 — decapsulate and IPv4 table lookup)

**Locator Block**: The common prefix shared by all locators in the SRv6 domain. Typically a /32 or /48 ULA or GUA prefix allocated for SRv6.

### 2. Segment Routing Header (SRH) — RFC 8754

The SRH is an IPv6 Routing Extension Header (type 4) that carries the segment list.

**SRH Format:**
```
├─ Next Header (1 byte): protocol after SRH
├─ Hdr Ext Len (1 byte): SRH length in 8-byte units (minus 1)
├─ Routing Type (1 byte): always 4 (Segment Routing)
├─ Segments Left (1 byte): index of current active segment
├─ Last Entry (1 byte): index of last segment (0-based)
├─ Flags (1 byte):
│   └─ Currently unused in SRv6 (reserved)
├─ Tag (2 bytes): packet classification/grouping
└─ Segment List[0..n] (16 bytes each): ordered from last to first
    ├─ Segment List[0] = final destination (last segment)
    ├─ Segment List[1] = second-to-last segment
    ├─ ...
    └─ Segment List[n] = first segment (current active)
```

**Segment processing at each SRv6 node:**
1. Packet arrives with IPv6 DA = local SID.
2. Router looks up the SID in its "My Local SID Table."
3. Executes the function associated with the SID.
4. If more segments remain: decrement Segments Left, copy Segment List[Segments Left] to IPv6 DA, forward.
5. If no more segments: execute the final function (decapsulate, lookup, deliver).

**Critical detail — segment list ordering**: The segment list is stored in REVERSE order. Segment List[0] is the final destination; Segment List[n] is the first segment. Segments Left starts at n and decrements. The IPv6 DA always contains the active segment — when Segments Left = k, IPv6 DA = Segment List[k].

**SRH may be absent**: If only one segment is needed (the destination itself), the SRH can be omitted entirely. The IPv6 DA is the SRv6 SID, and the destination node processes it directly. SRH is only required when multiple segments exist.

### 3. SRv6 Network Programming Functions (RFC 8986)

Each SRv6 SID has an associated function — the instruction executed when the packet reaches that node with that SID as the active segment.

**Transit functions** (processed by non-SRv6-aware routers):
- **T**: Transit. Normal IPv6 forwarding. Non-SRv6 routers forward based on DA without inspecting SRH. This is the "any IPv6 router can forward SRv6" property.

**Endpoint functions** (processed by SRv6-aware nodes):

| Function | Name | Behavior |
|----------|------|----------|
| End | Endpoint | Decrement Segments Left, update DA, forward. Basic SRv6 waypoint. |
| End.X | Endpoint with L3 cross-connect | Same as End but forward on a specific interface (like Adj-SID in SR-MPLS). |
| End.T | Endpoint with table lookup | Decrement SL, look up DA in a specific routing table. |
| End.DT4 | Endpoint with IPv4 decapsulation and table lookup | Pop SRv6 encapsulation, do IPv4 lookup. Used for L3VPN IPv4 service. |
| End.DT6 | Endpoint with IPv6 decapsulation and table lookup | Pop SRv6 encapsulation, do IPv6 lookup. Used for L3VPN IPv6 service. |
| End.DT46 | Endpoint with dual-stack decapsulation | Pop and look up in both IPv4 and IPv6 tables. |
| End.DX4 | Endpoint with IPv4 cross-connect | Pop SRv6, forward IPv4 packet to specific next-hop. L3VPN PE-CE. |
| End.DX6 | Endpoint with IPv6 cross-connect | Pop SRv6, forward IPv6 packet to specific next-hop. |
| End.DX2 | Endpoint with L2 cross-connect | Pop SRv6, forward L2 frame. L2VPN/EVPN service. |
| End.B6.Encaps | Endpoint with SRv6 encapsulation | Insert a new SRv6 encapsulation (binding SID behavior). |

**The "End" function family** replaces what MPLS does with different label stacks. Instead of transport label + VPN label + service label, SRv6 uses one SID per function, each encoding both the node and the action.

### 4. SRv6 Encapsulation Modes

SRv6 defines how the original packet is wrapped in the SRv6 IPv6 header:

**H.Encaps (Headend Encapsulation):**
- A new IPv6 header + SRH is prepended to the original packet.
- The original packet becomes the payload of the outer IPv6 packet.
- The outer IPv6 DA = first segment.
- Outer IPv6 SA = headend's address.

**H.Encaps.Red (Reduced Encapsulation):**
- Same as H.Encaps but the first segment is NOT in the SRH — it's only in the IPv6 DA. Saves 16 bytes.
- SRH Segments Left starts at n-1 instead of n.
- Optimization for reducing overhead when the first segment is always the DA.

**H.Insert (Headend Insertion):**
- An SRH is inserted into an existing IPv6 packet (no new outer header).
- The original IPv6 packet's DA is saved in the SRH, replaced with the first segment.
- Less overhead but modifies the original packet's header.
- Cannot be used for non-IPv6 payloads (IPv4, L2).

**H.Insert vs H.Encaps trade-off:**
- H.Insert: lower overhead (no outer IPv6 header), but only works for IPv6 payloads and modifies the original header.
- H.Encaps: higher overhead (40-byte outer IPv6 header), but works for any payload type and preserves the original packet intact.

### 5. IGP Extensions for SRv6

SRv6 locators and SIDs are advertised via IGP, similar to SR-MPLS.

**IS-IS Extensions:**
- **SRv6 Locator TLV (TLV 27)**: Advertises the SRv6 locator prefix and associated flags.
- **SRv6 End SID Sub-TLV**: Under TLV 27, advertises End function SIDs.
- **SRv6 SID Structure Sub-TLV**: Describes the locator/function/argument bit allocation.
- **SRv6 Adjacency SID**: Advertises End.X SIDs under Extended IS Reachability (TLV 22).

**OSPF Extensions:**
- SRv6 locators and SIDs advertised in Extended Prefix/Link Opaque LSAs, similar to SR-MPLS but with SRv6-specific sub-TLVs.

**BGP for SRv6 services:**
- BGP carries SRv6 SIDs in VPN routes using the SRv6 SID Information Sub-TLV within the Prefix-SID attribute.
- The BGP VPN route includes: the endpoint SID (e.g., End.DT4 SID) that the remote PE should use when decapsulating.

## Protocol Interactions

**SRv6 ↔ IGP**: Locators are advertised as IPv6 prefixes in the IGP. Normal IPv6 SPF provides reachability to SRv6 nodes. SRv6-specific TLVs carry function and SID structure information.

**SRv6 ↔ BGP**: BGP VPN routes carry SRv6 SIDs (End.DT4/DT6) as the service endpoint. The transport encapsulation uses H.Encaps with the SRv6 SID as the IPv6 DA. BGP doesn't need to know about MPLS — everything is IPv6.

**SRv6 ↔ IPv6 forwarding**: This is the key advantage. Any IPv6 router in the path can forward SRv6 packets based on the IPv6 DA without understanding SRv6 or SRH. Only segment endpoints need SRv6 awareness. This enables incremental deployment.

**SRv6 ↔ SR-MPLS**: Both can coexist. Interworking at domain boundaries uses gateway functions (encaps/decaps between MPLS and SRv6). Some operators run SR-MPLS in the core and SRv6 at the edge, or vice versa.

## Edge Cases & Gotchas

**1. 40-byte overhead per H.Encaps**: The outer IPv6 header adds 40 bytes minimum, plus SRH adds 8 + 16 bytes per segment. A 4-segment SRH over H.Encaps adds 40 + 8 + 64 = 112 bytes of overhead. MTU planning is critical.

**2. SRH and middleboxes**: Firewalls, load balancers, and DPI devices may not understand IPv6 extension headers. Some drop packets with extension headers entirely. This is a real deployment concern, especially at network boundaries.

**3. ECMP hashing with SRv6**: The IPv6 flow label (20 bits) and source/destination addresses provide entropy for ECMP. But some hardware only hashes on DA — which is the same for all packets in the same SRv6 tunnel. Verify platform ECMP behavior.

**4. Security of SRH**: An attacker injecting SRH into packets could redirect traffic. RFC 8754 Section 8 discusses security. In practice, SRH should be blocked at domain boundaries (ingress filtering). Internal SRv6 assumes trusted infrastructure.

**5. Locator prefix allocation**: SRv6 locators consume IPv6 address space. A /48 locator per node in a 500-node network needs 500 /48 prefixes. Plan address allocation carefully, especially if using GUA (globally unique) addresses.
