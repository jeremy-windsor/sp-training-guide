# 6.1 — SR-MPLS Fundamentals — Answer Key

These questions are IE-SP caliber — scenario-based, requiring you to apply multiple concepts.

## Question 1
> You're migrating a national backbone from LDP to SR-MPLS. The SRGB on your IOS-XR routers is 16000–23999 (default), but your Junos MX routers were deployed years ago with the old Junos default SRGB of 16000–17999. Both platforms have IS-IS as the IGP. What happens to SR-MPLS forwarding in this mixed SRGB environment, and what is your remediation plan?

### Answer

SR-MPLS forwarding will still work — but only because of how SID indexing operates. A Prefix-SID is advertised as an *index*, not an absolute label. Each router computes the local label as SRGB_base + index. If R1 (IOS-XR, SRGB 16000–23999) advertises Prefix-SID index 1, every router installs it as label 16001. If R5 (Junos, SRGB 16000–17999) receives this, it also computes 16000 + 1 = 16001. So far, identical.

The problem surfaces when SID indexes exceed the smaller SRGB. If you assign a Prefix-SID index of 2500 (label 18500 on the IOS-XR routers), the Junos routers can't accommodate it — their SRGB only goes to 17999. The Junos router will reject that SID advertisement. Traffic destined for that node will black-hole at any Junos hop because there's no label entry in the LFIB.

Remediation: expand the Junos SRGB to match IOS-XR. On Junos 18.1+, set `protocols source-packet-routing srgb start-label 16000 index-range 8000`. This is non-disruptive — Junos re-indexes existing SIDs into the expanded range. Verify with `show spring srgb`. Until the SRGB is aligned, keep all SID indexes below 2000 to stay within both ranges.

The real lesson: SRGB alignment should be in your Day 0 SR deployment template. Heterogeneous SRGBs are technically supported by the architecture (each router resolves indexes locally), but operationally they're a landmine. One engineer assigns index 3000, forgets about the Junos limitation, and now you're debugging intermittent black holes at 2 AM.

## Question 2
> A new P-router is added to your SR-MPLS backbone. IS-IS adjacencies form, routes are learned, but the router's Prefix-SID doesn't appear in any neighbor's LFIB. BGP VPN traffic transiting this router works fine via LDP (which is still running during migration). What are the most likely causes?

### Answer

The Prefix-SID not appearing in the LFIB while LDP works narrows this considerably. LDP doesn't care about SR configuration — it distributes labels independently. The SR-MPLS label isn't being installed, which means the problem is in the SR control plane, not basic connectivity.

**Cause 1: SR not enabled under the IGP instance.** On IOS-XR, you need `segment-routing mpls` under the IS-IS address-family. On Junos, you need `source-packet-routing` under the protocol. If SR isn't enabled in the IGP, the router won't advertise SR capabilities in its IS-IS Hello (IIH) packets, and neighbors won't treat it as an SR node. Check with `show isis protocol` (IOS-XR) or `show isis overview` (Junos) — look for "Segment Routing" in the output.

**Cause 2: No Prefix-SID assigned to the loopback.** Enabling SR under IS-IS is necessary but not sufficient. You must also assign an index to the specific prefix. On IOS-XR: `interface Loopback0 / address-family ipv4 unicast / prefix-sid index <N>`. On Junos: `protocols isis interface lo0.0 / node-segment ipv4-index <N>`. Without this, the router participates in SR (can forward SR-labeled packets) but doesn't advertise its own SID.

**Cause 3: SID index collision.** If the new router's Prefix-SID index matches an existing router's index, both advertisements exist in the IGP, but routers detect the conflict and may drop one or both. Check `show isis segment-routing prefix-sid-map` for conflicts. Each SID index must be unique network-wide.

**Cause 4: SRGB not configured or not active.** If the SRGB reservation failed (overlapping with static labels or dynamic ranges), the router can't install any SR labels. Verify with `show segment-routing local-block` and `show segment-routing global-block`.

Diagnostic sequence: verify SR is enabled in IGP → verify Prefix-SID is assigned → verify no index collision → verify SRGB is active → check neighbors' `show mpls forwarding` for the expected label.

## Question 3
> Explain the difference between PHP (Penultimate Hop Popping) and Explicit Null in SR-MPLS. Your backbone uses SR-MPLS with TI-LFA. A QoS engineer reports that DSCP markings are being lost on traffic entering PE routers. How is this related to PHP, and what's the fix?

### Answer

PHP means the second-to-last router pops the SR label before forwarding to the destination router. The destination receives a plain IP packet (or the next label in the stack). This is the default behavior for Prefix-SIDs — it reduces the load on the destination router by eliminating one label lookup.

Explicit Null means the penultimate router swaps the label to value 0 (IPv4 Explicit Null) or 2 (IPv6 Explicit Null) instead of popping it. The destination receives a packet with label 0/2, which it pops and processes. The label is meaningless for forwarding — the destination was going to process the IP packet anyway.

The QoS issue: when the penultimate router performs PHP and pops the MPLS label, the packet arrives at the PE as a plain IP packet. If the PE's ingress QoS policy was examining the MPLS EXP bits (Traffic Class field) to make classification decisions, those bits are gone — they were in the MPLS header that was just popped. The PE now sees IP DSCP, which may have different values than what was set in the MPLS TC field (especially if the SP uses a different DSCP-to-EXP mapping internally).

Fix: configure Explicit Null on the Prefix-SIDs. On IOS-XR: `segment-routing mpls / prefix-sid-map / address-family ipv4 / explicit-null`. On Junos: `protocols isis source-packet-routing explicit-null`. This preserves the MPLS header all the way to the PE, so the QoS policy can read the EXP bits. The trade-off is a marginal increase in PE processing (one extra label pop), which on modern hardware is negligible.

Alternative: reclassify based on IP DSCP at the PE ingress instead of MPLS EXP. This avoids the Explicit Null overhead but requires consistent DSCP preservation end-to-end, which is harder to guarantee across transit networks.

## Question 4
> You have a 50-router backbone running SR-MPLS with IS-IS. You need to steer traffic from PE-1 to PE-2 via a specific intermediate router R7 (bypassing the IGP shortest path). Describe how you'd accomplish this using only SR-MPLS label stacking — no PCE, no SR-TE policy, just raw label operations. What's in the label stack at PE-1's egress?

### Answer

This is source routing in its purest form — the whole reason SR exists.

PE-1 needs to push a label stack that forces the packet through R7 before reaching PE-2. The stack is built from bottom to top:

**Bottom label:** PE-2's Prefix-SID (say, 16002). This ensures that once the packet reaches R7, it follows the IGP shortest path from R7 to PE-2.

**Top label:** R7's Adjacency-SID from PE-1's perspective won't work here (Adj-SIDs are locally significant). Instead, use R7's Prefix-SID (say, 16007). This sends the packet to R7 via the IGP shortest path from PE-1 to R7.

So the label stack at PE-1's egress is: `{16007, 16002}` — top label is R7's Prefix-SID, bottom label is PE-2's Prefix-SID.

**Forwarding walk-through:**
1. PE-1 pushes `{16007, 16002}`, forwards toward R7 via IGP shortest path
2. Each transit router between PE-1 and R7 swaps the top label (16007) per its LFIB
3. The penultimate hop before R7 pops label 16007 (PHP), exposing label 16002
4. R7 receives the packet with label 16002, looks it up — it's PE-2's Prefix-SID — and forwards toward PE-2 via IGP shortest path
5. Each transit router between R7 and PE-2 swaps 16002
6. The penultimate hop before PE-2 pops 16002 (PHP), PE-2 receives the IP packet

If you need to force a *specific link* out of R7 (not just "get to R7"), you'd insert R7's Adj-SID for that link between the two Prefix-SIDs: `{16007, 24001, 16002}`. But Adj-SIDs are locally significant to R7, so you'd need to know R7's Adj-SID value for the desired outgoing interface.

This is exactly what SR-TE policies automate (Section 6.2) — but understanding the raw mechanics is critical for troubleshooting.

## Question 5
> During an SR-MPLS deployment, you notice that two routers (R3 and R8) have been accidentally assigned the same Prefix-SID index (index 3). Both are advertising this index via IS-IS. What happens to traffic in the network, and how does each vendor handle SID conflicts?

### Answer

This is one of the most dangerous misconfigurations in SR-MPLS because the failure mode is subtle. Both R3 and R8 advertise "Prefix-SID index 3" in their IS-IS LSPs, but for different prefixes (R3's loopback and R8's loopback). Every other router in the network now has a conflict: label 16003 (SRGB base + 3) maps to two different destinations.

**What happens:** The behavior depends on the implementation, but the general result is traffic black-holing or misrouting for one or both prefixes.

**IOS-XR behavior:** IOS-XR detects the conflict and installs neither SID. Both R3's and R8's Prefix-SIDs are withdrawn from the LFIB. Traffic that was using SR labels to reach either router falls back to LDP (if LDP is still running) or gets dropped. The router logs a conflict message: `%ROUTING-ISIS-4-SR_SID_CONFLICT`. You can see conflicts with `show isis segment-routing prefix-sid-conflicts`.

**Junos behavior:** Junos also detects the conflict. By default, it prefers the SID associated with the lower router-ID (the "winning" advertisement). The other prefix loses its SID. Traffic to the "losing" router's prefix will not have an SR label path. Check with `show isis spring prefix-sid` and look for conflict flags.

**The real danger:** If LDP is disabled (post-migration), there's no fallback. SR label paths to both conflicting prefixes are broken. BGP VPN traffic using those PEs as next-hops will black-hole. This is why SID index planning should be centralized and documented — treat SID indexes like IP addresses, not like something you make up on the fly.

**Resolution:** Change one router's SID index to a unique value. The IGP re-advertises, all routers recompute, and the LFIB converges within one SPF cycle (sub-second with fast IS-IS timers). Verify with `show mpls forwarding` on multiple routers to confirm both labels are now installed correctly.

**Prevention:** Maintain a SID index allocation spreadsheet or use a controller (PCE/NSO) to manage assignments. Some operators derive the index from the loopback IP (e.g., 10.0.0.X → index X) to avoid collisions by construction.
