# 6.5 — SRv6 Network Programming — Protocol Theory

## Protocol Overview

SRv6 Network Programming is the idea that a packet can carry not just a destination, but a sequence of behaviors to be executed by the network. In SR-MPLS, labels mostly identify forwarding instructions: go to this node, cross this link, enter this policy. In SRv6, a 128-bit SID can encode the node, the function, and optional arguments — turning the network into a distributed execution environment.

This idea is formalized in RFC 8986. The core claim is aggressive and kind of beautiful: routers are no longer just forwarding devices; they are programmable endpoints executing a chain of instructions carried in the packet itself. The SRv6 SID is both an IPv6 address and a program pointer.

SRv6 Network Programming extends simple transport steering into services:
- decapsulate into a VRF
- cross-connect to a CE
- insert a new SRH
- chain to a firewall / service function
- perform policy-specific lookup in a table
- compress multiple segments with uSID

This is where SRv6 stops being "MPLS with bigger labels" and becomes its own thing.

## Core Mechanisms

### 1. The My Local SID Table

Every SRv6-capable router maintains a **My Local SID Table**.

This is the local dispatch table mapping an incoming active SID to a behavior.

**Conceptually:**
- Packet arrives with IPv6 DA = local SRv6 SID
- Router looks up DA in My Local SID Table
- Matching entry says: execute behavior X with arguments Y
- Behavior updates SRH/DA, decapsulates, cross-connects, or forwards

**A My Local SID entry contains:**
- SID / SID block match
- Behavior (End, End.X, End.DT4, etc.)
- Optional VRF/table binding
- Optional next-hop/interface binding
- Flavor flags (PSP, USP, USD)
- Counters/state for monitoring

This table is the heart of SRv6 Network Programming. Without it, the SRH is just a fancy address list.

### 2. Network Programming Behaviors (RFC 8986)

RFC 8986 defines the standard behaviors. The important thing is not memorizing the names like a parrot — it's understanding the classes of actions they represent.

#### Endpoint Behaviors

These act on the segment list itself.

**End**
- Basic endpoint behavior
- If Segments Left > 0: decrement it, copy next SID into IPv6 DA, forward
- If Segments Left = 0: packet has reached final segment; continue processing as local delivery

This is the SRv6 equivalent of "node SID waypoint."

**End.X**
- Endpoint with Layer 3 cross-connect
- Like End, but instead of normal routing lookup, forward to a specific next-hop/interface
- SRv6 equivalent of an adjacency SID in SR-MPLS

#### Decapsulation Behaviors

These remove the SRv6 encapsulation and hand the inner payload to a service context.

**End.DT4**
- Decapsulate and perform IPv4 lookup in a specific table/VRF
- Main SRv6 behavior for IPv4 L3VPN services

**End.DT6**
- Decapsulate and perform IPv6 lookup in a specific table/VRF
- Main SRv6 behavior for IPv6 L3VPN services

**End.DT46**
- Decapsulate and perform dual-stack lookup (IPv4/IPv6) in a table
- Useful when the VRF contains both AFs

**End.DX4 / End.DX6**
- Decapsulate and cross-connect to a specific IPv4/IPv6 next-hop
- Often used for PE-CE service handoff

**End.DX2**
- Decapsulate and forward Layer 2 frame
- Used for EVPN/L2VPN services

#### Encapsulation / Binding Behaviors

These create a new SRv6 encapsulation from an existing packet.

**End.B6**
- Insert a new SRH into the packet
- Equivalent of pushing a new segment list

**End.B6.Encaps**
- Create a fresh outer IPv6 header + SRH around the packet
- Equivalent of an SRv6 binding SID that redirects traffic into another SR policy

These are what enable policy chaining and inter-domain steering.

### 3. Flavors: PSP, USP, USD

Flavors modify how endpoint behaviors treat the SRH and active segment at the end of a segment list.

These matter because the base End behavior is clean but not always optimal for overhead or service handoff.

#### PSP — Penultimate Segment Pop

PSP removes the SRH at the penultimate segment, before the final endpoint receives the packet.

**Why it exists:**
- Avoids making the final endpoint process a now-unnecessary SRH
- Saves a bit of overhead and parsing work
- Conceptually similar to PHP in MPLS

**When used:**
- Final endpoint doesn't need to inspect the SRH
- Simple transport or service destination

#### USP — Ultimate Segment Pop

USP keeps the SRH until the final segment, then the final endpoint removes it during local processing.

**Why it exists:**
- Final endpoint may need SRH context
- Cleaner semantics for some service behaviors

#### USD — Ultimate Segment Decapsulation

USD causes the final segment to remove the outer IPv6 + SRH and directly expose the inner payload.

**Why it matters:**
- Natural fit for service endpoints like End.DT4 / End.DT6
- Avoids extra parse/lookup steps after final segment processing

**SR-MPLS analogy:**
- PSP ≈ PHP behavior
- USP/USD ≈ explicit pop at the final service endpoint

### 4. uSID (Micro-SID) Compression

SRv6's biggest criticism is header overhead. A segment list of 5 SIDs is 80 bytes just for the SIDs, plus SRH and outer IPv6 header.

uSID compresses multiple micro-segments into one 128-bit SID container.

#### Core Idea

Instead of each SID being a full 128-bit address with locator+function, define a common locator block and pack multiple small function values into the remaining bits.

**Example conceptual model:**
- Locator block: `fcbb:0:0:1::/64`
- Remaining 64 bits divided into 16-bit micro-segments
- A single 128-bit address can carry 4 micro-segments after the locator

So instead of:
- SID1 = full 128-bit
- SID2 = full 128-bit
- SID3 = full 128-bit
- SID4 = full 128-bit

You can encode all four micro-instructions into one 128-bit container under the same locator block.

#### uSID Processing

At each uSID-capable node:
1. Inspect active DA
2. Consume the leftmost micro-segment
3. Shift the remaining micro-segments into place
4. Forward based on the next micro-segment

This drastically reduces SRH size and makes SRv6 practical for deep policy chains.

#### Why uSID matters

Without compression, SRv6 can become MTU-hostile fast.
With uSID, a complex policy that would need 4-6 full SIDs may fit in 1-2 containers.

This is why uSID is the thing that took SRv6 from "interesting academic overkill" to "actually deployable in a backbone without filing for a jumbo-frame religion." 

### 5. H.Encaps vs H.Insert in Programming Models

SRv6 network programming can manipulate packets in two main ways at the head-end or binding endpoint:

**H.Encaps**
- Add a new outer IPv6 header and optional SRH around the original packet
- Works for IPv4, IPv6, L2 payloads
- Preferred for service insertion and VPN use cases
- More overhead, cleaner semantics

**H.Insert**
- Insert an SRH into an existing IPv6 packet
- No new outer header
- Lower overhead
- Only valid for IPv6 payloads
- More intrusive — mutates the original packet header

**Why this matters operationally:**
- VPN and service edge cases usually use H.Encaps
- Native IPv6 service steering may use H.Insert or H.Insert.Red for efficiency

### 6. Service Chaining with SRv6

Service chaining means steering traffic through a sequence of network functions — firewall, IDS, NAT, proxy, optimizer, etc.

SRv6 can encode this chain directly in the segment list.

**Example conceptual chain:**
1. End at firewall function
2. End at IDS function
3. End at NAT function
4. End.DT4 at remote PE VRF

Each SID is not just "visit router X" — it's "execute service function X at router Y."

This is much richer than MPLS TE, where service chaining typically needs additional overlays (NSH, service-aware PBR, etc.).

**Advantages:**
- Service graph encoded directly in packet
- No per-flow state required in the core
- Policy can be distributed via BGP color or controller

**Trade-offs:**
- Header growth
- Service nodes must be SRv6-aware
- Troubleshooting becomes more like debugging a packet-carried program than following a path

### 7. SRv6 and EVPN Integration

SRv6 can replace MPLS labels as the transport/service identifier for EVPN and VPN services.

**In EVPN/L2VPN context:**
- BGP EVPN route carries an SRv6 SID instead of MPLS label stack semantics
- Remote PE encapsulates toward that SID
- Receiving PE executes End.DX2 / End.DT* behavior to deliver into the bridge-domain or VRF

**Why this matters:**
- Unified transport and service plane using IPv6
- No MPLS dependency
- Cleaner multi-service architecture if the network is already IPv6-native

This is one of SRv6's strongest strategic arguments: the same mechanism handles underlay, TE, L3VPN, L2VPN, and service chaining.

### 8. Locator Design and Function Allocation

SRv6 success or failure often comes down to address planning.

**Typical design dimensions:**
- Locator block per domain/region
- Locator per node or per service node
- Function ranges per service type
  - transport End functions
  - VPN End.DT functions
  - service chaining functions
  - binding behaviors
- Argument bits reserved for tenant/VRF/service metadata

**Bad design symptoms:**
- inconsistent locator lengths
- overlapping function spaces
- inability to summarize locators in IGP/BGP
- no room for future uSID compression model

SRv6 addressing is not "just IPv6 addressing." It is API design for the network.

## Protocol Interactions

**SRv6 Network Programming ↔ BGP**: BGP distributes service routes that carry SRv6 endpoint SIDs. The control plane tells remote PEs which function SID to target for a given VPN/EVPN destination.

**SRv6 Network Programming ↔ IGP**: IGP distributes locators and reachability. The IGP usually does not carry all service functions — it carries the locator structure, while specific service SIDs may be bound locally or signaled via BGP.

**SRv6 Network Programming ↔ Service Plane**: This is where SRv6 is unique. The segment list is not just a path — it can be a service graph. Firewalls, NATs, VRFs, L2 cross-connects, and policy chains all become addressable functions.

**uSID ↔ Hardware forwarding**: uSID requires hardware/software support for micro-segment parsing and shifting. Not every platform does this equally well. Compression is powerful, but implementation maturity matters.

## Edge Cases & Gotchas

**1. uSID is not universal yet**: It is strategically important, but platform and vendor support maturity varies. Don't design like every box in the network is already fluent in micro-segment sorcery.

**2. My Local SID table scale**: Every service SID consumes state. Large EVPN/VPN or service-chaining deployments can make the local SID table large quickly. This is a real scaling dimension, separate from the FIB.

**3. Flavors can break expectations**: PSP/USP/USD alter where and when headers are popped. If your service function expects to see an SRH and PSP stripped it earlier, your mental model just got mugged.

**4. Locator planning mistakes become architectural debt**: SRv6 addressing is hard to change later. A sloppy locator/function plan becomes permanent pain.

**5. Service chaining can create opaque troubleshooting paths**: Once packets carry service graphs in their headers, failure domains span routing, address planning, service programming, and policy logic all at once.
