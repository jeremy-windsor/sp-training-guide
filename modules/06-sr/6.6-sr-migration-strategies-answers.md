# 6.6 — SR Migration Strategies — Answer Key

## Question 1
> You've enabled SR-MPLS alongside LDP on all routers and configured `sr-prefer`. One VPN customer reports intermittent packet loss. `show cef` on the ingress PE shows the BGP next-hop resolved via SR. The problem PE's prefix-SID index is 500, SRGB is 16000-23999. What's your first check?

### Answer

**First check: verify the SR label is correct and consistent across the path.**

The expected label for prefix-SID index 500 with SRGB base 16000 is:

\[
16000 + 500 = 16500
\]

Your first check is whether **every router along the path** has:
1. The same SRGB (16000-23999)
2. Label 16500 correctly mapped to the egress PE's loopback
3. No label collision with the dynamic label range

```
IOS-XR (on each hop):
  show segment-routing mpls connected-prefix-sid-map
  ! Verify index 500 → label 16500 → correct egress PE loopback

  show mpls forwarding labels 16500
  ! Is the label installed? Does it point to the correct next-hop?

  show segment-routing mpls label-table
  ! Is 16500 in the SRGB range? Any conflict?
```

**Why "intermittent" is the key word:**

If the label were simply wrong or missing, traffic would be permanently broken, not intermittent. Intermittent loss with `sr-prefer` active suggests:

1. **ECMP path inconsistency** — some ECMP paths have correct SR labels, others don't. Traffic hashed to the working paths succeeds; traffic hashed to the broken path fails.

2. **SRGB mismatch with incorrect label programming** — heterogeneous SRGBs are supported (RFC 8402), but upstream nodes must impose/swap the label value expected by each next hop for the advertised SID index. If the control plane fails to program correct per-hop labels (e.g., stale SRGB advertisements, missing IGP updates), traffic transiting that P router gets misrouted or dropped. Best practice: standardize SRGBs across the domain for operational simplicity.

3. **SR/LDP label interaction during convergence** — `sr-prefer` means "use SR when available, fall back to LDP." During topology changes, there may be transient moments where some hops use SR labels and others still have LDP labels in their FIB. Label mismatch mid-path = packet drop.

**The answer is: verify SRGB consistency across every hop in the path.** That's where "intermittent" lives.

## Question 2
> A customer has 500 RSVP-TE tunnels with bandwidth reservations averaging 2 Gbps each. They want to migrate to SR-TE. How do you replicate the bandwidth guarantee behavior without RSVP RESV messages?

### Answer

You can't. Not natively.

SR-TE has **no built-in bandwidth reservation mechanism**. RSVP-TE's RESV message is what installs bandwidth reservation state on every transit router. SR-TE is stateless in transit — P routers don't know anything about the policies traversing them, let alone how much bandwidth each one "reserved."

So the question is really: **how do you approximate bandwidth-aware TE without RSVP admission control?**

### Approach 1: PCE with bandwidth-aware path computation

Deploy a PCE (Path Computation Element) that:
1. Collects real-time link utilization via **streaming telemetry** or **BGP-LS with bandwidth extensions**
2. Maintains a virtual bandwidth accounting model (how much each SR-TE policy "claims")
3. Computes paths that fit within available bandwidth — but **doesn't enforce it at the router level**

This is the closest replacement. The PCE acts as a centralized admission controller:
- When a new 2 Gbps policy is requested, PCE checks if the computed path has 2 Gbps available
- If yes, PCE installs the policy and updates its internal accounting
- If no, PCE either finds an alternate path or rejects the request

**Critical difference from RSVP:** This is **advisory, not enforced**. If traffic exceeds the "reserved" bandwidth, the router doesn't care — it forwards everything. Oversubscription is possible and invisible at the data plane.

### Approach 2: Interface QoS (the enforcement layer)

Map SR-TE traffic classes to QoS queues with guaranteed bandwidth:

```
IOS-XR:
  policy-map TE-GUARANTEE
   class GOLD-TE
    bandwidth percent 40
    priority
   class SILVER-TE
    bandwidth percent 30
   class class-default
    bandwidth remaining percent 100

  interface HundredGigE0/0/0
   service-policy output TE-GUARANTEE
```

This ensures that when congestion occurs, the "guaranteed" traffic classes get their bandwidth. It's not per-tunnel — it's per-class — but it provides the enforcement that SR-TE alone lacks.

### Approach 3: Telemetry + reactive rerouting

1. Deploy streaming telemetry on all core links
2. PCE monitors utilization in real-time
3. When a link approaches capacity, PCE proactively re-routes lower-priority SR-TE policies to alternate paths
4. This is the modern "intent-based" approach — instead of reserving bandwidth upfront, you monitor and react

### Recommended combined approach

For migrating 500 tunnels at 2 Gbps average:

1. **PCE with bandwidth modeling** — replaces RSVP admission control logically
2. **Interface QoS** — provides actual enforcement at congestion points
3. **Telemetry-driven reoptimization** — keeps utilization balanced
4. **Capacity planning** — if the network has headroom (50% utilization under normal), the lack of hard reservations is academic

**Reality check:** Most SP networks that migrated from RSVP-TE to SR-TE found that bandwidth reservations were more theoretical than practical anyway. Auto-bandwidth was constantly adjusting, reservations were stale, and the real protection came from capacity planning + QoS, not from RSVP RESV messages. SR-TE just makes the implicit reality explicit.

## Question 3
> After deploying SR Mapping Server for 10 LDP-only P-routers, 3 of them still don't have SR reachability from the SR domain. The Mapping Server config looks correct. What are the three most likely causes?

### Answer

SR Mapping Server (SRMS) creates prefix-SID mappings for non-SR nodes so that the SR domain can reach them using SR labels. If 7 out of 10 LDP-only routers are reachable via SR but 3 aren't, the SRMS itself is probably working — the issue is specific to those 3 nodes.

**1. The 3 routers' loopback prefixes aren't in the Mapping Server's prefix-to-SID map.**

SRMS maps specific prefixes to SID indices. If the 3 problem routers have loopbacks that don't match any SRMS entry (wrong /32 prefix, different address than expected, or simply missing from the map), they won't get SID mappings.

```
IOS-XR (on SRMS):
  show segment-routing mapping-server prefix-sid-map
  ! Are the 3 routers' loopbacks listed?
  ! Are the prefixes exact matches (/32)?

Junos (on SRMS):
  show isis source-packet-routing mapping-server entries
```

Verify: does the SRMS config include entries for 10.0.0.11/32, 10.0.0.12/32, and 10.0.0.13/32 (or whatever the 3 problem routers' loopbacks are)?

**2. The SRMS advertisements aren't reaching all SR-capable routers.**

SRMS distributes its mappings via IS-IS or OSPF TLVs. If the 3 problem prefixes are in a different IS-IS level or OSPF area than the SRMS, the mapping advertisements may not be flooding to all SR routers.

Check:
```
IOS-XR (on SR-capable routers):
  show isis database detail | include mapping-server
  show segment-routing mapping-server prefix-sid-map received
  ! Do the 3 problem prefixes appear in received mappings?
```

If the SR routers don't see the mapping for those 3 prefixes, it's a flooding/level/area boundary issue. The SRMS must be reachable and its TLVs must flood to all domains that need them.

**3. SID index conflict — the 3 prefixes have SID indices that collide with existing prefix-SIDs.**

If the SRMS assigns SID index 300 to router A's loopback, but an SR-capable router already has index 300 assigned to a different prefix, the conflict causes one or both to be rejected.

```
IOS-XR:
  show segment-routing mpls connected-prefix-sid-map
  ! Look for conflicts: two different prefixes with the same index
  
  show isis database detail | include SID
  ! Search for duplicate SID index values
```

SID index conflicts are silent killers. The SRMS config "looks correct" but the indices overlap with locally configured prefix-SIDs on SR-capable nodes. The 7 working routers got unique indices; the 3 broken ones hit collisions.

**Diagnostic priority:**
1. Verify the 3 prefixes are in the SRMS map (most common miss)
2. Verify flooding reaches all SR routers
3. Check for SID index collisions

Fix the missing/conflicting entries and the 3 routers should become SR-reachable.
