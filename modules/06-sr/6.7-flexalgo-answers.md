# 6.7 — Flexible Algorithm (FlexAlgo) — Answer Key

These questions are IE-SP caliber — scenario-based, requiring you to apply multiple concepts.

---

## Question 1
> Your backbone has 6 P-nodes. You enable FlexAlgo 128 (delay metric) on 4 of them but forget the other 2. What happens to traffic steered to Algorithm 128 that needs to transit the non-participating nodes?

### Answer

The two non-participating P-nodes do not advertise Algorithm 128 Prefix-SIDs (SR-MPLS) or Algorithm 128 SRv6 locators. From the perspective of the Algorithm 128 SPF computation, those nodes don't exist — they have no SIDs in that algorithm, so the IGP cannot build forwarding entries through them for Algorithm 128 traffic.

**What happens depends on topology:**

**Case 1 — Alternate path exists through participating nodes.** If the 4 participating P-nodes provide a connected path between every pair of PEs, the Algorithm 128 SPF tree simply routes around the 2 non-participating nodes. Traffic takes a longer path (higher cumulative delay), but it works. You get suboptimal forwarding — the whole point of delay-based FlexAlgo is to minimize latency, but you've forced traffic onto detour paths.

**Case 2 — Non-participating nodes are in the only path.** If any PE pair can only be reached by transiting a non-participating node, the Algorithm 128 SPF tree has no path to that destination. The result is a reachability black hole — traffic steered to Algorithm 128 SIDs destined for those PEs is dropped. Algorithm 0 traffic (default IGP) is unaffected because all 6 P-nodes participate in Algorithm 0.

**How to detect this:**

On IOS-XR:
```
show isis topology flex-algo 128
```
On Junos:
```
show isis spf flex-algorithm 128
```

Compare the reachable destinations in Algorithm 128 vs Algorithm 0. Any destination present in Algorithm 0 but absent from Algorithm 128 is a black hole for FlexAlgo traffic.

**The fix is simple but the lesson is critical:** every transit node in any algorithm's forwarding path must participate in that algorithm. This includes "dumb" P-nodes that you might think don't need FlexAlgo configuration because they "just forward." In SR-MPLS, they need a Prefix-SID index for Algorithm 128. In SRv6, they need a locator bound to Algorithm 128. No SID, no participation, no forwarding.

**Operational best practice:** Before enabling FlexAlgo steering on any VRF, run the topology check on every PE and verify full reachability within the algorithm. Automate this as a pre-deployment gate. A single missing P-node creates a silent, partial black hole that's extremely difficult to troubleshoot from the customer's perspective — some destinations work, others don't, and the pattern depends on which path the algorithm computes.

## Question 2
> Two teams independently configure FlexAlgo 128 with different definitions on different routers. What happens when both definitions flood through IS-IS? How does the IGP resolve the conflict?

### Answer

Both definitions flood through IS-IS as FAD (Flexible Algorithm Definition) sub-TLVs inside each router's LSP. Every router in the domain receives both definitions for Algorithm 128. The conflict resolution mechanism is defined in RFC 9350, Section 5.3 — it's deterministic, not random:

**Step 1 — Highest priority wins.** Each FAD advertisement includes a priority value (0–255). The definition with the higher priority is selected. If Team A's router advertises Algorithm 128 with priority 200 and Team B's router advertises it with priority 100, Team A's definition wins network-wide.

**Step 2 — If priorities are equal, highest System-ID / Router-ID wins.** This is the tiebreaker. The router with the numerically highest IS-IS System-ID (or OSPF Router-ID) wins the election. This is deterministic but arbitrary — it's not based on which definition is "better," just which router has a higher ID.

**The danger is subtle.** The resolution is deterministic, so every router in the domain agrees on which definition to use — there's no split-brain. But the losing definition is silently discarded. If Team B configured Algorithm 128 as "exclude submarine links" and Team A's "low latency" definition wins, Team B's submarine link avoidance simply doesn't happen. Traffic they assumed was avoiding submarine cables is, in fact, using them. No alarm, no error message, no log entry on most platforms.

**What makes this operationally dangerous:**

On IOS-XR, check which definition won:
```
show isis flex-algo 128
```
Look for the "source" field — it shows which router's definition is in use. If it's not the router you expected, you have a conflict.

On Junos:
```
show isis flex-algorithm 128 detail
```
The output shows the elected definition source and priority.

**Prevention:**
- Maintain a centralized algorithm registry mapping IDs to definitions, owners, and priority values
- Use only 2–3 designated routers (often route reflectors) to advertise definitions, configured with `advertise-definition`
- Set the priority explicitly on the advertising routers rather than relying on defaults (IOS-XR default priority is 128)
- Include algorithm ID ranges in your change management process — no ad-hoc allocation
- Monitor for multiple FAD sources per algorithm as a NOC alert condition

**Edge case:** If the winning definition router goes down, the remaining router's definition takes over. If the definitions are different, the algorithm's behavior silently changes when the primary advertiser fails. This can cause path shifts that look like a convergence issue but are actually a definition change. Always ensure all definition-advertising routers for the same algorithm have identical configurations.

## Question 3
> You're running delay-based FlexAlgo 128. During a traffic spike, queuing delay on several links increases by 5ms. The SPF tree for Algorithm 128 recomputes and shifts traffic to alternate paths, which then also become congested. Describe this feedback loop and how you'd prevent it.

### Answer

This is a classic oscillation problem — the networking equivalent of the Braess paradox. Here's the full feedback loop:

**The cycle:**

1. **Traffic spike** → queuing delay on links A, B, C increases by 5ms
2. **Delay advertisement update** → routers detect the delay change and flood updated delay TLVs via IS-IS
3. **SPF recomputation** → Algorithm 128 SPF runs with new delay values, computes different shortest paths that avoid links A, B, C
4. **Traffic shift** → all Algorithm 128 traffic moves to alternate links D, E, F
5. **New congestion** → links D, E, F are now overloaded, their queuing delay increases
6. **Second advertisement update** → new delay values flood for D, E, F
7. **Second SPF recomputation** → Algorithm 128 shifts traffic back toward A, B, C (which are now uncongested)
8. **Repeat** → the system oscillates indefinitely, never reaching steady state

Each cycle takes IGP-convergence-time (typically 1–5 seconds with fast timers), so you get traffic sloshing back and forth multiple times per minute. The result is worse than the original congestion — every path is intermittently overloaded, jitter goes through the roof, and every delay-sensitive application suffers.

**Prevention — layered approach:**

**Layer 1: Delay advertisement dampening.** Don't advertise every delay change. Configure a minimum change threshold — only update the advertised delay if it changes by more than X ms. On IOS-XR:
```
performance-measurement
 interface TenGigE0/0/0/0
  delay-measurement
   advertise-delay 1000
   threshold 3000
  !
 !
!
```
The `threshold` value (in microseconds) means delay changes smaller than 3ms are not advertised. This filters out queuing-induced noise while still tracking genuine topology changes (fiber cuts, route changes).

**Layer 2: SPF throttling per algorithm.** Configure aggressive SPF backoff timers for FlexAlgo. On IOS-XR:
```
router isis CORE
 address-family ipv4 unicast
  spf-interval initial-wait 50 secondary-wait 200 maximum-wait 5000
 !
!
```
The exponential backoff means rapid successive SPF triggers get progressively delayed. After the first recomputation, the next one waits 200ms, then 400ms, then 800ms, up to 5 seconds. This slows the oscillation cycle enough for the network to settle.

**Layer 3: Use static delay offsets instead of live measurements for FlexAlgo.** This is the nuclear option but the most stable. Instead of measuring real-time delay, manually configure the delay metric on each link based on geographic distance (approximately 5µs per km of fiber). Queuing delay is excluded entirely — the delay metric represents propagation delay only, which is constant. On IOS-XR:
```
performance-measurement
 interface TenGigE0/0/0/0
  delay-measurement
   advertise-delay 1000
   override 4500
  !
 !
!
```

**Layer 4: ECMP + weighted load balancing.** If the Algorithm 128 topology has multiple near-equal-delay paths, enable ECMP so traffic spreads across them rather than concentrating on a single "best" path. This reduces the severity of the congestion that triggers the feedback loop in the first place. Configure UCMP (Unequal-Cost Multi-Path) with a delay variance threshold to include paths within, say, 2ms of the best.

**The real-world answer:** Most production delay-based FlexAlgo deployments use static propagation delay (Layer 3) for exactly this reason. Live delay measurements are useful for detecting link degradation (fiber bends, optical issues), but queuing delay makes them too volatile for stable SPF computation. Use live measurements for monitoring and alarming, but base FlexAlgo path computation on stable propagation delay values.

## Question 4
> Explain why FlexAlgo + TI-LFA provides better protection than RSVP-TE FRR for constraint-based topologies. What's the failure mode when the constrained topology is too sparse for TI-LFA?

### Answer

**Why FlexAlgo + TI-LFA is superior:**

RSVP-TE FRR computes backup paths using a completely separate mechanism from the primary path constraints. When you build an RSVP-TE tunnel with affinity constraints (e.g., "exclude RED links"), the FRR backup path (facility backup or one-to-one backup) does not honor those constraints. The backup is computed by the PLR (Point of Local Repair) using standard IGP shortest path — it finds any path around the failure, regardless of link colors, TE metrics, or administrative constraints.

This means that during the ~50ms FRR switchover, your traffic that was carefully engineered to avoid submarine cables is suddenly riding a submarine cable. Your latency-optimized path suddenly detours through a high-latency link. The constraints you defined for the primary path are violated the moment a link fails.

**FlexAlgo + TI-LFA solves this by design.** TI-LFA computes backup paths per algorithm. When TI-LFA runs for Algorithm 128 (delay metric), it computes the post-convergence shortest path within the Algorithm 128 topology — using delay as the metric and honoring all algorithm constraints. When TI-LFA runs for Algorithm 129 (exclude RED), the backup path also excludes RED links.

The backup path is pre-computed and installed in the FIB before any failure occurs. When a link goes down, the PLR switches to the backup path in <50ms. The backup path respects all constraints of the algorithm. There is no window where constraints are violated.

**Implementation detail (IOS-XR):**
```
router isis CORE
 interface TenGigE0/0/0/0
  address-family ipv4 unicast
   fast-reroute per-prefix
   fast-reroute per-prefix ti-lfa
  !
 !
!
```
TI-LFA automatically applies to all algorithms — no per-algorithm TI-LFA configuration is needed. The IGP computes backup paths for Algorithm 0, Algorithm 128, Algorithm 129, etc., all independently.

**On Junos:**
```
set protocols isis backup-spf-options per-prefix-calculation
set protocols isis interface ge-0/0/0 level 2 post-convergence-lfa
```
Again, TI-LFA operates per algorithm automatically.

**The failure mode — sparse topologies:**

When the constrained topology has too few links, TI-LFA cannot find a loop-free backup path within the algorithm's constraints. Consider Algorithm 129 that excludes RED links. If the non-RED topology forms a tree (no redundancy), any single link failure partitions the Algorithm 129 topology. TI-LFA has no alternate path to compute.

**What happens:**
1. Link fails within the Algorithm 129 topology
2. TI-LFA has no pre-computed backup for Algorithm 129 (no loop-free alternate exists within the constraint set)
3. Traffic steered to Algorithm 129 is **dropped** until the IGP reconverges
4. After IGP convergence, the SPF for Algorithm 129 recomputes — if the destination is still reachable via remaining links, traffic resumes. If the failure partitioned the topology, the destination is unreachable via Algorithm 129 entirely.

During the IGP convergence window (typically 1–5 seconds with sub-second timers), Algorithm 129 traffic black-holes. Meanwhile, Algorithm 0 traffic (which has a richer topology) is protected by TI-LFA and converges in <50ms. The constrained algorithm suffers more downtime than the unconstrained one.

**Mitigation strategies:**

1. **Design for redundancy within each algorithm.** Every algorithm's topology should have at least 2-connected paths between any PE pair. If you're excluding links, make sure what remains is still redundant. Run `show isis topology flex-algo 129` and verify there's no single point of failure.

2. **Fallback to Algorithm 0.** Configure SR-TE policies with a fallback candidate path that uses Algorithm 0 if Algorithm 129 is unreachable. On IOS-XR:
```
segment-routing
 traffic-eng
  policy CUST-A
   candidate-paths
    preference 200
     dynamic
      sid-algorithm 129
     !
    !
    preference 100
     dynamic
      sid-algorithm 0
     !
    !
   !
  !
 !
!
```
If Algorithm 129 can't reach the tail-end, the policy falls back to preference 100 (Algorithm 0). Traffic violates the constraint, but at least it's delivered.

3. **Monitor TI-LFA coverage per algorithm.** Track the percentage of prefixes with TI-LFA backup paths for each algorithm. If Algorithm 129 only has backup coverage for 60% of prefixes (vs 99% for Algorithm 0), your constrained topology needs more links.

## Question 5
> Design a FlexAlgo scheme for a backbone with 3 service tiers: latency-sensitive (financial), standard (enterprise), and bulk (backup/replication). Define the algorithms, metric types, constraints, and VRF steering method.

### Answer

**Algorithm assignments:**

| Algorithm | Name | Metric Type | Constraints | Use Case |
|-----------|------|-------------|-------------|----------|
| 0 | Default | IGP metric | None | Standard enterprise traffic |
| 128 | Low-Latency | min-unidirectional-delay | None | Financial / real-time |
| 129 | High-Bandwidth | IGP metric | include-any: HIGH-BW (bit 3) | Bulk transfer / replication |

**Why three algorithms, not more:**

- **Algorithm 0 (Standard)** — already exists, zero additional configuration for transit nodes. Covers the vast majority of traffic. IGP metric provides optimal cost-based forwarding.
- **Algorithm 128 (Low-Latency)** — delay metric produces fundamentally different paths from IGP metric. Financial VRFs get the physically shortest path regardless of link bandwidth. No affinity constraints needed because the delay metric itself drives path selection.
- **Algorithm 129 (High-Bandwidth)** — uses IGP metric but restricted to links tagged HIGH-BW (your 100G+ core links). Bulk traffic only uses fat pipes, keeping it off 10G/25G metro aggregation links where it would cause congestion. The IGP metric within the HIGH-BW topology gives cost-optimal forwarding on the big links.

**Affinity color plan:**

| Bit | Color Name | Meaning | Applied To |
|-----|-----------|---------|------------|
| 3 | HIGH-BW | High-bandwidth core link (100G+) | All 100G and 400G backbone links |
| 10 | MAINT | Maintenance window | Applied ad-hoc before maintenance |

**Configuration — IOS-XR (definition-advertising routers, typically 2 RRs):**

```
router isis CORE
 flex-algo 128
  metric-type delay
  priority 200
  advertise-definition
 !
 flex-algo 129
  affinity include-any
   color HIGH-BW
  !
  priority 200
  advertise-definition
 !
!
```

**Configuration — Every P-node and PE (SR-MPLS Prefix-SIDs):**

```
router isis CORE
 flex-algo 128
 flex-algo 129
 interface Loopback0
  address-family ipv4 unicast
   prefix-sid index 1           ! Algorithm 0
   prefix-sid algorithm 128 index 1001  ! Low-Latency
   prefix-sid algorithm 129 index 2001  ! High-BW
  !
 !
!
```

**Affinity tagging on 100G+ links:**

```
router isis CORE
 interface HundredGigE0/0/0/0
  affinity flex-algo
   color HIGH-BW
  !
 !
!
```

**Delay measurement on all links (required for Algorithm 128):**

```
performance-measurement
 interface TenGigE0/0/0/0
  delay-measurement
   advertise-delay 1000
   threshold 3000
  !
 !
!
```

**Configuration — Junos equivalent (definition-advertising routers):**

```
set protocols isis flexible-algorithm 128 definition metric-type delay
set protocols isis flexible-algorithm 128 definition priority 200
set protocols isis flexible-algorithm 128 advertise

set protocols isis flexible-algorithm 129 definition admin-group include-any HIGH-BW
set protocols isis flexible-algorithm 129 definition priority 200
set protocols isis flexible-algorithm 129 advertise
```

**Junos — P-node and PE participation:**

```
set protocols isis source-packet-routing flex-algorithm 128 prefix-sid 1001
set protocols isis source-packet-routing flex-algorithm 129 prefix-sid 2001
set protocols isis interface ge-0/0/0 level 2 flex-algo-affinity-name HIGH-BW
```

**VRF steering — ODN with BGP color communities (recommended):**

This is the most operationally clean approach. Each VRF's BGP routes are tagged with a color community that maps to an algorithm.

| Service Tier | BGP Color Community | ODN Policy | Algorithm |
|-------------|-------------------|------------|-----------|
| Financial (Gold) | color 128 | on-demand color 128 | 128 (delay) |
| Enterprise (Silver) | No color | Default Algorithm 0 | 0 (IGP) |
| Bulk (Bronze) | color 129 | on-demand color 129 | 129 (HIGH-BW) |

**ODN configuration (IOS-XR):**

```
segment-routing
 traffic-eng
  on-demand color 128
   dynamic
    sid-algorithm 128
   !
  !
  on-demand color 129
   dynamic
    sid-algorithm 129
   !
  !
 !
!
```

**BGP color assignment (set on PE or RR via route-policy):**

```
route-policy SET-GOLD
  set extcommunity color (128)
end-policy
!
router bgp 65000
 vrf FINANCIAL
  address-family ipv4 unicast
   route-policy SET-GOLD out
  !
 !
!
```

**Why ODN over static VRF-to-algorithm binding:**
- Adding a new VRF only requires the correct BGP color community — no SR-TE policy changes
- VRFs can be moved between tiers by changing the color community at the RR level
- ODN auto-instantiates policies on demand — no pre-provisioning per destination PE
- Works seamlessly with inter-AS Option C where the color community transits across AS boundaries

**Capacity planning considerations:**

- Algorithm 129 (HIGH-BW) must have enough 100G+ links to form a connected topology between all PE pairs that serve bulk VRFs. Run the topology check before enabling bulk VRF steering.
- Algorithm 128 (Low-Latency) delay measurements should use static propagation delay (not live queuing delay) to avoid the oscillation problem described in Question 3.
- Standard enterprise traffic (Algorithm 0) gets the full physical topology — it's the most resilient tier by definition. This is correct: the "default" tier should be the most reliable, not the most constrained.
- Monitor per-algorithm TI-LFA coverage. Algorithm 129 is the most constrained (subset of links) and therefore the most likely to have TI-LFA coverage gaps. Ensure all HIGH-BW links have redundant peers within the HIGH-BW topology.
