# 8.2 — EVPN Fundamentals — Protocol Theory

## Protocol Overview

EVPN (Ethernet VPN, RFC 7432) is the modern control plane for Layer 2 VPN services. It replaces VPLS's data-plane MAC learning with BGP-based MAC/IP advertisement — every PE announces learned MAC addresses (and optionally IP bindings) via MP-BGP, so remote PEs know exactly where each MAC lives before the first data frame arrives. No more flooding to learn. No more full-mesh replication for known unicast. No more blackholing during failovers.

EVPN uses BGP address family AFI 25 (L2VPN), SAFI 70 (EVPN). It defines five core route types, each carrying specific information for different aspects of L2/L3 service delivery. The route types are the heart of EVPN — understanding what each carries and when it's generated is the difference between configuring EVPN and understanding EVPN.

EVPN was designed to fix every limitation of VPLS: active/active multi-homing (VPLS could only do active/standby), control-plane MAC learning (VPLS used data-plane flooding), ARP/ND suppression (VPLS flooded all ARP), and integrated L2+L3 service (VPLS was pure L2).

## Core Mechanisms

### 1. EVPN Route Types

EVPN defines five primary route types. Each is a distinct BGP NLRI encoding within the EVPN address family.

#### Route Type 1 — Ethernet Auto-Discovery (EAD)

**Purpose**: Enables multi-homing functions — fast convergence, aliasing, and split-horizon filtering.

**Two sub-types:**
- **Per-ES EAD**: Advertised per Ethernet Segment. Carries the ESI (Ethernet Segment Identifier). Used for split-horizon filtering — remote PEs use this to know which PEs share a multi-homed segment.
- **Per-EVI EAD**: Advertised per EVPN Instance (EVI) per Ethernet Segment. Carries the EVI-specific label. Used for aliasing — tells remote PEs they can load-balance traffic across all PEs attached to this ES.

**NLRI encoding:**
```
├─ Route Distinguisher (8 bytes)
├─ Ethernet Segment Identifier (10 bytes)
├─ Ethernet Tag ID (4 bytes)
└─ MPLS Label (3 bytes)
```

**When generated**: When a PE has a multi-homed CE (ESI ≠ 0). Single-homed CEs (ESI = 0) don't generate Type 1 routes.

#### Route Type 2 — MAC/IP Advertisement

**Purpose**: The workhorse. Advertises MAC addresses and optional IP bindings learned from CEs.

**NLRI encoding:**
```
├─ Route Distinguisher (8 bytes)
├─ Ethernet Segment Identifier (10 bytes)
├─ Ethernet Tag ID (4 bytes)
├─ MAC Address Length (1 byte): always 48
├─ MAC Address (6 bytes)
├─ IP Address Length (1 byte): 0, 32 (IPv4), or 128 (IPv6)
├─ IP Address (0, 4, or 16 bytes)
├─ MPLS Label 1 (3 bytes): L2 VNI / MPLS label
└─ MPLS Label 2 (3 bytes, optional): L3 VNI / MPLS label for routing
```

**Key properties:**
- Can carry MAC-only (pure L2) or MAC+IP (integrated L2/L3).
- Two labels possible: Label 1 for L2 forwarding (bridge domain), Label 2 for L3 forwarding (IP-VRF). This dual-label capability is what enables integrated routing and bridging (IRB).
- ESI field links the MAC to a multi-homed segment (or 0 for single-homed).

**When generated**: When the PE learns a MAC address from a local CE (via data-plane learning or static config). If ARP/ND snooping is enabled, the IP binding is included.

#### Route Type 3 — Inclusive Multicast Ethernet Tag (IMET)

**Purpose**: Sets up the BUM (Broadcast, Unknown unicast, Multicast) traffic tree. Each PE advertises its participation in an EVI, telling other PEs "send me BUM traffic for this service."

**NLRI encoding:**
```
├─ Route Distinguisher (8 bytes)
├─ Ethernet Tag ID (4 bytes)
├─ IP Address Length (1 byte)
└─ Originating Router's IP Address (4 or 16 bytes)
```

**Provider Multicast Service Interface (PMSI) Tunnel attribute:**
Type 3 routes carry a PMSI Tunnel attribute indicating the BUM replication method:
- **Ingress Replication**: Head-end replicates to each PE individually (unicast copies). Simple but doesn't scale for large BUM loads.
- **P2MP RSVP-TE/mLDP**: Uses MPLS multicast trees for efficient replication. Better for heavy BUM traffic.
- **PIM-based**: Uses IP multicast in the underlay.

**When generated**: When a PE joins an EVI. Always present — every PE in an EVI generates a Type 3 route.

#### Route Type 4 — Ethernet Segment Route

**Purpose**: Designated Forwarder (DF) election for multi-homed Ethernet Segments. PEs sharing an ES discover each other via Type 4 routes and elect a DF per EVI.

**NLRI encoding:**
```
├─ Route Distinguisher (8 bytes)
├─ Ethernet Segment Identifier (10 bytes)
├─ IP Address Length (1 byte)
└─ Originating Router's IP Address (4 or 16 bytes)
```

**When generated**: When a PE has a multi-homed CE (ESI ≠ 0). The Type 4 route tells other PEs "I'm attached to this Ethernet Segment."

#### Route Type 5 — IP Prefix Route

**Purpose**: Advertises IP prefixes for inter-subnet routing in EVPN. Enables pure L3 routing over EVPN without requiring a MAC binding.

**NLRI encoding:**
```
├─ Route Distinguisher (8 bytes)
├─ Ethernet Segment Identifier (10 bytes)
├─ Ethernet Tag ID (4 bytes)
├─ IP Prefix Length (1 byte)
├─ IP Prefix (4 or 16 bytes)
├─ GW IP Address (4 or 16 bytes)
└─ MPLS Label (3 bytes)
```

**When generated**: When a PE needs to advertise an IP prefix that isn't directly bound to a specific MAC. Used for external route injection (static routes, redistributed IGP) into the EVPN IP-VRF.

**Type 2 vs Type 5 for IP routing**: Type 2 carries MAC+IP (host route, /32 or /128). Type 5 carries IP prefix (any length, no MAC). Type 2 is for locally learned hosts. Type 5 is for aggregated or external prefixes.

### 2. EVPN Instances (EVI) and Service Types

**EVI**: An EVPN Instance is the service container — analogous to a VRF in L3VPN. Each EVI has its own set of Route Targets, route distinguisher, and bridge domain.

**EVPN service types:**
- **EVPN-MPLS**: Uses MPLS data plane. Traditional SP deployment.
- **EVPN-VXLAN**: Uses VXLAN data plane. Data center deployment.
- **EVPN-PBB**: Uses Provider Backbone Bridging (802.1ah) for MAC scalability.

The EVPN control plane is the same regardless of data plane — route types, BGP mechanics, and multi-homing all work identically. Only the encapsulation changes.

### 3. ARP/ND Suppression

One of EVPN's most impactful features. In VPLS, every ARP request is flooded to all PEs — pure broadcast. In large L2 domains, ARP flooding consumes significant bandwidth and PE CPU.

**How it works:**
1. PE learns MAC+IP binding from a local CE (via ARP/ND snooping).
2. PE advertises MAC+IP in a Type 2 route to all PEs.
3. When a remote CE sends an ARP request for that IP, the local PE intercepts it.
4. PE looks up the IP in its EVPN MAC/IP table.
5. PE generates a proxy ARP reply locally — the ARP request never crosses the MPLS core.

**Result**: ARP traffic is contained locally. The WAN sees only BGP control-plane updates, not broadcast ARP. In a VPLS with 1000 hosts, this can reduce cross-WAN ARP traffic by 95%+.

### 4. MAC Mobility

When a host moves from one PE to another (VM migration, physical re-cabling), EVPN handles it via the MAC Mobility Extended Community.

**Sequence:**
1. Host H1 (MAC aa:bb:cc:dd:ee:ff) was attached to PE1. PE1 advertised Type 2 route.
2. H1 moves to PE2. PE2 learns the MAC locally.
3. PE2 advertises a new Type 2 route for the same MAC with a MAC Mobility extended community containing a sequence number (incremented from PE1's value).
4. All PEs receive the update. The higher sequence number wins — they update their MAC tables to point to PE2.
5. PE1 withdraws its old Type 2 route.

**Loop detection**: If a MAC bounces between PEs rapidly (high sequence numbers in short time), EVPN detects it as a loop and can suppress the MAC. The threshold is configurable.

### 5. Integrated Routing and Bridging (IRB)

EVPN can provide both L2 (bridging) and L3 (routing) services simultaneously.

**Asymmetric IRB:**
- Ingress PE does both L2 lookup and L3 routing.
- Traffic is bridged into the destination subnet's VLAN at the ingress PE, then sent across the EVPN fabric as L2.
- The egress PE only bridges (no routing).
- Requires all subnets to be configured on all PEs.

**Symmetric IRB:**
- Ingress PE routes the packet into the IP-VRF.
- Traffic is sent across the fabric with both an L2 label (inner, for the bridge domain) and an L3 label (for the IP-VRF).
- The egress PE routes and then bridges to the destination host.
- Only the ingress and egress PEs need the relevant subnets configured.
- Scales better for large deployments.

**Type 2 dual-label**: Symmetric IRB uses the two-label capability of Type 2 routes. Label 1 = L2 domain, Label 2 = L3 VRF.

## Protocol Interactions

**EVPN ↔ MP-BGP**: EVPN is entirely BGP-based. Route types are BGP NLRI in AFI 25, SAFI 70. Route Targets control distribution. Route reflectors scale the control plane. All standard BGP machinery applies.

**EVPN ↔ MPLS/VXLAN**: EVPN is data-plane agnostic. The same control plane works over MPLS labels (SP networks) or VXLAN VNIs (data centers). The Extended Community "Encapsulation" indicates which data plane to use.

**EVPN ↔ L3VPN**: Type 5 routes can carry IP prefixes into an IP-VRF, providing L3VPN-like service within the EVPN framework. The L3 VRF in EVPN symmetric IRB is functionally identical to an L3VPN VRF.

**EVPN ↔ Legacy VPLS**: EVPN can interwork with VPLS during migration. Both can coexist for the same service, with EVPN PEs and VPLS PEs bridged together.

## Edge Cases & Gotchas

**1. Type 2 with and without IP**: A Type 2 route with MAC-only provides L2 forwarding. Adding IP enables ARP suppression and IRB. If ARP snooping is disabled, Type 2 routes carry MAC-only — ARP suppression won't work.

**2. Route Type 3 required even for known unicast**: Type 3 routes are always needed to establish the BUM replication tree. Without Type 3, PEs don't know about each other's EVI membership. Even if all traffic is known unicast, Type 3 must be present.

**3. MAC mobility storms**: Rapid MAC moves (e.g., misconfigured VM migration loop) cause rapidly incrementing sequence numbers. Without loop detection, this floods the BGP control plane with updates. Enable MAC mobility duplicate detection.

**4. Ingress replication doesn't scale for BUM-heavy workloads**: With 100 PEs and ingress replication, every broadcast frame becomes 99 unicast copies. For BUM-heavy environments, P2MP trees are essential.

**5. Type 5 and Type 2 interaction**: When both exist for the same prefix, Type 2 (host route, /32) is more specific and wins. This is correct behavior but can confuse operators who expect Type 5 prefix routes to override Type 2 host routes.
