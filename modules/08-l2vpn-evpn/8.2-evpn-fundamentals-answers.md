## Question 1
> **Scenario:** A customer reports intermittent connectivity drops lasting exactly 30 seconds in their EVPN service. You check the MAC table and see "Duplicate" status on several MACs with high sequence numbers. What is the most likely root cause, and what are three things you should check?

### Answer

The most likely root cause is a **MAC mobility storm** — the same MAC address is bouncing rapidly between two attachment points, causing EVPN to increment the **MAC Mobility sequence number** until the PE declares the MAC **duplicate** and freezes it.

The clue is the exact **30-second** outage window. That aligns with the common duplicate-MAC **freeze timer** discussed in the module. In other words, the control plane is doing what it is supposed to do: protecting the fabric from a pathological MAC-move condition.

### What is usually behind this

Typical real causes include:
- a CE-side Layer 2 loop,
- a server or hypervisor moving a MAC back and forth rapidly between sites / uplinks,
- a dual-homed CE with broken LACP / multi-homing behavior,
- accidental parallel attachments for the same host.

This is not a random EVPN failure. It is EVPN correctly detecting that the endpoint location is unstable.

### Three things to check

#### 1. Check where the MAC is moving between

You want to determine whether the MAC is being learned from:
- two different PEs,
- two different ESIs,
- or a local AC and a remote EVPN path.

#### IOS-XR
```cisco
show bgp l2vpn evpn route-type 2 mac-ip <mac>
show evpn evi mac detail | include <mac>
show evpn ethernet-segment detail
```

#### Junos
```junos
show route table bgp.evpn.0 match <mac>
show evpn database mac-address <mac>
show evpn instance extensive | match "esi|duplicate|sequence"
```

If the same MAC is alternating between two next-hops or two ESIs with rising mobility sequence numbers, you found the control-plane symptom.

#### 2. Check the CE / host side for an actual L2 or host-mobility problem

This is usually the real root cause:
- misconfigured NIC teaming / bonding,
- active/active attachment without matching EVPN multi-homing config,
- bad hypervisor bridge,
- accidental L2 loop,
- VM vMotion / host mobility gone pathological.

For dual-homed CE scenarios, confirm:
- same ESI on both PEs,
- correct all-active / single-active mode,
- LACP behaving correctly if all-active,
- no split-brain behavior at the CE.

#### 3. Check duplicate-MAC thresholds / freeze timer and whether the event is isolated or systemic

You do not want to “fix” this by blindly relaxing the timer first. But you do want to confirm:
- how often moves are happening,
- whether many MACs are affected or only a few,
- whether this is one bad host or a fabric-wide issue.

If dozens of MACs are in duplicate state, suspect a broader loop or multi-homing misconfiguration. If it is one MAC, suspect one host or one VM.

### Strong exam answer

> The most likely cause is a **MAC mobility loop / flapping endpoint**. EVPN sees repeated MAC moves, increments the MAC Mobility sequence, and eventually freezes the MAC as **duplicate** for about **30 seconds**. Check (1) the EVPN Type-2 routes / next-hop and ESI for the affected MAC, (2) CE or hypervisor behavior causing the moves, and (3) duplicate-MAC thresholds / whether the issue is isolated or widespread.

That shows you understand both the **protocol mechanism** and the **operational root cause**.

## Question 2
> **Design:** You're designing an EVPN-MPLS service for a customer with 50 sites, each with a single CE. The customer has heavy multicast traffic (video surveillance). Should you use ingress replication or P2MP mLDP for BUM delivery? What are the trade-offs?

### Answer

For **50 sites with heavy multicast / BUM-like traffic**, I would seriously consider **P2MP mLDP** rather than defaulting to ingress replication.

### Why ingress replication is the default in many EVPN deployments

Ingress replication is operationally simple:
- no multicast state in the core,
- no P2MP tree signaling to manage,
- works well when the number of PEs per EVI is modest,
- works very well when BUM volume is low.

That is why it dominates many EVPN-MPLS and almost all EVPN-VXLAN deployments.

### Why this scenario is different

Here the service has:
- **50 sites**, so each BUM frame may be copied to **49 remote PEs**,
- **video surveillance**, which strongly suggests non-trivial multicast / flood-like replication load.

With ingress replication, the ingress PE performs the replication work and the WAN carries many duplicate copies of the same packet. That is fine for ARP and the occasional unknown unicast. It is much less fine for sustained surveillance traffic.

### Recommendation

If the provider core already supports and operates **mLDP/P2MP** cleanly, I would recommend:

> **P2MP mLDP for BUM delivery in this specific EVI**, because it shifts replication into the core and reduces ingress PE load and WAN duplication.

### Trade-offs

#### Option A — Ingress Replication

**Pros**
- simplest to deploy and troubleshoot,
- no multicast state in the P routers,
- no mLDP dependency,
- operationally familiar.

**Cons**
- ingress PE must replicate N-1 copies,
- more WAN bandwidth consumed for the same BUM flow,
- scales poorly for large EVIs with sustained multicast-heavy traffic.

#### Option B — P2MP mLDP

**Pros**
- one packet enters the multicast tree instead of 49 separate copies,
- lower ingress PE replication burden,
- better WAN efficiency for large multicast-heavy services.

**Cons**
- multicast state in the MPLS core,
- more operational complexity,
- harder troubleshooting than ingress replication,
- dependency on underlay multicast / mLDP design maturity.

### What I would say on an exam

> For a 50-site EVPN-MPLS service with heavy surveillance traffic, **P2MP mLDP** is the better technical choice if the core supports it operationally, because ingress replication would require the ingress PE to create 49 copies per BUM packet. The trade-off is complexity: ingress replication is much simpler to operate, while mLDP improves replication efficiency at the cost of multicast state and harder troubleshooting.

Then add the adult caveat:

> If the SP does not already run multicast in the core competently, I would still choose ingress replication and accept the inefficiency rather than deploy a theoretically elegant but operationally fragile multicast underlay.

Beauty is nice. Operations pays the rent.

## Question 3
> **Troubleshooting:** You configured EVPN between PE1 and PE2. BGP shows the session up with routes exchanged, but CE-to-CE traffic fails. `show evpn evi mac` on PE1 shows remote MACs from PE2 with the correct labels. What should you check next, and why?

### Answer

If the EVPN control plane is up and PE1 already has the **remote MACs with the correct EVPN labels**, the next thing to check is the **data-plane underlay / transport resolution** — specifically the MPLS forwarding path to PE2.

### Why this is the next check

At this point, several things are already proven:
- BGP EVPN adjacency is working,
- Route Type 2 MAC advertisements are being exchanged,
- control-plane label information is present.

So the failure is very likely **not** in the EVPN control plane anymore. It is now most likely one of these data-plane issues:
- no valid MPLS LSP / transport label to the remote PE loopback,
- MPLS forwarding broken in the core,
- EVPN label installed but transport label missing or unresolved,
- MTU / label-stack forwarding issue in the core.

### What to check

#### 1. Verify the transport LSP / tunnel to the remote PE

#### IOS-XR
```cisco
show mpls forwarding labels
show mpls ldp bindings
show route <remote-pe-loopback>
show cef <remote-pe-loopback> detail
```

#### Junos
```junos
show route <remote-pe-loopback>
show route forwarding-table destination <remote-pe-loopback>
show mpls lsp extensive
show route table mpls.0
```

You want to confirm that packets for the remote EVPN service actually have a working **transport path** in addition to the service label.

#### 2. Check the CE-facing attachment circuit is in the correct bridge domain / EVI

A classic mistake is that the remote MAC is present in EVPN, but the local AC is not actually bound to the right bridge-domain/EVI/VLAN.

#### IOS-XR
```cisco
show l2vpn bridge-domain detail
show evpn evi detail
show interface <ce-facing-interface>
```

#### Junos
```junos
show evpn instance extensive
show bridge mac-table
show configuration routing-instances <instance>
```

If the CE-facing port is in the wrong VLAN or wrong EVI, the control plane can still look correct while forwarding fails.

#### 3. Check MTU if labels are stacking properly but frames still die

MPLS EVPN carries customer Ethernet plus MPLS overhead. If the core or handoff MTU is too small, control plane looks healthy while actual customer frames vanish.

### Best exam framing

> Because remote MACs and labels are already present, the EVPN control plane is likely fine. The next check is the **MPLS underlay/data plane** — verify the transport LSP and LFIB entries toward PE2, then verify the local AC is in the correct bridge domain/EVI. Control-plane success does not prove the service data plane is actually deliverable.

That is the key mental model: **BGP being happy is not the same thing as Ethernet traffic being forwardable.**
