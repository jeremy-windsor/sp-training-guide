# 8.3 — EVPN MPLS vs. VXLAN — Answer Key

---

## Question 1
> **Design:** A customer needs to extend their EVPN fabric from on-prem (MPLS core) to a public cloud VPC. The cloud provider only supports VXLAN encapsulation. What are the three most critical design considerations for the border gateway that must translate between these two encapsulations?

### Answer

The three most critical design considerations for the **EVPN border gateway** (BGW) translating between **MPLS** (on-prem) and **VXLAN** (cloud) are:

1. **Encapsulation rewrite and tunnel termination**
2. **BGP EVPN route type handling and next-hop manipulation**
3. **MTU and fragmentation handling across the encapsulation boundary**

### 1. Encapsulation rewrite and tunnel termination

The BGW must:
- **terminate MPLS LSPs** from the on-prem side and extract the inner Ethernet frame,
- **re-encapsulate into VXLAN** for the cloud side (and vice versa),
- maintain the **EVPN context** (EVI, VLAN, ESI) across the rewrite.

This is not just "strip MPLS, add VXLAN header." The BGW must:
- preserve the original Ethernet frame including VLAN tags,
- map EVPN service identifiers consistently between domains,
- handle any control-word or sequence number implications.

**IOS-XR example concept:**
```cisco
! MPLS side
l2vpn
 bridge group CLOUD
  bridge-domain CUSTOMER-1
   interface GigabitEthernet0/0/0/1.100
   evi 100
   
! VXLAN side  
evpn
 evi 100
  bgp
   route-target import 65000:100
   route-target export 65000:100
  segment-routing
   vxlan
    vni 10100
```

The BGW is the **only device** that needs to understand both encapsulations in the same forwarding path.

### 2. BGP EVPN route type handling and next-hop manipulation

The BGW must participate in **BGP EVPN on both sides** but with different next-hop behaviors:

**MPLS side:**
- BGP EVPN routes carry **MPLS labels** (service labels),
- next-hop is the **PE loopback** (usually /32),
- underlay is MPLS LSPs (SR-MPLS, LDP, or RSVP-TE).

**VXLAN side:**
- BGP EVPN routes carry **VNI values**,
- next-hop is the **VTEP IP** (underlay IP in the cloud),
- underlay is pure IP (no MPLS).

**Critical BGW behaviors:**
- **Re-originate routes:** The BGW must re-advertise EVPN routes between the two domains, rewriting the next-hop and encapsulation attributes.
- **Route Type handling:** 
  - RT-2 (MAC/IP): must preserve MAC, IP, ESI, and rewrite label→VNI
  - RT-5 (IP prefix): must preserve prefix and rewrite next-hop/encapsulation
  - RT-1 (EAD): may need filtering or translation depending on design
- **Split-horizon:** The BGW must prevent loops when the same EVI exists on both sides.

### 3. MTU and fragmentation handling

**MPLS EVPN typical overhead:**
- Customer Ethernet frame: up to 1518 bytes (or 1522 with 802.1Q tag)
- MPLS labels: 4 bytes per label (usually 2 labels: transport + service)
- Total: ~1526-1530 bytes for 1500-byte customer MTU

**VXLAN typical overhead:**
- Customer Ethernet frame: up to 1518 bytes
- VXLAN header: 8 bytes
- UDP header: 8 bytes
- IP header: 20 bytes
- Ethernet header: 14 bytes (outer)
- Total: ~1568 bytes for 1500-byte customer MTU

**The problem:** VXLAN adds ~40 bytes more overhead than MPLS.

**Design considerations:**
- The BGW-facing interfaces must support **jumbo frames** (at least 1600 bytes, preferably 9180),
- The cloud VPC connectivity (Direct Connect, ExpressRoute, etc.) must also support jumbo MTU,
- If the cloud does not support jumbo frames, the BGW must handle **fragmentation** or **PMTUD**,
- Customer must be informed that their effective MTU across the boundary may be reduced.

### Summary for exam

> The three critical BGW design considerations are: (1) **encapsulation rewrite** — terminating MPLS and re-encapsulating into VXLAN while preserving EVPN context; (2) **BGP EVPN route handling** — re-originating routes with appropriate next-hop and label/VNI translation between domains; and (3) **MTU management** — ensuring the BGW and cloud connectivity support jumbo frames or handling fragmentation, since VXLAN overhead exceeds MPLS overhead by ~40 bytes.

## Question 2
> **Troubleshooting:** You have an EVPN-VXLAN fabric using eBGP between leaf switches. Traffic between two hosts on the same VLAN works fine, but traffic to the anycast gateway IP fails. The gateway IP is configured on both leaf switches with the same anycast MAC. What is the most likely cause?

### Answer

The most likely cause is **missing or inconsistent EVPN Type-5 (IP prefix) routes** for the anycast gateway IP, or **missing IRB/SVI configuration** on one or both leaf switches.

### Why same-VLAN host traffic works but gateway fails

**Same-VLAN host-to-host traffic:**
- Uses **EVPN Type-2 (MAC/IP advertisement)** routes,
- Hosts ARP for each other's MAC addresses,
- Traffic is L2 bridged within the VLAN/VNI,
- No gateway involvement needed.

**Traffic to the anycast gateway IP:**
- Hosts ARP for the gateway IP,
- Expect to receive the **anycast gateway MAC**,
- Then send traffic to that MAC, which should be routed.

If the gateway IP is not properly advertised via **EVPN Type-5** (or if the IRB/SVI is not properly configured), the leaf switches may not know how to handle traffic destined to the gateway.

### Specific things to check

#### 1. Verify the anycast gateway MAC is being advertised

Both leaf switches should advertise the same **anycast MAC** for the gateway IP via EVPN Type-2.

**Junos:**
```junos
show evpn database mac-address <anycast-mac>
show route table bgp.evpn.0 match <anycast-mac>
```

**Cisco NX-OS:**
```cisco
show mac address-table anycast-gateway
show bgp l2vpn evpn route-type 2
```

If only one leaf is advertising the anycast MAC, hosts connected to the other leaf will not be able to resolve the gateway ARP.

#### 2. Verify IRB/SVI is configured and up on both leaf switches

**Junos:**
```junos
show interfaces irb.100
show configuration interfaces irb unit 100
```

**Cisco NX-OS:**
```cisco
show interface vlan 100
show running-config interface vlan 100
```

Common mistakes:
- IRB/SVI is configured but **administratively down**,
- IRB/SVI exists but **no IP address** is configured,
- IRB/SVI IP is configured but **different from the anycast gateway IP**.

#### 3. Verify EVPN Type-5 routes for the gateway subnet

If the gateway needs to route to other subnets (inter-VLAN or external), it must advertise **Type-5 (IP prefix)** routes.

**Junos:**
```junos
show route table bgp.evpn.0 match 0.0.0.0/0
show evpn ip-prefix-database
```

**Cisco NX-OS:**
```cisco
show bgp l2vpn evpn route-type 5
```

#### 4. Check for duplicate IP detection issues

If both leaf switches have the same anycast gateway IP but EVPN **duplicate IP detection** is triggered, the IP may be frozen or suppressed.

**Junos:**
```junos
show evpn ip-database duplicate
```

**Cisco NX-OS:**
```cisco
show bgp l2vpn evpn duplicate
```

### Best exam answer

> The most likely cause is **incomplete IRB/SVI configuration** or **missing EVPN Type-2 advertisement of the anycast gateway MAC** on one of the leaf switches. Same-VLAN traffic works because it uses MAC learning (Type-2), but gateway traffic requires the anycast MAC to be reachable via ARP. Verify that both leaf switches have: (1) IRB/SVI configured and up with the anycast gateway IP, (2) the same anycast MAC configured, and (3) EVPN Type-2 routes advertising that MAC from both leaves.

## Question 3
> **Scenario:** You're migrating from VPLS to EVPN. The customer has 200 sites with identical VLAN IDs (100-110) but overlapping MAC address ranges. How does EVPN solve the "MAC overlap" problem that VPLS cannot handle cleanly?

### Answer

EVPN solves the **MAC overlap** problem through the **Route Distinguisher (RD)** and **Route Target (RT)** mechanism, which creates **isolated VPN contexts** — something VPLS handles poorly at scale.

### The problem in VPLS

In **VPLS**:
- All sites share a single **broadcast domain** per VPLS instance,
- MAC addresses must be **globally unique** within that instance,
- If two different customers (or two sites of the same customer) have overlapping MAC ranges, VPLS cannot distinguish them,
- The result is **MAC flapping**, **traffic black-holing**, or **security leaks**.

VPLS has no native concept of **tenant isolation** at the control-plane level. You would need separate VPLS instances per customer/site, which scales poorly.

### How EVPN solves this

EVPN uses **BGP VPNv4/VPNv6-style addressing** with RD and RT:

#### 1. Route Distinguisher (RD) creates unique VPN routes

Each EVPN instance (EVI) gets a unique **RD** (typically `ASN:EVI` or `IP:EVI`).

Example:
- Customer A, EVI 100: RD = `65000:100`
- Customer B, EVI 200: RD = `65000:200`

Even if both customers use MAC `00:11:22:33:44:55` on VLAN 100, EVPN advertises:
- `65000:100:00:11:22:33:44:55` (Customer A)
- `65000:200:00:11:22:33:44:55` (Customer B)

These are **different BGP routes** and do not conflict.

#### 2. Route Target (RT) controls import/export

Each EVI has **import and export RTs** that determine which routes are accepted from and advertised to other PEs.

Example:
- Customer A EVI 100: RT import/export = `target:65000:100`
- Customer B EVI 200: RT import/export = `target:65000:200`

PEs serving Customer A only import routes with RT `65000:100`, completely isolating Customer B's overlapping MACs.

#### 3. VLAN normalization (Ethernet Tag ID)

EVPN also supports **VLAN normalization** via the **Ethernet Tag ID**:
- Customer A's VLAN 100 can be mapped to **ETI 100**,
- Customer B's VLAN 100 can be mapped to **ETI 200**,
- The same VLAN ID at different sites can be mapped to different ETIs per EVI.

This allows identical VLAN IDs to coexist without collision.

### Configuration example

**Junos:**
```junos
routing-instances {
    CUSTOMER-A {
        instance-type evpn;
        vlan-id 100;
        interface ge-0/0/1.100;
        route-distinguisher 65000:100;
        vrf-target target:65000:100;
    }
    CUSTOMER-B {
        instance-type evpn;
        vlan-id 100;
        interface ge-0/0/2.100;
        route-distinguisher 65000:200;
        vrf-target target:65000:200;
    }
}
```

**IOS-XR:**
```cisco
l2vpn
 bridge group CUSTOMER-A
  bridge-domain BD-100
   interface GigabitEthernet0/0/0/1.100
   evi 100
   
evpn
 evi 100
  bgp
   route-distinguisher 65000:100
   route-target import 65000:100
   route-target export 65000:100
```

### Summary for exam

> EVPN solves MAC overlap through **Route Distinguishers (RD)** and **Route Targets (RT)**, which create isolated VPN contexts similar to MPLS L3VPN. Each EVI has a unique RD, so identical MACs in different EVIs are different BGP routes. RTs ensure routes are only imported by PEs in the same VPN, providing tenant isolation. VPLS lacks this mechanism — all sites share a single broadcast domain, making MAC overlap a fundamental problem.

The key insight: **EVPN brings L3VPN-style isolation to L2 services**, which is why it can scale to multi-tenant and multi-service environments that VPLS cannot.
