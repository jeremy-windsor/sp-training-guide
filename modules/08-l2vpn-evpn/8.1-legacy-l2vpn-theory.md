# 8.1 — Legacy L2VPN: VPWS, VPLS, and H-VPLS — Protocol Theory

## Protocol Overview

Legacy L2VPN technologies provide Layer 2 connectivity across a Layer 3 MPLS backbone. Where L3VPN virtualizes routing tables, L2VPN virtualizes Ethernet segments — the customer sees a transparent Ethernet pipe between sites, unaware that traffic crosses an IP/MPLS core. The provider emulates a wire (point-to-point) or a switch (multipoint).

Three primary architectures emerged:

**VPWS (Virtual Private Wire Service)** — point-to-point pseudowires. One customer site connects to exactly one other customer site. The simplest L2VPN. Standardized in RFC 4447 (pseudowire signaling, later obsoleted by RFC 8077) and RFC 4448 (Ethernet over MPLS).

**VPLS (Virtual Private LAN Service)** — multipoint Layer 2 VPN. Multiple customer sites share a virtual Ethernet broadcast domain. Every PE in the VPLS learns MAC addresses and performs flooding/forwarding like a switch. Standardized in RFC 4761 (BGP-based VPLS) and RFC 4762 (LDP-based VPLS).

**H-VPLS (Hierarchical VPLS)** — a scaling optimization for VPLS. Introduces spoke pseudowires from MTU (multi-tenant unit) switches to hub PEs, reducing the full-mesh requirement. Not a separate standard — it's a deployment model using VPLS with QinQ or pseudowire aggregation.

These technologies are "legacy" because EVPN (Module 8.2) has largely superseded them with better control-plane MAC learning, multi-homing, and integration with modern data center fabrics. But legacy L2VPN still runs in production at scale — understanding the mechanics is essential.

## Core Mechanisms

### 1. Pseudowire Fundamentals

A pseudowire (PW) is a tunnel that emulates a Layer 2 connection across a packet-switched network. It's the fundamental building block of all L2VPN.

**Two-label stack (same concept as L3VPN):**
```
┌─────────────────┐
│ Transport Label  │  ← Outer: reaches the remote PE (LDP/SR/RSVP-TE)
├─────────────────┤
│    PW Label      │  ← Inner: identifies the pseudowire at the remote PE
├─────────────────┤
│  Ethernet Frame  │  ← Customer L2 frame (minus preamble/FCS)
└─────────────────┘
```

The PW label is functionally equivalent to the VPN label in L3VPN — it tells the remote PE which virtual circuit this frame belongs to.

**Pseudowire encapsulation (RFC 4448 for Ethernet):**
- The customer Ethernet frame is stripped of preamble and FCS.
- A 4-byte control word (optional but recommended) is prepended, carrying sequence numbers, flags, and padding info.
- The PW label and transport label are pushed.

**Control word**: The control word sits between the PW label and the Ethernet payload. It serves multiple purposes:
- Prevents ECMP misidentification (without it, the first 4 bytes of the Ethernet frame could look like an IP header to transit routers attempting IP-based ECMP hashing).
- Carries sequence numbers for ordered delivery (optional).
- Indicates payload length for padding (when the Ethernet frame is smaller than 64 bytes, it must be padded for MPLS transport).

### 2. Pseudowire Signaling — LDP (RFC 4447, obsoleted by RFC 8077)

The most common pseudowire signaling method uses targeted LDP (T-LDP).

**Setup process:**
1. PEs establish a targeted LDP session (TCP, port 646) between their loopbacks. This is separate from the interface-level LDP session used for transport labels.
2. PE1 sends a Label Mapping message with the FEC TLV containing:
   - PW Type (e.g., 0x0005 for Ethernet, 0x0004 for Ethernet VLAN)
   - PW ID (a 32-bit circuit identifier agreed between PEs)
   - Group ID (for bulk withdrawal)
   - Interface parameters (MTU, VCCV capabilities)
3. PE2 responds with its own Label Mapping for the same PW ID.
4. Both PEs now have bidirectional PW labels. The pseudowire is up.

**PW Status TLV (RFC 4447 Section 5.4.2):**
Pseudowires have an operational status signaled via PW Status TLV in LDP Notification messages:
- Status code 0x00000000: PW forwarding (up)
- Status code 0x00000001: PW not forwarding (standby)
- Status code 0x00000002: Local AC (attachment circuit) fault
- Status code 0x00000004: Remote AC fault
- Status code 0x00000008: Local PSN (packet-switched network) fault

This is how one PE tells the other "my local port is down — stop sending traffic this way."

**PW redundancy (RFC 6718/6870):**
For dual-homed sites, multiple pseudowires can be provisioned with primary/backup designation. PW Status TLV signals which PW is active. When the primary PW fails, the backup activates.

### 3. Pseudowire Signaling — BGP (RFC 4761)

BGP can also signal pseudowires, using the L2VPN address family (AFI 25, SAFI 65).

**BGP L2VPN NLRI:**
```
├─ Route Distinguisher (8 bytes)
├─ VE ID (2 bytes): Virtual Endpoint identifier (site ID)
├─ Label Block Offset (2 bytes)
├─ Label Block Size (2 bytes)
├─ Label Base (3 bytes): starting MPLS label for this block
```

**How BGP VPLS works:**
- Each PE advertises a label block — a range of labels, one per remote PE in the VPLS instance.
- Remote PEs compute which label from the block to use based on their VE ID.
- This eliminates per-PE label negotiation — each PE independently knows which label to use when sending to any other PE.

**Advantages of BGP over LDP for VPLS:**
- Scales better: BGP uses route reflectors, avoiding the full-mesh T-LDP requirement.
- Auto-discovery: BGP VPLS peers discover each other through BGP advertisements. LDP VPLS requires explicit neighbor configuration.
- RT-based: Like L3VPN, BGP VPLS uses Route Targets for membership, enabling policy-based provisioning.

### 4. VPLS — Multipoint L2VPN

VPLS creates a virtual Ethernet switch spanning multiple PEs. Each PE acts as a switch — learning MAC addresses from customer frames and forwarding/flooding based on MAC table lookups.

**VPLS forwarding behavior:**

**Known unicast**: PE looks up the destination MAC in its VPLS MAC table. If found, the frame is sent only on the pseudowire (or local AC) associated with that MAC. Standard bridge forwarding.

**Unknown unicast / Broadcast / Multicast (BUM traffic)**: PE floods the frame on ALL pseudowires in the VPLS instance AND all local ACs, except the one it arrived on. This is the expensive operation — it replicates traffic across the full mesh.

**MAC learning**: Data-plane learning. When a frame arrives on a PW or AC, the PE records (source MAC → incoming PW/AC) in its MAC table. Exactly like a physical Ethernet switch. No control-plane MAC distribution — this is why EVPN was invented.

**Split horizon**: To prevent loops in the full-mesh PW topology, VPLS enforces a split-horizon rule: **a frame received on a PW is NEVER forwarded to another PW**. It's only forwarded to local ACs. This works because the full mesh ensures every PE receives a direct copy — relaying through another PE would create duplicates.

### 5. VPLS Full-Mesh Requirement

VPLS requires a full mesh of pseudowires between all PEs in the VPLS instance. With N PEs, you need N×(N-1)/2 pseudowires.

**The scaling problem:**
- 10 PEs: 45 PWs
- 50 PEs: 1,225 PWs
- 100 PEs: 4,950 PWs

Each PW consumes label space, control plane state, and replication resources for BUM traffic.

### 6. H-VPLS (Hierarchical VPLS)

H-VPLS introduces a two-tier hierarchy to address the full-mesh problem:

**Hub PEs (N-PEs)**: Full-mesh of PWs between hub PEs. These are the core of the VPLS.

**Spoke devices (U-PEs/MTUs)**: Connect to one or two hub PEs via spoke PWs. Don't participate in the full mesh. Aggregation points.

**How it works:**
1. MTU/U-PE connects to hub PE via a spoke pseudowire (or QinQ tag).
2. Hub PE treats the spoke PW as a local AC — no split-horizon with other spoke PWs on the same hub.
3. Hub PE forwards/floods between its spoke PWs AND its core full-mesh PWs.
4. Result: spoke devices are shielded from the full mesh; only hub PEs carry the full-mesh burden.

**Trade-off**: H-VPLS breaks the pure split-horizon model. The hub PE must relay traffic between spoke PWs and core PWs, which can create loops if not carefully configured. The hub PE must apply split-horizon only among core PWs, not between spoke and core.

### 7. VPLS MAC Table Management

**MAC aging**: Learned MACs age out after a timer (default 300 seconds typically). Same as physical switches.

**MAC withdrawal (RFC 4762 Section 6.2)**: When a CE is removed or moves, the PE can send an LDP MAC withdrawal message to all peers, telling them to flush specific MACs (or all MACs for a given PW). This avoids waiting for MAC aging and prevents traffic blackholing during topology changes.

**MAC table size**: Each VPLS PE must hold MAC tables for all VPLS instances. In large deployments, MAC table size becomes a resource constraint. Hardware TCAM limits may apply.

## Protocol Interactions

**L2VPN ↔ MPLS transport**: Pseudowires use the same two-label stack as L3VPN. The transport label comes from LDP, RSVP-TE, or SR. L2VPN is transport-agnostic.

**VPLS ↔ STP**: VPLS replaces STP across the WAN — the full-mesh with split horizon provides a loop-free topology without spanning tree. But STP may still run on the CE side. PEs typically don't participate in CE STP — they forward BPDUs transparently.

**L2VPN ↔ EVPN**: EVPN replaces data-plane MAC learning with control-plane MAC advertisement. EVPN pseudowires can coexist with legacy PWs during migration. Many operators run both simultaneously.

## Edge Cases & Gotchas

**1. Control word mismatch**: If one PE sends with control word and the other expects without, the PW comes up but traffic is corrupted. Both ends MUST agree on control word usage during PW signaling. T-LDP negotiates this; misconfiguration causes silent forwarding failures.

**2. MTU mismatch**: If PW MTU negotiation fails (PEs advertise different interface MTUs), the PW won't come up. The error is in the LDP notification — check for MTU mismatch status codes.

**3. VPLS BUM flooding at scale**: In a VPLS with 50 PEs, every broadcast frame is replicated 49 times at the ingress PE. This is pure ingress replication — no multicast optimization by default. VPLS P2MP (RFC 7117) addresses this but adds complexity.

**4. MAC withdrawal delays**: Without MAC withdrawal, a CE failover causes traffic to blackhole for up to 300 seconds (MAC aging timer). Always ensure MAC withdrawal is enabled and functional.

**5. Split-horizon and H-VPLS interaction**: Misconfiguring split-horizon groups in H-VPLS creates loops. The hub PE must correctly distinguish spoke PWs (no split-horizon between them and core) from core PWs (split-horizon among themselves).
