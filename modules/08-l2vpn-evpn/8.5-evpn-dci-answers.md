# 8.5 — EVPN for DCI and Multi-Site — Answer Key

---

## Question 1
> **Design:** A customer wants to extend their EVPN fabric across three data centers for disaster recovery and workload mobility. They need both Layer 2 extension (for live VM migration) and Layer 3 routing between sites. Describe the three most common DCI models and when each is appropriate.

### Answer

The three most common EVPN DCI (Data Center Interconnect) models are:

1. **Unified Fabric (Single EVPN Domain)** — one EVPN instance spanning all sites
2. **Multi-Site with Border Gateways** — separate EVPN domains with BGWs interconnecting them
3. **L3-Only DCI with Type 5 Routes** — pure IP interconnect, no L2 extension

---

### Model 1: Unified Fabric (Single EVPN Domain)

**Design:**
- Single EVPN instance (single EVI or set of EVIs) spans all three data centers
- Same Route Distinguisher and Route Target space across all sites
- Full-mesh or RR-based BGP EVPN peering between all PEs

**When appropriate:**
- **Small scale:** Low number of sites (2-3), limited MAC scale per site
- **Tight coupling required:** Applications need seamless L2 mobility
- **Controlled latency:** Sites are geographically close (metro distances)
- **Single administrative domain:** Same team manages all sites

**Advantages:**
- Simplest design — no BGW complexity
- True L2 extension — VM migration with same IP/MAC
- Consistent policy across all sites

**Disadvantages:**
- **MAC scale explosion:** Every MAC from every site is learned everywhere
- **BUM replication across WAN:** Broadcast storms can propagate
- **Failure blast radius:** Site failure affects control plane everywhere
- **Distance limitations:** L2 extension doesn't work well over long distances

**Typical use case:** Metro DCI, active-active data centers within 100km.

---

### Model 2: Multi-Site with Border Gateways (BGWs)

**Design:**
- Each data center is a separate EVPN domain with its own RD/RT space
- **Border Gateways (BGWs)** interconnect the domains
- BGWs re-originate EVPN routes between domains, rewriting RD/RT and encapsulation

**When appropriate:**
- **Larger scale:** 3+ sites, high MAC scale per site
- **Geographic distribution:** Sites are regionally or globally distributed
- **Administrative separation:** Different teams per site
- **Failure isolation:** Want to contain site failures

**Advantages:**
- **MAC scale containment:** Each site learns only summarized/re-originated routes
- **Failure isolation:** Site failure is contained by BGW
- **Flexible interconnect:** Can use different underlays between sites
- **Policy control:** BGWs can filter, rate-limit, or transform routes

**Disadvantages:**
- **BGW complexity:** Additional devices to manage and troubleshoot
- **L2 extension limitations:** True L2 mobility limited to within a site
- **Convergence:** Additional hop through BGW adds latency

**BGW functions:**
- Re-originate EVPN routes with new RD/RT
- Translate encapsulation (MPLS ↔ VXLAN if needed)
- Implement inter-site policy (filtering, QoS)
- Provide failure boundary

**Typical use case:** Multi-region enterprise, global data centers, cloud on-ramp.

---

### Model 3: L3-Only DCI with Type 5 Routes

**Design:**
- No L2 extension between sites — pure Layer 3 interconnect
- EVPN Type 5 (IP Prefix) routes advertise subnets between sites
- Each site has independent L2 domains

**When appropriate:**
- **Maximum scale:** Very large number of sites
- **No L2 requirement:** Applications don't need live migration or L2 adjacency
- **Global distribution:** Sites worldwide with high latency
- **Cloud integration:** Connecting to cloud VPCs that don't support L2

**Advantages:**
- **Maximum scale:** No MAC learning across sites
- **Optimal routing:** Traffic takes shortest L3 path
- **Simplest WAN:** Standard IP routing, no L2 complexity
- **Cloud-friendly:** Works with any cloud provider

**Disadvantages:**
- **No live migration:** VMs cannot migrate between sites with same IP
- **Application redesign:** Apps requiring L2 adjacency must be rearchitected
- **DNS/GSLB complexity:** Need application-layer failover

**Typical use case:** DR with cold/warm standby, cloud-native applications, global anycast services.

---

### Summary comparison

| Model | L2 Extension | Scale | Complexity | Use Case |
|-------|-------------|-------|------------|----------|
| Unified Fabric | Yes | Low | Low | Metro DCI, 2-3 sites |
| Multi-Site BGW | Limited | Medium | Medium | Regional/global, controlled L2 |
| L3-Only | No | High | Low | Global, DR, cloud-native |

---

### Best exam answer

> The three common EVPN DCI models are: (1) **Unified Fabric** — single EVPN domain for metro distances with true L2 mobility but limited scale; (2) **Multi-Site with BGWs** — separate EVPN domains interconnected by Border Gateways that re-originate routes, providing scale and failure isolation while allowing controlled L2 extension; and (3) **L3-Only with Type 5** — pure IP interconnect using EVPN IP prefix routes, maximizing scale and global reach but requiring applications to handle L3 boundaries. Choose based on whether you need L2 mobility (Model 1 or 2) vs. maximum scale (Model 3), and whether sites are metro (Model 1) vs. global (Models 2 or 3).

## Question 2
> **Troubleshooting:** You have a multi-site EVPN deployment with Border Gateways. A VM migrates from Site A to Site B. The VM keeps its IP and MAC, but traffic from Site C to the VM is black-holed. The BGWs show the MAC route with the correct next-hop. What are the three most likely causes?

### Answer

Traffic black-holing after VM migration in a multi-site EVPN deployment typically indicates a **control-plane/data-plane mismatch** or **BGW forwarding issue**.

The three most likely causes are:

1. **BGW next-hop is not reachable in the underlay**
2. **BGW is not re-originating the route with correct encapsulation**
3. **MAC mobility sequence number not updated, causing route to be ignored**

---

### 1. BGW next-hop is not reachable in the underlay

**Root cause:** The BGW has the EVPN route with the correct next-hop (Site B's PE), but the **transport path to that next-hop is broken**.

**Common issues:**
- MPLS LSP to Site B PE is down
- BGP LU (Labeled Unicast) not advertising labels
- SR-MPLS SID not reachable
- Inter-site link failure

**How to check:**

**Junos:**
```junos
show route <site-b-pe-loopback>
show route forwarding-table destination <site-b-pe-loopback>
show mpls lsp
```

**IOS-XR:**
```cisco
show cef <site-b-pe-loopback>
show mpls forwarding-table <site-b-pe-loopback>
```

**Fix:** Repair the underlay connectivity. Ensure MPLS/SR-MPLS is working between BGWs.

---

### 2. BGW is not re-originating the route with correct encapsulation

**Root cause:** The BGW receives the MAC route from Site B but fails to re-originate it properly to Site C.

**Common issues:**
- BGW import policy blocks the route
- BGW export policy doesn't include the EVI
- Encapsulation mismatch (MPLS label vs. VNI)
- RD/RT translation error

**How to check:**

**Junos:**
```junos
show route table bgp.evpn.0 match <mac>
show evpn database mac-address <mac> extensive
show configuration policy-options policy-statement
```

**IOS-XR:**
```cisco
show bgp l2vpn evpn route-type 2 <mac>
show evpn evi mac <mac>
show route-policy
```

Verify:
- BGW sees the route from Site B
- BGW re-originates to Site C with correct RD/RT
- Encapsulation (label/VNI) is correct

**Fix:** Correct BGW import/export policies. Ensure EVI is included in inter-site advertisement.

---

### 3. MAC mobility sequence number not updated, causing route to be ignored

**Root cause:** The migrated MAC has a **lower or equal mobility sequence number** compared to the stale route, so the new location is not preferred.

**How MAC mobility works:**
- Each MAC move increments the sequence number
- Higher sequence = more recent location
- If the new route has lower sequence, it's rejected

**How to check:**

**Junos:**
```junos
show evpn database mac-address <mac> extensive
show route table bgp.evpn.0 match <mac> detail
```

Look for:
- Sequence numbers from different sites
- Which route is marked "preferred"

**IOS-XR:**
```cisco
show evpn evi mac <mac> detail
show bgp l2vpn evpn route-type 2 <mac> detail
```

**Fix:**
- Ensure the new site (Site B) advertises with **higher sequence number**
- If sequence is stuck, clear the stale route or wait for aging
- Check for duplicate MAC detection freezing the route

---

### Diagnostic workflow

1. **Verify the MAC route exists on all relevant PEs/BGWs:**
   ```junos
   show evpn database mac-address <mac>
   ```

2. **Check which next-hop is preferred:**
   - Should be Site B after migration
   - If still Site A, check MAC mobility sequence

3. **Verify underlay reachability to the preferred next-hop:**
   ```junos
   show route <next-hop>
   ping <next-hop>
   ```

4. **Check BGW re-origination:**
   - Does BGW see the route from Site B?
   - Does BGW advertise to Site C?
   - Are policies blocking?

5. **Check for duplicate MAC or freeze state:**
   ```junos
   show evpn database mac-address <mac> extensive | match duplicate
   ```

---

### Best exam answer

> Traffic black-holing after VM migration in multi-site EVPN typically indicates: (1) **underlay reachability failure** — the BGW cannot reach the new next-hop in Site B; (2) **BGW re-origination failure** — policies or configuration prevent the BGW from advertising the updated route to Site C; or (3) **MAC mobility sequence issues** — the new route has lower sequence number and is ignored, or duplicate MAC detection has frozen the route. Verify with `show evpn database`, `show route`, and BGW policy checks.

## Question 3
> **Scenario:** You're designing an EVPN-based DCI for a financial services customer. They have strict RPO/RTO requirements: RPO = 0 (no data loss), RTO = 5 minutes. They need synchronous replication between two data centers 50km apart. Which DCI model and specific EVPN features would you recommend?

### Answer

For **RPO = 0, RTO = 5 minutes, synchronous replication, 50km distance**, I recommend:

**DCI Model:** **Unified Fabric (Single EVPN Domain)** with **all-active multi-homing**

**Key EVPN features:**
1. All-active multi-homing with ESI
2. MAC Mobility for rapid convergence
3. BFD for sub-second failure detection
4. Mass withdrawal for fast MAC flush
5. Aliasing for ECMP

---

### Why Unified Fabric

**RPO = 0 (synchronous replication):**
- Requires **low latency** between sites (50km is acceptable for sync replication)
- Storage arrays need L2 adjacency for synchronous write acknowledgment
- Unified Fabric provides true L2 extension

**RTO = 5 minutes:**
- Requires **automated failover** — no manual intervention
- Unified Fabric with optimized EVPN can achieve sub-second convergence
- Well within 5-minute RTO

**50km distance:**
- Metro distance — acceptable for unified fabric
- Latency ~0.25ms each way — fine for sync replication
- Dark fiber or DWDM available at this distance

**Why not Multi-Site BGW:**
- BGW adds complexity and potential failure point
- L2 extension through BGW is more complex
- Convergence through BGW may exceed RTO

**Why not L3-Only:**
- L3-only cannot support synchronous storage replication
- RPO = 0 requires L2 adjacency for storage protocols

---

### Specific EVPN features

#### 1. All-active multi-homing with ESI

**Purpose:** Both data centers can actively forward traffic

**Configuration:**

**Junos:**
```junos
interfaces {
    ae0 {
        esi {
            00:00:00:00:00:00:00:00:01:01;
            all-active;
        }
        aggregated-ether-options {
            lacp {
                active;
                periodic fast;
            }
        }
    }
}
```

**Benefit:** Storage arrays can be dual-homed to both sites with active paths.

#### 2. MAC Mobility with optimized timers

**Purpose:** Rapid update when VMs or storage targets move

**Configuration:**

**Junos:**
```junos
routing-instances {
    EVPN-DC {
        protocols {
            evpn {
                mac-mobility;
                duplicate-mac-detection {
                    auto-recovery-time 30;
                }
            }
        }
    }
}
```

**Benefit:** MAC moves are accepted immediately with sequence number tracking.

#### 3. BFD for sub-second failure detection

**Purpose:** Detect link/node failures in ~300ms

**Configuration:**

**Junos:**
```junos
protocols {
    bgp {
        group EVPN-DC {
            bfd-liveness-detection {
                minimum-interval 100;
                multiplier 3;
            }
        }
    }
}
```

**Benefit:** Fast detection triggers rapid convergence.

#### 4. Mass withdrawal (Route Type 1)

**Purpose:** Immediate MAC flush on failure

**Mechanism:**
- PE detects failure
- Withdraws RT-1 (EAD per-ES)
- Remote PEs immediately flush all MACs for that ESI

**Benefit:** Sub-second MAC convergence instead of 300s aging.

#### 5. Aliasing (Route Type 1)

**Purpose:** Pre-established ECMP paths

**Benefit:** Traffic shifts to surviving path immediately without reconvergence.

---

### Additional design considerations

#### Storage replication

- Use **synchronous storage replication** (e.g., EMC SRDF, NetApp MetroCluster, IBM PPRC)
- Requires L2 extension for storage protocols (Fibre Channel over Ethernet or iSCSI)
- Unified Fabric provides this natively

#### Network redundancy

- **Dual dark fiber paths** between sites
- **Diverse physical routes** (different conduits/towers)
- **DWDM protection** (OLP - Optical Line Protection)

#### Monitoring and automation

- **BFD everywhere:** PE-PE, PE-CE, storage array links
- **Automated failover:** No manual steps in the 5-minute window
- **Monitoring:** Real-time visibility into EVPN route state, MAC mobility, DF election

#### Failure scenarios

| Scenario | Detection | Recovery | Time |
|----------|-----------|----------|------|
| PE failure | BFD (300ms) | Mass withdrawal + aliasing | <1s |
| Link failure | LACP fast (1s) | ECMP shift | <1s |
| Site failure | BFD + storage heartbeat | Automated DR | <5min |

---

### Summary for exam

> For RPO=0, RTO=5min, synchronous replication over 50km, I recommend **Unified Fabric (single EVPN domain)** with **all-active multi-homing**. Key EVPN features: **all-active ESI** for dual-homed storage, **MAC Mobility** for rapid MAC updates, **BFD (100ms)** for fast failure detection, **mass withdrawal** for immediate MAC flush, and **aliasing** for pre-established ECMP. This provides true L2 extension for synchronous storage replication with sub-second network convergence, well within the 5-minute RTO.
