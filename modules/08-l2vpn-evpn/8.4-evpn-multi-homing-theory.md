# 8.4 — EVPN Multi-Homing — Protocol Theory

## Protocol Overview

EVPN multi-homing is the mechanism that lets a CE connect to two or more PEs simultaneously with active/active or active/standby forwarding. This was VPLS's Achilles heel — VPLS could only do active/standby multi-homing via PW redundancy, wasting half the available bandwidth. EVPN solved this by introducing the Ethernet Segment (ES) concept and three critical mechanisms: Designated Forwarder (DF) election, aliasing, and split-horizon filtering.

The Ethernet Segment is identified by a 10-byte Ethernet Segment Identifier (ESI). All PEs attached to the same multi-homed CE share the same ESI. This shared identity is what enables the PEs to coordinate — they know they're looking at the same physical segment and can divide responsibilities.

EVPN multi-homing is defined in RFC 7432 (Sections 7, 8, 14, and 15) with significant enhancements in subsequent RFCs. It's one of the most complex aspects of EVPN but also the most powerful — active/active multi-homing with per-flow load balancing and sub-second failover.

## Core Mechanisms

### 1. Ethernet Segment Identifier (ESI)

The ESI is a 10-byte value that uniquely identifies a multi-homed Ethernet Segment.

**ESI encoding:**
```
├─ Type (1 byte): ESI type
└─ Value (9 bytes): type-specific encoding
```

**ESI Types:**

| Type | Name | Encoding | Use Case |
|------|------|----------|----------|
| 0 | Arbitrary | Manually configured 9-byte value | Operator-defined, any scenario |
| 1 | LACP-based | CE LACP system MAC + port key | Auto-derived from LACP on the CE LAG |
| 2 | L2 bridge protocol | CE bridge MAC + priority | STP-based CE multi-homing |
| 3 | MAC-based | PE system MAC + local discriminator | Auto-derived from PE MAC |
| 4 | Router ID-based | PE router ID + local discriminator | Auto-derived from PE router ID |
| 5 | AS-based | PE AS number + local discriminator | Auto-derived from PE ASN |

**Type 0 (arbitrary)** is the most common in SP deployments — operator manually configures the same ESI on both PEs.

**Type 1 (LACP-based)** is elegant: if the CE runs LACP toward both PEs, the ESI is auto-derived from the CE's LACP system ID and port key. Both PEs independently compute the same ESI because they see the same LACP parameters from the CE.

**ESI 0 (all zeros)**: Reserved for single-homed CEs. When ESI = 0, no multi-homing procedures apply.

### 2. Designated Forwarder (DF) Election

In a multi-homed ES, multiple PEs can receive and send BUM (Broadcast, Unknown unicast, Multicast) traffic. If all PEs forward BUM traffic to the CE simultaneously, the CE receives duplicate frames. The DF election ensures exactly one PE forwards BUM traffic to the CE per Ethernet Tag (often VLAN) within an EVI.

**DF election process:**
1. Each PE attached to the ES advertises a Type 4 (Ethernet Segment) route.
2. All PEs discover each other via these Type 4 routes — they now know which PEs share this ES.
3. PEs sort themselves by IP address (from Type 4 routes).
4. DF assignment is computed per Ethernet Tag. If 3 PEs share an ES and there are 30 Ethernet Tags participating on that ES, each PE is DF for roughly 10 tags.

**Default algorithm (mod-based, RFC 7432):**
```
DF for Ethernet_Tag_i = PE_list[ Ethernet_Tag_i mod len(PE_list) ]
```

This distributes DF responsibility evenly across PEs per ES.

**DF election algorithms (RFC 8584):**

RFC 8584 introduced improved DF election algorithms:

**Algorithm 0 — Default (mod-based)**: As described above. Simple but has issues: if one PE fails, all EVI-to-PE mappings shift, causing unnecessary churn.

**Algorithm 1 — HRW (Highest Random Weight)**: Uses a hash of (EVI, PE_IP) to compute a weight. Highest weight per EVI wins DF. When a PE fails, only EVIs assigned to that PE are reassigned — other EVIs are unaffected. Much better convergence behavior.

**Algorithm 2 — Preference-based (RFC 9785)**: Each PE advertises a preference value. Highest preference wins. Allows deterministic DF placement and non-revertive behavior. Useful when PEs have different capacities.

**DF role:**
- **DF PE**: Forwards BUM traffic from the fabric to the CE for this EVI. Also forwards BUM from the CE into the fabric.
- **Non-DF PE**: Does NOT forward BUM traffic to the CE for this EVI. Still forwards known unicast. Blocks BUM to prevent duplicates.

**Critical detail**: DF election only affects BUM traffic forwarding. Known unicast traffic is forwarded by ALL PEs in active/active mode (via aliasing). DF election prevents BUM duplication, not unicast duplication.

### 3. Aliasing

Aliasing enables active/active load balancing for known unicast traffic across all PEs in a multi-homed ES.

**The problem**: A host behind a multi-homed CE might only be learned on one PE (the one where the CE's LACP hash sends traffic for that MAC). Remote PEs only see the Type 2 route from that one PE. Without aliasing, all remote traffic for that MAC goes to a single PE — no load balancing.

**The solution — Type 1 per-EVI EAD routes:**
1. Each PE in the ES advertises a per-EVI EAD (Type 1) route for every EVI on the ES.
2. The per-EVI EAD route carries the PE's MPLS label for that EVI.
3. Remote PEs see: "PE1 and PE2 are both attached to this ES for this EVI."
4. Remote PEs install ECMP next-hops: traffic to any MAC on that ES can be sent to either PE1 or PE2.
5. The receiving PE (whichever gets the ECMP-hashed packet) delivers it to the CE via the LAG.

**Result**: Even though the MAC was only learned on PE1, PE2 can also deliver traffic to the CE because they share a LAG to the same CE. The per-EVI EAD route is the signal that says "I can deliver traffic for this ES."

### 4. Split-Horizon Filtering

When a BUM frame arrives from a multi-homed CE, all PEs in the ES receive it (via the LAG). Each PE floods it into the EVPN fabric. Remote PEs receive the BUM frame. But remote PEs must NOT send the frame back to PEs in the same ES — that would cause the CE to receive its own frame back.

**How it works:**
1. When PE1 floods a BUM frame from its local CE, it includes the ESI in the EVPN encapsulation.
2. Remote PEs receive the BUM frame. They check: is the ESI the same as any of my local ESIs?
3. If yes (this PE is also attached to the same ES): drop the frame. The CE already received it directly.
4. If no: forward normally.

**Per-ES EAD route's role**: The per-ES EAD (Type 1) route carries the ESI label. Remote PEs use this label to identify traffic originating from a specific ES. When they receive traffic with this ESI label, they know not to forward it back to CEs on the same ES.

### 5. Mass Withdrawal and Fast Convergence

When a PE loses connectivity to a multi-homed CE (link failure, PE failure), EVPN provides fast convergence via mass withdrawal.

**Without mass withdrawal**: The failed PE must withdraw every individual Type 2 (MAC) route it had advertised for that ES. If it had 10,000 MACs, that's 10,000 BGP withdrawals. Slow.

**With mass withdrawal (per-ES EAD route):**
1. PE detects ES failure (local link to CE goes down).
2. PE withdraws its per-ES EAD (Type 1) route — a single BGP withdrawal.
3. All remote PEs receive the withdrawal.
4. Remote PEs interpret: "PE1 is no longer attached to this ES. Remove PE1 from ECMP next-hops for ALL MACs on this ES."
5. All traffic immediately shifts to the remaining PE(s).

**Convergence time**: One BGP withdrawal → one FIB update per remote PE. Sub-second, regardless of how many MACs were on the segment. This is O(1) convergence instead of O(MACs).

### 6. Active/Active vs Active/Standby

**Active/Active:**
- Both PEs forward unicast traffic to/from the CE simultaneously.
- DF election determines which PE forwards BUM.
- Requires a LAG between CE and PEs (MC-LAG or EVPN-based).
- Full bandwidth utilization of both links.
- Aliasing enables load balancing from remote PEs.

**Active/Standby:**
- Only one PE (the DF) forwards all traffic — unicast and BUM.
- The standby PE monitors the ES but doesn't forward.
- Simpler — no need for aliasing or LAG.
- Half the bandwidth is wasted (standby link unused).
- Controlled via the per-ES EAD route's Active-Standby flag (bit in the ESI Label Extended Community).

**When to use active/standby**: When the CE doesn't support LAG toward both PEs (no MC-LAG capability). Or when traffic patterns make active/active ECMP hashing inefficient.

### 7. ES-Import Route Target

Each Ethernet Segment has an auto-derived RT called the ES-Import RT.

**Derivation**: The ES-Import RT is derived from the ESI itself — typically the first 6 bytes of the ESI value, formatted as a MAC-based RT.

**Purpose**: Type 4 (ES) routes carry the ES-Import RT. PEs only import Type 4 routes whose ES-Import RT matches one of their local ESIs. This scopes DF election to only the PEs that share the ES — other PEs in the network ignore the Type 4 route.

**Why this matters for scale**: In a network with 10,000 ESIs, a PE attached to only 5 ESIs should process DF election for only those 5. The ES-Import RT ensures the PE never sees Type 4 routes for the other 9,995 ESIs.

## Protocol Interactions

**Multi-homing ↔ LACP**: For active/active, the CE typically runs LACP toward both PEs. The PEs coordinate via EVPN (not LACP) — each PE runs LACP independently with the CE. The CE sees a single LAG. Type 1 ESI auto-derivation uses the CE's LACP system ID.

**Multi-homing ↔ BUM replication**: DF election interacts with the Type 3 (IMET) BUM replication tree. The non-DF PE participates in BUM replication for the EVI but blocks BUM toward its local ES. It still processes BUM from the fabric for other local CEs.

**Multi-homing ↔ MAC learning**: When active/active, both PEs may learn the same MAC from the CE (different LACP hash flows). Both advertise Type 2 routes. Remote PEs see two paths — this is expected and handled by ECMP (aliasing).

**Multi-homing ↔ IRB**: In symmetric IRB with multi-homing, the anycast gateway IP/MAC is the same on both PEs. The CE ARPs and gets the same gateway MAC regardless of which PE responds. This is essential for seamless active/active L3 forwarding.

## Edge Cases & Gotchas

**1. ESI mismatch between PEs**: If PE1 configures ESI 00:11:22:... and PE2 configures ESI 00:11:33:... for the same CE, they won't coordinate. No DF election, no aliasing. The CE gets duplicate BUM frames and potentially asymmetric forwarding. ESI must match exactly.

**2. DF election churn with mod-based algorithm**: If a PE in a 3-PE ES fails, the mod-based algorithm reassigns ALL EVIs (not just the failed PE's). This causes unnecessary traffic shifts. HRW (Algorithm 1) fixes this by only reassigning the failed PE's EVIs.

**3. Aliasing requires consistent EVPN labels**: For aliasing to work, the backup PE must be able to deliver traffic to the CE even if it hasn't locally learned the destination MAC. This only works if the CE is reachable via a LAG — if the CE has independent links (not bundled), aliasing breaks.

**4. Single-homed and multi-homed CEs on the same PE**: A PE can have both single-homed (ESI=0) and multi-homed (ESI≠0) CEs. Multi-homing procedures only apply to ESI≠0 interfaces. Don't configure ESI on single-homed ports.

**5. Mass withdrawal and convergence time**: Mass withdrawal is fast (one BGP message), but the remote PE still needs to reprogram its FIB. With 10,000 ECMP next-hop updates, FIB programming time may dominate. Platform ASIC speed matters.
