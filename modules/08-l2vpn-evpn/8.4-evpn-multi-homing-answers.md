# 8.4 — EVPN Multi-Homing (ESI, DF Election, Aliasing) — Answer Key

---

## Question 1
> **Design:** A customer wants all-active multi-homing for their EVPN service. Explain the roles of ESI, ES-Import RT, and DF election. What specific EVPN route types enable aliasing and mass withdrawal?

### Answer

All-active multi-homing in EVPN relies on three core mechanisms:

1. **ESI (Ethernet Segment Identifier)** — identifies the multi-homed CE segment
2. **ES-Import RT (Route Target)** — controls which PEs import ES routes for DF election
3. **DF (Designated Forwarder) election** — determines which PE forwards BUM traffic

The specific EVPN route types that enable aliasing and mass withdrawal are **Route Type 1 (Ethernet Auto-Discovery)**.

---

### 1. ESI (Ethernet Segment Identifier)

The ESI is a **10-byte identifier** that uniquely identifies a multi-homed Ethernet segment (the CE-side connection).

**Key properties:**
- All PEs connected to the same CE must configure the **same ESI** for that segment
- The ESI is encoded in EVPN routes to indicate multi-homing
- ESI values 0x00-0xFF are reserved; 0x00 means "single-homed"

**ESI types:**
- **Type 0 (0x00):** Single-homed (no multi-homing)
- **Type 1 (0x01):** LACP-based ESI (auto-generated from LACP system ID)
- **Type 2 (0x02):** LACP-based with CE-managed key
- **Type 3 (0x03):** MAC-based ESI
- **Type 4 (0x04):** Router-ID-based ESI
- **Type 5 (0x05):** AS-based ESI

**Typical configuration:**

**Junos:**
```junos
interfaces {
    ae0 {
        esi {
            00:00:00:00:00:00:00:00:01:01;
            all-active;
        }
    }
}
```

**IOS-XR:**
```cisco
interface Bundle-Ether1
 ethernet-segment
  identifier type 0 00.00.00.00.00.00.00.00.01.01
  bgp route-target import 00.00.00.00.00.00.00.00.01.01
```

---

### 2. ES-Import RT (Ethernet Segment Import Route Target)

The **ES-Import RT** is an automatically generated Route Target derived from the ESI. It controls which PEs participate in DF election for a given ESI.

**How it works:**
- Generated from the ESI (typically first 6 bytes)
- Used in **Route Type 4 (Ethernet Segment route)** import/export
- PEs import RT-4 routes with matching ES-Import RT to participate in DF election for that ESI

**Why it matters:**
- Only PEs connected to the same ESI need to run DF election
- Prevents unnecessary DF election state on PEs not serving that CE

---

### 3. DF (Designated Forwarder) Election

DF election determines **which PE forwards BUM (Broadcast, Unknown unicast, Multicast) traffic** toward the multi-homed CE.

**Why DF election is needed:**
- In all-active multi-homing, multiple PEs can forward unicast traffic to the CE
- But if **both PEs forward BUM**, the CE receives duplicates
- DF election ensures **only one PE forwards BUM** per VLAN/EVI

**DF election algorithm:**

The default algorithm is **modulo-based**:
```
DF = (VLAN_ID mod N) + 1
```
Where N = number of PEs in the ES, ordered by PE IP address.

**Example:**
- PE1: 10.0.0.1, PE2: 10.0.0.2 serving ES with ESI 00:00:00:00:00:00:00:00:01:01
- For VLAN 100: (100 mod 2) + 1 = 1 → PE1 is DF for VLAN 100
- For VLAN 101: (101 mod 2) + 1 = 2 → PE2 is DF for VLAN 101

**Alternative algorithms:**
- **HRW (Highest Random Weight):** Better load distribution
- **Preference-based:** Admin-configured priority

**Checking DF status:**

**Junos:**
```junos
show evpn ethernet-segment detail
show evpn ethernet-segment esi 00:00:00:00:00:00:00:00:01:01
```

**IOS-XR:**
```cisco
show evpn ethernet-segment detail
show evpn ethernet-segment esi 0000.0000.0000.0000.0101
```

---

### 4. Route Types for Aliasing and Mass Withdrawal

Both **aliasing** and **mass withdrawal** are enabled by **Route Type 1 (Ethernet Auto-Discovery)**.

#### Route Type 1A: EAD per-ES (per Ethernet Segment)

**Purpose:** Advertise reachability to the entire Ethernet Segment

**Used for:**
- **Aliasing** — remote PEs can ECMP to any PE in the ES
- **BUM forwarding** — identifies which PEs are in the ES for DF election

**Key fields:**
- ESI
- Ethernet Tag ID (usually 0 for per-ES)
- MPLS label or VNI

#### Route Type 1B: EAD per-EVI (per EVPN Instance)

**Purpose:** Advertise reachability to a specific VLAN/EVI on the ES

**Used for:**
- **Mass withdrawal** — when a PE fails, withdrawing RT-1B removes all MACs for that EVI
- **Aliasing per-VLAN** — more granular than per-ES

**Key fields:**
- ESI
- Ethernet Tag ID (specific VLAN)
- MPLS label or VNI

#### Aliasing explained

**Without aliasing:**
- Remote PE learns MAC X from PE1
- Traffic to MAC X goes only to PE1
- PE2 sits idle for that MAC

**With aliasing (via RT-1):**
- Remote PE learns MAC X from PE1 (RT-2)
- Remote PE also receives RT-1 from PE2 advertising the same ESI
- Remote PE can **ECMP to both PE1 and PE2** for MAC X
- Result: **true all-active forwarding**

#### Mass withdrawal explained

**Without mass withdrawal:**
- PE1 fails
- Each MAC learned from PE1 must age out individually (MAC mobility timeout)
- Traffic black-holed during convergence

**With mass withdrawal (via RT-1):**
- PE1 fails
- PE2 detects failure (BFD, LACP, etc.)
- PE2 withdraws RT-1 for the ESI
- Remote PEs immediately remove all MACs associated with that ESI
- Convergence is **sub-second** instead of minutes

---

### Summary for exam

> All-active multi-homing uses **ESI** to identify the multi-homed segment, **ES-Import RT** to scope DF election to participating PEs, and **DF election** to ensure only one PE forwards BUM. **Route Type 1 (Ethernet Auto-Discovery)** enables both **aliasing** (ECMP to all PEs in the ES) and **mass withdrawal** (fast removal of all MACs when a PE fails). RT-1A is per-ES for aliasing; RT-1B is per-EVI for granular mass withdrawal.

## Question 2
> **Troubleshooting:** In an all-active multi-homed setup, you see BUM traffic being delivered twice to the CE (duplicate frames). DF election appears to have run. What are the three most likely causes?

### Answer

Duplicate BUM frames in all-active multi-homing indicate that **both PEs are forwarding BUM** when only the DF should. Despite DF election appearing to run, something is bypassing or breaking the DF logic.

The three most likely causes are:

1. **Both PEs think they are DF** (split-brain DF election)
2. **ESI mismatch** between the PEs
3. **Non-DF PE is flooding due to unknown unicast or broadcast source**

---

### 1. Both PEs think they are DF (split-brain)

**Root cause:** The PEs cannot see each other's RT-4 (Ethernet Segment) routes, so each elects itself as DF.

**Common reasons:**
- **BGP EVPN session down** between the PEs
- **ES-Import RT mismatch** — different RTs mean routes aren't imported
- **Route policy blocking RT-4** — import policy filters ES routes
- **Different ESIs configured** — each PE thinks it's the only one on its ES

**How to check:**

**Junos:**
```junos
show evpn ethernet-segment detail
show route table bgp.evpn.0 match 00:00:00:00:00:00:00:00:01:01
```

Look for:
- Do you see the other PE's RT-4 route?
- Is the ES-Import RT consistent?

**IOS-XR:**
```cisco
show evpn ethernet-segment detail
show bgp l2vpn evpn route-type 4
```

**Fix:** Ensure both PEs have:
- Same ESI
- Same ES-Import RT (auto-derived from ESI)
- BGP EVPN adjacency working
- No import policies blocking RT-4

---

### 2. ESI mismatch between PEs

**Root cause:** The PEs are configured with **different ESIs** for the same CE.

**Result:**
- PE1 thinks it's on ES-A
- PE2 thinks it's on ES-B
- They don't recognize each other as multi-homing peers
- Both forward BUM independently

**How to check:**

**Junos:**
```junos
show configuration interfaces ae0 esi
show evpn ethernet-segment detail
```

**IOS-XR:**
```cisco
show running-config interface Bundle-Ether1 ethernet-segment
show evpn ethernet-segment detail
```

Compare the ESI values on both PEs. They must be **identical**.

**Fix:** Correct the ESI configuration to match on both PEs.

---

### 3. Non-DF PE is flooding due to unknown unicast or broadcast source

**Root cause:** Even with correct DF election, certain traffic types may bypass DF logic.

**Scenarios:**
- **Unknown unicast flooding:** If MAC learning is incomplete, the non-DF PE may flood
- **Broadcast from CE:** The CE may send broadcast that both PEs forward
- **Multicast:** Some multicast implementations bypass DF
- **Software bug:** Some early EVPN implementations had DF bypass bugs

**How to check:**

Capture and analyze the duplicate frames:
- Are they true BUM (broadcast destination MAC)?
- Are they unknown unicast?
- Is the source MAC learned on both PEs?

**Junos:**
```junos
show evpn database mac-address <source-mac>
show bridge mac-table | match <source-mac>
```

**IOS-XR:**
```cisco
show l2vpn bridge-domain mac-address <source-mac>
show evpn evi mac <source-mac>
```

**Fixes:**
- Ensure MAC learning is working (check RT-2 reception)
- Verify split-horizon is preventing looped BUM
- Check for platform-specific DF bypass bugs
- Consider **single-active** mode if all-active continues to have issues

---

### Diagnostic workflow

1. **Verify DF election state on both PEs:**
   ```junos
   show evpn ethernet-segment detail
   ```
   - Does each PE show the other as a peer?
   - Is DF status consistent per-VLAN?

2. **Verify ESI matches:**
   - Compare ESI config on both PEs
   - Must be byte-for-byte identical

3. **Verify RT-4 routes are exchanged:**
   ```junos
   show route table bgp.evpn.0 match <esi>
   ```
   - Both PEs should see each other's RT-4

4. **Capture traffic:**
   - Identify if duplicates are broadcast, unknown unicast, or multicast
   - Source MAC analysis

---

### Best exam answer

> Duplicate BUM in all-active multi-homing indicates both PEs are forwarding when only the DF should. Check: (1) **ESI mismatch** — verify both PEs have identical ESI configuration; (2) **split-brain DF election** — verify both PEs see each other's RT-4 routes and agree on DF status; (3) **non-DF flooding** — check if unknown unicast or broadcast is bypassing DF logic due to incomplete MAC learning or software issues. The key is verifying that DF election actually converged to a single DF per VLAN.

## Question 3
> **Scenario:** You're designing a data center fabric with EVPN. The customer requires sub-second convergence for link failures on multi-homed servers. Which specific EVPN mechanisms and timer tunings would you recommend?

### Answer

For **sub-second convergence** in EVPN multi-homing, you need to optimize both **detection** and **recovery** mechanisms.

### Recommended mechanisms

#### 1. BFD for fast failure detection

**Why:** Default BGP hold-down timers (180s/60s) are too slow. BFD provides **millisecond-level** failure detection.

**Configuration:**

**Junos:**
```junos
protocols {
    bgp {
        group EVPN {
            neighbor 10.0.0.2 {
                bfd-liveness-detection {
                    minimum-interval 300;
                    multiplier 3;
                }
            }
        }
    }
}
```

**IOS-XR:**
```cisco
router bgp 65000
 neighbor 10.0.0.2
  bfd fast-detect
  bfd minimum-interval 300
  bfd multiplier 3
```

**Result:** ~900ms failure detection (300ms × 3).

For sub-second, tune more aggressively:
- `minimum-interval 100`
- `multiplier 3`
- **Result:** ~300ms detection

#### 2. Mass withdrawal via Route Type 1

**Why:** Without mass withdrawal, MACs age out individually (default 300s). With mass withdrawal, all MACs for a failed ESI are removed immediately when RT-1 is withdrawn.

**Mechanism:**
- PE detects failure (BFD, LACP, link down)
- PE withdraws RT-1 (EAD per-ES and per-EVI)
- Remote PEs immediately flush all MACs for that ESI
- Convergence: **sub-second**

**Verification:**

**Junos:**
```junos
show evpn ethernet-segment detail
show evpn database mac-address | match <esi>
```

#### 3. Aliasing for pre-established ECMP

**Why:** With aliasing (RT-1), remote PEs already have ECMP paths to all PEs in the ES. When one fails, traffic shifts to remaining paths without reconvergence.

**Requirement:** Ensure RT-1 routes are advertised and imported.

**Junos:**
```junos
show route table bgp.evpn.0 match 1:...:0
```

#### 4. MAC Mobility sequence for rapid MAC moves

**Why:** If a server MAC moves between PEs (e.g., VM migration), MAC Mobility (RT-2 with sequence number) ensures rapid update.

**Default:** MAC moves are accepted immediately if sequence increases.

**Tuning:** Reduce duplicate-MAC freeze timer if needed (default 30s).

**Junos:**
```junos
routing-instances {
    EVPN-1 {
        protocols {
            evpn {
                duplicate-mac-detection {
                    auto-recovery-time 30;
                }
            }
        }
    }
}
```

#### 5. LACP fast timers for CE-side detection

**Why:** If the CE uses LACP to detect PE failure, standard LACP (30s/90s) is too slow.

**Configuration:**

**Junos:**
```junos
interfaces {
    ae0 {
        aggregated-ether-options {
            lacp {
                fast-hello;
                periodic fast;
            }
        }
    }
}
```

**IOS-XR:**
```cisco
interface Bundle-Ether1
 lacp period short
```

**Result:** LACP detection in ~1 second instead of 30+ seconds.

---

### Timer tuning summary

| Mechanism | Default | Sub-second tuning |
|-----------|---------|-------------------|
| BGP hold/keepalive | 180s/60s | BFD 100ms × 3 |
| LACP | 30s slow | LACP fast (1s) |
| MAC aging | 300s | Mass withdrawal (immediate) |
| Duplicate MAC freeze | 30s | Keep default or reduce to 10s |

---

### What NOT to rely on

- **BGP graceful restart:** Increases convergence time, not decreases it
- **BGP PIC:** Helps with prefix-independent convergence but not for EVPN MAC routes
- **MAC aging alone:** Too slow (300s default)

---

### Best exam answer

> For sub-second convergence in EVPN multi-homing, I recommend: (1) **BFD** with aggressive timers (100ms × 3) for fast failure detection; (2) **mass withdrawal via Route Type 1** for immediate MAC flush on failure; (3) **aliasing** pre-established via RT-1 for instant ECMP shift; (4) **LACP fast timers** for CE-side detection; and (5) tuned **MAC Mobility** with appropriate duplicate-MAC handling. The key is optimizing both detection (BFD/LACP) and recovery (mass withdrawal/aliasing) — not just one or the other.
