## Question 1
> You have a route-policy update that passes lint and simulation but fails in ring-1 due to unexpected route drops. What additional CI checks would you add to prevent this class of failure?

### Answer

The policy was syntactically valid (passed lint) and the simulation said it was fine. But simulation doesn't equal production, because the simulation didn't test with **real route data**.

**Additional CI checks:**

**1. Route-policy unit tests with production route snapshots.**

Capture a snapshot of actual routes hitting the policy (BGP RIB dump, filtered to the relevant address family and neighbor). Feed those routes through the updated policy in a test harness. Compare the output (accepted/rejected/modified routes) against the previous policy version. Any route that was previously accepted and is now rejected = a red flag requiring human review.

```yaml
# Example test case
test_name: "customer-X-import should accept 10.100.0.0/16"
input_route:
  prefix: 10.100.0.0/16
  as_path: "65001 65100"
  communities: ["65000:100", "65000:200"]
expected: accept
expected_local_pref: 200
```

**2. Diff-based impact analysis.**

Before pushing to ring-1, compute the exact set of routes that would be affected by the policy change. Run `show bgp policy <old-policy>` vs `show bgp policy <new-policy>` against the current RIB on a canary device (or a Batfish simulation with real routes). The diff tells you exactly which prefixes change behavior. If the diff is empty = safe. If the diff shows customer prefixes being dropped = investigate.

**3. Route count assertion gates.**

After deploying to ring-1, the post-deploy health check should assert that the BGP route count on the device stays within ±5% of the pre-deploy count. A sudden 20% drop in accepted routes = the policy is filtering too aggressively. This catches broad route drops that unit tests might miss if they don't cover every prefix.

**4. Customer-impact mapping.**

Cross-reference affected routes with the customer database. If the policy change affects routes in VRF-CustomerX, flag it as high-impact and require explicit approval from the customer-facing team. This isn't a technical check — it's a workflow check that prevents "we changed a policy and didn't realize it fed customer traffic."

**Bottom line:** Lint catches syntax. Simulation catches logic errors against synthetic data. What catches production-specific failures is **testing with real production data**. Add that step and this class of bug dies.

## Question 2
> Why is canary-by-role superior to canary-by-random-device in an SP network?

### Answer

An SP network has fundamentally different device roles with fundamentally different configurations, traffic patterns, and failure modes. A PE, P-router, route reflector, and border router all run different feature sets. A config change that works perfectly on a P-router might destroy a PE.

**Canary-by-random-device:**
- Pick 5 random devices from 2,000 → push config → validate
- Problem: all 5 might be P-routers. Your change has a PE-specific bug. You promote to ring-1 and 400 PEs break.
- Problem: none of the 5 might be in the region with the unusual topology feature (dual-homed eBGP, inter-AS link, peering edge). Your change works in 90% of the network and fails in the 10% that matters most.

**Canary-by-role:**
- Pick 1 PE, 1 P-router, 1 RR, 1 border router, 1 per unique topology role → push config → validate
- Each role exercises the config change against its specific feature set
- A PE canary tests VPN import/export policies. A P-router canary tests label forwarding. An RR canary tests route reflection with the full scale of BGP state.

**Additional reasons canary-by-role wins:**

1. **Different software versions.** SPs often run different IOS-XR or Junos versions across roles. A syntax that works on the P-router's software version might fail on the PE's older version.

2. **Different scale profiles.** An RR with 500 BGP peers responds to config changes differently than a PE with 3 peers. Resource-intensive changes (new route-policy with complex regex) might be fine on a PE and cause CPU spikes on an RR.

3. **Different failure blast radius.** A broken P-router affects all transit traffic. A broken PE affects one customer. A broken RR affects all PEs. The canary should test the highest-blast-radius role first, not discover it in ring-2.

4. **Different traffic patterns.** A border router handling 500Gbps of internet peering exercises QoS policies differently than a PE handling 10Gbps of VPN traffic. The same QoS config change can be fine on one and catastrophic on the other.

**The rule:** Your canary set must include one device from each unique operational role. If you have 8 roles and 2,000 devices, your ring-0 is 8 devices, not 5 random ones.

## Question 3
> A team relies on `commit confirmed` but still experienced a prolonged outage. What operational gaps could explain this?

### Answer

`commit confirmed` is a great safety net, but it's not a parachute that opens itself. Several real-world gaps can defeat it:

**Gap 1: The confirm-timeout was too short AND the team manually confirmed before validating.**

If the timeout is 60 seconds and the team's muscle memory is "commit, immediately confirm, then check" — they confirmed a broken change before the health checks ran. The rollback timer never fired because they disabled it with the premature confirming commit.

**Fix:** Set the confirm-timeout to 5–10 minutes minimum. Do NOT confirm until automated health checks pass. Train the team to treat the unconfirmed period as a mandatory validation window, not an annoyance to dismiss.

**Gap 2: The change broke out-of-band management, not in-band.**

`commit confirmed` rolls back if the confirming `<commit>` isn't received in time. But the router's management plane (SSH/NETCONF) might be fine while the data plane is broken. The team can connect, confirm, and make the change permanent — all while customer traffic is blackholed on the data plane.

**Fix:** Health checks must validate data plane health (traffic counters, BFD, ping from diverse sources), not just management plane reachability. If you can SSH to the router but customers can't route through it, `commit confirmed` doesn't help — you'll confirm a broken commit.

**Gap 3: The change was applied to multiple devices, and each one individually was fine.**

Each device's `commit confirmed` works perfectly. But the aggregate effect — 50 PEs all changing a route-policy simultaneously — causes a network-wide routing event that no single device's rollback can fix. The BGP route leak or withdrawal storm has already propagated beyond the changed devices.

**Fix:** Staged rollout (rings). Never push the same change to all devices within the same confirm-timeout window. Ring-0 first, wait, validate network-wide impact, then ring-1.

**Gap 4: Rollback itself caused an outage.**

The auto-rollback after timeout fires, reverting the config. But the rollback causes a process restart, which flaps BGP sessions, which causes route withdrawals, which causes a brief outage on top of the original issue. The team now has TWO incidents: the broken commit AND the rollback-induced flap.

**Fix:** Understand the rollback behavior of your platform. On IOS-XR, rollback is a full config replace, which may restart processes. On Junos, rollback is a `load override` + `commit`, which is gentler but still causes a commit event. For high-risk changes, extend the confirm-timeout enough to allow manual rollback (which can be done more surgically) instead of relying on the automatic rollback.

**Gap 5: Nobody was watching.**

The change was pushed by an automated pipeline at 3 AM. `commit confirmed` had a 10-minute timeout. The health check had a bug and returned "healthy" for all conditions. The confirming commit was sent. At 7 AM, the NOC noticed customers complaining. 4 hours of outage, despite `commit confirmed` being configured.

**Fix:** Never automate the confirming commit without robust, tested health validation. If the health check can't be trusted, don't auto-confirm — require human approval for the confirming commit during the first deployment cycle.

## Question 4
> How would you design telemetry promotion gates for SR-TE and L3VPN changes so that protocol-state-only false positives are reduced?

### Answer

**The false positive problem:** If your promotion gate is "all BGP sessions are Established," you'll block promotions every time a BGP peer flaps for unrelated reasons — maintenance elsewhere, a customer bouncing their CPE, or a transient TCP reset. The change you're deploying had nothing to do with the flap, but the gate says "FAIL" and you're stuck.

**Design principle:** Gates should measure **change-correlated health**, not absolute health.

**Gate design for SR-TE changes:**

| Gate | Metric | Pass Criteria | Why Not False Positive |
|------|--------|---------------|----------------------|
| 1 | SR-TE policy state | All policies with `color` matching the change are `Active` | Directly tests the changed policies, not all policies |
| 2 | SR-TE forwarding stats | Per-policy byte counters are non-zero AND trending consistent with pre-change baseline | Ensures traffic is actually flowing through the changed policies |
| 3 | PCE delegation status | All changed policies remain delegated (not fallen back to local) | Verifies the PCE accepted the new constraints |
| 4 | Candidate path preference | The intended candidate path is installed (not a fallback) | Confirms the right segment list is active |
| 5 | BFD sessions on SR-TE paths | BFD sessions for changed policy endpoints are UP | Validates end-to-end data plane reachability through the policy |

**What NOT to gate on for SR-TE:** Total IS-IS adjacency count, unrelated BGP peer count, or device CPU. These are background health metrics, not change-correlated gates.

**Gate design for L3VPN changes:**

| Gate | Metric | Pass Criteria | Why Not False Positive |
|------|--------|---------------|----------------------|
| 1 | VRF route count | Route count in affected VRFs is within ±5% of pre-change baseline | Catches route drops from policy errors. The ±5% tolerance absorbs normal churn. |
| 2 | PE-CE BGP session state | BGP sessions in the changed VRF are Established | Only checks the specific VRF being changed, not all VRFs |
| 3 | VRF traffic counters | Per-VRF byte counters are non-zero and trending within ±20% of pre-change rate | Validates actual forwarding, not just control plane |
| 4 | Customer-specific probes | ICMP/BFD probes from PE to known CE addresses in the changed VRF | End-to-end data plane validation specific to the change |
| 5 | RT import/export verification | Expected RT values present in the VRF config AND routes with those RTs are in the VRF RIB | Catches RT misconfiguration that would cause route leaking or missing routes |

**What NOT to gate on for L3VPN:** Global BGP table size, LDP session count, or IS-IS adjacency count. These don't correlate with VPN-specific changes.

**Implementation pattern:**

```python
def evaluate_gate(pre_snapshot, post_snapshot, changed_resources):
    for resource in changed_resources:
        delta = post_snapshot[resource] - pre_snapshot[resource]
        if abs(delta) > threshold[resource.type]:
            return GateResult.FAIL, f"{resource} changed by {delta}"
    return GateResult.PASS, "all within tolerance"
```

The key insight: **take a pre-change snapshot of the specific metrics you care about**, then compare post-change. This "delta-based" approach eliminates false positives from pre-existing conditions and unrelated events.

## Question 5
> Your network has heavy manual drift from emergency CLI changes. What sequence of controls would you implement to make CI/CD trustworthy again?

### Answer

You can't deploy CI/CD on top of config drift. If Git says one thing and the router says another, every automated deploy is a gamble. You need to reconcile the drift before CI/CD can be trusted.

**Sequence of controls (in order — don't skip steps):**

**Phase 1: Inventory and baseline (Week 1–2)**

1. **Full config backup of every device.** Pull `running-config` from all devices, store in a "reality" branch in Git. This is the actual state of the network, not the desired state.

2. **Diff reality vs intended.** Compare the pulled configs against whatever your source-of-truth claims the configs should be. Categorize the drift:
   - **Harmless drift:** Banner text, NTP server reordering, comment changes
   - **Intentional drift:** Emergency fixes that were correct but never backported to the SoT
   - **Dangerous drift:** Unknown changes that nobody can explain

3. **Reconcile.** For each drift category:
   - Harmless: Update SoT to match reality
   - Intentional: Review the emergency fix, validate it was correct, backport to SoT
   - Dangerous: Investigate. If nobody knows why it's there, document it, mark it for review, and don't touch it until you understand the impact

**Phase 2: Lock the front door (Week 3–4)**

4. **Enable configuration change logging.** Every device must log who changed what and when: `configuration commit auto-comment`, syslog for CLI changes, TACACS+ command accounting. You need to catch new drift as it happens.

5. **Config drift detection.** Scheduled job (hourly or daily) pulls running configs and diffs against SoT. Any deviation triggers an alert to the team. Don't auto-remediate yet — just detect and alert.

6. **Emergency change process.** Formalize it: emergency CLI changes are allowed, but the engineer must file a ticket within 24 hours documenting the change, and the change must be backported to the SoT within 48 hours. This legitimizes emergency changes while preventing them from becoming permanent drift.

**Phase 3: Build CI/CD trust (Week 5–8)**

7. **CI pipeline for all config changes.** Every change goes through Git PR → lint → simulation → peer review. No exceptions for "small changes" — that's where drift starts.

8. **Start with low-risk changes.** First CI/CD deploys should be things like NTP server updates, SNMP community changes, or banner updates. Build team confidence that the pipeline works before pushing BGP policies through it.

9. **Canary deployment with `commit confirmed`.** Every change deploys to ring-0 first with a 10-minute confirm timer. Automated health checks must pass before confirmation. Team observes the process for a few weeks.

**Phase 4: Enforce (Week 9+)**

10. **Config compliance enforcement.** The drift detection job now has teeth: any deviation from SoT that isn't filed as an emergency change ticket triggers automatic escalation to the team lead.

11. **Lock CLI access.** Reduce CLI write access to emergency-only. Regular changes must go through the pipeline. TACACS+ authorization can enforce this by denying `configure terminal` for non-emergency accounts.

12. **Continuous reconciliation.** Monthly "config audit" where the team reviews any remaining drift, reconciles it, and updates the SoT. This prevents slow-drip drift from re-establishing.

**The cultural change is harder than the technical change.** Engineers who've spent 20 years fixing networks via CLI will resist. The answer isn't "you can't use CLI anymore" — it's "CLI is for emergencies, and emergencies have a backport requirement." Give them an escape hatch so they don't go around the process.
