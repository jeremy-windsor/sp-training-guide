# 11.1 — Model-Driven Networking — Protocol Theory

## Protocol Overview

Model-driven networking replaces CLI screen-scraping and vendor-specific management with structured data models that define what a network device can do, and standardized protocols that interact with those models programmatically. The three pillars: **YANG** (the modeling language), **NETCONF** (the configuration protocol), and **gRPC/gNMI** (the high-performance telemetry and config interface).

YANG (RFC 7950, 2016) is a data modeling language that defines the structure, constraints, and semantics of configuration and operational state data. It doesn't carry data — it describes what valid data looks like. Think of YANG as the schema; NETCONF/gNMI as the protocols that read and write data conforming to that schema.

NETCONF (RFC 6241, 2011) is the configuration management protocol. It runs over SSH, uses XML-encoded RPCs, and provides transactional configuration operations (get, edit-config, commit, discard-changes, validate) against YANG-modeled datastores.

gRPC/gNMI is the modern alternative optimized for high-throughput telemetry streaming and programmatic configuration. It uses Protocol Buffers (protobuf) encoding, HTTP/2 transport, and supports streaming subscriptions for real-time state data.

Together, these technologies replace the CLI paradigm that has dominated network management for 30 years. The CLI was designed for humans typing commands. YANG/NETCONF/gNMI were designed for software managing networks.

## Core Mechanisms

### 1. YANG Data Modeling Language (RFC 7950)

YANG defines the structure of network device data using a hierarchical tree model.

**YANG tree structure:**

```
module ietf-interfaces {
  container interfaces {           ← container: grouping node
    list interface {               ← list: multiple instances (keyed)
      key "name";
      leaf name {                  ← leaf: single value
        type string;
      }
      leaf enabled {
        type boolean;
        default true;
      }
      leaf mtu {
        type uint16;
      }
      container statistics {       ← nested container
        leaf in-octets {
          type uint64;
        }
        leaf out-octets {
          type uint64;
        }
      }
    }
  }
}
```

**Key YANG node types:**

| Node Type | Purpose | Example |
|-----------|---------|---------|
| container | Grouping node, no value itself | `interfaces`, `statistics` |
| list | Keyed collection of entries, each identified by key(s) | `interface` (keyed by `name`) |
| leaf | Single data value | `mtu`, `enabled` |
| leaf-list | Collection of scalar values (ordered-by system default) | `dns-server` (list of IPs) |
| choice/case | Mutually exclusive options | `address-type: {ipv4 | ipv6}` |
| augment | Extends another module's tree | Vendor adds leaves to IETF model |
| grouping/uses | Reusable template of nodes | Common interface counters |

**YANG modules and namespaces**: Each YANG model is a module with a unique namespace (XML namespace URI). Models can import other models. The IETF publishes standard models (e.g., `ietf-interfaces`, `ietf-routing`). OpenConfig publishes vendor-neutral operator models. Vendors publish native models for platform-specific features.

**YANG constraints:**
- `type`: data type (string, uint32, boolean, enumeration, union, etc.)
- `mandatory`: must be present in configuration
- `must`: XPath expression that must evaluate to true (e.g., MTU must be ≥ 68)
- `when`: conditional presence (node only exists when condition is met)
- `min-elements/max-elements`: list entry count constraints

**Config vs state data**: YANG distinguishes between configuration data (read-write, what the operator sets) and operational state data (read-only, what the device reports). A `config false` statement marks a node as state-only. This separation enables the NETCONF concept of separate configuration and operational datastores.

### 2. NETCONF Protocol (RFC 6241)

NETCONF provides secure, transactional configuration management over SSH.

**NETCONF layers:**

```
┌────────────────────────────┐
│  Content (YANG data/XML)    │  ← What is being configured
├────────────────────────────┤
│  Operations (get, edit-config, commit, etc.) │  ← What action to take
├────────────────────────────┤
│  Messages (RPC, RPC-reply, notification)     │  ← Request/response framing
├────────────────────────────┤
│  Transport (SSH, port 830)  │  ← Secure channel
└────────────────────────────┘
```

**NETCONF operations:**

| Operation | Purpose |
|-----------|---------|
| `get` | Retrieve operational state and configuration data |
| `get-config` | Retrieve configuration data only (from a specific datastore) |
| `edit-config` | Modify configuration (merge, replace, create, delete, remove) |
| `copy-config` | Copy entire configuration between datastores |
| `delete-config` | Delete a configuration datastore |
| `lock / unlock` | Lock a datastore to prevent concurrent edits |
| `commit` | Apply candidate configuration to running (requires :candidate capability) |
| `discard-changes` | Discard uncommitted candidate changes |
| `validate` | Validate configuration without applying (requires :validate capability) |
| `close-session` | Gracefully close the NETCONF session |

**Datastores:**
- **running**: The active configuration currently applied to the device.
- **candidate**: A scratch configuration that can be edited without affecting the running config. Changes are applied atomically via `commit`. This provides transaction semantics.
- **startup**: The configuration loaded at boot. `copy-config running startup` saves the running config.

**NETCONF capabilities**: During session establishment, client and server exchange `<hello>` messages listing their capabilities (feature support). Capabilities include: `:candidate` (candidate datastore), `:confirmed-commit` (commit with automatic rollback timer), `:rollback-on-error`, `:validate`, `:xpath` (XPath filtering), `:with-defaults`.

**NETCONF message framing**: NETCONF messages are XML documents exchanged over SSH. Message boundaries are delimited by `]]>]]>` (NETCONF 1.0) or chunked framing with length prefixes (NETCONF 1.1, RFC 6242). NETCONF 1.1 is preferred — the old delimiter could appear in data.

### 3. YANG Model Ecosystem

**Three model families:**

**IETF models**: Standards-track models published as RFCs. Vendor-neutral. Cover core networking constructs.
- `ietf-interfaces` (RFC 8343): Interface configuration and state
- `ietf-ip` (RFC 8344): IPv4/IPv6 configuration
- `ietf-routing` (RFC 8349): Routing framework
- `ietf-yang-library` (RFC 8525): Model discovery

**OpenConfig models**: Operator-driven, vendor-neutral models published by the OpenConfig consortium (Google, AT&T, Microsoft, etc.). More opinionated than IETF models — designed for operational automation at scale.
- `openconfig-interfaces`: Interface config/state
- `openconfig-network-instance`: VRF, routing instances
- `openconfig-bgp`: BGP configuration
- `openconfig-telemetry`: Streaming telemetry

**Vendor-native models**: Platform-specific models covering features unique to a vendor's implementation. Most comprehensive coverage but not portable across vendors.
- Cisco: `Cisco-IOS-XR-*`, `Cisco-IOS-XE-*`
- Juniper: `junos-conf-*`, `junos-state-*`
- Nokia: `nokia-conf-*`, `nokia-state-*`

**Model discovery**: Devices advertise their supported YANG modules via `ietf-yang-library` (NETCONF) or gNMI Capabilities RPC. Automation tools discover available models before attempting configuration.

### 4. gRPC and gNMI

gRPC (Google Remote Procedure Call) is a high-performance RPC framework using HTTP/2 and Protocol Buffers.

**gNMI (gRPC Network Management Interface):**
gNMI is a gRPC-based protocol for network configuration and telemetry, defined by OpenConfig.

**gNMI operations:**

| RPC | Purpose |
|-----|---------|
| `Capabilities` | Discover supported models, encodings, gNMI version |
| `Get` | Retrieve configuration or state data at a path |
| `Set` | Modify configuration (update, replace, delete) |
| `Subscribe` | Stream telemetry data |

**gNMI paths**: gNMI uses structured paths (not XPath) to address YANG tree nodes:
```
/interfaces/interface[name=eth0]/state/counters/in-octets
/network-instances/network-instance[name=default]/protocols/protocol[identifier=BGP][name=bgp]/bgp/neighbors/neighbor[neighbor-address=10.0.0.1]/state
```

**Why gNMI over NETCONF:**
- **Performance**: protobuf encoding is more compact than XML. HTTP/2 multiplexing is faster than SSH.
- **Streaming**: gNMI Subscribe provides real-time telemetry streaming. NETCONF notifications exist but are less mature.
- **Simplicity**: gNMI has 4 RPCs. NETCONF has 12+ operations plus capabilities negotiation.
- **Trade-off**: NETCONF has richer transactional semantics (candidate datastore, lock, confirmed commit). gNMI Set is simpler but less feature-rich.

### 5. XPath and Filtering

NETCONF supports subtree filtering by default for `get` and `get-config` operations. XPath filtering is optional and requires the server to advertise the `:xpath` capability.

**Subtree filtering**: The default NETCONF filter. Specify an XML subtree; the device returns matching data. Simple but limited — can't express conditions like "all interfaces with MTU > 1500."

**XPath filtering**: More powerful. Requires the `:xpath` capability. Can express complex conditions:
```
/interfaces/interface[mtu > 1500]
/routing/bgp/neighbors/neighbor[state/session-state = 'ESTABLISHED']
```

**gNMI filtering**: gNMI uses path-based filtering inherently. Each Get or Subscribe targets a specific path. Wildcards are supported:
```
/interfaces/interface[name=*]/state/oper-status
```

## Protocol Interactions

**YANG ↔ NETCONF/gNMI**: YANG defines the data model. NETCONF and gNMI are the protocols that access YANG-modeled data. YANG is protocol-independent — the same model works with NETCONF (XML), gNMI (protobuf), or RESTCONF (JSON/XML).

**NETCONF ↔ SSH**: NETCONF runs over SSH (port 830). SSH provides authentication, encryption, and integrity. NETCONF's security model inherits SSH's key/password authentication.

**gNMI ↔ gRPC ↔ HTTP/2**: gNMI is defined as a gRPC service. gRPC uses HTTP/2 for transport and protobuf for serialization. TLS provides security.

**Model-driven networking ↔ Automation**: YANG models are the foundation for network automation tools (Ansible, Nornir, Terraform). The structured data model eliminates screen-scraping and enables idempotent configuration management.

## Edge Cases & Gotchas

**1. Model coverage gaps**: Vendor-native YANG models may not cover 100% of CLI features. Some features are CLI-only with no YANG model. Always verify model coverage for your specific features before committing to model-driven management.

**2. OpenConfig vs native model conflicts**: A device may support both OpenConfig and native models for the same feature. Configuring via one model and reading via the other can produce inconsistent views. Pick one model family and stick with it.

**3. NETCONF candidate commit failure**: If `commit` fails (validation error), the candidate datastore retains the failed changes. You must `discard-changes` before retrying. Forgetting this leaves stale config in the candidate.

**4. gNMI encoding differences**: gNMI supports multiple encodings (protobuf, JSON, JSON_IETF). Not all devices support all encodings. A client requesting protobuf encoding from a device that only supports JSON_IETF will fail.

**5. XPath performance**: Complex XPath filters can be computationally expensive on the device. Filtering `/interfaces/interface[statistics/in-errors > 0]` requires the device to evaluate every interface's error counters. On platforms with thousands of interfaces, this can be slow.
