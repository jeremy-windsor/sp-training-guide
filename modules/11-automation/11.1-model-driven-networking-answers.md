## Question 1
> **Scenario:** Your automation framework pushes BGP neighbor configs to 50 PE routers using NETCONF `<edit-config>` with `merge` operation. A decommissioned customer's BGP session is removed from your source-of-truth, but the session persists on the routers. Why? What operation should you use instead, and what are the risks of switching?

### Answer

**Why the session persists:** `merge` is additive. It adds or modifies elements that exist in the payload but **never removes** elements that are absent from the payload. When you delete the customer from your source-of-truth and re-push, the payload simply doesn't include that neighbor anymore. `merge` sees nothing to do for that neighbor and leaves it alone. The zombie BGP session lives forever.

This is the most common NETCONF automation mistake in production. It's also why "desired state" frameworks (NSO, Terraform) exist.

**What to use instead: `replace`**

`<edit-config>` with `operation="replace"` on the BGP neighbor list container. `replace` says "the payload IS the complete desired state — anything on the device that's not in the payload should be removed."

```xml
<edit-config>
  <target><candidate/></target>
  <config>
    <bgp xmlns="...">
      <neighbors xmlns:nc="urn:ietf:params:xml:ns:netconf:base:1.0" nc:operation="replace">
        <!-- Only active neighbors listed here -->
        <neighbor>
          <address>10.0.0.1</address>
          ...
        </neighbor>
        <!-- Decommissioned neighbor NOT listed = gets deleted -->
      </neighbors>
    </bgp>
  </config>
</edit-config>
```

**Risks of switching to `replace`:**

1. **Accidental mass deletion.** If your source-of-truth has a bug and generates an empty neighbor list, `replace` will delete ALL BGP sessions on all 50 PEs. With `merge`, a broken payload does nothing (additive of nothing = no change). With `replace`, a broken payload is a nuke. This is the "replace footgun."

2. **Scope matters enormously.** `replace` at the `<neighbors>` container level replaces all neighbors. `replace` at a specific `<neighbor>` entry level replaces just that neighbor's config. If you accidentally scope `replace` at the entire `<bgp>` container, you replace the entire BGP configuration — including address families, policies, and global settings.

3. **Ordering and dependencies.** If Neighbor A has a route-policy that references a prefix-set, and your replace payload removes the prefix-set (because it's defined elsewhere in your pipeline), the commit fails with a dependency error. `merge` never hit this because it never removed anything.

4. **Partial payload risk.** If your automation generates configs per-feature (one push for BGP neighbors, another for policies), a `replace` on neighbors must include ALL neighbors, not just the ones being changed. If Feature-Pipeline-A owns neighbors and Feature-Pipeline-B also pushes neighbors, they'll clobber each other with `replace`.

**The safe pattern:** Use `replace` with guardrails:
- Always validate the payload isn't empty before pushing (len(neighbors) > 0)
- Use `commit confirmed` with a 120-second timer — if you lose connectivity (because you just deleted the management BGP session), the config auto-rolls back
- Scope `replace` as narrowly as possible — per-neighbor if you can, per-container only when you need full reconciliation
- Run a pre-push diff (`get-config` → compare with intended payload) and alert if >10% of entries would be deleted

## Question 2
> **Design:** You're designing the management plane for a greenfield SP backbone (IOS-XR and Junos mixed). You need configuration management, real-time telemetry for traffic engineering, and a NOC dashboard. Which protocol (NETCONF/RESTCONF/gNMI) would you select for each use case, and why?

### Answer

| Use Case | Protocol | Why |
|----------|----------|-----|
| **Configuration management** | **NETCONF** | Transactional commits, candidate datastore, `commit confirmed`, rollback — none of the others match NETCONF's config safety features. Both IOS-XR and Junos have mature NETCONF implementations with full feature coverage. RESTCONF is simpler but lacks candidate/confirmed-commit on some platforms. gNMI Set works but has weaker transactional semantics. |
| **Real-time telemetry for TE** | **gNMI** | Streaming telemetry is gNMI's superpower. `SAMPLE` mode for interface counters at 10–30 second intervals, `ON_CHANGE` for BGP state and SR-TE policy changes. gNMI uses gRPC (HTTP/2 + protobuf) which is more efficient than NETCONF's XML-over-SSH for high-frequency data. Both IOS-XR (MDT) and Junos (JTI/OpenConfig) support gNMI streaming. |
| **NOC dashboard** | **gNMI → Prometheus/InfluxDB → Grafana** | The dashboard itself doesn't speak to routers — it reads from a time-series database. gNMI feeds gnmic or telegraf collectors, which write to Prometheus or InfluxDB, which feeds Grafana. The protocol choice here is really "what feeds the TSDB," and gNMI wins for efficiency and real-time capability. |

**Why NOT use one protocol for everything:**

- NETCONF is great for config but terrible for telemetry. XML parsing overhead at high frequency is painful, and NETCONF `<get>` is poll-based (no push/streaming).
- gNMI is great for telemetry but its `Set` operation lacks NETCONF's transactional depth (no candidate datastore, no confirmed commit in the gNMI spec — though some implementations add it).
- RESTCONF is the jack of all trades, master of none. Good for lightweight integrations and one-off queries, but neither the best config protocol nor the best telemetry protocol.

**Multi-vendor considerations:**

| Feature | IOS-XR | Junos |
|---------|--------|-------|
| NETCONF | Full (RFC 6241, candidate, confirmed commit) | Full (native candidate, commit confirmed) |
| gNMI | Full (MDT, OpenConfig + native models) | Full (JTI, OpenConfig) |
| RESTCONF | Supported but less common in SP deployments | Junos REST API available (not RFC 8040 RESTCONF — verify platform/release for standards compliance) |
| OpenConfig models | Wide coverage | Wide coverage |
| Native YANG models | `Cisco-IOS-XR-*` | `junos-conf-*` |

**Design recommendation:** NETCONF for config (with OpenConfig models where possible, native where needed), gNMI for telemetry, and accept that you're running two protocol stacks. The operational cost of two protocols is far less than the cost of using the wrong protocol for a critical function.

## Question 3
> **Troubleshooting:** A NETCONF session to a PE router hangs on `<commit>` for over 60 seconds and eventually times out. The candidate datastore has valid XML (passes `<validate>`). What are the likely causes, and how would you diagnose the issue?

### Answer

`<validate>` checks syntax and schema. `<commit>` applies the change to the running datastore, which involves process communication, hardware programming, and potentially distributed commits across line cards. That's where things stall.

**Likely causes, ordered by frequency:**

**1. Configuration lock contention.**

Another NETCONF session, CLI user, or system process holds a lock on the running datastore. Your `<commit>` is waiting for the lock to release. On IOS-XR, `show configuration lock` tells you who holds it. On Junos, `show system commit` or `show configuration users`.

Common culprits:
- A stuck NETCONF session from a previous automation run that didn't clean up
- An engineer logged in via CLI with uncommitted changes in `configure exclusive` mode
- NSO or another orchestrator holding a persistent session with a config lock

**Diagnosis:** From CLI, run `show configuration lock` (IOS-XR) or check `show system users` (Junos). Kill the blocking session.

**2. Large commit touching many objects.**

If the candidate has thousands of changes (e.g., replacing all prefix-sets, updating 500 BGP neighbors), the commit process takes time to program each change into the FIB, policy engine, and line cards. This is especially slow on older hardware or distributed platforms where line card commits are serialized.

**Diagnosis:** Check the size of the candidate diff (`show configuration changes diff` on Junos, or `show configuration commit changes last 1 diff` after the fact). If it's huge, break the change into smaller batches.

**3. Process restart triggered by the config change.**

Certain config changes trigger process restarts — changing the router-id, modifying ISIS/OSPF instance parameters, or enabling new features. The commit hangs while waiting for the process to restart and re-register.

**Diagnosis:** Check syslog during the commit window. Look for process restart messages (`show logging | include restart|respawn|kill`).

**4. Commit verification failure (post-apply check).**

IOS-XR performs post-commit verification to ensure the running datastore is consistent. If a line card fails to apply the change (hardware error, resource exhaustion), the commit hangs waiting for the line card ACK.

**Diagnosis:** Check line card status (`show platform`), look for OOR (Out of Resource) conditions on the hardware (`show controllers fia diagshell`).

**5. CDB (Configuration DataBase) contention on Junos.**

Junos stores config in a SQLite-based CDB. If the CDB is fragmented or the disk is slow (common on older RE/RE-x86 platforms), commit time increases. This is especially noticeable after many incremental commits without a `request system configuration rescue save`.

**Diagnosis:** `show system storage` — check /config partition utilization. `show system commit` — check if recent commit times are trending upward.

**General diagnostic approach:**

1. Open a parallel CLI session to the same router
2. Check for config locks
3. Check syslog for error/warning messages during the commit window
4. Check process health (`show processes | include bgp|isis|policy`)
5. Check line card health
6. If the commit eventually succeeds, check `show configuration commit list detail` for the commit duration and any warnings

**Prevention:** Always use NETCONF session keepalives and set reasonable timeouts. If your automation expects commits to complete in <30 seconds and they start taking >60 seconds, that's a signal to investigate — don't just increase the timeout.

## Question 4
> **Comparison:** Compare `ON_CHANGE` and `SAMPLE` gNMI subscription modes for monitoring BGP neighbor state on a route reflector with 500 peers. Which mode is appropriate, what's the resource impact of each, and what failure mode does each have?

### Answer

**`ON_CHANGE` — Event-driven, push when state changes:**

The router sends a gNMI notification only when a BGP neighbor's state actually changes (Established → Idle, or vice versa). On a stable route reflector with 500 peers, this produces almost zero traffic during steady state. When a peer flaps, you get a notification within milliseconds.

| Dimension | ON_CHANGE |
|-----------|-----------|
| Steady-state load | Near zero — 500 stable peers generate no updates |
| Event latency | Milliseconds — immediate notification on state change |
| Bandwidth | Minimal — only changed state transmitted |
| Router CPU impact | Low during steady state. Moderate during mass flap (500 peers bouncing = 500 rapid notifications) |
| Best for | State changes, fault detection, alarm triggering |

**Failure mode:** **Missed initial state.** If the gNMI subscription starts after the peer is already down, you never get an ON_CHANGE notification because nothing changed from the subscription's perspective. You need an initial sync (which gNMI provides as the first update in a subscription), and you need to handle reconnection carefully — re-subscribe after a collector restart and process the initial sync to avoid stale state.

**Second failure mode:** **Notification storm.** If 200 BGP peers flap simultaneously (route reflector loses an upstream link), you get 200+ rapid-fire notifications. Your collector pipeline needs to handle bursts without dropping events. If it can't, you miss state transitions.

**`SAMPLE` — Periodic polling:**

The router sends the complete BGP neighbor state table at fixed intervals (e.g., every 30 seconds) regardless of whether anything changed.

| Dimension | SAMPLE |
|-----------|--------|
| Steady-state load | Constant — 500 peers × 30s interval = ~17 updates/second |
| Event latency | Up to 1 full sample interval (worst case 30s to detect a peer down) |
| Bandwidth | High — full state table every interval, mostly redundant |
| Router CPU impact | Constant, proportional to peer count × sample frequency |
| Best for | Counters, metrics, utilization data — things that change continuously |

**Failure mode:** **Detection delay.** A peer goes down at T=1s, but your sample interval is 30s. You don't detect it until T=30s. For BGP neighbor state monitoring, 30 seconds is an eternity. You could reduce the interval to 5 seconds, but now you're pushing 100 updates/second from one RR with 500 peers — that's significant CPU and bandwidth overhead for mostly redundant data.

**Second failure mode:** **Counter aliasing.** If a peer flaps and recovers within one sample interval, you miss both the down and the up event. SAMPLE only shows point-in-time snapshots — it can't capture transient events.

**Which mode for BGP neighbor state on a 500-peer RR:**

**ON_CHANGE.** Unequivocally.

BGP neighbor state is a discrete state machine (Idle/Connect/Active/OpenSent/OpenConfirm/Established). It changes infrequently and you need to know about changes immediately. ON_CHANGE is designed exactly for this — low overhead during stability, instant notification during events.

SAMPLE mode for BGP state is like checking your smoke detector by walking through the house every 30 seconds instead of letting it alarm when it detects smoke. It wastes resources during normal operation and provides worse detection latency during actual events.

**When SAMPLE is the right choice:** Interface counters (`in-octets`, `out-octets`, utilization), optics PM (pre-FEC BER, power levels), and any continuously changing numeric metric. These change every instant — ON_CHANGE would fire constantly and overwhelm the collector. SAMPLE at 10–30 second intervals gives you trend data without notification storm.

## Question 5
> **Architecture:** A Tier-1 SP wants to implement "zero-touch provisioning plus confirmed commit" for 2,000 PE routers. They want every config change to auto-rollback if it causes a connectivity loss. Describe the workflow using NETCONF operations, including error handling for cases where the confirming NETCONF session itself is disrupted by the change.

### Answer

This is the hardest problem in network automation — how do you confirm a change worked when the change might break the path you'd use to confirm it?

**The workflow:**

**Step 1: Pre-flight checks.**
- Verify NETCONF reachability to the target PE
- Acquire candidate datastore lock (`<lock>`)
- Pull current running config backup (`<get-config>`)
- Load intended config into candidate (`<edit-config>`)
- Validate candidate (`<validate>`)
- Generate diff for audit trail

**Step 2: Confirmed commit with timeout.**
```xml
<commit>
  <confirmed/>
  <confirm-timeout>300</confirm-timeout>  <!-- 5 minutes -->
</commit>
```

This applies the change to running config BUT starts a rollback timer. If the automation doesn't send a confirming `<commit>` within 300 seconds, the router automatically reverts to the pre-change config.

**Step 3: Post-change health validation.**
- Wait 30 seconds for convergence
- Run health checks:
  - BGP neighbor count ≥ expected
  - IS-IS adjacency count = expected
  - BFD session count = expected
  - Interface up count = expected
  - Management reachability (ping/SSH from automation host)
- If ALL checks pass → send confirming `<commit>` to make the change permanent
- If ANY check fails → do nothing. The 5-minute timer will auto-rollback.

**Step 4: The confirming commit.**
```xml
<commit/>  <!-- No <confirmed/> tag = permanent commit -->
```

**The critical failure mode: The change breaks NETCONF reachability.**

Scenario: You push a config that changes the management VRF, ACL, or routing policy, which inadvertently blocks the NETCONF SSH session from the automation host to the PE. Your automation can't send the confirming `<commit>`. What happens?

**Answer: The router auto-rollbacks after 300 seconds.** This is exactly why `commit confirmed` exists. The timer is your dead man's switch. The change self-heals without human intervention.

But there are subtleties:

**Problem 1: The NETCONF session drops mid-commit.**

If the SSH session dies during the initial `<commit>` (not the confirming one), the commit may or may not have applied. You need to reconnect and check state:
- Can you reconnect? → Check `show configuration commit list` to see if the confirmed commit is pending
- Can't reconnect? → Wait for the timeout. If the commit applied with `confirmed`, it auto-rolls back. If it didn't apply at all, the router is unchanged.

**Problem 2: The auto-rollback itself fails.**

Rare but possible. If the rollback config has a dependency that was satisfied by the now-reverted change (circular dependency), the rollback can partially fail. Mitigation: always test rollbacks in lab before deploying the workflow.

**Problem 3: The health checks give a false positive.**

Your checks all pass at T+30s, you send the confirming commit, and at T+120s the network melts because the real impact was delayed (slow convergence, gradual route leak). Mitigation:
- Extend the confirm timeout to 10+ minutes for high-risk changes
- Run health checks at T+30s AND T+120s AND T+240s before confirming
- Use streaming telemetry to monitor continuously during the confirm window, not just point-in-time checks

**Architecture for 2,000 PEs:**

You can't do confirmed commit to 2,000 routers simultaneously. Use staged rollout:

```
Ring 0: 1 PE per role per region (5-10 PEs)
  → confirmed commit, 10-minute timer, full validation
  → confirm if healthy
  → wait 30 minutes, monitor

Ring 1: 10% of fleet (200 PEs)
  → confirmed commit in parallel batches of 50
  → each batch has its own confirm timer
  → independent health validation per batch
  → confirm batch-by-batch

Ring 2: Remaining 90%
  → same pattern, larger batches (100-200)
  → only if Ring 0 + Ring 1 show zero issues

Abort at any ring:
  → let confirmed commits timeout and auto-rollback
  → no manual intervention needed on the routers
  → investigate and fix the change before retrying
```

**The key insight:** `commit confirmed` is not just a safety net — it's an architectural building block. It turns every config push into a reversible experiment. The automation doesn't need to know how to rollback; the router handles it natively. The automation's job is deciding whether to confirm, not how to undo.
