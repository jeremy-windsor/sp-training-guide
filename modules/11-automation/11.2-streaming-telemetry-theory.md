# 11.2 — Streaming Telemetry — Protocol Theory

## Protocol Overview

Streaming telemetry is the paradigm shift from pull-based network monitoring (SNMP polling) to push-based real-time data streaming. Instead of a management station asking "what's your interface counter?" every 5 minutes and getting a single-point-in-time response, the network device continuously streams data at sub-second intervals — the device tells you what's happening, as it happens.

The motivation is simple: SNMP polling at 5-minute intervals misses micro-bursts, brief route flaps, transient congestion, and sub-second failure events. By the time the next poll arrives, the event is over. Streaming telemetry at 10-second or even 1-second intervals captures network behavior that was previously invisible.

Two primary protocols implement streaming telemetry: **gNMI Subscribe** (OpenConfig, gRPC-based) and **YANG-Push / Subscribed Notifications** (IETF, RFC 8639/8641 — building on earlier event notifications in RFC 5277). In practice, gNMI Subscribe has become the dominant approach due to its performance, simplicity, and ecosystem support. Model-Driven Telemetry (MDT) implementations from vendors (Cisco, Juniper, Nokia) typically support both gNMI and proprietary dial-in/dial-out modes.

## Core Mechanisms

### 1. Push vs Pull Paradigm

**SNMP polling (pull):**
1. Collector sends SNMP GET/WALK to device every N minutes.
2. Device processes request, retrieves data, sends response.
3. Collector receives response, stores data point.
4. Gap between polls = blind spot.

**Problems with polling at scale:**
- Device CPU load: each poll triggers MIB walks, often involving context switches between control and data plane.
- Collector load: polling 10,000 interfaces across 500 devices at 5-minute intervals = 33 polls/second sustained. At 30-second intervals (to catch more events) = 333 polls/second. Doesn't scale.
- Timing skew: polls arrive at slightly different times, making cross-device correlation imprecise.
- Information loss: events between polls are invisible.

**Streaming telemetry (push):**
1. Collector (or device, depending on mode) establishes a persistent session.
2. Device continuously streams specified data at configured intervals.
3. Data arrives in near-real-time — no polling overhead, no blind spots.
4. The device decides when to send (on-change or periodic), not the collector.

### 2. gNMI Subscribe

gNMI Subscribe is the primary streaming telemetry interface, defined by OpenConfig.

**Subscribe RPC:**
The client sends a SubscribeRequest specifying:
- One or more subscription paths (YANG model paths)
- Subscription mode per path
- Encoding (protobuf, JSON_IETF, JSON)
- Optional QoS marking

**Subscription modes:**

**STREAM**: Continuous streaming. The device sends data repeatedly based on the sub-mode:

- **SAMPLE**: Device sends current value at a fixed interval (e.g., every 10 seconds). Best for counters and gauges that change continuously. This is the workhorse mode for interface counters, CPU utilization, memory, optical power levels.

- **ON_CHANGE**: Device sends data only when it changes. Best for state data that changes infrequently: BGP session state, interface admin/oper status, route table entries. Massively reduces data volume compared to SAMPLE for stable state.

- **TARGET_DEFINED**: The device chooses the best mode per leaf. Counter leaves get SAMPLE; state leaves get ON_CHANGE. The device knows best which mode suits each data type.

**ONCE**: One-shot retrieval. Device sends current value for all subscribed paths, then closes the subscription. Equivalent to a gNMI Get but using the Subscribe RPC. Useful for point-in-time snapshots.

**POLL**: Client-triggered. The client sends a Poll message when it wants data. The device responds with current values. Bridges the gap between push and pull — the persistent session avoids connection setup overhead, but data is only sent on demand.

### 3. Dial-In vs Dial-Out

**Dial-in (dynamic subscription):**
- The collector initiates the gRPC/gNMI connection to the device.
- The collector sends the SubscribeRequest specifying paths and modes.
- The device streams data back over the same connection.
- If the connection drops, the subscription is lost — the collector must reconnect and resubscribe.

**Dial-out (configured subscription):**
- The subscription is configured on the device (via CLI, NETCONF, or YANG model).
- The device initiates the connection to the collector.
- If the connection drops, the device automatically reconnects and resumes streaming.
- More operationally resilient — survives collector restarts.
- Device configuration specifies: collector address/port, sensor paths, interval, encoding.

**Which to use:**
- Dial-in: preferred for dynamic, on-demand telemetry. Collector controls what's subscribed.
- Dial-out: preferred for production monitoring. Survives failures. Device-initiated.
- Many deployments use both: dial-out for baseline monitoring, dial-in for ad-hoc debugging.

### 4. Sensor Paths

A sensor path identifies the specific data to stream, expressed as a YANG model path.

**Examples:**
```
/interfaces/interface/state/counters
/network-instances/network-instance/protocols/protocol/bgp/neighbors/neighbor/state
/components/component/optical-channel/state/output-power
/system/cpus/cpu/state/total/instant
```

**Path granularity trade-offs:**
- **Broad paths** (e.g., `/interfaces/interface`): stream everything about every interface. Comprehensive but high data volume.
- **Narrow paths** (e.g., `/interfaces/interface[name=eth0]/state/counters/in-octets`): stream one counter for one interface. Minimal data but many subscriptions needed for full coverage.
- **Wildcard paths** (e.g., `/interfaces/interface[name=*]/state/oper-status`): stream specific data across all instances. Good balance.

**Vendor-specific sensor paths**: Despite YANG model standardization, sensor path syntax can vary between vendors. Cisco IOS-XR uses `Cisco-IOS-XR-*` native model paths. Juniper uses `/junos/` prefix paths. OpenConfig paths are the most portable.

### 5. Data Encoding

Streaming telemetry data must be serialized for transport. Three primary encodings:

**Protocol Buffers (protobuf):**
- Binary encoding. Compact, fast to serialize/deserialize.
- Requires a .proto schema definition (auto-generated from YANG models).
- Most efficient for high-volume streaming.
- Commonly used for high-volume gNMI streaming (JSON is the default when encoding is omitted per gNMI spec).

**JSON / JSON_IETF:**
- Text-based encoding. Human-readable.
- JSON_IETF follows RFC 7951 (JSON encoding of YANG data). JSON is a looser encoding.
- Larger than protobuf but easier to debug and process with standard tools.
- Preferred for development, debugging, and low-volume streams.

**Compact GPB (Google Protocol Buffers):**
- Cisco-specific optimization. Pre-compiled .proto files for specific sensor paths.
- Even more compact than standard protobuf — field names are replaced with numeric IDs.
- Fastest encoding but requires sensor-path-specific .proto files.

**Encoding impact on bandwidth:**
A device streaming 10,000 interface counters at 10-second intervals generates significant data. Protobuf encoding can reduce bandwidth by 5-10x compared to JSON. At scale, encoding choice directly affects collector capacity and network overhead.

### 6. Telemetry Collection Architecture

**Collection pipeline:**

```
Device (telemetry source)
  ↓ gNMI/gRPC stream (protobuf)
Collector / Ingestion Layer
  (telegraf, gnmic, pipeline)
  ↓ normalized data
Time-Series Database
  (InfluxDB, Prometheus, TimescaleDB)
  ↓ queries
Visualization / Alerting
  (Grafana, custom dashboards)
```

**Key collector tools:**
- **gnmic**: Open-source gNMI CLI and collector. Supports all gNMI operations, multiple output targets (Kafka, NATS, InfluxDB, file). The Swiss Army knife of gNMI.
- **Telegraf**: InfluxData's agent. Has gNMI input plugin. Feeds InfluxDB natively.
- **Cisco Pipeline**: Cisco's MDT collector. Supports TCP/gRPC dial-out.
- **Juniper jtimon**: Juniper's telemetry collector for JTI (Junos Telemetry Interface).

**Scaling considerations:**
- A single collector can handle hundreds of device streams.
- For large networks (1000+ devices), use multiple collectors with load distribution.
- Kafka or NATS as an intermediate message bus decouples collection from storage/processing.

### 7. NETCONF Notifications (RFC 5277)

The IETF's event-based telemetry mechanism, predating gNMI.

**How it works:**
1. Client sends `<create-subscription>` RPC to the device.
2. Device sends `<notification>` messages asynchronously when events occur.
3. Notifications contain YANG-modeled event data in XML.

**Subscription types:**
- **Stream-based**: Subscribe to a named event stream (e.g., "NETCONF" for config changes).
- **Filter-based**: XPath or subtree filter to receive only matching events.
- **Replay**: Request historical notifications from a device-side buffer.

**RFC 8639/8641 (Subscribed Notifications)**: Extended NETCONF notifications to support configured (dial-out) subscriptions and periodic/on-change modes — bringing NETCONF closer to gNMI's capabilities. But adoption has been slower than gNMI.

**NETCONF notifications vs gNMI Subscribe:**
- NETCONF notifications: XML, SSH, event-oriented, IETF standard.
- gNMI Subscribe: protobuf, gRPC/HTTP/2, stream-oriented, OpenConfig.
- gNMI wins on performance and ecosystem. NETCONF wins on standardization pedigree.

## Protocol Interactions

**Streaming telemetry ↔ YANG**: Sensor paths are YANG model paths. The data streamed conforms to YANG-defined schemas. YANG model coverage determines what can be streamed — if there's no YANG model for a feature's counters, those counters can't be streamed via model-driven telemetry.

**Streaming telemetry ↔ gRPC**: gNMI runs over gRPC, which provides the streaming infrastructure (HTTP/2, bidirectional streaming, TLS). gRPC handles connection management, multiplexing, and flow control.

**Streaming telemetry ↔ SNMP**: Streaming telemetry replaces SNMP polling for monitoring but doesn't replace SNMP traps entirely (some legacy systems still depend on traps). Many deployments run both during migration — SNMP for legacy tools, streaming telemetry for modern dashboards.

## Edge Cases & Gotchas

**1. Sample interval too aggressive**: Streaming interface counters at 1-second intervals across 10,000 interfaces generates enormous data volume and stresses device CPU. Start conservative (30s) and decrease only for specific troubleshooting.

**2. ON_CHANGE flood on flapping state**: If an interface flaps rapidly, ON_CHANGE subscriptions generate a notification for every state transition. A flapping interface can produce hundreds of notifications per second, overwhelming the collector. Dampening at the source or collector is essential.

**3. Sensor path support varies by platform**: Not every YANG model path is streamable on every platform. Some paths are "config-only" (not meaningful for telemetry). Some state paths are supported on one platform but not another. Verify sensor path support per device type.

**4. Clock synchronization**: Correlating telemetry data across devices requires synchronized clocks. Without NTP/PTP, timestamps from different devices may be offset by seconds, making cross-device analysis unreliable.

**5. gRPC connection limits**: Each gNMI session consumes resources on the device (memory, CPU, TCP connections). Devices have limits on concurrent gRPC sessions (typically 10-50). A collector connecting to every line card independently can exhaust session limits.
