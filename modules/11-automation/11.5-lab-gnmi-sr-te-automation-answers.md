## Question 1
> **Scenario:** Your daemon is firing but `show segment-routing traffic-eng policy name TO_PE2` still shows pref-200 as active after the NETCONF push. NETCONF logged `OK`. What are three things you would check first?

### Answer

NETCONF `OK` only means the RPC succeeded. It does **not** guarantee the intended candidate-path became active. Three things to check first:

**1. Did the config actually change the right object?**

The daemon may have pushed valid XML to the wrong policy name, wrong color, wrong endpoint, or wrong candidate-path preference node. NETCONF says `OK` because the payload was syntactically valid and committed — but it committed something you didn't care about.

**Check:**
- `show running-config segment-routing traffic-eng | include TO_PE2|preference`
- Verify the policy name is exactly `TO_PE2`
- Verify pref-200 was demoted (for example to pref-50) and pref-100 remained at 100
- Verify you're editing the active headend PE, not PE2 or a lab stub router

If the config didn't actually alter `TO_PE2`, the policy engine has no reason to switch candidates.

**2. Is pref-100 actually valid and installable?**

SR-TE will only switch to pref-100 if the backup candidate is viable. If pref-100 is unresolved, invalid, or missing segment resolution, the router keeps pref-200 active even if you lowered its preference — because the backup path is garbage.

**Check:**
- `show segment-routing traffic-eng policy name TO_PE2 detail`
- Look for candidate path status on both pref-200 and pref-100
- Verify pref-100 is `valid`, `resolved`, and not `inactive/no-path`
- Verify the explicit segment list for the backup path resolves (`{16004,16002}` in the lab)

A common failure: the backup candidate exists in config but the label stack can't be resolved due to a missing Node SID or stale IGP/SR state.

**3. Is operational state lagging because the policy never re-evaluated?**

Some platforms apply the config immediately but the policy doesn't flip until the SR-TE process re-runs candidate selection or until the existing active path becomes invalid. If the headend still considers pref-200 healthy and the preference change wasn't enough to trigger re-selection, you can get a successful commit with no operational switchover.

**Check:**
- SR-TE policy re-optimization / headend re-evaluation logs
- `show logging | include sr-te|xtc|policy`
- Whether the policy is set to dynamic re-evaluation or pinned to the current path
- Whether an explicit policy `shutdown/no shutdown` or candidate re-optimize was needed after the config push

**The practical triage order:**
1. Running-config diff — did I change the right thing?
2. Backup candidate validity — is pref-100 actually usable?
3. Policy operational re-evaluation — did SR-TE consume the change?

That catches 90% of these failures before you go off into the woods blaming NETCONF.

## Question 2
> **Design:** You want to run this daemon in a redundant (active/standby) HA deployment. What problems arise if both instances try to push NETCONF simultaneously? How would you prevent split-brain?

### Answer

If both daemons are active at once, you've built a fight club for config management.

**Problems if both push simultaneously:**

**1. Configuration flapping / oscillation.**
Daemon A sees utilization 81% and demotes pref-200. Daemon B, polling a slightly older Prometheus sample or reading a different cache, thinks utilization is 49% and restores pref-200. They alternate commits, the policy flips repeatedly, and traffic gets packet reordering plus control-plane churn. Congratulations, you've automated instability.

**2. NETCONF lock contention.**
Both instances open NETCONF sessions and attempt `<edit-config>` + `<commit>`. One acquires the candidate/running lock, the other blocks or fails. If your code treats lock timeout as retriable and immediately retries, you create a thundering herd against the router.

**3. Stale-state writes.**
Instance A reads state at T0 and decides "demote." Instance B reads state at T0+5s and decides "restore." Both decisions are locally rational, but neither accounts for the other's pending write. This is classic split-brain: both believe they are authoritative.

**4. Duplicate alerting and false telemetry.**
Both daemons emit logs, alerts, and metrics. Your NOC sees duplicate incidents and can't tell which automation decision actually changed the network.

**How to prevent split-brain:**

**A. Use an external leader election mechanism.**
One daemon is leader, the other is hot standby. Only the leader is allowed to write.

Options:
- **etcd/Consul lease** — daemon renews a short TTL lock every few seconds
- **Redis SETNX with expiry** — simple, good enough for small deployments
- **Kubernetes Lease object** — if the daemon runs in k8s

If the leader dies and stops renewing the lease, standby takes over after TTL expiry.

**B. Enforce single-writer semantics in the application.**
Even with leader election, every write should include a per-policy distributed lock:
- lock key: `srte:TO_PE2`
- TTL: 60–120 seconds
- only the holder can push NETCONF for that policy

This protects you during leader failover races.

**C. Make decisions idempotent and state-aware.**
Before every write:
1. Read actual policy state from the router
2. Compare with intended state
3. If already in desired state → do nothing
4. If not → write once, record generation/version

This prevents duplicate "demote" operations when the policy is already demoted.

**D. Add fencing.**
The old leader must be fenced off if it loses leadership. If it can't renew its lease, it must immediately disable its write path. Read-only mode is fine; write mode is not.

**Recommended pattern:**
- Active/standby using etcd/Consul lease
- per-policy write lock
- idempotent desired-state reconciliation
- metrics showing current leader identity

The rule is simple: **many readers, one writer.** Anything else is two drunk interns fighting over a route-policy.

## Question 3
> **Troubleshooting:** `fetch_metric()` returns `None` every cycle. gnmic is running and `gnmic capabilities` works. What is the most likely cause, and how do you fix it without restarting gnmic?

### Answer

If `gnmic capabilities` works, control-plane reachability and gNMI basic auth are fine. But `fetch_metric()` returning `None` every cycle means the daemon isn't finding the metric in Prometheus (or whatever local scrape endpoint it's reading from).

**Most likely cause:** the subscription path is wrong, or the metric name/labels the daemon expects don't match what gnmic is actually exporting.

This is the classic telemetry pipeline mismatch:
- gnmic is subscribed to path A
- Prometheus is exposing metric name B with labels C
- your daemon queries metric name D with labels E
- result = no matching series → `None`

`capabilities` only proves the router supports the model namespace. It does NOT prove you're subscribed to the right sensor path or that gnmic translated it into the metric name your Python code expects.

**How to fix without restarting gnmic:**

**1. Inspect the live Prometheus endpoint.**

```bash
curl -s http://localhost:9273/metrics | grep -i 'interface\|octets\|policy\|sr_te'
```

Find the actual exported metric names and labels. You might discover your code expects:
- `interface_utilization_percent{device="PE1",interface="Gi0/0/0/0"}`

but gnmic is exporting:
- `gnmic_Cisco_IOS_XR_infra_statsd_oper_interfaces_interface_latest_generic_counters_rates_output_data_rate{source="192.168.100.1:57400",interface_name="GigabitEthernet0/0/0/0"}`

Same data, different names. Your query returns nothing because the labels don't line up.

**2. Fix the daemon query, not gnmic.**

Since you asked for a no-restart fix, change the query in the Python daemon to match the actual metric name and labels gnmic is already exporting. If the daemon supports config reload or dynamic query update, use that. Otherwise edit the query source/config file and trigger a lightweight daemon reload.

**3. If the issue is the subscribed path but gnmic supports hot config reload, reload the subscription config.**
Some gnmic deployments support SIGHUP or file-watcher-based config reload. Update the sensor path in the gnmic config and reload it without full process restart. If hot reload isn't supported, then yes, you'd need a restart — but the question says avoid restarting gnmic, so first assume the path is fine and the query is wrong.

**The most likely root cause in lab reality:** interface naming mismatch. Your daemon asks for `Gi0/0/0/0`, but IOS-XR exports `GigabitEthernet0/0/0/0`, or it keys by `name` instead of `interface`. One wrong label and the time series disappears.

## Question 4
> **Scaling:** Your network grows to 50 PE routers. The daemon currently monitors one interface on one router. Describe the architectural changes needed — what breaks at scale, and what pattern would you replace the single polling loop with?

### Answer

The single-router polling loop dies immediately at scale. It works in a lab because the world is small and forgiving. Fifty PEs is not forgiving.

**What breaks:**

**1. Single-threaded polling latency.**
If the daemon polls one metric every 30 seconds and now has to poll 50 routers × multiple interfaces × multiple policies, the loop duration itself exceeds the polling interval. By the time it finishes checking router 50, router 1's data is already stale.

**2. No state partitioning.**
One global in-memory state machine for 50 routers means a bug or exception in one router's logic can halt decisions for all routers. That's a terrible blast radius.

**3. No per-router hysteresis or dwell tracking.**
At scale, each PE needs its own cooldown timers, last-action timestamps, and failure counters. A single global timer doesn't make sense when PE1 is congested and PE37 is idle.

**4. NETCONF session churn.**
Opening and closing NETCONF sessions for 50 routers every cycle is expensive and fragile. SSH handshakes, auth, candidate locking — multiplied by 50 — creates unnecessary load and longer mean reaction time.

**5. Observability collapse.**
One logfile with mixed messages from 50 routers becomes unreadable. One Prometheus metric for "daemon state" is useless. You need per-router and per-policy metrics.

**What pattern to replace it with:**

**Event-driven controller with a work queue**, not a polling loop.

Architecture:

```text
gNMI / Prometheus
      ↓
Metric ingestion layer
      ↓
Per-router evaluator workers
      ↓
Decision queue
      ↓
Single-writer config workers (NETCONF)
      ↓
Per-router state store
```

**Recommended design elements:**

**A. Separate ingestion from decision-making.**
One process ingests metrics and normalizes them into a common schema (`router, interface, policy, utilization, timestamp`). Don't let each router worker query Prometheus independently — centralize metric ingestion.

**B. Per-router workers (or shards).**
Assign each PE to a worker/shard. Each worker maintains independent hysteresis state, dwell timers, and last-action records for its assigned routers. If worker 3 crashes, only routers 21–30 are impacted, not the whole fleet.

**C. Intent queue.**
Workers don't push NETCONF directly. They emit intents: `router=PE17, action=demote, policy=TO_PE2`. A dedicated config worker consumes intents and serializes writes per router. This prevents overlapping config pushes.

**D. Persistent state store.**
Store last-known action, dwell timer start, and flip counters in Redis/Postgres/etc. If the daemon restarts, it shouldn't forget it already demoted PE17 two minutes ago and immediately oscillate again.

**E. Long-lived NETCONF connection pools.**
Maintain reusable sessions per router or per worker, with health checks. Reconnect only on failure. This cuts reaction time and reduces SSH overhead.

**F. Rate limits and blast-radius controls.**
Global rule: no more than N simultaneous policy flips network-wide. If a shared traffic pattern causes 20 PEs to cross threshold at once, you don't want 20 simultaneous TE flips unless you're sure that's safe.

**The replacement pattern:** from **synchronous poll-loop** to **distributed event-driven control plane**. Same logic, grown-up architecture.

## Question 5
> **Design:** The war story described a 4-minute oscillation cycle. Without changing the 30-second Prometheus poll interval, what minimum dwell timer would prevent this specific oscillation from recurring? Show your math.

### Answer

The oscillation period is **4 minutes = 240 seconds**.

The daemon polls every **30 seconds**.

If the system is flipping every 240 seconds, the dwell timer must be **strictly longer than the oscillation period** to prevent the next reversal from being allowed when the traffic pattern swings back.

### Math

- Oscillation period = 240 seconds
- Poll interval = 30 seconds
- Number of polls per oscillation cycle:

```text
240 / 30 = 8 polls
```

If you set dwell = 240 seconds exactly, you can still get an edge-case flip at the first poll that lands right on or just after expiry, depending on how the daemon evaluates `>= dwell` and when the previous action timestamp was recorded.

So the **minimum safe dwell timer** should be the **next poll boundary above the oscillation period**:

```text
240 seconds + 30 seconds = 270 seconds
```

That guarantees the daemon cannot reverse the previous action within the same 4-minute traffic oscillation window.

### Practical answer

- **Mathematical minimum safe dwell:** **270 seconds (4.5 minutes)**
- **Operationally sane setting:** **300 seconds (5 minutes)**

Why 300s is better:
- Aligns to a clean human-readable timer
- Gives one extra poll of safety margin
- Matches the war story's production fix
- Reduces sensitivity to clock skew, Prometheus scrape lag, or delayed NETCONF completion timestamps

So if you're asking for the **smallest value that actually prevents this specific 4-minute loop**, it's **270 seconds**.
If you're asking what I'd configure in production, it's **300 seconds** because I enjoy sleeping at night.
