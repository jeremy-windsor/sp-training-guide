# 11.3 — SR-TE Controller Integration — Answer Key

---

## Question 1
> **A carrier operates SR-TE with 400 headend routers. BGP-LS sessions are established between RRs and the PCE, but the PCE's TED shows only 60% of the expected links. What are three likely causes?**

### Answer

**Cause 1: Incomplete BGP-LS redistribution from the IGP.**

BGP-LS depends on the IGP (IS-IS or OSPF) distributing its link-state database into BGP. If `distribute link-state` (IOS-XR) or `traffic-engineering` (Junos) isn't enabled on all IS-IS/OSPF instances or levels, some parts of the topology won't appear in BGP-LS.

Common sub-problems:
- `distribute link-state` enabled on Level-2 but not Level-1 — inter-area links from L1-only areas are missing
- Multi-instance IGP (separate IS-IS instances for different regions) and only some instances redistribute into BGP-LS
- OSPF areas with no ABR configured for BGP-LS export

**Diagnosis:** On the BGP-LS speaker (RR), check `show bgp link-state link-state summary` — compare the number of node/link NLRIs against the IGP's LSDB size (`show isis database summary`). If LSDB has 400 nodes and 2,000 links but BGP-LS only shows 240 nodes and 1,200 links, you have a redistribution gap.

**Cause 2: BGP-LS address family not enabled on all RR→PCE paths.**

The PCE receives BGP-LS from one or more route reflectors. If the BGP-LS address family (`address-family link-state link-state`) isn't configured on every BGP session in the chain from IGP speaker → RR → PCE, links are dropped.

Common sub-problems:
- Multiple RR tiers (edge RR → core RR → PCE) and one tier is missing the link-state AFI
- BGP-LS configured on the RR but not on the specific neighbor statement toward the PCE
- ORF (Outbound Route Filtering) or route-policy on the RR→PCE session filtering out some link-state NLRIs

**Diagnosis:** On each RR in the chain, verify `show bgp neighbor <PCE-IP> | include link-state` shows the address family as negotiated and active. Check for any outbound route-policies on the BGP-LS session.

**Cause 3: TE attributes missing on some interfaces → links not exported.**

BGP-LS for SR-TE requires TE attributes on the interfaces — TE metric, admin-group, SRLG, delay. If IS-IS TE extensions aren't enabled on some interfaces (`isis interface <x> address-family ipv4 unicast` without `mpls traffic-eng` or equivalent), those links appear in the IGP LSDB as plain IS-IS adjacencies but don't carry TE TLVs. The BGP-LS speaker may still export them, but without SIDs, or the PCE may ignore links without SR capability information.

Also: if SR is not enabled on some nodes (`segment-routing mpls` under IS-IS), those nodes' prefix SIDs won't appear in BGP-LS, and the PCE can't compute segment lists through them — effectively making those links "invisible" for SR-TE path computation even if the links themselves are in the TED.

**Diagnosis:** On a suspicious node, check `show isis adjacency detail` — verify TE attributes and SID advertisements. On the PCE, check if the missing links have SR-capable nodes on both ends.

## Question 2
> **SR policy `color=100` is delegated to the PCE, but after a link failure, traffic continues on the old path for 3 minutes before rerouting. The PCEP session is up. What would you check?**

### Answer

3 minutes is insane for PCE-delegated reroute. Normal behavior is seconds: PCC sends PCRpt (path down) → PCE recomputes → PCE sends PCUpd (new segment list) → PCC programs FIB. This should complete in under 5 seconds. 3 minutes means something in the feedback loop is broken.

**Check 1: Is the PCC actually sending PCRpt when the link fails?**

If the headend router doesn't detect the link failure promptly, it won't report the path change to the PCE. Check:
- **BFD on the SR-TE policy path.** If there's no BFD or BFD is configured with long intervals (1000ms × 3 multiplier = 3 seconds detect), the headend relies on IGP convergence to detect the failure, which can take seconds. Without BFD on the affected interface, the IGP hold-timer (typically 30-40 seconds for IS-IS, 40 seconds for OSPF) must expire before the link is declared down.
- **IGP convergence time.** Check IS-IS/OSPF timers. If SPF hold-down is set to 30 seconds (some conservative configs), the headend won't update its local TED for 30 seconds after the link goes down.

But 3 minutes suggests something worse than slow detection.

**Check 2: Is the PCE actually receiving the topology change?**

The PCE's TED is fed by BGP-LS from the RRs. If the BGP-LS update for the failed link is delayed:
- BGP MRAI (Minimum Route Advertisement Interval) on the RR→PCE session might be set to 30 seconds or more
- BGP-LS batching/dampening on the RR
- The RR itself hasn't received the IGP update yet (see Check 1)

**Diagnosis:** Check PCE logs for when it received the link-down BGP-LS withdrawal. If there's a 2.5-minute gap between the physical link failure and the PCE learning about it, the problem is in the IGP→BGP-LS→PCE pipeline.

**Check 3: PCE recomputation delay.**

Some PCE implementations have configurable recomputation timers — a hold-down to batch multiple topology changes before recomputing paths. If this is set to 60 seconds or more, the PCE intentionally waits before sending PCUpd.

**Diagnosis:** Check PCE configuration for recomputation timer/hold-down settings. Cisco SR-PCE has `pce timers` configuration. If the recompute timer is 180 seconds... there's your 3 minutes.

**Check 4: PCEP delegation state.**

Even though the PCEP session is up, is the specific policy actually delegated? Check `show segment-routing traffic-eng policy color 100 detail` on the headend — look for `Delegation: PCE` vs `Delegation: none`. If the policy lost delegation (perhaps after a previous PCEP flap) and fell back to local computation, the headend is computing its own path using its local TED, which may not have the topology change yet.

**Check 5: PCUpd is sent but the headend isn't programming it.**

The PCE sends a new segment list via PCUpd, but the headend has an issue programming the new path:
- Label stack validation failure (stale SIDs from the previous topology)
- FIB programming delay on hardware platforms with slow TCAM updates
- The new segment list references a SID that the headend can't resolve

**Diagnosis:** On the headend, check `show segment-routing traffic-eng policy color 100 candidate-path` — look for the new candidate path from PCE and its installation status.

## Question 3
> **Two SR policies in disjoint group 1 are supposed to follow node-disjoint paths. After a topology change, they start sharing an intermediate node. The PCE logs show successful PCUpd messages. What could explain this?**

### Answer

The PCE computed paths that it believes are node-disjoint, sent them successfully (PCUpd logged as OK), but the paths actually share a node. Something in the disjointness computation or topology model is wrong.

**Explanation 1: The PCE relaxed the disjoint constraint.**

Most PCE implementations have a disjointness priority or fallback behavior. If the topology change made it impossible to find strictly node-disjoint paths (e.g., the only path for Policy B now must transit a node that Policy A also uses), the PCE may **relax** the constraint and return the "best available" paths — which are link-disjoint or partially disjoint instead of fully node-disjoint.

The PCUpd succeeds because the PCE considers this a valid solution with degraded constraints. It doesn't fail — it degrades gracefully.

**Diagnosis:** Check PCE logs for "disjoint constraint relaxed" or "fallback to link-disjoint" messages. Check the disjoint association configuration — if `strict` mode isn't set, the PCE is allowed to relax. Set `association-group disjoint strict` to force failure instead of relaxation.

**Explanation 2: Association group mismatch between the two policies.**

Disjoint computation requires both policies to be in the same association group. If one policy references `association-group 1` and the other references `association-group 1` but with different parameters (e.g., one says `node-disjoint`, the other says `link-disjoint`), the PCE may compute them independently rather than jointly.

Or worse: one policy was re-created after the topology change and the association group membership wasn't re-established on the PCE. The PCE computes each policy independently, and they happen to share a node.

**Diagnosis:** On both headend PEs, verify the SR policy association group configuration matches exactly. On the PCE, verify both policies are associated in the same disjoint group.

**Explanation 3: The shared node was added to the topology by the topology change.**

Before the topology change, the two paths were node-disjoint because there were enough nodes for two independent paths. The topology change (link failure) removed a link, and the PCE's recomputation found that the only feasible path for one policy now goes through a node that the other policy already uses.

If the PCE isn't configured for strict disjointness, it returns the "least overlapping" paths. The shared node is the only viable transit point after the topology change.

**Diagnosis:** Visualize the topology before and after the change. Identify whether node-disjoint paths still exist. If they don't, this is a topology design problem — you need more nodes/links to maintain disjointness under failure.

**Explanation 4: Stale TED on the PCE.**

The PCE's TED didn't fully converge after the topology change. It computed paths based on a partially updated topology where one link was removed but another wasn't yet added. The resulting computation used a path that shares a node because the PCE didn't know about an alternative path that would have avoided it.

**Diagnosis:** Compare the PCE's TED (`show pce topology`) with the actual IGP LSDB on the routers. If they differ, you have a BGP-LS convergence delay problem (see Question 2).

## Question 4
> **An operator wants to steer VPN traffic into SR-TE policies without pre-provisioning policies per VPN. Which feature enables this, and what BGP mechanism carries the intent?**

### Answer

**Feature: Automated Steering (AS) with On-Demand Nexthop (ODN).**

Also called "On-Demand Color" or "BGP-driven SR-TE." The idea: instead of pre-creating an SR-TE policy for every VPN destination, the headend PE automatically instantiates an SR-TE policy when it receives a BGP VPN route with a color extended community.

**How it works:**

1. The egress PE (or a route-policy on the RR) attaches a **color extended community** to VPN prefixes:
   ```
   route-policy SET-COLOR-100
     set extcommunity color 100
   end
   ```

2. The ingress PE receives the VPN route with `color:100` and next-hop `PE-B`.

3. The ingress PE checks if an SR-TE policy with `color=100, endpoint=PE-B` exists.
   - If yes → steer traffic into that policy
   - If no → **ODN creates one automatically**, delegating path computation to the PCE (or using a local template)

4. The PCE computes a constrained path (e.g., color 100 = "low-latency" → min-delay metric) and returns the segment list.

5. Traffic is steered into the dynamically created SR-TE policy. No pre-provisioning needed.

**The BGP mechanism:**

- **Color Extended Community (RFC 9012):** A transitive extended community that carries a 32-bit color value. The color maps to a steering intent (policy template, constraint set, or PCE optimization objective).
- **BGP next-hop resolution:** The PE resolves the VPN next-hop via the SR-TE policy that matches `(color, endpoint)`. This is called "color-aware next-hop resolution" or "transport-class routing."

**IOS-XR config example:**

```
! On egress PE — tag VPN routes with color
route-policy CUSTOMER-URLLC-OUT
  set extcommunity color 100
end

router bgp 65000
  vrf CUSTOMER-URLLC
    address-family ipv4 unicast
      redistribute connected route-policy CUSTOMER-URLLC-OUT

! On ingress PE — ODN template for color 100
segment-routing
  traffic-eng
    on-demand color 100
      dynamic
        pcep
        metric type latency  ! min-latency constraint
```

When the ingress PE receives a VPN route with `color:100` and next-hop `PE-B`, it automatically creates an SR-TE policy `color=100, endpoint=PE-B`, delegates to the PCE for a min-delay segment list, and steers all matching VPN traffic into it.

**Why this scales:** Without ODN, a 400-PE network with 50 color intents = 400 × 50 = 20,000 pre-provisioned policies. With ODN, you configure 50 templates and policies are created only when needed — typically a fraction of the full mesh.

## Question 5
> **During planned maintenance on the primary PCE, all delegated SR policies fall back to local headend CSPF. Some critical policies show "no path" because the headend lacks full TE topology. How should this have been designed?**

### Answer

The headend's local CSPF (Constrained Shortest Path First) uses the IGP TE database — which only contains the local area/level's topology. If the SR-TE policy spans multiple areas or uses constraints that require global topology visibility (SRLG avoidance across areas, inter-area disjointness), the headend can't compute a path. "No path" is the correct answer for a question the headend doesn't have enough information to answer.

**How this should have been designed:**

**1. Redundant PCEs (active/standby or active/active).**

This is the #1 answer. You should never have a single PCE for 400 headend routers.

| Design | Failover |
|--------|----------|
| Active/standby PCE pair | PCC has PCEP sessions to both. Standby takes over in <10s when primary goes down. All delegated policies transfer. |
| Active/active PCE pair | PCC load-balances policy delegation across both PCEs. Each PCE has full TED. Losing one PCE = other handles all policies. |
| Geographically diverse PCE pair | PCE-1 in DC-East, PCE-2 in DC-West. Survives site failure. |

**IOS-XR PCC config for redundant PCEs:**
```
segment-routing
  traffic-eng
    pcc
      source-address ipv4 1.1.1.1
      pce address ipv4 10.0.0.200  ! Primary PCE
        precedence 10
      pce address ipv4 10.0.0.201  ! Standby PCE
        precedence 20
```

With this, planned maintenance on PCE-1 → PCCs seamlessly delegate to PCE-2. No local CSPF fallback needed.

**2. Pre-computed backup segment lists cached on the headend.**

Even with redundant PCEs, defense-in-depth means the headend should have a locally computable fallback path. Configure each critical SR-TE policy with:
- Candidate path preference 200: PCE-delegated (dynamic, optimal)
- Candidate path preference 100: Locally configured explicit segment list (static, suboptimal but valid)

If the PCE is unreachable, the policy falls back to the pre-configured static segment list. Not optimal, but the traffic flows.

**3. BGP-LS on the headend PE.**

If the headend PE receives BGP-LS from the RR (same feed the PCE gets), it has full global topology for local CSPF computation. This eliminates the "headend lacks full TE topology" problem:

```
router bgp 65000
  address-family link-state link-state
  neighbor <RR-IP>
    address-family link-state link-state

segment-routing
  traffic-eng
    ! Note: IOS-XR headend CSPF uses the IGP TED, not BGP-LS directly.
    ! BGP-LS consumption is a PCE function. For local fallback, ensure
    ! the headend PE has IGP adjacencies covering the required topology.
```

Now when the PCE is down, local CSPF uses the IGP TED for path computation. The catch: this increases memory usage on every headend PE (storing the full global TED) and adds computation load. For 400 headend PEs, this is a significant scale consideration.

**4. Graceful PCE maintenance procedure.**

Before taking the primary PCE offline:
1. Verify all PCC sessions have established to the standby PCE
2. Manually trigger delegation transfer (`pce policy delegate` on the standby)
3. Verify all policies are active on standby PCE
4. THEN take primary offline

Never just `shutdown` a PCE and hope for the best.

**The design hierarchy:** Redundant PCEs (mandatory) → static fallback paths (defense-in-depth) → BGP-LS on headends (belt-and-suspenders). All three for critical infrastructure.
