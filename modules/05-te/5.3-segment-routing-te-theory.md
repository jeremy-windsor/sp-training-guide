# 5.3 — Segment Routing TE — Protocol Theory

## Protocol Overview

Segment Routing Traffic Engineering (SR-TE) replaces RSVP-TE with a source-routing paradigm. Instead of signaling per-hop state along the path, the head-end encodes the entire path as a stack of segment identifiers (SIDs) in the packet header. Each transit router processes only the top segment — no per-LSP state, no refresh messages, no signaling protocol.

SR-TE was designed to solve RSVP-TE's fundamental scalability problem: per-hop, per-LSP state. In a network with 10,000 TE tunnels, every transit router maintains 10,000 RSVP-TE path/reservation states, processes thousands of refresh messages per second, and must coordinate soft-state timers. SR-TE eliminates all of this. The head-end encodes the path; transit routers just follow instructions.

SR-TE is defined primarily in RFC 9256 (SR Policy Architecture) and builds on the SR-MPLS foundation (RFC 8402 architecture, RFC 8667 IS-IS extensions, RFC 8665/8666 OSPF extensions). The control plane uses either static configuration, a local CSPF computation, or an external PCE via PCEP (RFC 8664) to determine segment lists.

## Core Mechanisms

### 1. SR-TE Policy Architecture (RFC 9256)

An SR-TE policy is the fundamental construct. It defines traffic steering intent: "traffic to endpoint E with color C should follow this path."

**Policy identification — the tuple:**
- **Headend**: The router originating the policy.
- **Color**: A 32-bit value representing the intent (e.g., low-latency, avoid-region-X, gold-SLA).
- **Endpoint**: The destination address (typically a PE loopback).

The (headend, color, endpoint) tuple uniquely identifies an SR-TE policy.

**Policy structure:**

```
SR-TE Policy (headend, color, endpoint)
├── Candidate Path 1 (preference 200)
│   ├── Segment List 1 (weight 1)
│   │   └── SID1 → SID2 → SID3
│   └── Segment List 2 (weight 1)
│       └── SID4 → SID5
├── Candidate Path 2 (preference 100)
│   └── Segment List 1
│       └── SID6 → SID7
└── BSID: 15001
```

**Candidate paths**: Each policy can have multiple candidate paths, ranked by preference (higher = better). The highest-preference valid candidate path is active. Others are backup.

**Segment lists**: Each candidate path can have multiple segment lists. When multiple segment lists exist, traffic is load-balanced across them (weighted ECMP).

**Candidate path sources:**
- **Static/local**: Operator-configured segment list.
- **Dynamic**: Computed by a local CSPF or an external PCE.
- **BGP SR Policy SAFI**: Distributed via BGP from a controller (AFI 1, SAFI 73).

### 2. Segment Types

SR-TE uses several segment types, each identified by a SID:

**Type A — Node Segment (Prefix-SID):**
- Identifies a node in the SR domain.
- Globally significant within the domain — same SID value everywhere.
- Traffic follows the shortest IGP path to that node.
- Encoded as SRGB base + index (e.g., SRGB 16000 + index 1 = label 16001).
- The workhorse of SR-TE — most segment lists are sequences of node SIDs.

**Type B — Adjacency Segment (Adj-SID):**
- Identifies a specific link (adjacency) from a router.
- Locally significant — only meaningful at the router that allocated it.
- Forces traffic onto a specific link, even if it's not the IGP shortest path.
- Used for strict path control when node SIDs don't provide enough specificity (e.g., parallel links to the same neighbor).

**Type C — IPv4/IPv6 Prefix with Algorithm:**
- A prefix SID associated with a specific algorithm (e.g., Flex-Algo).
- Allows steering into algorithm-specific topologies.

**Type F — Binding SID:**
- A SID that maps to another SR-TE policy.
- Used for policy chaining and inter-domain stitching.

**Type H — Binding SID of a candidate path:**
- More specific BSID reference to a particular candidate path.

### 3. Binding SID (BSID)

The Binding SID is a local label that represents an entire SR-TE policy. Any packet arriving with the BSID as its top label gets steered into the associated policy's active segment list.

**Why BSID matters:**

**Scalability**: Instead of encoding a full segment list in every packet, an upstream router can just push the BSID. The local router expands it into the full segment list. This keeps label stacks shallow.

**Service chaining**: Multiple policies can be chained via BSIDs. Policy A's segment list includes Policy B's BSID. Traffic follows Policy A's path, arrives at the BSID router, and gets steered into Policy B's path.

**Inter-domain TE**: Each domain assigns a BSID to its internal TE policy. The head-end's segment list contains only the inter-domain BSIDs — it doesn't need to know the internal topology of remote domains.

**Resilience**: If the active candidate path changes (e.g., primary fails, backup activates), the BSID stays the same. Upstream routers don't need to update — they keep pushing the same BSID, and the local router transparently switches to the new path.

### 4. On-Demand Next-Hop (ODN)

ODN automates SR-TE policy creation based on BGP route color.

**Mechanism:**
1. A BGP route is received with a Color Extended Community (e.g., color 100 = low-latency).
2. The head-end router checks if an SR-TE policy exists for (headend, color=100, endpoint=BGP-next-hop).
3. If no policy exists, ODN automatically creates one using a predefined template.
4. The template specifies how to compute the path: dynamic CSPF with specific constraints, PCE delegation, or Flex-Algo.
5. Traffic matching the BGP route is automatically steered into the newly created SR-TE policy.

**ODN templates** define:
- Constraint type (latency, TE metric, affinity, Flex-Algo)
- Path computation method (local CSPF, PCE)
- Protection requirements
- ECMP behavior

**Why ODN matters**: Without ODN, every SR-TE policy must be manually configured for every (color, endpoint) pair. In a network with 200 PEs and 5 colors, that's 1,000 policies per head-end. ODN creates them automatically based on intent expressed through BGP color communities.

### 5. PCE and PCEP for SR-TE

A Path Computation Element (PCE) is an external server that computes paths on behalf of routers. PCEP (Path Computation Element Communication Protocol) is the interface between routers and the PCE.

**PCEP for SR-TE (RFC 8664):**
- PCC (Path Computation Client — the router) sends a computation request to the PCE.
- The request includes: endpoint, constraints (bandwidth, latency, affinity), optimization objective.
- PCE computes the path using its own TED (populated via BGP-LS from the network).
- PCE returns a segment list to the PCC.
- PCC installs the segment list as the active candidate path for the SR-TE policy.

**Stateful PCE (RFC 8231):**
- PCE maintains state of all LSPs/policies in the network.
- PCE can proactively update or reroute policies (PCE-initiated, RFC 8281).
- Enables global optimization: PCE sees all policies and can reoptimize holistically rather than each head-end optimizing independently.

**PCE and multi-domain**: PCE is the recommended solution for inter-area and inter-AS SR-TE. Each domain has a PCE; PCEs cooperate to compute end-to-end paths. The head-end gets a single segment list that spans domains.

### 6. Flexible Algorithm (Flex-Algo)

Flex-Algo allows the IGP to compute multiple independent topologies, each with its own metric type and constraints. SR-TE policies can steer traffic into a specific Flex-Algo topology without explicit segment lists.

**Core concept:**
- Standard IGP SPF uses algorithm 0 (shortest path, default metric).
- Flex-Algo defines additional algorithms (128-255) with custom properties.
- Each algorithm specifies: metric type (IGP, TE, latency, etc.), include/exclude admin groups, and calculation type.
- Routers participating in a Flex-Algo compute a separate SPF for that algorithm and allocate Prefix-SIDs specific to it.

**Flex-Algo definition (advertised via IGP):**
```
Algorithm 128:
  Metric type: minimum-delay
  Include-any: 0x00000002 (metro links only)
  Calculation type: SPF
```

All routers participating in Algo 128 run SPF using delay metrics over only metro-colored links. They advertise algorithm-specific Prefix-SID indices — the SRGB label block itself is shared, but the Prefix-SID association carries the algorithm identifier.

**Steering into Flex-Algo**: A packet with a Flex-Algo Prefix-SID follows the path computed by that algorithm. No explicit segment list needed — just one SID. This is dramatically simpler than explicit SR-TE segment lists for intent-based routing.

**Flex-Algo + ODN**: BGP color maps to a Flex-Algo. ODN creates a policy that uses the Flex-Algo Prefix-SID for the endpoint. One SID, one lookup per hop, automatic path computation. This is the simplest possible TE deployment.

**Flex-Algo limitations:**
- Each algorithm produces one shortest path (or ECMP set). You can't express arbitrary paths — only metric-optimized paths through constrained topologies.
- All routers must participate in the algorithm for end-to-end reachability.
- Algorithm definitions must be consistent across the network (advertised via IGP, but operational errors can create inconsistency).

### 7. SR-TE vs RSVP-TE Comparison

| Property | RSVP-TE | SR-TE |
|----------|---------|-------|
| Per-hop state | Yes (PATH/RESV at every transit) | No (source-routed, transit is stateless) |
| Refresh overhead | Yes (soft state, periodic refresh) | No |
| Signaling protocol | RSVP (IP protocol 46) | None (segment list in packet header) |
| Bandwidth reservation | Explicit (RSVP reserves on each link) | No native reservation (use Flex-Algo or controller-based) |
| FRR | RSVP-TE FRR (bypass/detour tunnels) | TI-LFA (pre-computed segment lists) |
| Path computation | CSPF at head-end or PCE | CSPF at head-end, PCE, or Flex-Algo |
| Scalability | Limited by per-hop state (tens of thousands) | Limited only by label stack depth |
| Inter-domain | Complex (stitching, nesting) | BSID chaining (simpler) |
| Bandwidth guarantee | Native | Requires external mechanism (PCE, controller) |

**SR-TE's trade-off**: Eliminates state and signaling overhead but loses native bandwidth reservation. In RSVP-TE, every transit router knows the bandwidth of every LSP and can perform admission control. In SR-TE, transit routers have no per-policy state — they can't reject a policy for exceeding bandwidth. Bandwidth management in SR-TE requires either PCE-based bandwidth accounting or Flex-Algo topology constraints.

## Protocol Interactions

**SR-TE ↔ BGP (color communities)**: BGP routes carry Color Extended Communities that signal TE intent. The head-end maps color to SR-TE policy (via ODN or static config). This is the primary service-to-TE binding mechanism — the color is the contract between the service layer and the transport layer.

**SR-TE ↔ IGP (Flex-Algo)**: Flex-Algo definitions are advertised via IGP. Each Flex-Algo computes an independent topology. SR-TE policies can use Flex-Algo Prefix-SIDs instead of explicit segment lists, delegating path computation to the IGP.

**SR-TE ↔ PCE (PCEP)**: PCE computes segment lists and installs them on routers via PCEP. Stateful PCE can optimize globally, rebalance traffic, and coordinate inter-domain paths. PCEP is the control interface between centralized intelligence and distributed forwarding.

**SR-TE ↔ TI-LFA**: TI-LFA provides sub-50ms protection for SR-TE policies. When a link/node fails, TI-LFA's pre-computed backup segment list takes over. No bypass tunnels, no RSVP FRR — just a different segment list pre-installed in the FIB.

## Edge Cases & Gotchas

**1. Label stack depth limits**: SR-TE segment lists can require 3-8+ labels (node SIDs + adj SIDs + VPN label + TI-LFA backup). Some platforms limit label stack depth to 5 or 6. Deep stacks also increase MTU requirements. Always verify platform label stack limits before deploying complex SR-TE policies.

**2. Flex-Algo participation gaps**: If a router in the middle of the network doesn't participate in Flex-Algo 128, there's a routing hole. Traffic using Algo 128 SIDs can't transit that router. All routers on the intended path must participate. This is an operational consistency challenge.

**3. BSID collision**: BSIDs are locally allocated. If two policies accidentally get the same BSID (misconfiguration), traffic intended for one policy may enter the other. Use dynamic BSID allocation from a dedicated range to prevent this.

**4. ODN template mismatch**: If an ODN template's constraints are too restrictive for some endpoints, ODN will fail to create policies for those endpoints. The BGP routes are received, the color is present, but no SR-TE policy exists — traffic falls back to IGP shortest path without TE. Monitor ODN policy creation failures.

**5. PCE single point of failure**: If the PCE fails and all policies were PCE-computed, routers have no path computation capability. Policies may go stale or fail to adapt to topology changes. Deploy redundant PCEs and configure local CSPF as fallback.

**6. SR-TE has no native bandwidth reservation**: Unlike RSVP-TE, transit routers don't track per-policy bandwidth. You can oversubscribe a link with SR-TE policies without any admission control failure. Bandwidth management requires external tooling (PCE, controller, monitoring).
