# 5.1 — TE Fundamentals — Answer Key

## Question 1
> You configure a TE tunnel requesting 5 Gbps with `affinity 0x5 mask 0x5`. The tunnel fails to come up. TED shows all links have affinity `0x00000003`. Why does CSPF fail, and what's the fix?

### Answer

**The affinity constraint is pruning every link in the topology.**

Let's decode it:
- Tunnel affinity: `0x5` = binary `0101` (bits 0 and 2 set)
- Tunnel mask: `0x5` = binary `0101` (bits 0 and 2 are significant)
- Link affinity: `0x3` = binary `0011` (bits 0 and 1 set)

The affinity check formula (include-all style with mask): `(link_affinity & mask) == (tunnel_affinity & mask)`

Evaluating:
- `link_affinity & mask` = `0x3 & 0x5` = `0x1` (binary `0001`)
- `tunnel_affinity & mask` = `0x5 & 0x5` = `0x5` (binary `0101`)
- `0x1 ≠ 0x5` → **FAIL**

The tunnel requires bit 2 to be set on all traversed links (because of the mask). No links in the network have bit 2 set — they only have bits 0 and 1. CSPF prunes every link, finds no valid path, and the tunnel stays down.

**Fix options:**

1. **Change the tunnel constraint** to match what exists:
```
! Only require bit 0 (which all links have)
tunnel mpls traffic-eng affinity 0x1 mask 0x1
```

2. **Add bit 2 to the appropriate links:**
```
interface TenGigabitEthernet0/0/0
 mpls traffic-eng attribute-flags 0x00000007
 ! Sets bits 0, 1, and 2
```

3. **Use include-any instead of include-all** if the intent is "at least one of these attributes" rather than "all of these attributes."

**Exam tip:** Affinity math is pure bitmask logic. Draw out the binary every time. Don't try to do hex in your head during an exam — you will get bitten.

## Question 2
> Your backbone has 400G core links and 100G edge links. You want TE tunnels to prefer the 400G path even when the 100G direct path has fewer hops. How would you configure metrics to achieve this without changing IGP routing?

### Answer

Use **TE metric** independently of IGP metric. TE metric only affects CSPF path computation for TE tunnels — it does not change the IGP routing table.

**Design:**
- Assign **low TE metric** to 400G core links (e.g., 10)
- Assign **high TE metric** to 100G edge links (e.g., 100)
- Leave IGP metric unchanged (so normal IGP routing is unaffected)

A 2-hop path through 400G links has TE cost = 20. The direct 1-hop 100G path has TE cost = 100. CSPF picks the 400G path.

**IOS-XE:**
```
! 400G core links
interface HundredGigE0/0/0
 mpls traffic-eng administrative-weight 10

! 100G edge links
interface TenGigabitEthernet0/1/0
 mpls traffic-eng administrative-weight 100
```

**Junos:**
```
! 400G core links
set protocols isis interface et-0/0/0.0 level 2 te-metric 10

! 100G edge links
set protocols isis interface xe-0/1/0.0 level 2 te-metric 100
```

**Why not just use IGP metric?** Because IGP metric affects ALL traffic — not just TE tunnels. Changing IGP metric to prefer 400G paths would reroute everything, including non-tunneled traffic, potentially breaking traffic engineering policies that depend on IGP cost. TE metric is a separate knob specifically for this purpose.

**Design pattern this enables:** Use IGP metric for capacity-based shortest-path (reference bandwidth / link speed), and TE metric for whatever constraint matters for tunnels (latency, cost, admin preference). A 100G terrestrial link and 100G submarine link might have the same IGP metric but wildly different TE metrics.

## Question 3
> A TE tunnel shows `Oper: up, Path: valid` but traffic isn't flowing through it. The destination prefix still uses the IGP next-hop. What's missing?

### Answer

**The tunnel needs to be injected into the routing table.** A TE tunnel being operationally up only means RSVP signaling succeeded and the LSP is established. It does NOT automatically mean traffic uses it. You need to tell the routing table to use the tunnel as a next-hop.

Three mechanisms, and at least one is missing:

**1. Autoroute announce (most common):**
```
IOS-XE:
  interface Tunnel100
   tunnel mpls traffic-eng autoroute announce
   ! Injects tunnel as next-hop for all prefixes reachable via tunnel tail-end

Junos:
  set protocols mpls label-switched-path to-PE5 install <prefix>/32
  ! Or for all prefixes:
  set protocols mpls label-switched-path to-PE5 install 0.0.0.0/0
```

**2. Forwarding adjacency:**
```
IOS-XE:
  interface Tunnel100
   tunnel mpls traffic-eng forwarding-adjacency
   ! Advertises tunnel as an IGP link — other routers can use it too
```

**3. Static route pointing to tunnel:**
```
IOS-XE:
  ip route 10.0.0.5 255.255.255.255 Tunnel100
```

**4. Policy-based routing (PBR):**
Route-map matching specific traffic and setting next-hop to the tunnel.

**The most likely missing piece is `autoroute announce`.** It's the standard approach for PE-to-PE TE tunnels. Without it, the tunnel is up but the routing table doesn't know it exists as a valid forwarding path.

**Verification:**
```
IOS-XE:
  show ip route 10.0.0.5
  ! Should show "via Tunnel100" not "via <physical interface>"
  show ip cef 10.0.0.5
  ! Verify CEF points to tunnel

Junos:
  show route 10.0.0.5/32
  ! Check next-hop is the LSP, not direct IGP path
```

## Question 4
> You're deploying a full-mesh of TE tunnels across 200 PE routers. Calculate the number of tunnels and RSVP sessions. What mechanisms would you use to keep this manageable?

### Answer

**Full-mesh tunnel count:** n × (n-1) = 200 × 199 = **39,800 tunnels**

Each tunnel requires an RSVP session (PATH + RESV state). In full-mesh, each PE originates 199 tunnels and terminates 199 tunnels. That's **398 RSVP sessions per PE** (199 ingress + 199 egress).

But the real problem is on **P routers** — transit nodes carry state for every tunnel that passes through them. A central P router might transit 30,000+ of those 39,800 tunnels, each requiring RSVP refresh messages every 30 seconds.

**Mechanisms to manage this:**

**1. RSVP Refresh Reduction (RFC 2961) — mandatory**
```
IOS-XE: ip rsvp signalling refresh reduction
Junos:  set protocols rsvp interface all aggregate (summary refresh)
```
Bundles multiple refresh messages into a single packet. Reduces RSVP control-plane load by ~10x.

**2. RSVP Authentication + Summary Refresh**
Summary refresh replaces individual PATH/RESV refreshes with a compact message listing all active sessions by ID. Dramatically reduces per-session overhead.

**3. Hierarchical TE / Hub-and-Spoke**
Instead of full-mesh, use a hierarchical model:
- PEs build tunnels only to core P routers (hub nodes)
- P routers build tunnels between each other
- PE count: 200 PEs × 8 P-hubs = 1,600 PE tunnels
- P-P mesh: 8 × 7 = 56 P-P tunnels
- Total: ~1,656 tunnels vs 39,800. **24x reduction.**

**4. Auto-mesh**
```
IOS-XE: mpls traffic-eng auto-tunnel mesh
Junos:  set protocols mpls auto-policing / mesh-group
```
Auto-mesh creates tunnels on demand to discovered PEs, reducing manual config but not state.

**5. Consider SR-TE migration**
SR-TE eliminates per-hop RSVP state entirely. 39,800 SR-TE policies create zero state on transit P routers. This is the strongest argument for SR-TE at scale.

## Question 5
> Explain why hold priority should always be numerically ≤ setup priority (hold stronger than setup). What happens if setup=3 and hold=5?

### Answer

RSVP-TE priorities range from 0 (highest) to 7 (lowest). **Lower number = higher priority.**

- **Setup priority**: The priority used when CSPF checks if bandwidth is available and when the tunnel is first established. It competes against other tunnels' **hold** priorities.
- **Hold priority**: The priority used to defend the tunnel against preemption by new tunnels.

**Why hold ≤ setup (numerically, meaning hold is at least as strong):**

If a tunnel can SET UP at priority 3 but only HOLDS at priority 5, here's what happens:

1. Tunnel A establishes at setup=3, claiming bandwidth
2. Tunnel A now holds at priority 5 (weaker)
3. Tunnel B comes along with setup=4
4. Tunnel B's setup (4) beats Tunnel A's hold (5) → Tunnel A is **preempted**
5. Tunnel A tries to re-establish at setup=3 → it preempts Tunnel B (setup 3 beats hold of whatever B has)
6. Tunnel B re-establishes, preempts A again...

**This creates a preemption loop (thrashing).** Tunnels endlessly preempt each other because they're strong enough to establish but too weak to hold. The network oscillates, generating constant RSVP signaling storms and traffic disruption.

**Correct design:**
- `priority 3 3` — setup and hold are equal. Tunnel can establish and defend at the same level.
- `priority 3 2` — tunnel holds stronger than it sets up. Once established, it's harder to kick out. This is sometimes used for critical tunnels that shouldn't be preempted once running.

**Never `priority 3 5`** — that's a self-destructing tunnel configuration. Most modern implementations will warn or reject this, but older IOS versions will happily let you shoot yourself in the foot.
