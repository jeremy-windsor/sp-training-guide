# 5.2 — RSVP-TE Advanced — Protocol Theory

## Protocol Overview

RSVP-TE Advanced covers the mechanisms that transform basic RSVP-TE from a static tunnel signaling protocol into a dynamic, self-optimizing, failure-resilient system. The base RSVP-TE specification (RFC 3209) provides explicit path setup. The advanced features — auto-bandwidth, soft preemption, inter-area/inter-AS TE, reoptimization, SRLG-aware protection, and path diversity — are what make RSVP-TE viable at SP scale.

These features address the operational reality that an SP network is not static. Traffic patterns shift hourly, links fail unpredictably, maintenance windows require graceful traffic drain, and the network spans multiple IGP areas or autonomous systems. Without these features, every TE tunnel would require manual provisioning, manual bandwidth adjustment, and manual rerouting — defeating the purpose of automation.

## Core Mechanisms

### 1. Auto-Bandwidth

Auto-bandwidth dynamically adjusts an LSP's reserved bandwidth based on measured traffic utilization.

**Mechanism:**
1. The head-end router samples the LSP's traffic rate at a configured interval (e.g., every 5 minutes).
2. At the end of an adjustment interval (e.g., every 24 hours, or every 1 hour), the head-end calculates the maximum, average, or percentile traffic rate observed during the interval.
3. If the calculated rate differs from the current reservation by more than a threshold, the head-end re-signals the LSP with the new bandwidth using Make-Before-Break.
4. CSPF runs with the new bandwidth constraint, potentially finding a different path if the current path can't accommodate the new reservation.

**Key parameters:**
- **Collection interval**: How often traffic rate is sampled (seconds to minutes).
- **Adjustment interval**: How often bandwidth is recalculated and potentially changed (minutes to hours).
- **Minimum bandwidth**: Floor — never reserve less than this, even if traffic drops to zero.
- **Maximum bandwidth**: Ceiling — never reserve more than this, even during traffic spikes.
- **Adjustment threshold**: Minimum change (absolute or percentage) required to trigger re-signaling. Prevents oscillation from minor fluctuations.
- **Overflow/underflow**: Optional rapid adjustment when traffic exceeds or drops below current reservation by a large margin, without waiting for the full adjustment interval.

**Overflow detection**: If the measured rate exceeds the current reservation by an overflow threshold for N consecutive samples, trigger immediate bandwidth increase. This handles sudden traffic surges (DDoS, flash crowds) without waiting for the normal adjustment cycle.

**Underflow detection**: If the measured rate drops below a threshold for N consecutive samples, trigger immediate bandwidth decrease. Frees capacity for other LSPs.

**Stability considerations**: Auto-bandwidth can oscillate if parameters are too aggressive. A traffic pattern that cycles between high and low rates will cause continuous re-signaling. The adjustment threshold and minimum change interval prevent this. Some implementations also use dampening — requiring the new bandwidth to be stable for a period before re-signaling.

### 2. Soft Preemption (RFC 5712)

Standard RSVP-TE preemption is hard: when a higher-priority LSP needs bandwidth, lower-priority LSPs on the same link are torn down immediately. Traffic on preempted LSPs is disrupted until the head-end re-signals on a new path.

Soft preemption provides a graceful alternative.

**Mechanism:**
1. A higher-priority LSP requests bandwidth on a link.
2. The link determines that a lower-priority LSP must be preempted.
3. Instead of tearing down the lower-priority LSP, the router sends a PathErr message with a "soft preemption" indication to the lower-priority LSP's head-end.
4. The head-end has a grace period to re-signal the LSP on a new path using MBB.
5. During the grace period, the preempted LSP continues to carry traffic on the original path.
6. Once MBB completes, the old path is torn down and bandwidth is released.
7. If the grace period expires without successful reroute, hard preemption occurs.

**Benefits:**
- Zero traffic loss during preemption (if MBB succeeds within grace period)
- Graceful capacity management
- Allows controlled bandwidth rebalancing

**Grace period**: Configurable, typically 30-120 seconds. Must be long enough for CSPF + RSVP signaling + MBB on the new path.

### 3. Reoptimization

RSVP-TE LSPs may follow suboptimal paths over time as network conditions change. Reoptimization periodically re-evaluates paths.

**Timer-based reoptimization:**
- Head-end periodically runs CSPF (e.g., every 300 seconds).
- If a better path exists (shorter TE metric, more bandwidth headroom), the LSP is re-signaled via MBB.
- If no better path exists, nothing changes.

**Event-triggered reoptimization:**
- A link comes up (new capacity available) → head-end re-evaluates.
- IGP-TE update shows increased bandwidth on a previously constrained link → head-end re-evaluates.
- Operator manually triggers reoptimization.

**Reoptimization and MBB**: Reoptimization always uses MBB to avoid traffic disruption. The old LSP carries traffic until the new path is fully established.

**Reoptimization scope**: Some implementations allow configuring which LSPs are eligible for reoptimization. Low-priority best-effort LSPs might reoptimize frequently; high-priority latency-sensitive LSPs might be pinned to specific paths.

### 4. SRLG (Shared Risk Link Group)

An SRLG identifies a set of links that share a common physical fate — failure of one means failure of all.

**Common SRLGs:**
- Links in the same fiber conduit
- Links on the same DWDM system
- Links crossing the same bridge or tunnel
- Links powered by the same generator

**SRLG encoding**: Each link is assigned one or more SRLG values (32-bit integers). These are advertised via IGP-TE extensions and stored in the TED.

**SRLG-aware CSPF**: When computing a backup path (FRR bypass or diverse path), CSPF can exclude links that share an SRLG with the primary path. This ensures the backup survives the same physical failure that takes out the primary.

**SRLG-aware FRR**: When building a bypass tunnel for FRR, the bypass path must not share an SRLG with the protected link. Otherwise, a fiber cut takes out both the primary and the bypass simultaneously.

**IS-IS SRLG advertisement**: Carried in sub-TLV under TLV 22 (link SRLG, RFC 5307) or via a dedicated TLV for SRLG constraints.

**OSPF SRLG advertisement**: Carried in sub-TLV of the TE Link TLV in Type 10 Opaque LSAs.

### 5. Inter-Area TE

Standard CSPF operates within a single IGP area because the TED only contains intra-area link information. TE across area boundaries requires special handling.

**Loose hop expansion:**
1. Head-end computes an ERO with strict hops within its area and a loose hop to the destination (or ABR) in the remote area.
2. The ABR receives the PATH message with the loose hop.
3. The ABR runs CSPF within its area to expand the loose hop into strict hops.
4. The ABR forwards the PATH with the expanded ERO.
5. This continues area by area until the tail-end is reached.

**Limitations:**
- Each area computes independently — the overall path may not be globally optimal.
- Bandwidth constraints may be violated if areas have inconsistent views.
- No globally optimal path guarantee — the head-end lacks cross-area TE visibility. RSVP still reserves bandwidth on the actual signaled path, but without coordinated computation (BRPC/PCE) the result may be suboptimal.

**PCE-based inter-area TE (RFC 5441)**: A Path Computation Element (PCE) in each area cooperates to compute a globally optimal constrained path. The head-end sends a computation request to its local PCE, which cooperates with PCEs in other areas using PCEP (Backward-Recursive PCE-based Computation, BRPC).

### 6. Inter-AS TE

TE across autonomous system boundaries is even more complex than inter-area because ASes don't share IGP topology.

**Three approaches:**

**Contiguous LSP**: A single RSVP-TE LSP spans both ASes. Requires RSVP-TE adjacency at the AS boundary and mutual trust (sharing TE topology or using loose hops). Rare in practice — ASes typically don't trust each other's RSVP signaling.

**Stitched LSP**: Separate LSPs in each AS, stitched at the boundary. Each AS independently computes and signals its segment. At the AS boundary router, the two LSPs are joined by label stitching (swap the tail-end label of AS1's LSP with the head-end label of AS2's LSP). Provides per-AS autonomy.

**Nested LSP**: AS1's LSP tunnels through AS2's LSP. The transit AS provides a transport tunnel; the end-to-end LSP rides inside it. AS2 only needs to provide a tunnel — it doesn't need to know about AS1's TE requirements.

**Reality**: Most inter-AS TE uses either stitched LSPs with PCEP coordination or SR-TE (where each AS computes its own segment list and the head-end combines them). Full inter-AS RSVP-TE with contiguous LSPs is rare outside of single-operator multi-AS networks.

### 7. Path Diversity

Path diversity ensures that multiple LSPs between the same endpoints follow physically different paths — surviving any single failure.

**Diversity types:**
- **Link diverse**: No shared links between primary and backup paths.
- **Node diverse**: No shared transit nodes (stronger — also implies link diversity for transit links).
- **SRLG diverse**: No shared SRLGs (strongest — covers shared physical infrastructure).

**Implementation:**
1. Compute and signal the primary LSP.
2. For the diverse LSP, add the primary path's links/nodes/SRLGs to CSPF's exclusion list.
3. Run CSPF on the constrained topology.
4. If a diverse path exists, signal it.

**Secondary standby LSPs**: Some implementations support a standby LSP that is fully signaled (labels allocated, state installed at each hop) but doesn't carry traffic. On primary failure, traffic switches to the standby instantly — no CSPF or signaling delay. More expensive in state but faster failover than computing a new path on demand.

## Protocol Interactions

**Auto-bandwidth ↔ CSPF**: Auto-bandwidth re-signals LSPs with new bandwidth, triggering CSPF. If the current path can't accommodate increased bandwidth, CSPF finds a new path. This can cause LSPs to jump between paths as traffic fluctuates — dampening parameters prevent oscillation.

**Soft preemption ↔ MBB**: Soft preemption depends on MBB to reroute the preempted LSP without traffic loss. If MBB fails (no alternative path), hard preemption follows. MBB's SE style is essential — it allows the old and new LSP to share bandwidth during transition.

**SRLG ↔ FRR**: SRLG-aware FRR bypass tunnels are the gold standard for physical protection. Without SRLG awareness, a bypass tunnel may share the same fiber conduit as the protected link, providing no actual protection.

**Inter-area TE ↔ PCE**: For optimal inter-area paths, PCE with BRPC is the recommended approach. Per-domain loose hop expansion is simpler but produces locally optimal, globally suboptimal paths.

## Edge Cases & Gotchas

**1. Auto-bandwidth oscillation**: If traffic fluctuates around the adjustment threshold, the LSP constantly re-signals. Each re-signal triggers MBB, CSPF, IGP-TE flooding. At scale (thousands of auto-bandwidth LSPs), this creates a network-wide oscillation storm. Set adjustment thresholds high enough (10-20%) and minimum change intervals long enough (minutes) to prevent this.

**2. Soft preemption grace period too short**: If the grace period expires before MBB completes (CSPF takes time, signaling traverses many hops), hard preemption occurs anyway. Size the grace period for worst-case CSPF + signaling time in your network.

**3. SRLG data quality**: SRLG protection is only as good as the SRLG data. If the fiber plant database is incomplete or wrong, links that share a conduit may not share an SRLG value, and SRLG-diverse paths may not actually be diverse. SRLG data maintenance is an ongoing operational task.

**4. Inter-area TE bandwidth fragmentation**: When each area independently reserves bandwidth for a cross-area LSP, the reservation in one area may succeed while the other area doesn't have capacity. The result is a partial reservation that wastes bandwidth in the area that succeeded. PCE coordination mitigates this.

**5. Diverse path impossibility**: In some topologies, truly diverse paths don't exist (single fiber route between two cities). CSPF will fail to find a diverse path, and the operator must accept the risk or invest in physical infrastructure. The algorithm can't create paths that don't physically exist.
