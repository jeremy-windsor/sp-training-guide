# 5.3 — Segment Routing TE — Answer Key

## Question 1
> You have an SR-TE policy with color 100 to endpoint 10.0.0.5. A BGP VPN route arrives with next-hop 10.0.0.5 and color community 100. The route is not being steered into the SR policy. What are the three most likely causes?

### Answer

**1. Color-extended community is not being processed or matched correctly.**

The BGP route must carry the exact color extended community that matches the SR-TE policy color. Verify:
```
IOS-XR: show bgp vpnv4 unicast <prefix> detail
        ! Look for "Color: 100" in extended communities

Junos:  show route <prefix> extensive
        ! Check for color:0:100 in communities
```

If the color community isn't attached to the route, the BGP VPN prefix has no signal to steer into the SR policy. Check the route-policy at the originating PE — is it attaching `extcommunity color(100)` or `community color:0:100`?

**2. SR-TE policy is not in an active/operational state.**

The policy exists in config but may not have a valid candidate path. If no candidate path is active (all segment lists failed validation, PCE returned no path, or dynamic computation found no valid path), the policy is "down" and BGP won't steer traffic into it.

```
IOS-XR: show segment-routing traffic-eng policy color 100 endpoint ipv4 10.0.0.5
        ! Check: Admin: up, Operational: up, candidate path status

Junos:  show spring-traffic-engineering lsp detail
        ! Verify LSP state is "Up"
```

**3. On-Demand Next-Hop (ODN) or automatic steering is not enabled.**

BGP Automatic Steering (AS) must be configured for the address-family. Without it, the router receives the colored BGP route but doesn't look up a matching SR-TE policy.

```
IOS-XR:
  segment-routing traffic-eng
   on-demand color 100
    dynamic
     pcep
     ! or: metric type latency / igp
   !
  router bgp 65000
   address-family vpnv4 unicast
    ! BGP AS is implicit in XR when SR-TE policies exist

Junos:
  set protocols source-packet-routing use-transport-class
  ! Or explicit steering via routing-policy
```

**Other less common causes:**
- Next-hop mismatch: the BGP next-hop is 10.0.0.5 but the SR policy endpoint resolves to a different address (e.g., IPv6 endpoint vs IPv4 next-hop)
- Policy preference: a higher-preference candidate path exists but points to a different color
- The policy has a binding SID but autoroute/steering isn't configured to use it

## Question 2
> Compare deploying 1,000 RSVP-TE tunnels vs 1,000 SR-TE policies across a 50-node backbone. What's the control plane state difference on transit routers? Where does each approach hit scaling limits first?

### Answer

### RSVP-TE: 1,000 Tunnels

**State on transit P routers:**
- Each tunnel creates **PATH state** and **RESV state** at every transit hop
- A central P router transiting 800 of those 1,000 tunnels maintains: 800 PATH states + 800 RESV states = **1,600 RSVP sessions**
- Each session requires periodic refresh (default every 30 seconds)
- With summary refresh: ~32 refresh bundles per cycle (50 sessions per bundle)
- Without summary refresh: **1,600 individual refresh messages every 30 seconds** on a single P router. That's 53 messages/second just for refresh.

**Total control-plane state across backbone:**
Average path length ~4 hops × 1,000 tunnels = ~4,000 transit session instances network-wide.

**Where it hits limits first:**
- **RSVP session count on central P routers** — most platforms start struggling at 10K-50K sessions. At 1,000 tunnels you're fine, but growth to 5,000-10,000 tunnels starts hurting.
- **RSVP refresh processing** — even with summary refresh, the CPU cost of maintaining state is proportional to tunnel count × average path length.
- **Convergence time** — when a link fails, every tunnel transiting that link needs re-signaling. 500 simultaneous re-signals is a storm.

### SR-TE: 1,000 Policies

**State on transit P routers:**
- **Zero.** Transit routers have no per-policy state. They perform normal MPLS label operations based on the SRGB (Segment Routing Global Block). The label stack is imposed at the head-end; transit nodes just swap prefix-SIDs.
- No RSVP sessions, no refresh messages, no PATH/RESV state.

**State on head-end routers:**
- Each policy has a candidate path with one or more segment lists — this is **local state only**.
- 1,000 policies on a single head-end is trivially manageable (it's just 1,000 forwarding entries).

**Where SR-TE hits limits first:**
- **Label stack depth (MSD — Maximum SID Depth)**: If policies require many SIDs (e.g., 8-10 segment strict paths), the hardware ASIC may not support deep enough stacks. Most modern platforms support 6-10 SIDs. Older hardware may only support 3-5.
- **Head-end forwarding table size**: With 1,000 policies × multiple segment lists, the head-end needs corresponding LFIB entries. This is rarely a problem on modern hardware.
- **PCE scalability**: If using centralized PCE for all 1,000 policies, the PCE server becomes the bottleneck — not the routers.

### Summary

| Metric | RSVP-TE (1,000) | SR-TE (1,000) |
|--------|-----------------|---------------|
| Transit router state | ~1,600 sessions per central P | **Zero** |
| Refresh overhead | 53 msg/sec per central P | **None** |
| Failure convergence | Re-signal 500+ tunnels | TI-LFA (local repair, instant) |
| Bandwidth guarantees | Native admission control | External (controller-based) |
| First scaling wall | Transit RSVP session count | Label stack depth (MSD) |

SR-TE wins on scalability by an absurd margin. RSVP-TE wins on bandwidth guarantees. At 1,000 policies, RSVP-TE is manageable. At 10,000, SR-TE is the only sane answer.

## Question 3
> PCE computes a path with 9 SIDs. Packets are being dropped at the 4th transit node. How do you diagnose this? What's the fix that doesn't require hardware replacement?

### Answer

**Most likely cause: the 4th transit node's ASIC has a forwarding/parser limit that can't handle a 9-deep label stack.** Note: advertised MSD (RFC 8491) is primarily an imposition capability used by the head-end to constrain segment list depth. A mid-path drop from a deep stack is a platform forwarding limitation, not directly the node's advertised MSD value.

**Diagnosis:**

**1. Check the MSD of the dropping node:**
```
IOS-XR: show isis database detail <node-id> | include MSD
        show segment-routing traffic-eng topology | include MSD

Junos:  show isis database <node-id> extensive | match "Maximum SID"
        show spring-traffic-engineering ted database
```
If MSD = 5 and the packet arrives with 6+ labels remaining, the node can't forward it.

**2. Verify packet drops:**
```
IOS-XR: show mpls forwarding drop
        show controllers np counters all | include drop|discard

Junos:  show pfe statistics traffic | match discard
        show interface queue counters
```

**3. Trace the label stack at each hop:**
```
IOS-XR: traceroute segment-routing mpls policy name <policy> 
        ! Shows label stack depth at each hop

Junos:  traceroute mpls segment-routing <policy>
```

**Fix without hardware replacement — reduce the SID count:**

**1. Use prefix-SIDs (node SIDs) instead of adjacency-SIDs where possible.**
Prefix-SIDs are "loose" — they route via IGP shortest path to the node, potentially skipping multiple hops with one SID. Adjacency-SIDs are "strict" — one SID per hop. Converting strict hops to loose hops can collapse 9 SIDs to 3-4.

Example:
```
Before (9 SIDs, all adjacency):
  [adj-1, adj-2, adj-3, adj-4, adj-5, adj-6, adj-7, adj-8, adj-9]

After (4 SIDs, mix of node + adjacency):
  [node-P2, adj-critical-link, node-P7, node-PE9]
```

**2. Use Binding SID (BSID) to split the path into segments.**
Configure an intermediate node with a BSID that encapsulates the remaining path:
```
Node P4 BSID policy:
  BSID: 24500 (local label)
  Segment list: [SID-5, SID-6, SID-7, SID-8, SID-9]

Head-end policy becomes:
  Segment list: [SID-1, SID-2, SID-3, 24500]  ← only 4 SIDs!
```
When the packet reaches P4, the BSID pops and a new 5-SID stack is imposed. Neither the head-end nor P4 exceeds their MSD.

**3. Request PCE to respect MSD constraints.**
Configure PCE to advertise and honor MSD:
```
IOS-XR:
  segment-routing traffic-eng
   pce
    segment-routing
     max-sid-depth 6
```
PCE will then compute paths that never exceed the minimum MSD along the path, using BSID stitching or node-SID optimization automatically.

## Question 4
> Explain how binding SID enables multi-domain SR-TE without any domain needing visibility into another domain's topology. Draw the label stack at each hop for a 3-domain path.

### Answer

**Binding SID (BSID)** is a locally significant label allocated to an SR-TE policy. When a packet arrives with the BSID as the top label, the router pops it and steers the packet into the associated policy (imposing that policy's segment list).

This creates **opacity between domains**. Domain 1 doesn't need to know Domain 2's topology — it just knows that pushing BSID-X at the domain boundary will get the packet through Domain 2 correctly.

### 3-Domain Path Example

```
Domain 1 (AS 65001)     Domain 2 (AS 65002)     Domain 3 (AS 65003)
PE1 → P2 → ASBR1  ───  ASBR2 → P5 → ASBR3 ── ASBR4 → P7 → PE8
```

**Policies configured:**

On **PE1** (head-end):
- SR-TE Policy to PE8
- Segment list: `[16002, 24001]`
  - 16002 = prefix-SID of ASBR1 (traverse Domain 1)
  - 24001 = BSID on ASBR1 (handles everything beyond Domain 1)

On **ASBR1** (Domain 1/2 border):
- SR-TE Policy: BSID 24001
- Segment list: `[16502, 24002]`
  - 16502 = prefix-SID of ASBR3 in Domain 2
  - 24002 = BSID on ASBR3 (handles Domain 3)

On **ASBR3** (Domain 2/3 border):
- SR-TE Policy: BSID 24002
- Segment list: `[16707, 16708]`
  - 16707 = prefix-SID of P7 in Domain 3
  - 16708 = prefix-SID of PE8 in Domain 3

### Label Stack at Each Hop

| Hop | Top Label | Stack (top→bottom) | Action |
|-----|-----------|---------------------|--------|
| **PE1** (ingress) | 16002 | `[16002, 24001, VPN]` | Push 3-label stack. 16002 routes to ASBR1 via Domain 1 IGP |
| **P2** (transit, Domain 1) | 16002 | `[16002, 24001, VPN]` | Swap 16002 → PHP or next label for ASBR1 |
| **ASBR1** | 24001 | `[24001, VPN]` | Pop 24001 (BSID). Impose Domain 2 stack: `[16502, 24002, VPN]` |
| **ASBR2** (Domain 2 entry) | 16502 | `[16502, 24002, VPN]` | Forward toward ASBR3 via Domain 2 IGP |
| **P5** (transit, Domain 2) | 16502 | `[16502, 24002, VPN]` | Swap/PHP toward ASBR3 |
| **ASBR3** | 24002 | `[24002, VPN]` | Pop 24002 (BSID). Impose Domain 3 stack: `[16707, 16708, VPN]` |
| **ASBR4** (Domain 3 entry) | 16707 | `[16707, 16708, VPN]` | Forward toward P7 |
| **P7** (transit, Domain 3) | 16708 | `[16708, VPN]` | PHP — pop 16708 |
| **PE8** (egress) | VPN | `[VPN]` | Pop VPN label, deliver to customer VRF |

**What each domain sees:**
- **Domain 1**: Knows about prefix-SID 16002 (ASBR1) and forwards normally. Has no idea Domains 2 or 3 exist.
- **Domain 2**: Knows about prefix-SID 16502 (ASBR3) and BSID 24002. No knowledge of Domain 1 or 3 internals.
- **Domain 3**: Knows about prefix-SIDs 16707 and 16708. No knowledge of anything upstream.

**This is the killer feature of BSID:** each domain is a black box to the others. You can change Domain 2's entire internal topology, re-number nodes, add/remove links — as long as ASBR1's BSID 24001 still points to a working policy, PE1 never knows or cares.

## Question 5
> You define flex-algo 128 with metric-type `min-unidirectional-delay` and exclude admin-group "submarine." Three out of five nodes in the shortest delay path don't participate in algo 128. What happens to the flex-algo SPF computation?

### Answer

**Non-participating nodes are excluded from the flex-algo topology entirely.** They are invisible to the algo 128 SPF computation.

When a node does not advertise participation in flex-algo 128 (via FAD — Flex-Algo Definition advertisement in IS-IS/OSPF), all participating nodes **prune that node and all its links** from the algo 128 topology before running SPF.

So the effect is:

1. The 5-node shortest-delay path exists in algo 0 (default)
2. For algo 128, 3 of those 5 nodes are removed from the topology
3. Algo 128 SPF runs on the **pruned topology** (only nodes that participate)
4. The result is either:
   - **A longer but valid path** through only participating nodes — this path may have higher delay than the algo-0 optimal path, but it's the best achievable within the participating topology
   - **No path at all** — if the non-participating nodes were the only connectivity between participating nodes, the algo 128 topology is partitioned and some destinations become unreachable in that algorithm

**What does NOT happen:**
- Non-participating nodes are NOT used as transit even if they'd provide a shorter path
- The algorithm does NOT fall back to algo 0 for those nodes — flex-algo is strict
- Traffic steered into algo 128 prefix-SIDs will NOT traverse non-participating nodes

**Practical implications:**
- You must ensure **all nodes along at least one viable path** participate in the flex-algo
- Partial deployment creates islands. If your low-latency algo only covers 60% of nodes, some PE-to-PE paths simply won't exist in that algo
- Verify with: `show isis flex-algo 128` (IOS-XR) or `show isis spring flex-algorithm 128` (Junos) — check which nodes are participating and whether the topology is connected

**Design rule:** Deploy flex-algo participation on ALL core/transit nodes first, then PEs. An algo that only some P routers participate in will have path gaps that are painful to debug.
