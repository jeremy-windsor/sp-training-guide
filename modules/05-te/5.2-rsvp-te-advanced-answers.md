# 5.2 — RSVP-TE Advanced — Answer Key

## Question 1
> Auto-bandwidth on Tunnel100 shows a current sample of 2 Gbps, but the reserved bandwidth is still at the initial 500 Mbps. The adjustment interval is 86400s with 43200s remaining. What mechanism would cause an immediate adjustment, and what config enables it?

### Answer

**Overflow detection** would cause an immediate adjustment without waiting for the full 86400s interval to expire.

Overflow detection triggers when traffic exceeds the current reservation by a configured threshold for a specified number of consecutive samples. In this case, traffic (2 Gbps) is 4x the reservation (500 Mbps) — a 300% overshoot. If overflow is configured with, say, a 15% threshold and a count of 3, the system would detect the overflow within 3 sampling periods and immediately re-signal the tunnel via make-before-break.

**IOS-XE config to enable overflow:**
```
interface Tunnel100
 tunnel mpls traffic-eng auto-bw
  frequency 86400
  min-bw 100000
  max-bw 5000000
  overflow-threshold 15 3
  ! If traffic exceeds reservation by 15% for 3 consecutive samples → immediate adjust
  collect-bw
```

**Junos config:**
```
set protocols mpls label-switched-path Tunnel100 auto-bandwidth
set protocols mpls label-switched-path Tunnel100 auto-bandwidth adjust-interval 86400
set protocols mpls label-switched-path Tunnel100 auto-bandwidth overflow-limit 3
set protocols mpls label-switched-path Tunnel100 auto-bandwidth adjust-threshold-overflow-limit 15
set protocols mpls label-switched-path Tunnel100 auto-bandwidth monitor-bandwidth
```

**What happens when overflow triggers:**
1. Auto-bandwidth samples detect traffic > reservation + threshold for N consecutive periods
2. Head-end immediately initiates MBB with the new measured bandwidth
3. New LSP is signaled (CSPF re-runs with new bandwidth constraint)
4. If CSPF succeeds, old LSP torn down, new LSP active with higher reservation
5. The adjustment-interval timer resets

**Why not just use a shorter adjustment interval?** Because short intervals (e.g., 300s) cause constant MBB churn across all tunnels, not just the one that's overloaded. Overflow detection is surgical — it only fires for tunnels that are genuinely under-provisioned, leaving stable tunnels alone.

**The symmetric feature — underflow detection** — does the same in reverse: if traffic drops well below reservation for N samples, reclaim the bandwidth early.

## Question 2
> You have a dual-homed PE connected to two P routers via separate fiber conduits. How would you ensure the primary and secondary TE tunnels to a remote PE never share the same conduit? Show the config for both IOS-XE and Junos.

### Answer

Use **SRLG (Shared Risk Link Group)** to tag both links in the same conduit with the same SRLG value, then configure the secondary tunnel to exclude the primary's SRLGs.

**Concept:**
```
PE1 ──── Conduit A ────── P1
PE1 ──── Conduit B ────── P2

Conduit A links: SRLG 100
Conduit B links: SRLG 200

Primary tunnel: PE1 → P1 → ... → PE5 (uses Conduit A)
Secondary tunnel: must exclude SRLG 100 → forced through P2 / Conduit B
```

**IOS-XE:**
```
! Tag interfaces with SRLG values
interface TenGigabitEthernet0/0/0
 description Link to P1 via Conduit-A
 mpls traffic-eng srlg 100
 mpls traffic-eng srlg 300
 ! SRLG 100 = Conduit A, 300 = additional shared risk

interface TenGigabitEthernet0/0/1
 description Link to P2 via Conduit-B
 mpls traffic-eng srlg 200

! Primary tunnel — no SRLG constraint needed
interface Tunnel100
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng path-option 1 dynamic
 tunnel mpls traffic-eng autoroute announce

! Secondary tunnel — exclude primary's SRLGs
interface Tunnel101
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng path-option 1 dynamic attributes SRLG-EXCLUDE
 tunnel mpls traffic-eng path-option protect

! Attribute set with SRLG exclusion
mpls traffic-eng lsp attributes SRLG-EXCLUDE
 path-selection exclude srlg tunnel100
 ! Exclude all SRLGs used by Tunnel100
```

**Junos:**
```
# Define fate-sharing groups (Junos' SRLG mechanism)
set protocols mpls fate-sharing group CONDUIT-A members ge-0/0/0.0
set protocols mpls fate-sharing group CONDUIT-A members ge-0/0/1.0
# All links in Conduit A
set protocols mpls fate-sharing group CONDUIT-A cost 10000

set protocols mpls fate-sharing group CONDUIT-B members ge-0/1/0.0
set protocols mpls fate-sharing group CONDUIT-B cost 10000

# Primary LSP
set protocols mpls label-switched-path to-PE5-primary to 10.0.0.5
set protocols mpls label-switched-path to-PE5-primary bandwidth 1g

# Secondary LSP — SRLG-diverse from primary
set protocols mpls label-switched-path to-PE5-secondary to 10.0.0.5
set protocols mpls label-switched-path to-PE5-secondary bandwidth 1g
set protocols mpls label-switched-path to-PE5-secondary secondary via-P2
set protocols mpls label-switched-path to-PE5-secondary fate-sharing exclude to-PE5-primary
```

**Why SRLG, not just node/link exclusion?** Because the same conduit might carry links to multiple different nodes. Excluding a single node doesn't protect against the backhoe that cuts the conduit containing links to three different P routers. SRLG captures the physical shared fate that topology diagrams hide.

## Question 3
> After enabling soft preemption, you notice tunnels are still being hard-preempted. The preemption log shows "cleanup timer expired." What's happening and how do you fix it?

### Answer

**The victim tunnel is failing to reroute within the soft preemption grace period.**

Here's the sequence:
1. Higher-priority tunnel arrives, needs bandwidth on a link
2. Transit node identifies the victim (lower-priority tunnel)
3. Soft preemption kicks in: sends PathErr with "Reroute Request Soft" to the victim's head-end
4. Victim head-end gets the signal and starts MBB to find an alternate path
5. **But CSPF can't find a valid alternate path** (bandwidth exhausted elsewhere, affinity constraints too tight, or topology is too constrained)
6. Cleanup timer expires (default ~30 seconds)
7. **Hard preemption occurs** — victim is torn down, traffic drops

"Cleanup timer expired" means the victim was given its chance and couldn't make it happen in time.

**Fixes:**

**1. Increase the cleanup timer** (buys more time for CSPF):
```
IOS-XE:
  mpls traffic-eng pce timers cleanup 120
  ! Give victim 120 seconds instead of 30

Junos:
  set protocols rsvp preemption soft cleanup-timer 120
```

**2. Ensure alternate capacity exists** — this is the real fix. If the victim can't find any alternate path with available bandwidth, more time won't help. You need:
- More physical capacity
- Better bandwidth distribution across links
- Lower max-reservable on some links to keep headroom

**3. Reduce victim's bandwidth requirement** or set `min-bw` on auto-bandwidth so the rerouted tunnel can accept a lower bandwidth reservation to fit on the alternate path.

**4. Check that soft preemption is enabled on ALL transit nodes**, not just the head-end. If a transit node doesn't support or isn't configured for soft preemption, it will hard-preempt directly regardless of head-end config.

**5. Verify CSPF can actually find paths** — the victim head-end may have constraints (affinity, explicit hops) that make alternate paths impossible. Relax constraints on backup path-options.

## Question 4
> Your network has 500 TE tunnels with auto-bandwidth at 300-second adjust intervals. During a topology change, you see 200 simultaneous MBB events. What's the impact and how would you tune this?

### Answer

**Impact of 200 simultaneous MBB events:**

1. **RSVP signaling storm**: Each MBB creates a new LSP (PATH + RESV) before tearing down the old one. 200 MBBs = 200 new PATH messages + 200 RESV messages propagating through the core. Transit P routers process all of them.

2. **Double bandwidth reservation**: During MBB, both old and new LSP coexist. Using SE (shared-explicit) style, bandwidth is shared on common segments. But diverging segments have both reservations active simultaneously. If 200 tunnels all try to re-signal at once, bandwidth pools may appear exhausted, causing cascading CSPF failures.

3. **CPU spike on head-ends and transit**: CSPF computation for 200 tunnels, RSVP message processing, LFIB reprogramming. On some platforms this can push CPU to 100% and delay other control-plane functions (IGP, BFD, BGP keepalives).

4. **Potential cascading failures**: If the MBB storm causes BFD sessions to time out due to CPU starvation, you get secondary topology changes, which trigger more MBBs. Classic feedback loop — the same pattern as the "Auto-Bandwidth DDoS Amplifier" war story.

**Tuning recommendations:**

**1. Increase adjust interval to 3600-86400 seconds:**
```
IOS-XE:
  tunnel mpls traffic-eng auto-bw
   frequency 86400
   ! 24-hour adjustment cycle, not 5-minute

Junos:
  set protocols mpls label-switched-path <name> auto-bandwidth adjust-interval 86400
```
300 seconds is way too aggressive for production. 24 hours is standard.

**2. Add adjustment threshold:**
```
  adjustment-threshold 15 10
  ! Only adjust if traffic changed by 15%, minimum 10% absolute
```
This prevents tunnels from MBB'ing over insignificant traffic fluctuations.

**3. Enable RSVP refresh reduction** (should already be on):
```
  ip rsvp signalling refresh reduction
```

**4. Stagger MBB events:**
Some platforms support MBB rate-limiting — cap concurrent MBBs to 20-50 per second. This prevents the storm pattern.

**5. Use event-driven reoptimization with damping:**
Instead of timer-based, only reoptimize when a topology change creates a better path, with a hold-down timer to prevent rapid-fire reopt.

**Bottom line:** 300s auto-bandwidth with 500 tunnels is a time bomb. Change the interval to hours, add thresholds, and rate-limit MBB events.

## Question 5
> Explain the difference between RSVP SE (shared-explicit) and FF (fixed-filter) styles. Why is SE critical for make-before-break?

### Answer

**Fixed-Filter (FF):**
Each sender gets its own dedicated bandwidth reservation. If two senders (LSPs) share a link, each gets a separate reservation. Bandwidth is NOT shared even if they're part of the same tunnel.

**Shared-Explicit (SE):**
Multiple senders listed in the RESV message share a single bandwidth reservation. The reservation is "shared" among the explicitly listed senders.

**Why SE is critical for MBB:**

During make-before-break, two LSPs temporarily coexist for the same tunnel:
- **Old LSP**: LSP-ID 1, carrying traffic
- **New LSP**: LSP-ID 2, being established

Both share the same Tunnel-ID but have different LSP-IDs. They likely share many common links (especially in the core).

**With FF style:**
```
Link X carries both old and new LSP:
  Old LSP reservation: 1 Gbps
  New LSP reservation: 1 Gbps
  Total reserved on link: 2 Gbps  ← DOUBLED!
```
This double-booking can cause CSPF to fail for the new LSP (insufficient bandwidth), or it can cause other tunnels to be preempted to make room. Either way, MBB breaks.

**With SE style:**
```
Link X carries both old and new LSP:
  RESV lists both LSP-ID 1 and LSP-ID 2 as shared senders
  Shared reservation: 1 Gbps  ← SHARED on common segments
  Total reserved on link: 1 Gbps  ← No double-booking!
```
On common segments, the old and new LSP share the reservation. The new LSP only needs additional bandwidth on segments where it diverges from the old path.

**This is why SE is the default for TE tunnels** and why MBB works seamlessly — the two coexisting LSPs don't fight each other for bandwidth on the links they share. Without SE, every MBB event (reoptimization, auto-bandwidth adjustment, FRR revert) would either double-book bandwidth or fail to signal.

**Practical note:** You almost never configure this explicitly — SE is the default. But understanding why it's the default is the difference between knowing TE and understanding TE.
