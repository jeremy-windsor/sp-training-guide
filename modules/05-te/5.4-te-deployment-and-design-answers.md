# 5.4 — TE Deployment & Design — Answer Key

## Question 1
> You have a 75-PE backbone. Calculate the tunnel count for full-mesh. Then design a hierarchical model using 8 core P routers and calculate the reduced tunnel count. What's the scaling improvement?

### Answer

### Full-mesh model
For **N PEs**, full-mesh tunnel count is:

\[
N \times (N - 1)
\]

For **75 PEs**:

\[
75 \times 74 = 5550
\]

So a full-mesh RSVP-TE design requires **5,550 tunnels**.

That is just the tunnel count. The real pain is the RSVP state on transit P routers and the refresh traffic those 5,550 tunnels generate.

---

### Hierarchical model with 8 core P routers
Use a two-tier / hub-and-spoke TE model:

- **Tier 1:** Core P-to-P tunnels between the 8 core P routers
- **Tier 2:** PE-to-core-P tunnels from each PE to one or more core P routers
- **Tier 3:** LDP provides PE-to-PE service reachability over the TE transport

#### Core P-to-P mesh
For 8 P routers in full mesh:

\[
8 \times 7 = 56
\]

So the core mesh requires **56 P-to-P tunnels**.

#### PE-to-core tunnels
If each of the 75 PEs builds a TE tunnel to each of the 8 core P routers:

\[
75 \times 8 = 600
\]

So the PE-to-core layer requires **600 tunnels**.

#### Total hierarchical tunnels
\[
600 + 56 = 656
\]

So the hierarchical model uses **656 tunnels** total.

---

### Scaling improvement
Reduction in tunnel count:

\[
5550 - 656 = 4894
\]

Percentage reduction:

\[
\frac{4894}{5550} \times 100 \approx 88.2\%
\]

So the hierarchical design gives about an **88% reduction** in tunnel count.

Another way to say it: full-mesh uses about **8.46× more tunnels** than the hierarchical design.

---

### Why this matters operationally
Full-mesh is easy to understand and terrible to scale.

With 5,550 RSVP-TE tunnels:
- central P routers carry huge PATH/RESV state
- refresh reduction becomes mandatory
- auto-bandwidth and reoptimization events create RSVP storms
- troubleshooting gets ugly fast

With the hierarchical model:
- fewer tunnels
- fewer RSVP sessions on transit routers
- larger, strategic TE transport pipes instead of PE-to-PE spaghetti
- services ride via LDP over TE (LDPoRSVP), which is how real SP networks have done this for years

**Best answer:**
- Full mesh = **5,550 tunnels**
- Hierarchical = **656 tunnels**
- Improvement = **4,894 fewer tunnels**, or about **88% reduction**

## Question 2
> After a fiber cut, FRR activates and traffic reroutes via bypass tunnels. You notice 200ms of packet loss before FRR kicks in. What's the most likely cause, and what do you add to reduce it to <50ms?

### Answer

**Most likely cause: failure detection is too slow.**

FRR itself is fast — once the router knows the protected link/node failed, it can push the bypass label and reroute almost immediately. If you're seeing **200 ms of packet loss**, the usual culprit is not bypass setup time. It's the delay between:

1. fiber cut occurs
2. router detects failure
3. FRR activates

If detection is driven by slow link-state notification, IGP hello dead timers, or delayed physical fault propagation, you lose packets during that gap.

### What to add
Add **BFD on all core interfaces**.

BFD gives fast liveliness detection independent of the IGP timer model. Once BFD declares the adjacency down, FRR can activate immediately.

### Target settings
Use something like:
- **3 × 10 ms** for very fast convergence on capable hardware
- **3 × 50 ms** for a safer production default if the platform can't sustain 10 ms cleanly

That yields detection in roughly:
- **30 ms** with 10 ms timers
- **150 ms** with 50 ms timers

FRR activation after detection is near-instant, so with BFD + FRR you can get to **sub-50 ms** total on a healthy design.

### IOS-XE example
```
interface TenGigabitEthernet0/0/0
 bfd interval 10 min_rx 10 multiplier 3
 mpls traffic-eng tunnels
```

And make sure FRR is enabled on the tunnel/interface:
```
interface Tunnel100
 tunnel mpls traffic-eng fast-reroute
 tunnel mpls traffic-eng fast-reroute node-protect
```

### Junos example
```
set protocols bfd interface ge-0/0/0.0 minimum-interval 10
set protocols bfd interface ge-0/0/0.0 multiplier 3
set protocols mpls label-switched-path to-PE5 fast-reroute
set protocols mpls label-switched-path to-PE5 node-link-protection
```

### Also verify
- bypass tunnel exists and is **Ready** / active in FRR database
- node/link protection is actually available for the protected hop
- no SRLG issue makes the bypass invalid during the same failure event

**Bottom line:** FRR isn't the slow part. **Detection is.** Add **BFD** and you stop losing packets while the router stands there thinking about its life choices.

## Question 3
> Your LDPoRSVP setup shows a 3-label stack (RSVP + LDP + VPN) on most prefixes, but some VPN prefixes only show a 2-label stack. What's different about those prefixes?

### Answer

Those prefixes are most likely **not being transported over an RSVP-TE tunnel for that portion of the path** — or they are hitting a case where the transport label is not needed at the point you're observing.

In classic **LDP-over-RSVP** for L3VPN, the normal stack is:

1. **RSVP transport label** (top)
2. **LDP transport/service reachability label**
3. **VPN label**

So why would some prefixes only show **2 labels**?

### Most common reasons

#### 1. The destination prefix is directly resolved via LDP without RSVP transport
Not every prefix necessarily rides an RSVP tunnel. If the route to the egress PE loopback is not using the RSVP LSP — for example because autoroute/install only covers some destinations or the LSP is not active for that egress — then the stack might be:

- **LDP label**
- **VPN label**

No RSVP label, because there is no RSVP transport in use for that FEC.

#### 2. You are observing packets after PHP / near the merge point
On the penultimate hop or after PHP of the RSVP label, the outer RSVP label may already be removed. At that observation point, the visible stack becomes:

- **LDP label**
- **VPN label**

This is normal if you're looking close to the tail end of the RSVP LSP.

#### 3. The prefix is the tunnel endpoint / special-resolution case
Some destination loopbacks or specific prefixes may resolve directly via the TE tunnel endpoint behavior rather than needing both transport layers in the way you expect.

### What is different about those prefixes?
**They are resolving differently in the forwarding plane.** Either:
- they are **not using RSVP as transport**, or
- they are being observed at a point where the **RSVP label has already been popped**.

### How to verify

**IOS-XE:**
```
show ip route <egress-loopback>
show ip cef <vpn-prefix> detail
show mpls forwarding-table <vpn-prefix>
show mpls traffic-eng tunnels
```

Check whether the egress PE loopback is actually installed via the TE tunnel (`autoroute announce`) or plain IGP/LDP.

**Junos:**
```
show route <egress-loopback>/32 detail
show route <vpn-prefix> extensive
show route forwarding-table destination <vpn-prefix> detail
show mpls lsp
```

### Exam-quality answer
If most prefixes show **RSVP + LDP + VPN** but some show only **LDP + VPN**, then those prefixes are **not using RSVP transport end-to-end in that forwarding path**, or the **outer RSVP label has already been popped** due to PHP / observation point.

That difference is not magic. It's resolution.

## Question 4
> Explain why forwarding adjacency can cause routing loops. Give a specific topology example where this occurs and how to prevent it.

### Answer

**Forwarding adjacency (FA)** advertises a TE tunnel as if it were a normal IGP link. Other routers can then run SPF over that advertised link.

That sounds elegant until recursion shows up with a shotgun.

### Why loops happen
If the FA tunnel is itself built over the same part of the topology that later starts routing back into the FA, you can create **recursive forwarding**:
- IGP sees the FA as a shortcut
- traffic is sent into the FA
- but the tunnel's actual path depends on topology that, after SPF recalculation, now points back toward the FA
- packets chase their own tail

### Example topology

```
      P2
     /  \
PE1       P3 ---- PE5
     \  /
      P4
```

Assume:
- A TE tunnel from **PE1 to PE5** is signaled via **PE1 → P2 → P3 → PE5**
- That TE tunnel is advertised into the IGP as a **forwarding adjacency**
- Other routers now see a new IGP link: **PE1 → PE5 (low metric)**

Now suppose SPF on P4 recalculates and decides the best route toward PE5 is:
- go to PE1
- then use the FA tunnel to PE5

But traffic entering PE1 and taking the FA tunnel may still depend on an underlay that, after failure or metric change, routes portions of the path back via P4.

Now you have:
- P4 forwards toward PE1 because the FA looks best
- PE1 forwards into the FA
- the underlay used by the FA leads back toward P4 or through a topology dependent on P4
- congratulations, you've built a routing Möbius strip

### More realistic failure mode
The most common case is when the FA metric is set **too low**, so the IGP strongly prefers the tunnel over underlying physical links. Once multiple routers start preferring the FA as part of SPF, the tunnel underlay and IGP topology become coupled in ugly ways.

### How to prevent it

#### 1. Avoid FA unless you really need it
Most operators prefer:
- `autoroute announce`
- `autoroute destination`
- static route into tunnel
- policy steering

These are simpler and safer.

#### 2. Set FA metric carefully
Do **not** advertise the tunnel with a metric lower than the physical topology in a way that attracts widespread recursion.

#### 3. Keep the FA design topologically clean
Use FA only in controlled, hierarchical topologies where the tunnel abstraction aligns with actual traffic flow.

#### 4. Validate recursion explicitly
Check:
- route to tunnel endpoint
- underlay path of tunnel
- IGP route learned from FA
- whether any of those point back into each other

#### 5. Prefer hierarchical TE instead of FA-heavy designs
Real SPs usually scale TE with **PE→core tunnels + LDP over TE**, not with a spiderweb of forwarding adjacencies.

### Best concise answer
Forwarding adjacency can cause loops because a TE tunnel advertised as an IGP link can become part of SPF in a way that recursively attracts traffic into a tunnel whose own underlay depends on the same topology. Prevent it by avoiding FA where possible, keeping metrics conservative, and using hierarchical TE / autoroute instead.

## Question 5
> You're migrating from RSVP-TE to SR-TE on a live 100-PE backbone. Describe a phased migration plan that maintains FRR protection at every stage. What's the coexistence model?

### Answer

You do this in **phases**, not with a hero move. Hero moves are how outage reports get adjectives.

### Goal
Move from:
- RSVP-TE transport with RSVP FRR / facility backup

to:
- SR-MPLS underlay with SR-TE policies and **TI-LFA** protection

while preserving fast reroute the whole time.

---

## Phase 1 — Prepare the underlay
1. Enable **SR-MPLS** in the IGP across the core
2. Allocate SRGB / prefix-SIDs consistently
3. Verify all core nodes advertise SIDs correctly
4. Keep **existing RSVP-TE tunnels and FRR fully in place**

At this stage:
- forwarding still uses RSVP/LDP transport where it does today
- FRR remains **RSVP facility backup**
- SR exists but is not yet carrying production traffic

### Protection model
- Existing services: **RSVP FRR**
- SR underlay validation traffic: not yet relied on for production

---

## Phase 2 — Enable TI-LFA in the SR domain
1. Turn on **TI-LFA** for the SR-enabled IGP
2. Validate repair coverage for all core links/nodes
3. Confirm post-convergence paths and repair segments are installed

Now the SR underlay has its own fast protection mechanism.

### Protection model
- RSVP traffic: **RSVP FRR**
- SR traffic: **TI-LFA**

This is the coexistence period. Two TE universes, same backbone, both armed.

---

## Phase 3 — Introduce SR-TE policies on selected head-ends
1. Build SR-TE policies for a small set of PE-to-PE or PE-to-core intents
2. Start with non-critical traffic classes or pilot PEs
3. Steer selected prefixes / colors / VPNs into SR-TE policies
4. Leave RSVP-TE tunnels in place as fallback during validation

Use:
- explicit policy steering
- color-based steering / ODN
- limited blast radius

### Protection model
- Traffic still on RSVP tunnels: **RSVP FRR**
- Traffic now using SR-TE policies: **TI-LFA**

---

## Phase 4 — Migrate transport in waves
1. Migrate additional PE pairs or service classes from RSVP-TE to SR-TE
2. Keep LDP/L3VPN service resolution stable while transport moves underneath
3. Monitor:
   - path intent
   - failover time
   - stack depth / MSD
   - CPU / control-plane load

Do this wave by wave:
- region by region
- service class by service class
- head-end by head-end

Never all-at-once unless you enjoy paging systems.

---

## Phase 5 — Remove RSVP dependencies
Once all production transport is on SR-TE or native SR paths:
1. remove unused RSVP-TE tunnel configs
2. remove RSVP FRR bypass tunnels
3. disable RSVP where no longer needed
4. keep TI-LFA as the steady-state protection mechanism

Now the backbone is SR-native.

---

## Coexistence model
During migration, the correct coexistence model is:

- **RSVP-TE and SR-MPLS both enabled in the core**
- existing TE tunnels continue operating normally
- new SR-TE policies are introduced incrementally
- service steering determines whether traffic uses RSVP-TE or SR-TE
- FRR protection depends on the transport in use:
  - **RSVP transport → RSVP FRR**
  - **SR transport → TI-LFA**

That is the key exam answer.

### Why this works
- RSVP-TE is stateful and already proven in your network
- SR-TE is introduced without removing the safety net first
- you validate SR forwarding and TI-LFA before trusting it broadly
- rollback is easy: stop steering traffic into SR-TE and it falls back to RSVP/legacy model

### Best-practice notes
- Use PCE only after base SR-TE is stable
- Validate max SID depth before large policy rollout
- Keep LDP where needed during intermediate transport phases if services still depend on it
- Do not remove RSVP until you have confirmed equivalent or better protection with TI-LFA everywhere

### Best concise answer
Phased migration:
1. deploy SR underlay
2. enable TI-LFA
3. introduce SR-TE policies on a subset of traffic
4. migrate in waves while RSVP-TE remains active
5. remove RSVP only after SR-TE + TI-LFA is proven everywhere

**Coexistence model:** RSVP-TE and SR-TE run side by side; traffic is steered per service/prefix, and protection is RSVP FRR for RSVP traffic and TI-LFA for SR traffic.
