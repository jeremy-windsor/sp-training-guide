# 10.2 — FlexE (Flexible Ethernet) — Protocol Theory

## Protocol Overview

FlexE (Flexible Ethernet) is a shim layer between the Ethernet MAC and the Ethernet PHY that decouples the client data rate from the physical interface rate. In standard Ethernet, a 100GbE MAC is permanently bonded to a 100G PHY — the rates are locked together. FlexE breaks this bond, enabling operators to create Ethernet connections at arbitrary rates that don't match any standard PHY speed.

FlexE was developed by the OIF (Optical Internetworking Forum), first published as OIF-FLEXE-01.0 (2016), with subsequent revisions through OIF-FLEXE-02.1 (2019). It was driven by two converging demands: 5G xHaul needing precise bandwidth slicing (e.g., 25G, 50G, or 150G connections over 400G links), and service providers wanting hard bandwidth isolation between services on shared infrastructure without deploying separate physical interfaces for each rate.

The key innovation: FlexE operates entirely at Layer 1.5 — below Ethernet switching, above the physical layer. No IP, no MPLS, no VLAN tags. It's invisible to everything above the MAC. A router or switch sees standard Ethernet interfaces at whatever rate FlexE presents, unaware that the physical link is carrying multiple FlexE clients multiplexed together.

## Core Mechanisms

### 1. FlexE Architecture

FlexE introduces three concepts between the MAC and PHY:

**FlexE Group**: A bundle of one or more physical Ethernet PHYs that form a single logical transport pipe. The aggregate bandwidth of the group is the sum of all member PHYs.

Example: A FlexE Group of 4 × 100GbE PHYs provides 400G of aggregate bandwidth.

**FlexE Client**: A virtual Ethernet interface presented to the MAC layer. Each client has a specific bandwidth, allocated from the FlexE Group's aggregate capacity. Multiple clients share a single FlexE Group.

Example: From a 400G FlexE Group, create three clients: 150G, 200G, and 50G.

**FlexE Shim**: The protocol layer that maps FlexE Clients to FlexE Group PHYs using a calendar-based TDM mechanism. The shim sits between the MAC and PCS (Physical Coding Sublayer) layers.

```
┌────────────────────────────────────┐
│  Ethernet MAC (Client A: 150G)     │
│  Ethernet MAC (Client B: 200G)     │
│  Ethernet MAC (Client C: 50G)      │
├────────────────────────────────────┤
│  FlexE Shim (calendar-based mux)   │  ← Layer 1.5
├────────────────────────────────────┤
│  PCS / PHY (4 × 100GbE)           │
└────────────────────────────────────┘
```

### 2. Calendar Slot Mechanism

FlexE uses a TDM-like calendar to allocate bandwidth from the FlexE Group to FlexE Clients.

**How it works:**

Each 100G PHY in the FlexE Group is divided into 20 calendar slots of 5G each. A FlexE Group with 4 × 100G PHYs has 80 total slots (400G capacity in 5G granularity).

**Calendar structure:**
- The calendar is a repeating frame of N slots (N = 20 per 100G PHY).
- Each slot is assigned to a specific FlexE Client.
- The assignment repeats every calendar cycle.
- Bandwidth = number of assigned slots × 5G.

**Example allocation on a 4 × 100G group (80 slots):**
- Client A: 30 slots = 150G
- Client B: 40 slots = 200G
- Client C: 10 slots = 50G
- Total: 80 slots = 400G (fully utilized)

**Slot distribution across PHYs**: Slots for a single client are distributed across all PHYs in the group for load balancing. Client A's 30 slots might be spread as 7-8 slots per PHY rather than concentrated on a few PHYs. This distributes the data evenly and ensures no single PHY becomes a bottleneck.

**Sub-calendar granularity**: The base granularity is 5G (one slot per 100G PHY). For finer granularity, some implementations support sub-slot allocation, but 5G is the standard unit defined by OIF.

### 3. FlexE Overhead Frame

FlexE uses a dedicated overhead frame to communicate calendar assignments and group configuration between endpoints.

**FlexE overhead structure:**
- Transmitted in-band, using a fixed overhead pattern inserted into the 66B-encoded data stream.
- **Overhead frame period**: One overhead frame per 1,023 × 20 × 66B blocks (approximately every ~100 microseconds at 100G).
- **Calendar**: The primary content — which slots map to which clients.
- **Group configuration**: Which PHYs are members of the group, PHY numbering, group ID.
- **CRC**: Error protection for the overhead frame itself.
- **Management channel**: A small bandwidth allocation for FlexE management communication.

**Calendar negotiation**: Both endpoints must agree on the calendar. Changes to client bandwidth (adding, removing, or resizing clients) require calendar renegotiation. During renegotiation, a hitless calendar swap mechanism allows changes without traffic disruption.

### 4. FlexE Client Types

**FlexE Channelized (sub-rate)**:
The primary use case. Multiple clients share a FlexE Group, each getting a fraction of the bandwidth. A 400G link becomes three separate Ethernet interfaces at 150G, 200G, and 50G.

**FlexE Bonded (super-rate)**:
Multiple PHYs are bonded to create a single client larger than any individual PHY. Four 100G PHYs bonded into a single 400G FlexE Client. Similar to LAG but at Layer 1.5 — no hashing, deterministic interleaving. Lower latency than LAG-based bonding.

**FlexE Unaware**:
A single PHY carries a single client at the PHY's native rate. FlexE overhead is still present but the calendar assigns all slots to one client. Functionally equivalent to standard Ethernet, but FlexE-compatible endpoints can later reconfigure to channelized mode without hardware changes.

### 5. FlexE and Network Slicing

FlexE provides hard bandwidth isolation — each FlexE Client gets a guaranteed, deterministic bandwidth allocation at the physical layer. This is fundamentally different from QoS-based soft isolation (where bandwidth is shared and prioritized) or VLAN-based logical isolation (where all VLANs share the same physical bandwidth).

**Hard isolation properties:**
- A FlexE Client's bandwidth is reserved in the calendar. Other clients cannot consume it, even if they're idle.
- Jitter and latency for each client are independent — traffic bursts on Client A don't affect Client B's timing.
- Failure of one client's MAC or protocol stack doesn't affect other clients on the same FlexE Group.

**5G xHaul use case:**
- Fronthaul: 25G FlexE Client for eCPRI (constant bit rate, latency-sensitive)
- Midhaul: 50G FlexE Client for F1 interface
- Backhaul: remaining bandwidth as a FlexE Client for IP/MPLS traffic
- All three share a single physical fiber pair with FlexE providing the hard slicing

### 6. FlexE vs Alternatives

**FlexE vs LAG:**
- LAG bonds links at Layer 2 with hash-based distribution. FlexE bonds at Layer 1.5 with calendar-based distribution.
- LAG has flow-level granularity and potential hash imbalance. FlexE has 5G granularity and deterministic allocation.
- LAG cannot create sub-rate interfaces. FlexE can.

**FlexE vs Channelized interfaces:**
- Traditional channelized interfaces (e.g., channelized OC-48 into DS3s) are fixed at manufacturing. FlexE is software-reconfigurable.
- FlexE supports arbitrary rates in 5G increments. Traditional channelization has fixed hierarchical rates.

**FlexE vs OTN ODUflex:**
- Both provide flexible bandwidth allocation. ODUflex operates at the OTN layer; FlexE operates at the Ethernet layer.
- FlexE is simpler (no OTN overhead) but lacks OTN's monitoring and protection.
- In converged networks, FlexE may ride over OTN, or replace OTN for Ethernet-only services.

## Protocol Interactions

**FlexE ↔ Ethernet**: FlexE is transparent to the Ethernet MAC. Each FlexE Client presents as a standard Ethernet interface. The MAC doesn't know it's running over a FlexE shim — it sees a normal Ethernet port at whatever rate the calendar provides.

**FlexE ↔ OTN**: FlexE can operate below OTN (FlexE clients mapped into ODUflex) or alongside OTN (FlexE providing Ethernet-layer slicing while OTN provides transport-layer monitoring). ITU-T G.8312 defines the interworking.

**FlexE ↔ Network Slicing**: FlexE provides the physical-layer hard isolation that network slicing architectures require. Combined with SR-TE or Flex-Algo for routing isolation and VRF/EVPN for service isolation, FlexE completes the end-to-end slicing stack.

## Edge Cases & Gotchas

**1. 5G granularity may be too coarse**: For services needing 1G or 2G slices, FlexE's 5G minimum is wasteful. You'd allocate 5G for a 1G service, wasting 4G. Sub-slot mechanisms exist but aren't universally implemented.

**2. Both endpoints must support FlexE**: FlexE is not backward-compatible with standard Ethernet PHYs. Both ends of the link must be FlexE-capable and configured with matching group and calendar parameters.

**3. Calendar changes require coordination**: Resizing a FlexE Client means changing the calendar on both endpoints simultaneously. Without hitless calendar swap support, this causes a brief traffic disruption during reconfiguration.

**4. PHY failure in a bonded group**: If one PHY in a FlexE Group fails, the aggregate bandwidth decreases. Clients mapped to slots on the failed PHY lose connectivity unless the calendar is reconfigured to redistribute those clients across remaining PHYs. This is not automatic in all implementations.

**5. FlexE overhead bandwidth consumption**: The FlexE overhead frame consumes a small amount of bandwidth (~200 ppm). For most applications this is negligible, but in precision timing or exact-rate applications, account for it.
