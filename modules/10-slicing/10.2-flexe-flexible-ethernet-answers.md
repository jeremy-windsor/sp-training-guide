# 10.2 — FlexE (Flexible Ethernet) — Answer Key

---

## Question 1
> **Scenario:** You're designing a 5G transport network for a Tier 2 operator. They have three slice types: eMBB (bulk data), URLLC (autonomous vehicles), and mMTC (smart meters). The URLLC slice has a contractual SLA guaranteeing < 1ms one-way latency and zero bandwidth contention with other slices. The eMBB slice needs 80% of total bandwidth. Budget is constrained. What isolation model would you recommend, and where (if anywhere) would you deploy FlexE?

### Answer

The words "zero bandwidth contention" and "budget is constrained" are pulling in opposite directions. Here's how to satisfy both.

**Isolation model: Hybrid — FlexE at aggregation, Flex-Algo + QoS in the core.**

**The rationale:** "Zero bandwidth contention" is a legal/contractual term that means hard isolation — QoS scheduling alone won't satisfy it because a sufficiently aggressive eMBB burst could theoretically cause queuing delay on a shared interface, even with strict-priority for URLLC. You need FlexE channelization somewhere. But FlexE *everywhere* on a Tier 2 budget is insane — the hardware cost, operational complexity, and capacity rigidity would bankrupt the project.

**Where to deploy FlexE — aggregation links only:**

```
Cell Sites          Aggregation         Core
┌─────┐           ┌─────────────┐     ┌──────┐
│O-RU │──25GE──→ │  Agg Router  │──→ │ Core │
│O-DU │           │             │     │      │
└─────┘           │  FlexE here │     │      │
                  │  ↓          │     │      │
┌─────┐          │ Ch1: URLLC  │     │      │
│O-RU │──25GE──→ │ Ch2: eMBB   │──→ │      │
│O-DU │           │ Ch3: mMTC   │     │      │
└─────┘           └─────────────┘     └──────┘
                  100G uplink split:
                  10G URLLC (hard)
                  80G eMBB
                  10G mMTC
```

The aggregation-to-core links are where all three slice types converge and contention risk is highest. Channelize the 100G (or 400G) uplinks into FlexE clients: 10% URLLC (hard-isolated, guaranteed — eMBB can't touch these calendar slots), 80% eMBB, 10% mMTC.

**In the core — Flex-Algo + strict-priority QoS:**

The core has abundant bandwidth (400G+ links), and contention risk is low. Deploy Flex-Algo 129 (min-delay) for URLLC path computation, default algo for eMBB/mMTC, and strict-priority QoS for URLLC traffic. This provides enhanced-soft isolation that's effectively equivalent to hard isolation when links are operating below capacity — which core links should always be.

**Why NOT FlexE in the core:** A 200-node core with FlexE on every link means every capacity change (new customer, traffic shift, link upgrade) requires FlexE calendar reconfiguration across the entire path. That's an operational tax you can't afford on a Tier 2 budget. Flex-Algo + QoS gives you 99.9%+ of the isolation benefit with 10% of the operational cost.

**The <1ms latency target:** This is achievable without FlexE — it's a path computation problem, not an isolation problem. Flex-Algo 129 min-delay with strict-priority queuing keeps URLLC latency well under 1ms on a properly designed metro network (typical metro latency is 200–500 μs). FlexE guarantees the *bandwidth* isolation; Flex-Algo + QoS guarantees the *latency*.

**Budget reality:** FlexE-capable line cards cost 20–40% more than standard cards. Deploying at 10–20 aggregation nodes is affordable. Deploying at 200 core nodes is not.

## Question 2
> **Design:** An operator has a 400G coherent DWDM link between two cities. The transponder is configured for DP-16QAM, delivering 300G effective rate. How would you use FlexE sub-rating to efficiently connect this to a 400G router port? What happens to the unused capacity?

### Answer

This is the textbook FlexE sub-rating use case. The transponder delivers 300G, but your router has a 400G port. Without FlexE, the router tries to push 400G into a 300G pipe and you get drops. With FlexE, you match rates precisely.

**Configuration:**

Both ends need 400G PHYs in their FlexE groups — all PHYs in a FlexE group must operate at the same rate end-to-end between FlexE shims (per OIF-FLEXE-02.1 clause 6.1). The group has 80 calendar slots (400G ÷ 5G per slot = 80 slots). You allocate 60 slots to a FlexE client, creating a 300G logical interface:

```
400G Router Port (80 calendar slots)
┌────────────────────────────────────────────────────────────────────────────────┐
│████████████████████████████████████████████████████████████░░░░░░░░░░░░░░░░░░░░│
│← 60 slots = 300G FlexE Client (active) →│← 20 slots = unavailable (100G) →│
└────────────────────────────────────────────────────────────────────────────────┘
```

The rate mismatch between 400G PHY and 300G optical information rate is handled by FlexE-aware transport (OIF-FLEXE-02.1 clause 8.3): the ingress transport node marks 20 slots as unavailable based on optical capacity, and the egress node restores them. The router sees a 300G interface. IP/MPLS routes traffic to it as a 300G link. The IGP advertises 300G bandwidth.

**What happens to the unused 100G (20 slots):**

Those 20 calendar slots are marked as **unavailable** in the FlexE overhead calendar. They carry idle blocks. The capacity is genuinely unused — it cannot carry additional client traffic because the optical path only supports 300G of information rate.

**Important constraint:** You cannot simply create a second 100G FlexE client on the remaining 20 slots for a different service — the optical link only supports 300G total. The unused slots exist because the FlexE shim operates at 400G PHY rate while the optical information rate is 300G.

**Why this matters for optical transport:** PCS (probabilistic constellation shaping) means transponders increasingly deliver non-standard rates — 275G, 350G, 450G — optimized for the specific fiber route's reach and margin. Without FlexE sub-rating, every non-standard rate forces the router to provision the next larger standard Ethernet rate and waste the difference. FlexE sub-rating eliminates this waste by precisely matching the logical interface rate to the available optical capacity.

## Question 3
> **Troubleshooting:** After a FlexE group comes up with 4 member PHYs, one of the three channelized clients shows as DOWN while the other two are UP. The group itself is healthy, all PHYs are UP, and overhead CRC errors are zero. What are the three most likely causes, and how would you diagnose each?

### Answer

Group is healthy. All 4 PHYs are up. Overhead CRC clean. Two clients up, one client down. The problem is in the calendar mapping or client configuration, not the physical layer.

**Cause 1: Calendar mismatch between local and remote endpoints.**

The FlexE calendar must be identical on both sides — same slot-to-client mapping, same number of slots per client. If the local end maps Client 3 to slots 50–65 but the remote end maps Client 3 to slots 50–60 (fewer slots), the client can't establish because the shim layer detects a frame size mismatch.

**Diagnosis:** Compare calendar configurations on both endpoints:
- Junos: `show interfaces fe-x/y/z flexe-group calendar`
- IOS-XR: `show controllers FlexEGroup <rack/slot/instance> calendar`
- NX-OS: Not typically applicable (FlexE is SP-class feature)

Look for mismatched slot counts or slot ranges for the DOWN client. Fix: align calendar configurations on both endpoints.

**Cause 2: Client not provisioned on the remote end.**

The most embarrassing and most common cause. Two clients were configured on both sides. The third client was configured locally but never committed on the remote router. The FlexE group comes up (it only needs the group-level overhead to sync), the two configured clients come up, and the third has no peer.

**Diagnosis:** SSH to the remote end and check if the FlexE client exists:
- `show flexe client <id>` — if it returns "not found" or "not configured," there's your answer.
- Check for pending config that wasn't committed (`show configuration commit changes`).

**Cause 3: Client-level fault signaling (CS-LF or CS-RF).**

Even with the group healthy, individual FlexE clients carry per-client fault signaling in the FlexE overhead. If the remote end's client detects a local fault (laser failure on specific lanes, PCS alignment loss on the client's allocated slots), it sends Client Signal - Local Fault (CS-LF) toward you. Your end receives it and marks the client as DOWN.

**Diagnosis:** Check per-client fault counters:
- Look for CS-LF or CS-RF indications on the DOWN client
- Check if the remote end's client shows any alarms
- On the remote end, check the specific PHY lanes that carry the DOWN client's calendar slots — one PHY may have a partial lane failure that only affects this client's slot allocation

**Bonus unlikely cause:** If the DOWN client's calendar slots span a specific PHY that has marginal BER (below the group alarm threshold but above the client sensitivity), the client-level data integrity check may fail intermittently. The group stays up because most traffic is fine, but the client whose slots happen to live on the bad PHY bounces. Diagnosis: check per-PHY BER and verify which PHYs carry the DOWN client's calendar slots.

## Question 4
> **Comparison:** Explain why FlexE provides "hard" isolation while Flex-Algo provides "enhanced-soft" isolation. Under what failure scenarios can Flex-Algo isolation break down that FlexE would survive?

### Answer

**FlexE = hard isolation.** Calendar slots are TDM-assigned. Client A's slots physically cannot carry Client B's traffic. This is enforced in hardware at the PCS layer — it's not a scheduling decision, it's a physical partitioning of the Ethernet frame. There is no software bug, no misconfiguration, no traffic burst that can cause Client A's traffic to leak into Client B's bandwidth. It's the same principle as SONET/SDH time slots. The isolation is absolute and unconditional.

**Flex-Algo = enhanced-soft isolation.** Flex-Algo computes separate topologies per algorithm — different SPF trees, different forwarding entries, different SID stacks. Traffic for Algo 128 follows Algo 128's computed paths. Traffic for Algo 129 follows Algo 129's paths. This is *logically* isolated — different forwarding decisions, potentially different physical links. But the isolation depends on:
- Correct control plane computation (IS-IS/OSPF SPF)
- Correct data plane programming (FIB entries)
- Correct QoS enforcement (scheduling and policing)
- Sufficient physical capacity on the shared links

**Failure scenarios where Flex-Algo isolation breaks down but FlexE survives:**

**1. Link oversubscription during failure.**

Normal state: Flex-Algo 128 (eMBB) and Flex-Algo 129 (URLLC) use different links. A fiber cut removes Algo 129's primary path. TI-LFA reroutes URLLC traffic to a backup path that shares a physical link with Algo 128. Now both algos' traffic converges on the same physical interface. QoS scheduling determines who gets priority, but under heavy load, URLLC may experience queuing delay because eMBB traffic is competing for the same output buffer.

**FlexE survives this** because the URLLC FlexE channel has dedicated calendar slots on every link. Even if traffic reroutes, the URLLC slots are reserved end-to-end. eMBB traffic literally cannot occupy those time slots.

**2. QoS misconfiguration or policy bug.**

A network-wide QoS policy push accidentally removes strict-priority for the URLLC class on 3 nodes. Flex-Algo still routes URLLC traffic on the correct topology, but at those 3 nodes, URLLC packets compete with eMBB in a weighted-fair queue instead of getting priority scheduling. Latency spikes.

**FlexE survives this** because FlexE isolation is below the QoS layer. Calendar slot assignment doesn't depend on packet classification, DSCP markings, or queue scheduling. The isolation is in the Ethernet PCS, not the router's traffic manager.

**3. Microloop during convergence.**

A link flap causes a brief microloop where packets intended for Flex-Algo 129 bounce between two nodes for a few milliseconds before the SPF reconverges. During this microloop, the packets consume bandwidth on links shared with other Flex-Algos, and some may be dropped due to TTL expiry or buffer exhaustion.

**FlexE survives this** because even during a microloop, each packet stays within its FlexE channel. The channel's bandwidth is reserved regardless of how many times the packet transits the link.

**4. Software bug in the Flex-Algo computation.**

An IOS-XR or Junos bug causes a node to install incorrect Flex-Algo FIB entries, routing Algo 129 traffic onto Algo 128's topology. Traffic that should be isolated ends up on the wrong path, sharing links and queues with the wrong slice.

**FlexE survives this** because FlexE operates at Layer 1.5 (PCS shim). Even if the router's FIB is wrong and sends traffic out the wrong logical interface, the FlexE channelization ensures the traffic stays within its allocated bandwidth slice on the physical link. The traffic may go to the wrong *destination*, but it won't steal bandwidth from other channels.

**The bottom line:** Flex-Algo isolation depends on correct software behavior at L3 (routing) and L2 (QoS). FlexE isolation depends on correct hardware behavior at L1 (PCS calendar). Hardware is simpler, less configurable, and therefore less likely to fail. That's why FlexE is "hard" and Flex-Algo is "enhanced-soft."

## Question 5
> **Architecture:** A network architect proposes deploying FlexE channelization on every link in a 200-node SP backbone to implement hard slicing for all traffic. What operational challenges would this create, and what alternative design would you propose?

### Answer

This architect has been reading too many vendor whitepapers and not enough change management tickets.

**Operational challenges of universal FlexE:**

**1. Capacity rigidity.**

Every link is pre-sliced into fixed calendar allocations. If eMBB needs 60% today and 80% tomorrow (Beyoncé dropped an album, everyone's streaming), you can't dynamically shift capacity — you need to reconfigure the FlexE calendar on every link in the path. That's a planned change window, not a traffic engineering knob. Statistical multiplexing is dead; you're back to circuit-switched capacity planning.

In contrast, Flex-Algo + QoS lets traffic naturally burst into available bandwidth. eMBB can use URLLC's unused capacity during off-peak hours. With FlexE, that unused capacity sits idle in time slots that nobody can touch.

**2. Calendar management at scale.**

200 nodes × average 6 links per node × 3 slices per link = 3,600 FlexE calendar configurations. Every capacity change, every new service, every link upgrade requires coordinated calendar updates on both endpoints of every affected link. Miss one side and you get asymmetric calendars → client DOWN.

There's no IGP for FlexE calendars. No RSVP-TE for FlexE path setup. No PCE for FlexE optimization. Calendar management is either manual (per-device CLI) or requires a custom orchestrator that doesn't exist in most SP toolchains. The operational burden is enormous.

**3. Failure handling.**

When a link fails, traffic reroutes via TI-LFA or IGP reconvergence. But the backup link must have pre-provisioned FlexE calendar slots for the rerouted traffic. If the backup link's URLLC channel is sized for its normal load (10G), it can't absorb the failed link's URLLC load (another 10G) — the calendar doesn't stretch.

You'd need to either (a) pre-provision spare capacity in every channel on every backup path (doubling your capacity cost), or (b) dynamically resize calendars during failures (which requires sub-second calendar reconfiguration that current hardware doesn't support).

**4. Multi-vendor interop.**

FlexE interop between vendors is still maturing. Juniper, Nokia, and Cisco implement FlexE, but calendar exchange, fault signaling, and management channel behavior have subtle interop issues. On 200 nodes, you're guaranteed to hit them.

**5. Cost.**

FlexE-capable line cards command a 20–40% premium. Multiply that across 200 nodes × 6 ports each = 1,200 line card slots. That's potentially millions in additional capex for hard isolation that's unnecessary on core links with abundant bandwidth.

**The alternative design:**

**Tiered isolation architecture:**

| Tier | Location | Isolation Method | Why |
|------|----------|-----------------|-----|
| **Access/aggregation** (20–40 nodes) | Cell site to agg router | FlexE channelization | High contention risk, low link count, critical for SLA compliance |
| **Core** (160+ nodes) | Agg router to core, core-to-core | Flex-Algo + strict-priority QoS | Low contention risk, abundant bandwidth, operational simplicity |
| **Peering/DCI** (5–10 nodes) | Core to external/DC | FlexE or dedicated ports | Hard isolation at trust boundaries |

FlexE where it matters (30–50 nodes), Flex-Algo + QoS everywhere else. You get contractual "hard isolation" at the network segments that face customers and auditors, with operationally simple "enhanced-soft isolation" in the core where the engineering team needs flexibility.

**Tell the architect:** "Hard slicing everywhere is an architecture for a network that never changes. We run a network that changes every day. FlexE at the edges, Flex-Algo in the core — right tool, right place."
